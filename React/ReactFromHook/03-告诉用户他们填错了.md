# 第三章：告诉用户他们填错了

> [上一章](02-你的第一个表单.md)我们让表单跑起来了。但一个真正的表单不只是"能提交"——它要在用户犯错的瞬间，温和而准确地告诉他哪里出了问题。本章深入 react-hook-form 的校验系统。

---

## 3.1 先想清楚：为什么要校验

你填过一个表单——辛辛苦苦填了 15 个字段，点提交，服务器返回"用户名已存在"，所有字段被清空。那种感觉，你知道的。

==校验的根本目的是把错误拦在提交按钮前面，让用户一次过。==

但你**什么时候**告诉用户"填错了"？这里有真正的权衡：

| 时机 | 用户体验 | 性能代价 |
|------|----------|----------|
| 敲每个字符时就提示 | "烦死了，我还没打完！" | 高——每次按键都校验整个表单 |
| 点提交时才提示 | "早说啊，又要回去改..." | 低——只校验一次 |
| **离开输入框时提示**（失焦） | 折中："不打字了→该检查了" | 中——只在焦点切换时校验 |

react-hook-form 默认选择**提交时校验**。这不是偷懒——先追求==正确性（默认 onSubmit），再按需加体验（改为 onBlur 或 onChange）==。而不是反过来：先把所有校验都开成实时的，然后想办法做性能优化。

## 3.2 内置校验规则速览

所有校验通过 `register` 的第二个参数传入：

```tsx
register('fieldName', {
  required: '必填',            // 不能为空
  min: { value: 1, message: '最小值为 1' },
  max: { value: 100, message: '最大值为 100' },
  minLength: { value: 2, message: '至少 2 个字符' },
  maxLength: { value: 20, message: '最多 20 个字符' },
  pattern: { value: /正则表达式/, message: '格式不正确' },
})
```

记忆技巧：
- 带 `Length` 的 → 判断**字符串长度**（`minLength`、`maxLength`）
- 不带 `Length` 的 → 判断**数值大小**（`min`、`max`）

### 错误消息的两种写法

```tsx
// 简洁写法：required 后直接跟消息字符串
{ required: '此处必填' }

// 完整写法：需要额外参数时用对象
{ minLength: { value: 2, message: '至少 2 个字符' } }
```

## 3.3 一个带完整校验的字段

```tsx
<input
  {...register('name', {
    required: '名字不能为空',
    minLength: {
      value: 2,
      message: '名字至少 2 个字符，再写长一点',
    },
    maxLength: {
      value: 20,
      message: '名字最多 20 个字符，太长了',
    },
  })}
/>
{errors.name && <p className="error">{errors.name.message}</p>}
```

`errors.name` 只有在 `name` 字段校验失败时才存在。它的结构：

```typescript
// errors.name 的结构
{
  type: 'required' | 'minLength' | 'maxLength' | 'pattern' | 'validate' | ...,
  message: '你定义的错误消息',
  ref: <对应的 DOM 元素>,
}
```

## 3.4 validate：万能自定义校验

内置规则很棒，但覆盖不了所有场景——"确认密码必须和密码一致""用户名不能是 admin""异步去服务器查重"。用 `validate`：

```tsx
register('confirmPassword', {
  validate: (value, allValues) =>
    value === allValues.password || '两次密码不一致',
})
```

规则很简单：返回 `true` → 通过；返回 `false` 或字符串 → 不通过，字符串就是错误消息。

### 多个自定义校验？用对象

```tsx
register('username', {
  validate: {
    notForbidden: (v) => v !== 'admin' || '该用户名已被占用',
    noSpaces: (v) => !v.includes(' ') || '用户名不能有空格',
    exists: async (v) => {
      // 支持异步校验——去服务器查重
      const res = await fetch(`/api/check-username?name=${v}`);
      const { taken } = await res.json();
      return !taken || '用户名已被注册';
    },
  },
})
```

### 校验函数能拿到什么

```tsx
validate: (value, formValues) => {
  // value      → 当前字段的值
  // formValues → 整个表单当前所有字段的值
}
```

`formValues` 让你可以写跨字段校验——比如"结束日期必须大于开始日期"。不过当校验规则多到一定程度，手动写 `validate` 会在 JSX 里变得臃肿。[下一章](04-当表单长大以后.md)会介绍更好的方式。

## 3.5 valueAsNumber：input 值的类型陷阱

`<input>` 的值**永远是字符串**——即使 `type="number"`。

```tsx
<input type="number" {...register('age')} />
// onSubmit 拿到的 data.age 是 "18"（字符串），不是 18（数字）
```

后端的 API 期望数字，你用字符串 `"18"` 发过去，要么被拒，要么静默出错。加 `valueAsNumber` 解决：

```tsx
register('age', { valueAsNumber: true })
// data.age 现在是 18（数字）
// 如果用户输入的不是数字，值是 NaN
```

同理：`valueAsDate: true` 用于 `<input type="date">`，把日期字符串转为 Date 对象。

## 3.6 调整校验时机：mode 参数

```tsx
const form = useForm({
  mode: 'onBlur', // 用户离开输入框时校验
});

// 全部选项：
// 'onSubmit'  → 默认：只在校验时触发，零渲染输入
// 'onBlur'    → 失焦时校验
// 'onChange'  → 每次按键都校验（慎重：每个字符触发全表单校验）
// 'onTouched' → 第一次失焦后开始追踪，之后每次变化都校验
// 'all'       → onBlur + onChange 一起
```

> **建议**：大部分情况保持默认的 `onSubmit`。只有在你确信"让用户更早看到错误"能明显提升体验时，才考虑 `onBlur`。`onChange` 几乎永远是过度设计——用户还在打字，你急着告诉他"填错了"是一种干扰。

## 3.7 错误对象的完整遍历

当你有多个字段时，`errors` 对象的完整结构需要理解清楚：

```tsx
// errors 的形状（假设 email 和 name 各有一个错误）
{
  email: {
    type: 'pattern',
    message: '邮箱格式不正确',
    ref: <input>  // 对应的 DOM 元素引用
  },
  name: {
    type: 'required',
    message: '名字不能为空',
    ref: <input>
  }
}
```

显示错误的标准模式：

```tsx
{errors.email && <span role="alert">{errors.email.message}</span>}
```

`role="alert"` 让屏幕阅读器在错误出现时主动读出——这是无障碍访问的基础。

## 本章要点

- 校验是**帮用户**，不是防用户；先追求正确性（默认 onSubmit），再按需加体验
- 内置规则覆盖常见场景：`required`、`min/max`、`minLength/maxLength`、`pattern`
- `validate` 是万能自定义口——支持多个校验函数和异步
- `valueAsNumber` 解决 input 值永远是字符串的类型陷阱
- 校验时机用 `mode` 控制，`onSubmit` 是默认且最推荐的
- `errors.fieldName.message` 拿错误消息；加 `role="alert"` 保证无障碍

## 思考题

1. 本章推荐默认 `onSubmit` 校验。但有些表单（比如 Google 的注册页）会在你失焦时立刻显示错误。它们这样做的理由可能是什么？在什么条件下 `onBlur` 比 `onSubmit` 更好？
2. 异步校验函数（`validate` 里的 `async`）会阻塞表单提交吗？如果用户快速点击提交两次，会发生什么？
3. `valueAsNumber` 把 `""`（空字符串）转成了 `NaN`。这合理吗？你的 onSubmit 函数应该怎么防御这种情况？
