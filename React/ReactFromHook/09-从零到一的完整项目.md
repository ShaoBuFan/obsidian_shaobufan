## 9.1 模式一：用户注册（标准表单）

串联 Zod Schema、内置校验、跨字段校验、loading 状态、服务端错误：

```tsx
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { z } from 'zod';

// ─── 规则：独立于组件 ───
const schema = z
  .object({
    username: z.string().min(2, '至少 2 个字符').max(20, '最多 20 个字符'),
    email: z.string().email('邮箱格式不正确'),
    password: z.string().min(8, '密码至少 8 位'),
    confirmPassword: z.string(),
  })
  .refine((d) => d.password === d.confirmPassword, {
    message: '两次密码不一致',
    path: ['confirmPassword'],
  });

type FormData = z.infer<typeof schema>;

function RegisterForm() {
  const {
    register,
    handleSubmit,
    setError,
    formState: { errors, isSubmitting },
  } = useForm<FormData>({
    resolver: zodResolver(schema),
  });

  const onSubmit = async (data: FormData) => {
    try {
      const res = await fetch('/api/register', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(data),
      });
      if (!res.ok) {
        const err = await res.json();
        if (err.code === 'EMAIL_EXISTS') {
          setError('email', { type: 'server', message: '该邮箱已被注册' });
          return;
        }
        throw new Error(err.message);
      }
      // 成功
    } catch {
      setError('root', { type: 'server', message: '注册失败，请稍后重试' });
    }
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)} noValidate>
      {errors.root && <div role="alert" style={{ color: 'red' }}>{errors.root.message}</div>}

      <Field label="用户名" error={errors.username?.message}>
        <input {...register('username')} placeholder="2-20 个字符" />
      </Field>

      <Field label="邮箱" error={errors.email?.message}>
        <input {...register('email')} placeholder="example@mail.com" />
      </Field>

      <Field label="密码" error={errors.password?.message}>
        <input type="password" {...register('password')} placeholder="至少 8 位" />
      </Field>

      <Field label="确认密码" error={errors.confirmPassword?.message}>
        <input type="password" {...register('confirmPassword')} />
      </Field>

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? '注册中...' : '创建账号'}
      </button>
    </form>
  );
}

// ─── 可复用的字段包装器 ───
function Field({ label, error, children }: {
  label: string;
  error?: string;
  children: React.ReactNode;
}) {
  return (
    <div style={{ marginBottom: 16 }}>
      <label>{label}</label>
      {children}
      {error && <p style={{ color: 'red', fontSize: 14 }} role="alert">{error}</p>}
    </div>
  );
}
```

注意 `noValidate`——它禁用浏览器的原生校验弹窗。浏览器原生校验的样式和行为不可控，会和 react-hook-form 的校验冲突。

## 9.2 模式二：编辑已有数据

```tsx
function EditUser({ userId }: { userId: string }) {
  const { register, handleSubmit, reset, formState: { isDirty } } = useForm();

  // 加载已有数据
  useEffect(() => {
    fetch(`/api/user/${userId}`)
      .then(r => r.json())
      .then(reset); // reset 回填所有字段，并设为"未修改"的基准
  }, [userId, reset]);

  const onSubmit = async (data) => {
    await fetch(`/api/user/${userId}`, {
      method: 'PUT',
      body: JSON.stringify(data),
    });
    reset(data); // 保存成功——当前值变成新基准，isDirty 回到 false
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('name')} />
      <button disabled={!isDirty}>保存</button> {/* 没有修改时按钮不可点 */}
    </form>
  );
}
```

`reset` → 用户编辑 → `isDirty` 变 true → 保存 → 再 `reset(data)` → `isDirty` 变回 false。这个循环是编辑表单的标准模式。

## 9.3 模式三：搜索防抖

```tsx
function SearchBar({ onSearch }: { onSearch: (kw: string) => void }) {
  const { register, watch } = useForm();
  const keyword = watch('keyword');

  useEffect(() => {
    const timer = setTimeout(() => onSearch(keyword), 300);
    return () => clearTimeout(timer); // 清理上一个 timer——防抖的核心
  }, [keyword, onSearch]);

  return <input {...register('keyword')} placeholder="搜索..." />;
}
```

`watch` 产生渲染 → 渲染触发 effect → effect 里设置 300ms 的定时器 → 下一次按键取消上一个定时器。这就是 ==300ms 防抖==。注意这里用 `watch` 是合理的——因为页面不需要实时重渲染大量的内容，只在 300ms 后触发搜索。

## 9.4 模式四：条件显示字段

```tsx
function PaymentForm() {
  const { register, watch } = useForm();
  const method = watch('method');

  return (
    <form>
      <select {...register('method')}>
        <option value="wechat">微信支付</option>
        <option value="card">银行卡</option>
      </select>

      {method === 'card' && (
        <div>
          <input
            {...register('cardNumber', { required: '请输入卡号' })}
            placeholder="16 位卡号"
          />
        </div>
      )}
    </form>
  );
}
```

这里用 `watch('method')` 是因为 UI 需要在 method 变化时更新——==条件渲染天然需要 watch==。这不是性能问题，而是必要的需求。

## 9.5 目录结构建议

当你的项目有多个表单时，一致性比任何具体的目录方案都重要。以下是一个可参考的结构：

```
src/
└── features/
    └── user/
        ├── schemas.ts              # 所有 Zod schema 集中管理
        ├── RegisterForm.tsx        # 注册表单
        ├── EditUserForm.tsx        # 编辑用户
        └── fields/                 # 可复用字段组件
            ├── FormField.tsx        # label + error 包装器
            ├── EmailField.tsx       # 邮箱字段（含校验 + 无障碍属性）
            └── PasswordField.tsx    # 密码字段
```

`schemas.ts` 是"单一真相来源"——所有校验规则和类型在这里定义，所有表单文件从这里引用。新同事想了解"用户表单有哪些规则"，看这一个文件就够了。

## 9.6 最佳实践清单

### 性能
1. 默认 `onSubmit` 校验——只有体验确实需要、且字段数量少时才改 `onBlur`
2. 给 `watch` 传具体字段名——`watch('name')` 而非 `watch()`（全局监听等于退化为受控）
3. 不需要 UI 变化的判断用 `getValues` 而非 `watch`
4. 原生元素用 `register`——==能用 register 绝不用 Controller==

### 架构
5. Schema 集中放在独立文件（`schemas.ts`），不散落在 JSX 里
6. 可复用字段包装成组件——至少 label + error 的组合抽出来
7. 巨型表单拆分用 `FormProvider` + `useFormContext`——不要在 props 里层层传递 methods
8. 服务端校验错误通过 `setError` 注入，保持错误来源统一

### 防坑
9. `formState` 必须解构订阅——`const { formState: { errors } } = useForm()`
10. `useFieldArray` 的 key 必须用 `field.id`
11. `<input type="number">` 配合 `valueAsNumber: true` 使用
12. `<form>` 上加 `noValidate` 禁用浏览器原生校验，和 react-hook-form 不冲突

## 9.7 常见反模式：这些代码不要写

```tsx
// ❌ 反模式 1：注册表单用 Controller 包原生 input
<Controller name="email" control={control} render={({ field }) => (
  <input {...field} />
)} />
// ✅ 应该直接用 register——免去 Controller 的额外渲染开销

// ❌ 反模式 2：全局 watch
const values = watch(); // 任何字段变化都重渲染——表单退化为受控
// ✅ 监听具体字段
const email = watch('email');

// ❌ 反模式 3：手动维护类型 + 校验两套东西
interface User { name: string; email: string; } // 写了一边
<input {...register('email', { required: '必填' })} /> // 又写了一遍
// ✅ Zod schema 定义一次，z.infer 推导类型

// ❌ 反模式 4：key 用 index
{fields.map((field, index) => <div key={index}>...</div>)}
// ✅ key 用 field.id
{fields.map((field, index) => <div key={field.id}>...</div>)}
```

## 全书回顾

react-hook-form 学到这儿，核心就三件事：

1. **别让 React 管 input 的值**（非受控模式）— 性能的根本。DOM 是值的权威来源，React 只在提交时介入。
2. **在合适的时机校验**（默认提交时，按需提前）— 体验的平衡。先正确后体验，不要反着来。
3. **规则和 UI 分开**（Zod Schema）— 代码的整洁。定义一次，类型和校验永远同步。

工具会更新、API 会变化，但这三个原则是稳定的。忘掉具体 API 名字没关系——记住这三个"为什么"，查文档就能找回"怎么做"。
