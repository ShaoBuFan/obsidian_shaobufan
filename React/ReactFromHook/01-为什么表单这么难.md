# 第一章：为什么表单这么难

> 在学任何工具之前，先回到问题的原点：没有这个工具会怎样？它到底解决了什么根本问题？这不是哲学思辨——这是让你在遇到任何新场景时都能独立判断"该怎么做"的底层思维框架。

---

## 1.1 从一个输入框说起

打开浏览器，页面上有一个 `<input>` 标签。你敲一下键盘，字符出现了。

这时候发生了两件事：
1. **DOM 记住了你敲的字符**——这是浏览器自己的行为，天生就会
2. **React 可能知道也可能不知道**——这取决于你怎么写

这两件事之间的关系，就是 React 表单一切问题的起源。

## 1.2 让 React"知道"一切——受控组件

初学者（包括曾经的你我）会这样写：

```tsx
function Form() {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  return (
    <form>
      <input
        value={name}
        onChange={(e) => setName(e.target.value)}
      />
      <input
        value={email}
        onChange={(e) => setEmail(e.target.value)}
      />
      <button type="submit">提交</button>
    </form>
  );
}
```

这种写法叫 ==受控组件（Controlled Component）==。名字唬人，意思很简单：**值存在 React 的 state 里，input 只是一个傀儡——React 说什么它就显示什么。**

### 每一步发生了什么

你敲了一个 `h`：
1. `onChange` 触发 → `setName('h')`
2. React 标记这个组件需要更新
3. React 重新执行整个函数 → 生成新虚拟 DOM → 对比旧虚拟 DOM → 更新真实 DOM
4. 页面显示 `h`

接着你敲了 `i`（想打 `hi`）：同样的流程再跑一遍。

**你敲了 2 次键盘，React 重新渲染了 2 次。**

> 这里的关键不是"渲染了 2 次"。而是每次渲染，**整个组件的 JSX 都重新执行了**——包括其他所有字段的 JSX。即使其他字段的值完全没变。

现在想象一个 30 个字段的注册表单。每敲一个字符，所有 30 个字段对应的 JSX **全部重新执行一遍**。99% 的计算都是在做无用功——其他 29 个字段根本没变。

> **根本问题**：React 的受控组件设计假设"渲染够快，不需要担心"。但对于表单，这个假设不成立——用户连续快速输入，每次按键都触发全量渲染，输入延迟肉眼可见。这不是"React 优化不够好"，而是**这个模式本身就制造了不必要的渲染**。

## 1.3 换个思路：DOM 自己就能存值

等一下。浏览器的 `<input>` 天生就能自己存值——不需要 JavaScript，不需要 React state。

你在纸上填表的时候，纸自己就"记住"了内容，不需要有人站在旁边反复抄写。那为什么 React 要"记住"每次按键呢？

==非受控组件（Uncontrolled Component）==就是让 DOM 自己管理值，React 只在需要的时候才去读。

### 一个类比把它说透

| | 受控组件 | 非受控组件 |
|---|---|---|
| 想象成 | 实时监控：你每写一笔，旁边有个人就抄一遍，然后大声念出来 | 收卷：你写你的，老师最后收卷时一起看 |
| 值存在哪 | React state | DOM |
| 每次按键是否渲染 | 是 | 否 |
| 什么时候拿值 | 随时（state 里一直有） | 提交时（去 DOM 里读） |

react-hook-form 选择了**收卷模式**。你写你的，提交的时候一次性收。

## 1.4 react-hook-form 的三段式设计

```
注册阶段              输入阶段              提交阶段
───────             ────────             ────────
register('name')    用户打字             handleSubmit(onSubmit)
告诉表单：            DOM 自己存值         集中读取所有字段的值
"有个字段叫 name"     React 不介入         校验 → 回调
```

**只在两个时间点做事——注册时建立索引，提交时读取数据。中间什么都不干。** 这就是它快的根本原因。

> 为什么其他库不默认这么做？因为"非受控"有一个代价——你不能实时知道值是什么。如果你需要根据一个字段的值控制另一个字段的显示（比如选"其他"才显示输入框），就得额外处理。react-hook-form 提供了 `watch` 函数应对这种场景——它不是设计缺陷，而是刻意的取舍：**默认零渲染，需要时再开。** 这个开关由你主动选择，不是库替你决定。

## 1.5 三行代码感受这个理念

```tsx
import { useForm } from 'react-hook-form';

function HelloForm() {
  const { register, handleSubmit } = useForm();

  const onSubmit = (data) => {
    console.log(data); // 提交时才拿到的值——之前 React 完全不知道
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('name')} />
      <button type="submit">提交</button>
    </form>
  );
}
```

`useForm()` 给你一套工具。`register` 绑定字段——告诉表单"有个 input 叫 name"。`handleSubmit` 收集数据——到点了，把所有值交给你。

注意这个 input 上没有 `value`，没有 `onChange`。它是一个**普通的、自由的 input**——在 DOM 里自己呼吸，不需要每敲一个键都去向 React 汇报。

## 本章要点

- ==受控组件==的根本问题是**每次按键都触发整个表单重渲染**——不是 React 优化不够，是这个模式本身就制造了不需要的渲染
- ==非受控组件==信任 DOM 自己管理值，只在需要时（提交）才读取
- react-hook-form 的三段式设计：**注册时建立索引 → 输入时零渲染 → 提交时集中读取**
- 非受控的代价是"不能实时知道值"，`watch` 是按需开启的补偿——这是刻意取舍，不是设计缺陷
- `{...register('name')}` 就是往原生元素上绑几个属性——你的 input 还是你的 input

## 思考题

1. 如果一个表单只有 2 个字段，受控组件和非受控组件的性能差距有意义吗？为什么 react-hook-form 仍然选择非受控作为默认？
2. 找一个线上的长表单（比如某个网站的注册页），想象一下如果每个字段完全受控，每次按键会发生什么。这个表单有多少个字段？每次按键渲染多少次？
3. "DOM 自己存值"听起来很简单。为什么 React 社区花了这么久才广泛接受非受控模式？这和 React 早年的设计潮流有什么关系？
