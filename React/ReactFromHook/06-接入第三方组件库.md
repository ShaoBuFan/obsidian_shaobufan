# 第六章：接入第三方组件库

> 此前我们一直在用原生 `<input>`。但现实项目中你大概率在用 MUI、Ant Design、React Select 这类组件库。它们不是原生元素——你没办法把 `...register()` 展开到它们上面。本章解决这个对接问题。

---

## 6.1 为什么 `...register()` 在第三方组件上会"失灵"

`register` 返回的东西长这样：

```tsx
{ ref, onChange, onBlur, name }
```

展开到 `<input>` 上完全没问题——input 天生就接收这些属性。但 `<TextField>`（MUI）或 `<Select>`（Ant Design）呢？

这些组件的内部是一个 `<input>` 被一堆 `<div>` 包着，值存在组件自己的 state 里。它们用的是 **受控模式**——`value` + `onChange`——而不是 `ref` + `onChange`。

**所以你没办法把 `register` 展开到它们上面。它们需要的接口是 `value={x} onChange={setX}`，不是 `ref={someRef}`。**

---

## 6.2 Controller：翻译官

`Controller` 的作用：==把 react-hook-form 的非受控信号翻译成受控组件能理解的信号==。

```
react-hook-form（非受控）  →  Controller（翻译）  →  第三方组件（受控）
  内部用 ref 跟踪值          转换为 value + onChange       "我的 value 是 xxx"
                              fieldState 提取错误          "我有错误，显示红色框"
```

### MUI 的完整例子

```tsx
import { useForm, Controller } from 'react-hook-form';
import TextField from '@mui/material/TextField';

function MuiForm() {
  const { control, handleSubmit } = useForm();

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <Controller
        name="email"
        control={control}
        rules={{ required: '邮箱必填' }}
        render={({ field, fieldState }) => (
          <TextField
            {...field}                          // { value, onChange, onBlur, ref }
            label="邮箱地址"
            error={!!fieldState.error}
            helperText={fieldState.error?.message}
          />
        )}
      />
      <button type="submit">提交</button>
    </form>
  );
}
```

### 解剖 render 的两个参数

```tsx
render={({ field, fieldState }) => {
  // field = {
  //   value: any,          // 字段当前值
  //   onChange: Function,  // 更新值（接收事件或直接值）
  //   onBlur: Function,    // 标记字段被触碰
  //   ref: callback ref,   // 绑定 DOM 节点
  //   name: string,        // 字段名
  // }

  // fieldState = {
  //   error: FieldError | undefined,   // 校验错误
  //   isTouched: boolean,             // 失去过焦点
  //   isDirty: boolean,               // 不同于 defaultValue
  //   invalid: boolean,               // 有校验错误（!!error 的简写）
  //   isValidating: boolean,          // 正在执行异步校验
  // }
}}
```

你把 `field` 展开到组件上，Controller 就完成了 ==非受控 → 受控==的翻译。

> **设计哲学**：Controller 用 render prop 而不是高阶组件。render prop 给你完全的渲染控制权——你想怎么传 props、加什么样式、用什么组件，都是你决定。库不干预你的 UI。这和 `register` 的设计理念一致：库退后一步，让你完全拥有自己的组件。

## 6.3 三种常见 UI 库的写法

### Ant Design

Ant Design 的 Input 对 `value` 和 `onChange` 的处理和原生 input 兼容，直接 `...field` 展开即可：

```tsx
import { Input } from 'antd';

<Controller
  name="username"
  control={control}
  render={({ field }) => (
    <Input {...field} placeholder="请输入用户名" />
  )}
/>
```

### React Select

React Select 的 `onChange` 返回的是整个 option 对象，不是事件。需要手动映射：

```tsx
import Select from 'react-select';

<Controller
  name="city"
  control={control}
  render={({ field: { value, onChange, ref } }) => (
    <Select
      ref={ref}
      value={value}
      onChange={(selectedOption) => onChange(selectedOption?.value)}
      options={[
        { label: '北京', value: 'beijing' },
        { label: '上海', value: 'shanghai' },
      ]}
    />
  )}
/>
```

关键：不能用 `...field`——`field.onChange` 期望原生事件，React Select 的 `onChange` 给的是对象。手动用 `onChange(selectedOption?.value)` 把 option 对象解成值。

### MUI DatePicker

日期选择器的值不是字符串，是 Dayjs 对象：

```tsx
import { DatePicker } from '@mui/x-date-pickers';
import dayjs from 'dayjs';

<Controller
  name="birthDate"
  control={control}
  render={({ field }) => (
    <DatePicker
      {...field}
      label="出生日期"
    />
  )}
/>
```

MUI DatePicker 接受 dayjs 对象作为 value，`field` 直接展开即可——MUI 已经处理了和 react-hook-form 的兼容。

## 6.4 useController：无需 render prop 的 Controller

如果你不喜欢 render prop 的嵌套，react-hook-form 提供了 ==`useController`==——以 hook 形式使用，和 `Controller` 共享相同的底层实现：

```tsx
import { useForm, useController } from 'react-hook-form';

function MuiField() {
  const { control } = useForm();

  const { field, fieldState } = useController({
    name: 'email',
    control,
    rules: { required: '邮箱必填' },
  });

  return (
    <TextField
      {...field}
      error={!!fieldState.error}
      helperText={fieldState.error?.message}
    />
  );
}
```

和 `Controller` 的区别：`Controller` 是**组件**——用 render prop 让你渲染任何东西；`useController` 是 **hook**——直接返回值，由你自己展开。没有谁更好，只是风格差异：render prop 适合多字段共用一个模板，hook 适合单个自定义逻辑。

> 注意：`useController` 和 `Controller` 一样，会在字段值变化时触发所在组件的重渲染——它本质上是受控模式。所以在原生元素上，==用 register 而非 useController==，这是同样的原则。

## 6.5 register vs Controller：判断标准

一个简单规则让你不用纠结：

```tsx
// 原生 HTML 元素 → register（零额外开销）
<input {...register('name')} />
<select {...register('country')}>...</select>
<textarea {...register('bio')} />

// 第三方受控组件 → Controller（必要的翻译层）
<Controller name="name" control={control} render={...} />
```

**判断标准**：你用的组件文档里，props 里是不是有 `value` 和 `onChange`？如果是 → Controller。如果它直接接受 `ref` → register。如果不确定，看组件的 TypeScript 类型——如果有 `ref` prop 且类型是 `React.Ref<HTMLInputElement>`，大概率可以用 register。

> react-hook-form 的立场：==能用 register 的地方，不要用 Controller。== Controller 是妥协——它是为那些"不收 ref"的第三方组件准备的翻译层。Controller 内部自己管理了状态，带来了额外的渲染。这不是 bug，而是必要的代价。但如果你在原生 input 上用 Controller，就是在浪费性能。

## 6.6 Checkbox 和 Radio 的特殊处理

原生的 checkbox 和 radio 在 react-hook-form 中需要特别注意，因为它们的值不在 `value` 里——在 `checked` 里：

```tsx
// 原生 checkbox
<input
  type="checkbox"
  {...register('agreeToTerms')}
/>
// 提交时 data.agreeToTerms 是 boolean——react-hook-form 自动处理

// Controller 版本的 checkbox（用 MUI Checkbox）
<Controller
  name="agreeToTerms"
  control={control}
  render={({ field: { value, onChange, ...rest } }) => (
    <Checkbox
      {...rest}
      checked={value}
      onChange={(e) => onChange(e.target.checked)}
    />
  )}
/>
```

## 本章要点

- 第三方组件是受控模式，接口是 `value + onChange`；`register` 展开（`ref + onChange`）不生效
- `Controller` 用 render prop 做 ==翻译——非受控信号转为受控信号==
- `field` = `{ value, onChange, onBlur, ref, name }`；`fieldState` = `{ error, isTouched, isDirty, invalid, isValidating }`
- 大多数情况 `...field` 直接展开即可；React Select 等特殊组件需手动映射 onChange
- ==能用 register 绝不用 Controller==——Controller 是必要的妥协，不是推荐方案
