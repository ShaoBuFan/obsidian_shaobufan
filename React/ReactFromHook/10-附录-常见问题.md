## A.1 属性展开式 vs 包装组件式：两种库哲学

react-hook-form 的 `register` 和 Formik 的 `<Field>` 代表了两种根本不同的库设计哲学。

### 包装组件式（Formik / Ant Design Form）

```tsx
// 库"拥有"了你的 input——它被包裹在一个库组件内部
<Form.Item name="email" rules={[...]}>
  <Input />
</Form.Item>
```

### 属性展开式（react-hook-form）

```tsx
// 库对你的 input "毫不知情"——它只返回了四个属性
<input {...register('email')} />
```

`register('email')` 返回的只是：`{ ref, onChange, onBlur, name }`。`{...register('email')}` 和手动写 `ref={...} onChange={...} onBlur={...} name="email"` 完全等价。react-hook-form 没有创建任何额外的 DOM 层级。

### 为什么这很重要

| | 包装组件式 | 属性展开式 |
|---|---|---|
| 库的角色 | **框架**——定义你怎么写表单 | **工具**——帮你省事 |
| 对你的假设 | 你需要被引导 | 你知道自己在做什么 |
| 灵活性 | 库允许什么你就能做什么 | 库不管你怎么用 |
| 迁移成本 | 高（换库 = 所有 `<Form.Item>` 要重写） | 低（删掉 `...register()`，input 还是那个 input） |
| 额外 DOM 节点 | 有（每个字段多一层 wrapper） | 零 |

> react-hook-form 的核心设计原则：==做最小的介入，保持最大的透明。== 它不试图成为你表单的"框架"，不定义你的组件结构。`{...register()}` 的本质是——库退后一步，让你完全拥有自己的 input。

Controller 恰好是包装组件式的例外——它存在是因为第三方组件不接受单独的 `ref`，要 `value + onChange`。这种翻译必须有一个组件来管理状态。但 react-hook-form 的立场很清楚：==能用 register 解决的问题，不要用 Controller==。Controller 是妥协，不是推荐。

## A.2 受控 vs 非受控：根本区别是什么

从第一性原理理解这个问题：

浏览器原生 `<input>` 天生就能自己维护值——用户打字，字符出现。这个过程不需要任何 JavaScript。

React 的"受控组件"是用 state **接管**了这个天然能力：每次按键 → `onChange` → `setState` → React 把新值写回 input。原本 input 自己就能做好的事，变成了 React 每次都插手。

类比：
- **受控组件**：实时监控——你每写一笔，旁边有人抄一遍，然后大声念出来
- **非受控组件**：收卷——你写你的，老师最后一起看

react-hook-form 的选择：==默认非受控==。只在提交时集中读一次 DOM。期间零渲染。代价是不能实时知道值——需要时用 `watch` 开启。但这个开关是你主动选的，不是默认行为。

## A.3 关注点分离：表单的三层架构

每个表单做了三件事。混在一起是面条代码，分开就是清晰的架构：

```
业务规则层              交互逻辑层              视图层
─────────────────      ────────────────       ──────────────
定义"什么是对的"         管理"什么时候做"         负责"长什么样"

"名字 2-20 个字符"      "失焦时校验？提交时？"    <input> 还是 <TextField>
"邮箱必须合法"          "错误怎么收集和分发"     红色边框还是弹窗
"两次密码必须一致"       "提交后怎么 reset"       placeholder 写什么
"年龄 1-150"           "isDirty 怎么追踪"      原生还是 MUI
```

### 三层在代码中的对应

```
Zod Schema            react-hook-form         JSX / 第三方组件
（业务规则层）           （交互逻辑层）            （视图层）

z.object({            useForm({               <form>
  name: z.string()     resolver: ...           <input {...register('name')} />
})                    })                      </form>

可独立测试              可切换模式               可自由更换组件
多端复用                mode/errors/watch       完全由你控制
```

> 校验规则不应该关心它校验的值来自哪个 input，input 也不应该关心它展示的是谁的校验错误。两者唯一的联系是一个字段名——`'email'`。这个字符串就是它们之间的接口。

## A.4 单一真相来源（Single Source of Truth）

对于任何一项知识，系统中只应存在**一个权威定义**。如果同一项知识出现在多个地方，它们迟早会不一致——不是"可能"，是"迟早"。

### 表单场景的例子

没有 SSoT 时，"邮箱字段的规则"散布在三个地方——TypeScript 类型、`register` 的校验参数、后端的验证逻辑。三份独立的定义，没有编译器能保证它们一致。改了类型忘了改校验规则 → 编译通过，运行时才炸。

有了 Zod 之后：

```typescript
// schemas.ts——整个项目中 email 字段规则的唯一权威来源
const emailField = z.string().email('邮箱格式不正确');

const RegisterSchema = z.object({ email: emailField });

// 类型从这里推导——不用再写 interface
type FormData = z.infer<typeof RegisterSchema>;
```

回答"email 字段的规则是什么"→ 看 `schemas.ts` 一个文件就够了。

> 任何问题，你都应该能指着一个地方说"答案在这"。如果你的回答是"要看情况，看 A 文件但是 B 组件里可能覆盖了..."——那你就没有单一真相来源。

## A.5 DRY 原则：不是"别写重复代码"

DRY（Don't Repeat Yourself）是被误解最深的原则之一。它的原始定义（出自《程序员修炼之道》）：

> Every piece of knowledge must have a single, unambiguous, authoritative representation within a system.

关键词是 ==knowledge（知识）==，不是 code（代码）。两段一模一样的代码，如果表达的是不同知识——不是重复。两段不同的代码，如果表达的是同一个知识——就是重复。

### 在表单中的三层体现

**第一层（代码级）**：消除样板——把 label + error 的组合抽成 `<FieldWrapper>`

**第二层（知识级）**：同一个知识只定义一次——邮箱校验规则放在 Zod schema 里，三个表单引用同一份

**第三层（架构级）**：跨端共享——前端和后端消费同一套校验规则定义（理想状态）

> DRY 不是"别写重复代码"，是=="同一项知识，不该有两个表达"==。代码碰巧一样不是 DRY 的信号——知识重复才是。

### 什么时候不该 DRY

强行 DRY 会让两个独立的知识被错误地绑在一起：

```typescript
// 注册表单：名字 2-20 个字符
// 收货地址表单：名字 2-50 个字符
// → 它们是不同的知识——即使都叫 "name"
const registerName = z.string().min(2).max(20);
const addressName = z.string().min(2).max(50);
// 不该合并！改一个不应该影响另一个
```

## A.6 DRY、SSoT、SoC 三者怎么配合

三个原则看同一个问题，从不同角度：

| 原则 | 问的问题 | 解决什么 |
|------|----------|----------|
| DRY | "这个知识出现了几次？" | 知识重复 |
| SSoT | "如果多次出现，哪个是权威？" | 权威模糊 |
| SoC | "这个知识属于哪一层？" | 职责混乱 |

在表单里的协作：

```
SoC 告诉你：校验规则应该独立于 UI
    ↓
SSoT 告诉你：校验规则只应该定义一次
    ↓
DRY 告诉你：不要在三个表单里各写一遍邮箱正则
    ↓
结果：一个 Zod schema 文件，定义所有规则，所有表单引用它
```

> 两者缺一不可：只有 SSoT 没有 SoC → 规则只有一份但和 UI 混在一起；只有 SoC 没有 SSoT → 层分开了但每层各有一份规则的副本。

## A.7 Controller 的 field 和 fieldState 完整字段

在 [第六章](06-接入第三方组件库.md) 我们看到 Controller 的基本用法。这里是 render 回调的完整 API 参考：

### field（ControllerRenderProps，6 个属性）

| 属性 | 类型 | 说明 |
|------|------|------|
| `value` | `any` | 字段当前值。非受控模式下为 `undefined` 直到提交 |
| `onChange` | `(...event: any[]) => void` | 更新值。标准事件直接传；React Select 等返回 option 的手动映射 |
| `onBlur` | `() => void` | 触发失焦，库内部标记 `isTouched` |
| `ref` | `React.Ref<any>` | **callback ref**——可以和你自己的 `useRef` 共存 |
| `name` | `string` | 字段名 |
| `disabled` | `boolean` | 从 Controller 的 `disabled` prop 传入 |

**ref 为什么用 callback ref？** 因为 callback ref 不创建新对象——只是一个函数调用。你可以同时绑定自己的 `useRef` 和 Controller 的 `ref`，两者不冲突。如果 react-hook-form 用的是 `useRef` 返回的对象 ref，你就没法再绑自己的 ref 了——一个元素只能有一个 `ref` 属性。

### fieldState（ControllerFieldState，5 个属性）

| 属性 | 类型 | 说明 |
|------|------|------|
| `error` | `FieldError \| undefined` | 当前字段的校验错误——结构 `{ type, message, ref }` |
| `isTouched` | `boolean` | 字段失去过焦点 |
| `isDirty` | `boolean` | 字段当前值 ≠ `defaultValue` |
| `invalid` | `boolean` | 有校验错误——`!!error` 的简写 |
| `isValidating` | `boolean` | 正在执行异步校验 |

### 状态组合的 UI 决策

```
fieldState           →    渲染决策
────────                  ────────
isTouched = false          正常输入框（不显示错误状态）
isTouched = true
  invalid = false          绿色边框（合法）
  invalid = true           红色边框 + 错误消息
isValidating = true        loading spinner
isDirty = true             未保存标记
```

## A.8 常见误区速查

| 误区 | 正确理解 |
|------|----------|
| "react-hook-form 不支持受控组件" | 支持——Controller 就是为受控组件准备的。只是默认推荐非受控 |
| "formState 不解构也能用" | 不解构拿到的是旧值——Proxy 需要知道你在乎什么 |
| "Controller 比 register 更通用，干脆全用 Controller" | Controller 每字段多一层组件实例和 render 调用——在原生元素上纯属浪费 |
| "watch() 全量监听很方便" | 等于把表单退化回受控模式——每次按键全表单渲染 |
| "Zod 和 Yup 差不多，用哪个都无所谓" | 新项目用 Zod，TypeScript 体验好一档。但已有 Yup 的项目不必迁移 |
| "把校验规则写在 register 里更方便——不用多建一个文件" | 3 个字段时方便的，30 个字段时是噩梦。技术决策要考虑规模增长 |
| "useFieldArray 的 key 用 index 就行，Demo 里经常这么写" | Demo 里不删不减所以没有 bug。真实项目中用户会删——用 index 会在删除后导致值和位置错位 |
