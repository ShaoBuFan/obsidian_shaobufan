# 第11章：状态管理测试

## 学习目标

- 理解状态管理的测试立场——测试行为而非容器
- 掌握 React Context 的测试模式：wrapper 注入与 value 变更
- 能直接测试 Zustand store 逻辑并在组件中集成验证
- 熟练配置 TanStack Query 测试环境并验证缓存、乐观更新等行为
- 能封装 `renderWithProviders` 处理多 Provider 嵌套

---

## 11.1 状态管理的测试立场

### 核心原则

状态管理（Context、Zustand、TanStack Query）是组件之间的"隐形管道"。用户不关心你的状态存在哪里、怎么流动——他们只关心**页面上看到的内容是否正确**。

```
用户视角：
  点击按钮 → 页面数据更新

测试视角：
  点击按钮 → 状态变更 → 组件重渲染 → DOM 变化

不应该测试：
  点击按钮 → dispatch action → reducer 返回新 state  → store 更新
  （这些是实现细节）
```

因此状态管理测试的第一原则：**通过组件行为间接验证状态逻辑**，而非直接测试状态容器本身。

### 何时可以直接测试 Store

直接测试 store（不通过组件）只有在以下场景才有价值：

1. **复杂推导逻辑**：store 中有 selector 或 computed 值，用纯函数测试更高效
2. **中间件行为**：验证 persist、immer、devtools 等中间件的效果
3. **纯数据层测试**：store 作为"无 UI 的数据处理管道"

除这些场景外，都应该通过渲染组件、模拟用户交互来验证状态管理的正确性。

## 11.2 React Context 测试

### Context 的测试挑战

Context 的本质是 React 的依赖注入机制。测试 Context 依赖的组件时，有两个关键问题：

1. **Provider 缺失**：组件渲染时如果找不到 Provider，会使用默认值或抛出错误
2. **Value 变更**：Provider 的 value 变化后，消费者组件应自动重渲染

### 基础模式：wrapper 注入

```typescript
import { render, screen } from '@testing-library/react'
import { createContext, useContext, type ReactNode } from 'react'

// ---- 被测试的代码 ----

interface User {
  name: string
  role: 'admin' | 'user'
}

const UserContext = createContext<User | null>(null)

function useUser(): User {
  const ctx = useContext(UserContext)
  if (!ctx) throw new Error('useUser must be used within UserProvider')
  return ctx
}

function WelcomeMessage() {
  const user = useUser()
  return <h1>Welcome, {user.name}</h1>
}

// ---- 测试 ----

describe('WelcomeMessage', () => {
  it('renders user name from context', () => {
    const user: User = { name: 'Alice', role: 'admin' }

    render(<WelcomeMessage />, {
      wrapper: ({ children }: { children: ReactNode }) => (
        <UserContext.Provider value={user}>{children}</UserContext.Provider>
      ),
    })

    expect(screen.getByRole('heading', { name: /welcome, alice/i })).toBeInTheDocument()
  })

  it('throws when rendered without provider', () => {
    // 注意：在测试中捕获 React 错误需要特殊处理
    // React 会把 thrown error 转为 console.error
    const consoleSpy = vi.spyOn(console, 'error').mockImplementation(() => {})

    expect(() => render(<WelcomeMessage />)).toThrow(
      'useUser must be used within UserProvider'
    )

    consoleSpy.mockRestore()
  })
})
```

> **自我验证说明**：`render` 的 `wrapper` 选项接受一个 React 组件，该组件接收 `{ children: ReactNode }` 并返回包裹后的内容。TypeScript 签名：`wrapper?: React.JSXElementConstructor<{ children: React.ReactNode }>`。使用时注意 Provider 必须是 `children` 的直接父级，不能有额外的样式容器。

### 修改 Context Value 验证消费者响应

```typescript
import { render, screen } from '@testing-library/react'
import {
  createContext,
  useContext,
  useState,
  type ReactNode,
  type Dispatch,
  type SetStateAction,
} from 'react'
import userEvent from '@testing-library/user-event'

// ---- 被测试的代码 ----

interface ThemeContextType {
  theme: 'light' | 'dark'
  toggle: () => void
}

const ThemeContext = createContext<ThemeContextType | null>(null)

function useTheme(): ThemeContextType {
  const ctx = useContext(ThemeContext)
  if (!ctx) throw new Error('useTheme must be used within ThemeProvider')
  return ctx
}

function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<'light' | 'dark'>('light')
  const toggle = () => setTheme((t) => (t === 'light' ? 'dark' : 'light'))

  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  )
}

function ThemeButton() {
  const { theme, toggle } = useTheme()
  return (
    <button onClick={toggle} aria-label="Toggle theme">
      Current: {theme}
    </button>
  )
}

// ---- 测试 ----

describe('ThemeButton', () => {
  it('toggles theme when clicked', async () => {
    const user = userEvent.setup()

    render(<ThemeButton />, {
      wrapper: ({ children }) => <ThemeProvider>{children}</ThemeProvider>,
    })

    const btn = screen.getByRole('button', { name: /toggle theme/i })
    expect(btn).toHaveTextContent('Current: light')

    await user.click(btn)
    expect(btn).toHaveTextContent('Current: dark')

    await user.click(btn)
    expect(btn).toHaveTextContent('Current: light')
  })

  it('responds to external theme change', () => {
    // 通过 wrapper 直接控制传入的 value
    const setTheme = (): void => {}
    const toggle = vi.fn()

    const { rerender } = render(<ThemeButton />, {
      wrapper: ({ children }) => (
        <ThemeContext.Provider value={{ theme: 'light', toggle }}>
          {children}
        </ThemeContext.Provider>
      ),
    })

    expect(screen.getByRole('button')).toHaveTextContent('light')

    rerender(<ThemeButton />, {
      wrapper: ({ children }) => (
        <ThemeContext.Provider value={{ theme: 'dark', toggle }}>
          {children}
        </ThemeContext.Provider>
      ),
    })

    expect(screen.getByRole('button')).toHaveTextContent('dark')
  })
})
```

> **自我验证说明**：第二个测试模式在实际项目中很有用——当你想测试消费者组件对 context value 变化的响应时，可以直接控制传入的 value 而无需通过完整的 Provider 逻辑。`rerender` 会重新渲染组件并触发 context 的更新。

## 11.3 Zustand Store 测试

### 测试策略

Zustand store 是独立于 React 组件树的——它是一个纯 TypeScript 对象，可以在不渲染组件的情况下直接测试。这给了我们两种测试路径：

```
路径 A（直接测试）：store.getState() → store.getState() 验证
路径 B（组件测试）：渲染组件 → 用户交互 → 断言 DOM 变化
```

### 直接测试 Store 逻辑

```typescript
import { createStore } from 'zustand/vanilla'
import { describe, expect, it } from 'vitest'

// ---- Store 定义 ----

interface CartItem {
  id: string
  name: string
  price: number
  quantity: number
}

interface CartState {
  items: CartItem[]
  total: number
  addItem: (item: Omit<CartItem, 'quantity'>) => void
  removeItem: (id: string) => void
  updateQuantity: (id: string, quantity: number) => void
}

function createCartStore() {
  return createStore<CartState>((set, get) => ({
    items: [],
    total: 0,

    addItem: (item) => {
      const existing = get().items.find((i) => i.id === item.id)
      if (existing) {
        get().updateQuantity(item.id, existing.quantity + 1)
      } else {
        set((state) => ({
          items: [...state.items, { ...item, quantity: 1 }],
        }))
      }
      // 更新 total
      const items = get().items
      set({ total: items.reduce((sum, i) => sum + i.price * i.quantity, 0) })
    },

    removeItem: (id) => {
      set((state) => ({
        items: state.items.filter((i) => i.id !== id),
      }))
      const items = get().items
      set({ total: items.reduce((sum, i) => sum + i.price * i.quantity, 0) })
    },

    updateQuantity: (id, quantity) => {
      set((state) => ({
        items: state.items.map((i) => (i.id === id ? { ...i, quantity } : i)),
      }))
      const items = get().items
      set({ total: items.reduce((sum, i) => sum + i.price * i.quantity, 0) })
    },
  }))
}

// ---- 直接 Store 测试 ----

describe('CartStore', () => {
  it('adds item with quantity 1', () => {
    const store = createCartStore()

    store.getState().addItem({ id: '1', name: 'Widget', price: 10 })

    const state = store.getState()
    expect(state.items).toHaveLength(1)
    expect(state.items[0]).toMatchObject({ id: '1', quantity: 1 })
    expect(state.total).toBe(10)
  })

  it('increments quantity when adding existing item', () => {
    const store = createCartStore()

    store.getState().addItem({ id: '1', name: 'Widget', price: 10 })
    store.getState().addItem({ id: '1', name: 'Widget', price: 10 })

    const state = store.getState()
    expect(state.items).toHaveLength(1)
    expect(state.items[0].quantity).toBe(2)
    expect(state.total).toBe(20)
  })

  it('removes item and updates total', () => {
    const store = createCartStore()

    store.getState().addItem({ id: '1', name: 'A', price: 10 })
    store.getState().addItem({ id: '2', name: 'B', price: 20 })
    store.getState().removeItem('1')

    const state = store.getState()
    expect(state.items).toHaveLength(1)
    expect(state.items[0].id).toBe('2')
    expect(state.total).toBe(20)
  })

  it('updates quantity', () => {
    const store = createCartStore()

    store.getState().addItem({ id: '1', name: 'Widget', price: 10 })
    store.getState().updateQuantity('1', 5)

    const state = store.getState()
    expect(state.items[0].quantity).toBe(5)
    expect(state.total).toBe(50)
  })
})
```

> **自我验证说明**：`createStore` 来自 `zustand/vanilla`，返回一个不依赖 React 的 store 实例。`store.getState()` 获取当前快照，`store.subscribe()` 监听变化——都不需要 React 环境。测试文件顶部加 `// @vitest-environment node` 即可在 Node 环境下运行，比 jsdom 更快。

### 组件集成测试

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { create } from 'zustand'
import { describe, expect, it, beforeEach } from 'vitest'
import { type ReactNode } from 'react'

// ---- 被测试的代码 ----

interface CounterState {
  count: number
  increment: () => void
  reset: () => void
}

const useCounterStore = create<CounterState>((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  reset: () => set({ count: 0 }),
}))

function CounterDisplay() {
  const count = useCounterStore((s) => s.count)
  return <p role="status">Count: {count}</p>
}

function CounterControls() {
  const { increment, reset } = useCounterStore()
  return (
    <div>
      <button onClick={increment}>+</button>
      <button onClick={reset}>Reset</button>
    </div>
  )
}

function CounterWidget() {
  return (
    <div>
      <CounterDisplay />
      <CounterControls />
    </div>
  )
}

// ---- 测试 ----

describe('CounterWidget', () => {
  beforeEach(() => {
    // 每条测试前重置 store 到初始状态
    useCounterStore.setState({ count: 0 })
  })

  it('displays initial count', () => {
    render(<CounterWidget />)
    expect(screen.getByRole('status')).toHaveTextContent('Count: 0')
  })

  it('increments count on click', async () => {
    const user = userEvent.setup()
    render(<CounterWidget />)

    await user.click(screen.getByRole('button', { name: '+' }))

    expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
  })

  it('resets count on reset click', async () => {
    const user = userEvent.setup()
    useCounterStore.setState({ count: 5 }) // 预设状态
    render(<CounterWidget />)

    expect(screen.getByRole('status')).toHaveTextContent('Count: 5')

    await user.click(screen.getByRole('button', { name: /reset/i }))

    expect(screen.getByRole('status')).toHaveTextContent('Count: 0')
  })
})
```

> **自我验证说明**：Zustand 组件测试的关键是**每个测试前重置 store 状态**。由于 `useCounterStore` 是模块级单例，不同测试之间会共享同一个 store 实例。`useCounterStore.setState({ count: 0 })` 直接将状态重置到初始值，效果等价于调用 `reset()`。如果不重置，测试 B 会受到测试 A 的副作用污染。

### Mock 初始状态进行特定场景测试

## 11.4 TanStack Query 集成测试

### QueryClientProvider Wrapper

TanStack Query（React Query v5）的所有 hook 都依赖 `QueryClient` 提供的上下文。测试时的核心配置：

```typescript
import { render, screen, waitFor } from '@testing-library/react'
import {
  QueryClient,
  QueryClientProvider,
  useQuery,
} from '@tanstack/react-query'
import { http, HttpResponse } from 'msw'
import { server } from '../mocks/server'
import { type ReactNode } from 'react'

// ---- 被测试的 Hook ----

function useUser(id: number) {
  return useQuery({
    queryKey: ['user', id],
    queryFn: async () => {
      const res = await fetch(`/api/users/${id}`)
      if (!res.ok) throw new Error('Failed to fetch')
      return res.json() as Promise<{ id: number; name: string }>
    },
  })
}

function UserProfile({ userId }: { userId: number }) {
  const { data, isLoading, error } = useUser(userId)

  if (isLoading) return <p>Loading...</p>
  if (error) return <p role="alert">Error: {error.message}</p>
  if (!data) return null

  return <h1>{data.name}</h1>
}

// ---- 测试辅助 ----

function createTestQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        retry: false, // 测试中禁止重试，避免超时
        gcTime: 0,    // 关闭垃圾回收时间，防止缓存干扰
      },
    },
  })
}

function renderWithClient(ui: React.ReactElement) {
  const queryClient = createTestQueryClient()
  return render(ui, {
    wrapper: ({ children }: { children: ReactNode }) => (
      <QueryClientProvider client={queryClient}>
        {children}
      </QueryClientProvider>
    ),
  })
}

// ---- 测试 ----

describe('UserProfile', () => {
  it('shows loading state initially', () => {
    renderWithClient(<UserProfile userId={1} />)
    expect(screen.getByText(/loading/i)).toBeInTheDocument()
  })

  it('displays user name after fetch', async () => {
    renderWithClient(<UserProfile userId={1} />)

    // findByRole 内部等待异步数据加载完成
    const heading = await screen.findByRole('heading', { name: /alice/i })
    expect(heading).toBeInTheDocument()
  })

  it('handles fetch error', async () => {
    // 覆盖 MSW handler 返回 500
    server.use(
      http.get('/api/users/:id', () => {
        return new HttpResponse(null, { status: 500 })
      })
    )

    renderWithClient(<UserProfile userId={1} />)

    const alert = await screen.findByRole('alert')
    expect(alert).toHaveTextContent(/error/i)
  })
})
```

> **自我验证说明**：`createTestQueryClient` 的关键配置：`retry: false` 防止失败的查询自动重试（默认重试 3 次，每次间隔翻倍，会大幅拖慢测试），`gcTime: 0` 防止不同测试间缓存污染。这两个配置是 TanStack Query 组件测试的标配。注意 v5 中移除了 `cacheTime`，替换为 `gcTime`。

> **Jest 对比：TanStack Query 测试环境搭建**
>
> 在 Jest 中测试 TanStack Query 组件，需要同样的 `QueryClientProvider` 封装，但环境搭建方式不同：
>
> ```typescript
> // Jest — 需要手动 mock fetch 或使用 jest-fetch-mock
> global.fetch = jest.fn() as jest.Mock
>
> // 或使用 jest-fetch-mock 库
> import fetchMock from 'jest-fetch-mock'
> fetchMock.enableMocks()
> ```
>
> Vitest + MSW 的组合优势在于：MSW 在网络层拦截请求（而不是 mock `fetch` 函数本身），这意味着即使组件的 fetch 调用方式发生变化（比如从 `fetch` 换成 `axios`），测试代码不需要改动。MSW handler 与 `QueryClient` 的配置完全独立，测试逻辑更清晰。Jest 生态中 MSW 同样可用（`msw/node`），但需要额外配置 `transformIgnorePatterns` 和 `testEnvironment`，而 Vitest 继承 Vite 的 plugin 体系，开箱即用。

> **为什么：测试缓存行为比测试渲染更重要**
>
> 新手往往只验证"数据是否正确渲染"，而忽略了 TanStack Query 最核心的价值——缓存管理。缓存行为决定了：
>
> 1. **性能**：重复请求是否被缓存命中？`staleTime` 是否合理配置？如果缓存没命中，每次重新挂载都会触发网络请求，这对用户体验是灾难
> 2. **一致性**：`invalidateQueries` 后数据是否自动刷新？这是 TanStack Query 区别于手动 `fetch + useState` 的核心能力——数据变更后无需手动同步
> 3. **用户体验**：乐观更新是否先展示变化、失败后回滚？这直接影响用户感知的响应速度，是"看起来快"和"真的快"的区别
>
> 测试缓存行为（如上例：验证相同 queryKey 只发一次请求）验证的是状态管理系统的核心契约，而非一次性的 UI 表现。**如果你只测渲染不测缓存，等于只测了 TanStack Query 的皮毛。**

### 缓存行为测试

```typescript
describe('query caching', () => {
  it('does not send duplicate requests for the same key', async () => {
    const fetchSpy = vi.fn()

    server.use(
      http.get('/api/users/:id', async ({ request }) => {
        fetchSpy(request.url)
        return HttpResponse.json({ id: 1, name: 'Alice' })
      })
    )

    // 第一次渲染：触发请求
    const { unmount } = renderWithClient(<UserProfile userId={1} />)
    await screen.findByRole('heading', { name: /alice/i })

    // 卸载后重新挂载：查询缓存应该命中，不触发新请求
    unmount()
    renderWithClient(<UserProfile userId={1} />)
    await screen.findByRole('heading', { name: /alice/i })

    // 应该只发了 1 次请求（第二次来自缓存）
    expect(fetchSpy).toHaveBeenCalledTimes(1)
  })
})
```

### 乐观更新（Optimistic Update）测试

乐观更新是 TanStack Query 最强大的模式之一：在请求完成前先更新 UI，请求失败后回滚。

```typescript
import {
  useMutation,
  useQueryClient,
  type QueryClient,
} from '@tanstack/react-query'

// ---- 被测试的 Hook ----

interface UpdateUserData {
  name: string
}

function useUpdateUser() {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: async ({
      id,
      data,
    }: {
      id: number
      data: UpdateUserData
    }) => {
      const res = await fetch(`/api/users/${id}`, {
        method: 'PATCH',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(data),
      })
      if (!res.ok) throw new Error('Update failed')
      return res.json() as Promise<{ id: number; name: string }>
    },

    // 乐观更新：请求发出前先更新缓存
    onMutate: async ({ id, data }) => {
      // 取消正在进行的查询（避免覆盖乐观更新）
      await queryClient.cancelQueries({ queryKey: ['user', id] })

      // 保存旧数据以便回滚
      const previousData = queryClient.getQueryData<{ id: number; name: string }>(
        ['user', id]
      )

      // 立即更新缓存
      queryClient.setQueryData(['user', id], (old: unknown) => ({
        ...(old as object),
        ...data,
      }))

      // 返回旧数据，用于 onError 回滚
      return { previousData }
    },

    // 请求失败：回滚到旧数据
    onError: (_err, { id }, context) => {
      if (context?.previousData) {
        queryClient.setQueryData(['user', id], context.previousData)
      }
    },
  })
}

function UserEditor({ userId }: { userId: number }) {
  const { data } = useUser(userId)
  const updateUser = useUpdateUser()

  const handleUpdate = () => {
    updateUser.mutate({ id: userId, data: { name: 'New Name' } })
  }

  return (
    <div>
      <h1>{data?.name ?? 'Loading...'}</h1>
      <button onClick={handleUpdate} disabled={updateUser.isPending}>
        Update
      </button>
      {updateUser.isError && (
        <p role="alert">Failed to update: {updateUser.error.message}</p>
      )}
    </div>
  )
}

// ---- 测试 ----

describe('UserEditor optimistic update', () => {
  it('shows new name immediately before server responds', async () => {
    const user = userEvent.setup()

    renderWithClient(<UserEditor userId={1} />)
    await screen.findByRole('heading', { name: /alice/i })

    // 点击更新按钮——UI 应该立即变化（不等服务器响应）
    await user.click(screen.getByRole('button', { name: /update/i }))

    // 乐观更新后 UI 立即显示新名字
    expect(screen.getByRole('heading')).toHaveTextContent('New Name')
  })

  it('rolls back to original name on failure', async () => {
    const user = userEvent.setup()

    renderWithClient(<UserEditor userId={1} />)
    await screen.findByRole('heading', { name: /alice/i })

    // 覆盖 handler 返回 500
    server.use(
      http.patch('/api/users/:id', () => {
        return HttpResponse.json(
          { message: 'Server error' },
          { status: 500 }
        )
      })
    )

    await user.click(screen.getByRole('button', { name: /update/i }))

    // 错误发生后，UI 应该回滚到原来的名字
    await screen.findByRole('heading', { name: /alice/i })

    // 错误提示应该出现
    expect(screen.getByRole('alert')).toHaveTextContent(/failed to update/i)
  })
})
```

> **自我验证说明**：乐观更新测试的关键是时序控制。`onMutate` 是同步执行的（`cancelQueries` 虽然是 async，但这里我们 await 它），所以乐观更新的 `setQueryData` 在 `mutate` 调用后立即生效。测试中不需要等待任何异步操作——UI 的改变在 `await user.click` 之前就已经发生。失败回滚时，需要等到 mutation 的 Promise reject 后，`onError` 中的 `setQueryData` 才会执行，此时 `findByRole` 因为 DOM 被重渲染，需要等待。

## 11.5 多 Provider 嵌套

真实项目中，一个组件通常依赖多个 Provider：`BrowserRouter`、`QueryClientProvider`、`ThemeProvider`、`UserProvider` 等。每次都手动嵌套 `wrapper` 是反模式。

### renderWithProviders 封装

```typescript
import { render, type RenderOptions, type RenderResult } from '@testing-library/react'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'
import { MemoryRouter } from 'react-router-dom'
import { type ReactElement, type ReactNode } from 'react'

// ---- 被封装的 Provider ----

interface ProvidersOptions {
  initialEntries?: string[]
  queryClient?: QueryClient
}

function createProviders({ initialEntries, queryClient }: ProvidersOptions = {}) {
  const qc = queryClient ?? new QueryClient({
    defaultOptions: {
      queries: { retry: false, gcTime: 0 },
    },
  })

  // 返回 Wrapper 组件
  return function Providers({ children }: { children: ReactNode }) {
    return (
      <MemoryRouter initialEntries={initialEntries}>
        <QueryClientProvider client={qc}>
          {children}
        </QueryClientProvider>
      </MemoryRouter>
    )
  }
}

// ---- 增强的 render 函数 ----

interface RenderWithProvidersOptions extends Omit<RenderOptions, 'wrapper'> {
  initialEntries?: string[]
  queryClient?: QueryClient
}

function renderWithProviders(
  ui: ReactElement,
  options: RenderWithProvidersOptions = {}
): RenderResult {
  const { initialEntries, queryClient, ...renderOptions } = options

  return render(ui, {
    wrapper: createProviders({ initialEntries, queryClient }),
    ...renderOptions,
  })
}

export { renderWithProviders, createProviders }
```

### 使用

```typescript
import { renderWithProviders } from './test-utils'

describe('PageWithMultipleProviders', () => {
  it('renders within router and query context', async () => {
    renderWithProviders(<UserProfile userId={1} />, {
      initialEntries: ['/users/1'],
    })

    expect(await screen.findByRole('heading', { name: /alice/i }))
      .toBeInTheDocument()
  })

  it('supports custom query client for specific test', () => {
    const queryClient = new QueryClient({
      defaultOptions: {
        queries: {
          retry: 3,
          retryDelay: 100,
          gcTime: 5000,
        },
      },
    })

    renderWithProviders(<SlowLoadingComponent />, { queryClient })
  })
})
```

> **自我验证说明**：`renderWithProviders` 的返回值与 `render` 完全一致（TypeScript 中 `RenderResult` 类型相同），所以可以无缝替换。`createProviders` 使用闭包创建 Wrapper 组件，每次调用 `renderWithProviders` 都会创建新的 `QueryClient` 和 `MemoryRouter`，确保测试间状态隔离。这个模式是第12章的主要内容。

> **渐进式示例：Provider 封装的三步进化**
>
> 在实际项目中，Provider 嵌套通常会经历三个阶段：
>
> **阶段 1：内联 wrapper（适合 1-2 个测试）**
> ```typescript
> render(<Component />, {
>   wrapper: ({ children }) => (
>     <QueryClientProvider client={queryClient}>
>       {children}
>     </QueryClientProvider>
>   ),
> })
> ```
> 优点：直接、无须额外函数；缺点：每个测试文件都要写一遍，Provider 增多后嵌套越来越深。
>
> **阶段 2：文件级 renderWithProviders（适合单个组件文件）**
> ```typescript
> function renderWithProviders(ui: ReactElement) {
>   return render(ui, {
>     wrapper: ({ children }) => (
>       <MemoryRouter>
>         <QueryClientProvider client={createTestQueryClient()}>
>           {children}
>         </QueryClientProvider>
>       </MemoryRouter>
>     ),
>   })
> }
> ```
> 优点：消除重复，返回类型与 `render` 一致；缺点：每个测试文件仍需要定义一次，标准不统一。
>
> **阶段 3：全局导出（适合项目级，推荐）**
> ```typescript
> // src/test/test-utils.tsx — 项目级工具文件
> export { renderWithProviders, createProviders }
> ```
> 所有测试文件从 `test-utils` 导入，统一修改。这是 12.2 节展开的内容。
>
> 建议的过渡策略：从阶段 2 开始（每个组件文件自己封装），当发现多数文件使用相同模式时升级到阶段 3。过早全局化可能导致过度抽象。

### 反模式 1：直接测试 Context 的 Provider 渲染

```typescript
// ❌ 测试 Context Provider 本身的渲染

  render(<ThemeButton />, {
    wrapper: ({ children }) => <ThemeProvider>{children}</ThemeProvider>,
  })
  expect(screen.getByRole('button')).toHaveTextContent('light')

```

### 反模式 2：测试中直接 import store 的单例修改

```typescript
// ❌ 直接修改模块级 store，污染其他测试
import { useCounterStore } from './counter-store'

it('test 1', () => {
  useCounterStore.setState({ count: 5 })
  // 测试通过，但 test 2 的初始状态也被改了
})

// ✅ 在 beforeEach 中重置 store
beforeEach(() => {
  useCounterStore.setState({ count: 0 })
})
```

### 反模式 3：在 TanStack Query 测试中忘记禁用 retry

```typescript
// ❌ 默认 retry: 3 会让失败的测试等待非常久
const queryClient = new QueryClient()
// 查询失败后会重试 3 次，每次间隔递增

// ✅ 测试中必须禁用重试
const queryClient = new QueryClient({
  defaultOptions: {
    queries: { retry: false },
  },
})
```

### 反模式 4：在 mutation 的 onSuccess 中 invalidate 所有 query

```typescript
// ❌ 全量 invalidate 造成非必要的重请求
onSuccess: () => {
  queryClient.invalidateQueries()
}

// ✅ 只 invalidate 相关 query

  queryClient.invalidateQueries({ queryKey: ['user', id] })
}
// 测试中也需要验证具体的 queryKey
```

## 11.8 本章总结

- 状态管理测试的第一原则：通过组件行为间接验证状态逻辑
- React Context 测试用 `wrapper` 注入 Provider，通过 `rerender` 模拟 value 变化
- Zustand 可以直接用 `createStore` 测试纯逻辑，组件测试前用 `setState` 重置
- TanStack Query 测试需要 `QueryClientProvider` wrapper，必须配置 `retry: false`
- 乐观更新的测试验证：onMutate 后 UI 立即变化，onError 后 UI 回滚
- 多 Provider 嵌套封装为 `renderWithProviders`，返回类型与 `render` 一致
- 每个测试后清理状态（store、query cache、MSW handlers）是测试隔离的基础
