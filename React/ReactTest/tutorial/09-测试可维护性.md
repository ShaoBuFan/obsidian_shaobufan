# 第九章：测试代码的可维护性

## 学习目标

- 理解测试代码也需要维护，需要相同的工程纪律
- 掌握测试命名约定与文件组织策略
- 掌握何时抽象、何时保留重复的判断标准
- 能设计可复用的测试工具库

---

## 9.1 测试即代码

一个常见的误区：测试代码是「二等代码」，对它的质量要求可以比生产代码低。

但测试代码的维护成本往往**高于**生产代码——生产代码重构时测试代码也需要同步更新。如果测试写得乱，每次重构都会花费大量时间在修测试而非改功能上。

### 测试代码的质量标准

1. **可读性优先于简洁**：一个陌生人在代码审查时应该能 30 秒内理解测试在测什么。为此多写几行代码是值得的。
2. **一个测试只测一个行为**：如果一个测试名中出现了「和」字，考虑拆成两个。
3. **独立性**：一个测试的失败不应该导致其他测试失败。
4. **确定性**：同一个测试跑 10 次，应该 10 次得到相同的结果。

## 9.2 测试命名约定

### 测试名称

```typescript
// ✅ 描述行为的命名
it('未登录用户访问 /admin 时重定向到登录页', () => {})
it('搜索返回空结果时显示「未找到」提示', () => {})
it('订单金额超过 1000 时自动应用 9 折优惠', () => {})

// ❌ 描述实现的命名
it('setState is called with correct value', () => {})
it('useEffect triggers on mount', () => {})
it('callback is invoked once', () => {})
```

### describe 层级

```typescript
describe('ProductList', () => {
  describe('数据加载', () => {
    it('显示加载指示器直到数据返回', () => {})
    it('API 返回 500 时显示错误信息', () => {})
    it('API 返回空数组时显示空状态', () => {})
  })

  describe('用户交互', () => {
    it('点击商品进入详情页', () => {})
    it('点击排序按钮切换排序方式', () => {})
  })

  describe('边界情况', () => {
    it('网络断开时显示离线提示', () => {})
    it('组件卸载时取消进行中的请求', () => {})
  })
})
```

## 9.3 文件组织

### 方案一：Co-located（推荐）

```
src/components/SearchForm/
  SearchForm.tsx
  SearchForm.test.tsx     # 紧邻源文件
  SearchForm.stories.tsx
```

### 方案二：独立 tests 目录

```
src/
  components/SearchForm/
    SearchForm.tsx
  __tests__/
    components/
      SearchForm.test.tsx
```

**推荐 Co-located**，原因：
- 新增组件时不会忘记创建对应的测试目录
- 删除组件时测试同步删除
- 代码审查时能在一个 diff 中同时看到实现和测试

### 测试工具放哪

测试工具（test-utils, factories, mocks）应该放在统一的 `src/test/` 目录下——它们被多个测试文件共享，不能 co-locate。

## 9.4 DRY 在测试中的应用与边界

DRY（Don't Repeat Yourself）在生产代码中是美德，但在测试中需要**有节制地使用**。

### 应该抽象的：重复的 setup

```typescript
// 多个测试文件都写这段 → 抽象到 test-utils
function renderUserPage(userId: string) {
  const queryClient = createTestQueryClient()
  queryClient.setQueryData(['user', userId], createTestUser({ id: userId }))

  return renderWithProviders(<UserPage />, {
    queryClient,
    route: `/users/${userId}`,
  })
}
```

### 应该保留的：测试内部的重复

```typescript
it('邮箱为空时显示错误', async () => {
  const user = userEvent.setup()
  render(<LoginForm />)

  await user.type(screen.getByLabelText('邮箱'), '')
  await user.type(screen.getByLabelText('密码'), 'password')
  await user.click(screen.getByRole('button', { name: '登录' }))

  expect(screen.getByText('请输入邮箱')).toBeInTheDocument()
})

it('密码少于 6 位时显示错误', async () => {
  // 这段 setup 与上面类似，但保留重复让每个测试自包含且可独立阅读
  const user = userEvent.setup()
  render(<LoginForm />)

  await user.type(screen.getByLabelText('邮箱'), 'test@example.com')
  await user.type(screen.getByLabelText('密码'), '123')
  await user.click(screen.getByRole('button', { name: '登录' }))

  expect(screen.getByText('密码至少 6 位')).toBeInTheDocument()
})
```

如果把上面的两个测试「抽象」成一个共享的 helper，会导致：
- 读者需要跳到另一个文件才能理解测试
- 修改一个测试的 setup 可能影响其他测试
- 构建 helper 的复杂度超过它节省的代码

**经验法则**：如果某个 setup 被 5 个以上的测试文件使用，提取为公共工具。否则，在测试内部保持自包含。

## 9.5 让测试文档化

好的测试是**可执行的文档**。当你需要了解一个组件的所有可能行为时，测试文件是最可靠的信息源——比 README 更准确，因为它会被 CI 验证。

```typescript
// 这个测试文件本身就是 ProductCard 的行为规格说明书

describe('ProductCard', () => {
  it('显示商品名称、价格和图片', () => {})
  it('库存为 0 时显示「已售罄」标签', () => {})
  it('促销商品显示折扣价和原价', () => {})
  it('VIP 用户看到会员专享价', () => {})
  it('点击商品图片放大预览', () => {})
  it('点击「加入购物车」后按钮变为「已加入」', () => {})
  it('未登录时点击购买跳转到登录页', () => {})
})
```

把这些测试名称连起来读，就是一份完整的产品行为文档。

## 练习与思考

1. 审查你的测试代码，找出：
   - 5 个可以改进的测试名称
   - 3 个应该抽象为公共工具的重复 setup
   - 2 个过度抽象、应该回退到重复的地方
2. 为你的项目制定一份测试代码风格指南（参考本章内容）
3. 找一个超过 200 行的测试文件，思考如何拆分

---

## 本章总结

- 测试代码需要同等质量标准的维护
- 测试命名描述行为（而非实现），可连读为行为文档
- 文件组织首选 co-located 模式
- DRY 在测试中要节制：shared setup 值得抽象，局部重复保留自包含性
- 经验法则：5 个以上文件使用同一 setup → 提取到 test-utils
