# 第一章：测试环境搭建 — 从零到运行

## 学习目标

- 完成 Vite + React + TypeScript + Vitest + RTL + MSW 的完整环境搭建
- 理解 vitest.config.ts 中每个配置项的含义
- 掌握 setup 文件（全局测试配置）的写法
- 了解 MSW 在测试环境中的生命周期管理
- 能够排查常见的环境配置问题

---

## 1.1 创建 Vite + React + TypeScript 项目

如果你已有项目，可以跳过本节。如果从零开始：

```bash
npm create vite@latest my-app -- --template react-ts
cd my-app
npm install
```

确保 `tsconfig.json` 包含以下关键设置：

```json
{
  "compilerOptions": {
    "strict": true,
    "jsx": "react-jsx",
    "esModuleInterop": true,
    "moduleResolution": "bundler",
    "module": "ESNext",
    "target": "ES2022",
    "types": ["vitest/globals"]
  }
}
```

> **自我验证说明**：`moduleResolution: "bundler"` 是 Vite 推荐的方式，配合 `"module": "ESNext"` 确保与 Vitest 的模块解析完全一致。`types` 中加入 `vitest/globals` 使 TypeScript 识别 `vi`、`describe`、`it`、`expect` 等全局变量。

## 1.2 安装依赖

```bash
# 测试运行器 + 覆盖率
npm install -D vitest @vitest/coverage-v8 @vitest/ui

# DOM 环境
npm install -D jsdom

# React Testing Library 生态
npm install -D @testing-library/react @testing-library/jest-dom @testing-library/user-event

# MSW
npm install -D msw

# ESLint 插件（可选但推荐）
npm install -D eslint-plugin-testing-library eslint-plugin-jest-dom
```

各依赖的职责：

| 包 | 用途 |
|---|---|
| `vitest` | 测试运行器、断言库、mock 工具 |
| `@vitest/coverage-v8` | 基于 V8 的代码覆盖率 |
| `@vitest/ui` | 可视化测试浏览器 |
| `jsdom` | 在 Node.js 中模拟浏览器 DOM 环境 |
| `@testing-library/react` | 组件渲染、查询、断言 |
| `@testing-library/jest-dom` | DOM 状态自定义匹配器 |
| `@testing-library/user-event` | 模拟真实用户交互 |
| `msw` | 网络层 API mock |

## 1.3 配置 vitest.config.ts

Vitest 可以从 `vite.config.ts` 继承配置，也可以使用独立的 `vitest.config.ts`。推荐后者，因为它更清晰地分离了构建和测试的配置。

```typescript
// vitest.config.ts
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'
import path from 'node:path'

export default defineConfig({
  plugins: [react()],
  test: {
    // 1. 全局 API：无需每次 import describe/it/expect/vi
    globals: true,

    // 2. DOM 环境：在 Node 中模拟浏览器
    environment: 'jsdom',

    // 3. 全局 setup 文件：在每个测试文件之前执行
    setupFiles: ['./src/test/setup.ts'],

    // 4. 测试文件匹配模式
    include: ['src/**/*.{test,spec}.{ts,tsx}'],

    // 5. 排除目录
    exclude: ['node_modules', 'dist'],

    // 6. CSS 处理：设为 true 保证 import './style.css' 不报错
    css: true,

    // 7. 覆盖率配置
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html', 'lcov'],
      include: ['src/**/*.{ts,tsx}'],
      exclude: [
        'src/**/*.test.{ts,tsx}',
        'src/**/*.spec.{ts,tsx}',
        'src/test/**',
        'src/**/types.ts',
        'src/**/*.d.ts',
      ],
      thresholds: {
        lines: 70,
        functions: 70,
        branches: 70,
        statements: 70,
      },
    },
  },
  // 8. 路径别名：与 vite.config.ts 保持一致
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },
})
```

逐项解释：

**(1) `globals: true`**
开启后，`describe`、`it`、`test`、`expect`、`vi`、`beforeEach` 等直接在测试文件中使用，无需导入。这需要配合 `tsconfig.json` 中的 `"types": ["vitest/globals"]`。

**(2) `environment: 'jsdom'`**
在 Node.js 进程中模拟一套浏览器 DOM API。jsdom 是最常用的选择；`happy-dom` 是更轻量的替代，但部分 API 可能缺失。

**(3) `setupFiles`**
指定一个或多个文件，**在每个测试文件之前**于同一进程中执行。注意与 `globalSetup` 的区别——后者只在主线程中运行一次，无法访问测试进程的作用域。

**(4) `include`**
控制哪些文件被视为测试文件。默认已覆盖常用模式，这里显式声明以增强可读性。

**(5) CSS 处理**
设为 `true` 确保组件中 `import './styles.css'` 这样的语句不会在测试中报错。如果需要断言具体的 CSS 类名效果，可以使用 `toHaveClass` 等匹配器。

**(6) 覆盖率**
`v8` provider 比传统的 `istanbul` 更快，因为它直接利用 V8 引擎的内建覆盖率收集。`thresholds` 是可选的——建议等项目有一定测试规模后再启用。

**(7) resolve.alias**
与 `vite.config.ts` 保持完全一致是关键。如果两边解析出的路径不同，测试可能通过但生产构建失败。

> **自我验证说明**：上述配置基于 Vitest 3.x 的官方文档，配置项 `globals`、`environment`、`setupFiles`、`css`、`coverage` 均来自 `vitest/config` 的 `defineConfig` 类型定义。`@vitejs/plugin-react` 的导入路径和用法与 Vite 官方文档一致。

## 1.4 编写 setup 文件

setup 文件是测试环境的「入口」，负责全局配置。创建 `src/test/setup.ts`：

```typescript
// src/test/setup.ts
import '@testing-library/jest-dom/vitest'
import { cleanup } from '@testing-library/react'
import { afterEach } from 'vitest'

// 每个测试后自动卸载 React 组件树，确保测试隔离
afterEach(() => {
  cleanup()
})
```

> **自我验证说明**：`@testing-library/jest-dom/vitest` 是 `@testing-library/jest-dom` v6.x 提供的 Vitest 专用入口，自动注册 `toBeInTheDocument`、`toHaveTextContent`、`toBeDisabled` 等自定义匹配器到 Vitest 的 `expect`。`cleanup` 来自 `@testing-library/react`，内部调用 `unmount()` 卸载所有被 render 的组件。

如果同时使用 MSW（大多数项目都应该用），完整的 setup 文件会在 1.6 节展示。

## 1.5 编写第一个测试

创建 `src/utils/__tests__/math.test.ts` 验证环境：

```typescript
// src/utils/__tests__/math.test.ts

// 工具函数
export function add(a: number, b: number): number {
  return a + b
}

export function divide(a: number, b: number): number {
  if (b === 0) throw new Error('Division by zero')
  return a / b
}

// 测试
describe('add', () => {
  it('returns the sum of two positive numbers', () => {
    expect(add(2, 3)).toBe(5)
  })

  it('handles negative numbers', () => {
    expect(add(-2, 3)).toBe(1)
  })

  it('handles zero', () => {
    expect(add(0, 5)).toBe(5)
    expect(add(0, 0)).toBe(0)
  })
})

describe('divide', () => {
  it('divides two numbers', () => {
    expect(divide(6, 2)).toBe(3)
  })

  it('handles floating point results', () => {
    expect(divide(5, 2)).toBeCloseTo(2.5)
  })

  it('throws on division by zero', () => {
    expect(() => divide(5, 0)).toThrow('Division by zero')
  })
})
```

运行测试：

```bash
npx vitest              # 运行一次
npx vitest --watch      # watch 模式
npx vitest --ui         # 可视化界面
npx vitest --coverage   # 带覆盖率报告
```

如果一切正常，你应该看到绿色的测试结果。

创建一个组件测试来验证 React 环境也正常。创建 `src/components/__tests__/Hello.test.tsx`：

```typescript
// src/components/__tests__/Hello.test.tsx
import { render, screen } from '@testing-library/react'

function Hello({ name }: { name: string }) {
  return <h1>Hello, {name}!</h1>
}

describe('Hello', () => {
  it('renders the greeting with the provided name', () => {
    render(<Hello name="World" />)
    expect(
      screen.getByRole('heading', { name: 'Hello, World!' })
    ).toBeInTheDocument()
  })

  it('updates when the name prop changes', () => {
    const { rerender } = render(<Hello name="Alice" />)
    expect(
      screen.getByRole('heading', { name: 'Hello, Alice!' })
    ).toBeInTheDocument()

    rerender(<Hello name="Bob" />)
    expect(
      screen.getByRole('heading', { name: 'Hello, Bob!' })
    ).toBeInTheDocument()
  })
})
```

> **自我验证说明**：`render` 和 `screen` 来自 `@testing-library/react`。`getByRole('heading', { name: '...' })` 是最高优先级的查询方式，`name` 选项匹配元素的 `aria-label` 或文本内容。`toBeInTheDocument` 来自 `@testing-library/jest-dom/vitest`。

## 1.6 MSW 集成（完整 setup）

现在将 MSW 集成到测试环境中。创建 MSW 的目录结构：

```
src/
  test/
    setup.ts         # 全局 setup（已有，需要升级）
    mocks/
      server.ts      # setupServer 实例
      handlers/
        index.ts     # barrel export
```

先创建 handlers：

```typescript
// src/test/mocks/handlers/index.ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  http.get('/api/health', () => {
    return HttpResponse.json({ status: 'ok' })
  }),
]
```

创建 server：

```typescript
// src/test/mocks/server.ts
import { setupServer } from 'msw/node'
import { handlers } from './handlers'

export const server = setupServer(...handlers)
```

升级 `src/test/setup.ts`：

```typescript
// src/test/setup.ts
import '@testing-library/jest-dom/vitest'
import { cleanup } from '@testing-library/react'
import { afterAll, afterEach, beforeAll } from 'vitest'
import { server } from './mocks/server'

// 在所有测试开始前启动 MSW
beforeAll(() => {
  server.listen({ onUnhandledRequest: 'error' })
})

// 每个测试后重置 handler，防止跨测试污染
afterEach(() => {
  server.resetHandlers()
  cleanup()
})

// 所有测试结束后关闭 MSW
afterAll(() => {
  server.close()
})
```

**关键配置 `onUnhandledRequest: 'error'`**：当一个请求发出但没有匹配到任何 handler 时，MSW 会抛出错误。这非常重要——它确保你不会「不小心」让测试中的请求穿透到真实网络，也不会遗漏需要 mock 的 API。

> **自我验证说明**：`setupServer` 从 `msw/node` 导入，返回 `SetupServerApi` 实例。`.listen()`、`.close()`、`.resetHandlers()` 的签名和调用顺序参考 MSW 官方文档和 mswjs/examples 中的 with-vitest 示例。`onUnhandledRequest` 接受 `'bypass'`、`'warn'`、`'error'` 三个值。

## 1.7 浏览器端的开发 Mock（可选但推荐）

MSW 的另一个强大能力是：同一套 handler 可以同时用于**测试**和**浏览器开发**。

```typescript
// src/test/mocks/browser.ts
import { setupWorker } from 'msw/browser'
import { handlers } from './handlers'

export const worker = setupWorker(...handlers)
```

在应用入口中按需启动：

```typescript
// src/main.tsx
import React from 'react'
import ReactDOM from 'react-dom/client'
import App from './App'

async function enableMocking() {
  // 仅在开发模式下启用 MSW
  if (import.meta.env.DEV) {
    const { worker } = await import('./test/mocks/browser')
    // worker.start() 返回 Promise，必须 await
    return worker.start({ onUnhandledRequest: 'bypass' })
  }
}

enableMocking().then(() => {
  ReactDOM.createRoot(document.getElementById('root')!).render(
    <React.StrictMode>
      <App />
    </React.StrictMode>
  )
})
```

还需要生成 MSW 的 Service Worker 脚本：

```bash
npx msw init public/ --save
```

这会生成 `public/mockServiceWorker.js`。`--save` 标志将其路径写入 `package.json`，开箱即用。

> **注意**：`worker.start()` 中的 `onUnhandledRequest: 'bypass'` 是合适的——在开发环境中，未拦截的请求应该穿透到真实后端。测试环境中用 `'error'` 更合适。

## 1.8 常见环境问题排查

### 问题 1：`ReferenceError: TextEncoder is not defined`

Jest + 旧版 jsdom 中常见，Vitest + jsdom 26+ 一般不需要，但如果出现：

```typescript
// 在 setup.ts 最顶部
import { TextEncoder, TextDecoder } from 'node:util'
Object.defineProperties(globalThis, {
  TextEncoder: { value: TextEncoder },
  TextDecoder: { value: TextDecoder },
})
```

### 问题 2：`ReferenceError: ResizeObserver is not defined`

jsdom 不包含 ResizeObserver（它是真正的浏览器 API）。组件库如 MUI、Ant Design、或图表库可能依赖它。

```typescript
// 在 setup.ts 的 beforeAll 或顶层
vi.stubGlobal('ResizeObserver', vi.fn(() => ({
  observe: vi.fn(),
  unobserve: vi.fn(),
  disconnect: vi.fn(),
})))
```

### 问题 3：路径别名 `@/` 不生效

确保 `vitest.config.ts` 的 `resolve.alias` 配置与 `vite.config.ts` 完全一致，且 `tsconfig.json` 中也配置了 `paths`：

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

还需要安装 `vite-tsconfig-paths` 插件让 Vitest 读取 tsconfig 的 paths。

### 问题 4：MSW handler 没有拦截到请求

检查清单：
1. `server.listen()` 是否在 `beforeAll` 中调用？
2. handler 的 URL 是否与代码中发起的请求 URL **完全匹配**（包括协议、域名、路径）？
3. 是否在 handler 中**忘记 return**？忘记 return 会让 handler fall through
4. `onUnhandledRequest: 'error'` 是否开启？看控制台是否有错误

### 问题 5：测试中 `userEvent` 的操作看起来没有生效

userEvent v14+ 所有方法返回 Promise。确保全部 `await`：

```typescript
const user = userEvent.setup()
// 错误：缺少 await
user.click(screen.getByRole('button'))
// 正确
await user.click(screen.getByRole('button'))
```

## 本章总结

- Vitest 通过约 18 行配置即可完成完整测试环境设置
- `globals: true` + `environment: 'jsdom'` + `setupFiles` 是最小可用配置
- setup 文件负责 MSW 生命周期管理（listen → resetHandlers → close）
- `onUnhandledRequest: 'error'` 是测试环境的推荐配置
- 同一套 MSW handler 可同时用于测试和浏览器开发
- 优先用 `@testing-library/jest-dom/vitest` 导入 matchers，而非旧版的 `@testing-library/jest-dom`
