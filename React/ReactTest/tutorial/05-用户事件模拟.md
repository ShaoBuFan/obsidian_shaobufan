## 5.1 userEvent vs fireEvent

### fireEvent：直接触发，但过于"诚实"

`fireEvent` 是 React Testing Library 最底层的事件触发方式。它对一个 DOM 元素直接调用 `dispatchEvent`：

```typescript
import { fireEvent } from '@testing-library/react'

// fireEvent 只触发这一个事件
fireEvent.click(button)
// 内部逻辑等价于：
button.dispatchEvent(new MouseEvent('click', { bubbles: true, cancelable: true }))
```

它"诚实"地只做你要求的事——你说 `click`，它就只发一个 `click` 事件。但真实用户的点击不是这样的。真实点击涉及一个完整的事件序列：

```
用户点击一个按钮时，浏览器实际触发：
pointerover → mouseover → pointerenter → mouseenter
→ pointerdown → mousedown
→ pointerup → mouseup
→ click
```

某些组件依赖这个序列中的中间事件。例如一个日期选择器在 `mousedown` 时开始选择、`mouseup` 时结束选择——用 `fireEvent.click` 完全绕过这两个事件，测试通过但实际功能是坏的。

### userEvent：模拟完整交互序列

`@testing-library/user-event` 模拟的是"用户操作"而非"事件分派"：

```typescript
import userEvent from '@testing-library/user-event'

const user = userEvent.setup()
await user.click(button)
// 这一行触发了：pointerover → mouseover → pointerenter → mouseenter
// → pointerdown → mousedown → pointerup → mouseup → click
```

> **Jest 对比：** `userEvent.setup()` 的 API 在 v14 中引入，同时适用于 Jest 和 Vitest。但 Vitest 的一项关键优势是 `advanceTimers` 选项——当你在测试中使用 `vi.useFakeTimers()` 后，可以将 `vi.advanceTimersByTimeAsync` 传给 `userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })`，使得 `userEvent.type()` 等内部延迟也能通过假定时器快速推进。在 Jest 中实现相同行为需要额外的版本依赖和 `advanceTimers: jest.advanceTimersByTime` 的显式配置。

> **为什么：** userEvent 需要模拟完整事件序列而不是单个 click 事件，原因在于现代 Web 应用中大量使用了更底层的事件监听。例如 Material UI 的 `<Button>` 通过监听 `keydown`（不是 `click`）来实现键盘无障碍，Element Plus 的 dropdown 通过 `pointerdown` 来决定触发时机。如果你用 `fireEvent.click(el)` 测试这些组件，你跳过了完整序列，测试会通过但浏览器中实际功能是坏的。`userEvent` 模拟的是用户的手指/鼠标行为，而非单个 DOM 事件的语法糖。

### 对比表格

| 维度 | fireEvent | userEvent |
|------|-----------|-----------|
| **事件粒度** | 单一事件直接 dispatch | 完整事件序列模拟 |
| **返回类型** | `boolean`（同步） | `Promise<void>`（异步） |
| **输入模拟** | 不模拟逐字符输入 | `type()` 逐字符触发 keydown/keypress/input/keyup |
| **焦点管理** | 不自动管理 | 自动 focus/blur 交互元素 |
| **无障碍检查** | 无 | 检查元素可交互性（disabled、hidden 等） |
| **可信度** | 低（用户不会这样操作） | 高（模拟真实用户行为） |
| **性能** | 快（无额外开销） | 稍慢（需要模拟完整序列） |

> **自我验证说明**：userEvent v14 的事件序列基于 [W3C UI Events 规范](https://www.w3.org/TR/uievents/) 和浏览器实际行为。`click` 触发的事件顺序在 `pointer` + `mouse` 都存在时如上所示。实际序列取决于输入设备类型（鼠标/触摸/笔）。参考 [userEvent 文档 - click behavior](https://testing-library.com/docs/user-event/concepts)。

### 唯一仍应使用 fireEvent 的场景

虽然 userEvent 是绝大多数情况下的正确选择，但仍有少数场景需要 fireEvent：

```typescript
// 场景 1：测试窗口 resize 事件
fireEvent(window, new Event('resize'))

// 场景 2：测试 scroll 事件
fireEvent.scroll(container, { target: { scrollTop: 100 } })

// 场景 3：测试自定义 DOM 事件
fireEvent(element, new CustomEvent('my-event', { detail: { key: 'value' } }))

// 场景 4：极低性能要求下的海量事件触发
// （但通常这种测试应该重新考虑设计）
```

> **自我验证说明**：userEvent 没有 `resize`、`scroll` 等窗口级事件的 API。这些事件是浏览器自动触发的，不存在对应的"用户操作"。测试这些事件时使用 fireEvent 是正确做法。参考 [RTL 文档 - fireEvent](https://testing-library.com/docs/dom-testing-library/api-events/)。

## 5.2 userEvent.setup() v14 API

### 为什么必须 setup()

在 userEvent v13 中，可以直接调用模块导出的方法：

```typescript
// v13（旧）
import userEvent from '@testing-library/user-event'
userEvent.click(button)
```

v14 引入了 `setup()` API。每个通过 `setup()` 创建的实例都有独立的内部状态（焦点位置、键盘修饰键状态、指针状态等）。这避免了测试之间互相污染：

```typescript
// v14（新）
import userEvent from '@testing-library/user-event'

const user = userEvent.setup()
await user.click(button)
```

### 完整签名

```typescript
import userEvent from '@testing-library/user-event'

function setup(options?: UserEventOptions): UserEventInstance
```

### UserEventOptions

```typescript
interface UserEventOptions {
  // 所有 API 的默认延迟（毫秒），默认值因 API 而异
  delay?: number

  // 自定义时间推进函数（配合 vi.useFakeTimers 使用）
  advanceTimers?: (delay: number) => void

  // 键盘映射表（高级用法，自定义键位）
  keyboardMap?: KeyboardKeyOptions[]

  // 跳过可访问性检查（默认 false）
  skipAccessibilityCheck?: boolean

  // 自定义 document 对象（用于非标准 DOM 环境）
  document?: Document

  // 点击 disabled 元素时的行为
  // 'error'：抛错（默认） | 'skip'：跳过 | 'ignore'：静默执行
  onDisabledClick?: 'error' | 'skip' | 'ignore'

  // 写操作（如 type）时的延迟分布策略
  delayDistribution?:
    | 'default'
    | 'uniform'
    | 'normal'
    | ((step: number, totalSteps: number) => number)
}
```

### 常见配置模式

```typescript
// 模式 1：默认配置（推荐）
const user = userEvent.setup()

// 模式 2：带延迟模拟——更接近真实用户输入速度
const user = userEvent.setup({ delay: 100 })

// 模式 3：配合假定时器
import { vi } from 'vitest'

beforeEach(() => vi.useFakeTimers({ shouldAdvanceTime: true }))

const user = userEvent.setup({
  delay: 100,
  advanceTimers: vi.advanceTimersByTime,
})

// 模式 4：允许点击 disabled 按钮（默认会抛错）
const user = userEvent.setup({ onDisabledClick: 'skip' })
```

> **自我验证说明**：`advanceTimers` 选项的设计目的是解决 `delay > 0` 时 userEvent 内部使用 `setTimeout` 与 `vi.useFakeTimers` 的冲突。当传递 `vi.advanceTimersByTime` 时，userEvent 在等待延迟时调用该函数推进假定时器，而不会卡在等待真实时间上。如果不设置此选项且使用了假定时器，userEvent 的延迟型操作会永远 pending。参考 [userEvent 文档 - fake timers](https://testing-library.com/docs/user-event/options/#advancetimers)。

> **为什么：** v14 引入 `setup()` API 的核心动机是状态隔离。在 v13 中，所有 userEvent 方法共享一个全局内部状态机——一个测试中的焦点位置、按下的修饰键、指针状态都会泄漏到下一个测试。如果一个测试按下了 Shift 键但没有释放，下一个测试的所有点击都会变成 Shift+点击。`setup()` 为每个测试创建独立的内部状态实例，从根本上解决了状态污染问题。这个设计模式与 RTL 的"每个测试独立渲染"哲学一致：渲染独立、状态独立、互不干扰。

## 5.3 核心交互方法

### click / dblClick / tripleClick

```typescript
// 方法签名
interface UserEventInstance {
  click(target: Element, options?: PointerOptions): Promise<void>
  dblClick(target: Element, options?: PointerOptions): Promise<void>
  tripleClick(target: Element, options?: PointerOptions): Promise<void>
}
```

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

function Counter() {
  const [count, setCount] = useState(0)
  return (
    <div>
      <p role="status">Count: {count}</p>
      <button onClick={() => setCount((c) => c + 1)}>+1</button>
      <button onDoubleClick={() => setCount((c) => c + 2)}>+2</button>
    </div>
  )
}

describe('click variations', () => {
  it('increments by 1 on single click', async () => {
    const user = userEvent.setup()
    render(<Counter />)

    await user.click(screen.getByRole('button', { name: '+1' }))
    expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
  })

  it('increments by 2 on double click', async () => {
    const user = userEvent.setup()
    render(<Counter />)

    await user.dblClick(screen.getByRole('button', { name: '+2' }))
    expect(screen.getByRole('status')).toHaveTextContent('Count: 2')
  })
})
```

> **自我验证说明**：`dblClick` 触发的事件序列与真实双击一致：`mousedown → mouseup → click → mousedown → mouseup → click → dblclick`。注意两次 `click` 事件在 `dblclick` 之前已经触发。如果组件在 `click` 中也有逻辑，双击时会先执行两次单击逻辑再执行双击逻辑。

### type / clear

```typescript
// 方法签名
interface UserEventInstance {
  type(
    target: Element,
    text: string,
    options?: TypeOptions
  ): Promise<void>

  clear(target: Element): Promise<void>
}

interface TypeOptions {
  delay?: number                    // 每字符间隔（ms）
  skipAccessibilityCheck?: boolean
  initialSelectionStart?: number    // 初始选区起点
  initialSelectionEnd?: number      // 初始选区终点
}
```

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

function LoginForm() {
  const [email, setEmail] = useState('')
  return (
    <form>
      <label htmlFor="email">Email</label>
      <input id="email" type="email" value={email}
        onChange={(e) => setEmail(e.target.value)}
      />
      <button type="submit">Login</button>
    </form>
  )
}

describe('type and clear', () => {
  it('types text into input field', async () => {
    const user = userEvent.setup()
    render(<LoginForm />)

    const input = screen.getByLabelText(/email/i)

    await user.type(input, 'alice@test.com')
    expect(input).toHaveValue('alice@test.com')
  })

  it('clears input field', async () => {
    const user = userEvent.setup()
    render(<LoginForm />)

    const input = screen.getByLabelText(/email/i)

    await user.type(input, 'alice@test.com')
    expect(input).toHaveValue('alice@test.com')

    await user.clear(input)
    expect(input).toHaveValue('')
  })
})
```

> **自我验证说明**：`type()` 逐个字符触发 `keydown → keypress → input → keyup` 序列。这意味着如果组件在 `onKeyDown` 中过滤字符（如只允许数字），`type()` 会忠实地模拟这个过程。`clear()` 触发的事件序列为：`focus → select → delete`（选中全部内容后删除），比直接设置 `value = ''` 更接近用户操作。

### keyboard

`keyboard()` 是 userEvent v14 新增的核心 API。它不依赖特定目标元素，在**当前 focus 的元素**上触发键盘事件：

```typescript
// 方法签名
interface UserEventInstance {
  keyboard(
    text: string | KeyboardInput[],
    options?: KeyboardOptions
  ): Promise<void>
}
```

```typescript
// 字符串格式：特殊键用花括号包裹
await user.keyboard('Hello')                 // 输入 "Hello"
await user.keyboard('{Enter}')               // 按回车
await user.keyboard('{Shift>}A{/Shift}')     // 按 Shift + A（输出大写 A）
await user.keyboard('{Control>}v{/Control}') // 按 Ctrl + V

// 数组格式：更精细控制
await user.keyboard([
  { key: 'a', ctrlKey: true },   // Ctrl + A
  { key: 'c', ctrlKey: true },   // Ctrl + C
])
```

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

describe('keyboard', () => {
  it('handles keyboard shortcut', async () => {
    const user = userEvent.setup()
    const onSave = vi.fn()

    render(
      <div onKeyDown={(e) => {
        if (e.ctrlKey && e.key === 's') {
          e.preventDefault()
          onSave()
        }
      }} tabIndex={0}>
        Press Ctrl+S to save
      </div>
    )

    // 先 focus 元素
    await user.click(screen.getByText(/press ctrl\+s/i))
    await user.keyboard('{Control>}s{/Control}')

    expect(onSave).toHaveBeenCalledTimes(1)
  })
})
```

> **自我验证说明**：`keyboard()` 的字符串格式支持以下特殊键标记：
> - `{Enter}`, `{Escape}`, `{Tab}`, `{Backspace}`, `{Delete}`, `{ArrowUp}`, `{ArrowDown}`, `{ArrowLeft}`, `{ArrowRight}`, `{Home}`, `{End}`, `{PageUp}`, `{PageDown}`
> - 修饰键按下/释放：`{Shift>}` 按下 Shift，`{/Shift}` 释放 Shift。同样适用于 `{Control>}`, `{Alt>}`, `{Meta>}`
> - 所有文本字符直接输入。花括号 `{` 和 `}` 本身用 `{{}` 和 `{}}` 转义
>
> 参考 [userEvent 文档 - keyboard](https://testing-library.com/docs/user-event/keyboard)。

### Keyboard 组合操作在无障碍测试中的应用

键盘无障碍（keyboard accessibility）是 WCAG 2.1 的核心要求——所有交互功能应该仅通过键盘即可完成。userEvent 的 `keyboard()` 是测试键盘无障碍的首选工具：

```typescript
// 场景 1：测试下拉菜单的键盘导航
function SimpleSelect() {
  const [open, setOpen] = useState(false)
  const [selected, setSelected] = useState('')
  const [focusedIdx, setFocusedIdx] = useState(-1)
  const options = ['Apple', 'Banana', 'Cherry']

  return (
    <div>
      <button
        aria-haspopup="listbox"
        aria-expanded={open}
        onClick={() => setOpen(!open)}
      >
        {selected || 'Choose fruit'}
      </button>
      {open && (
        <ul role="listbox" aria-label="Fruits">
          {options.map((opt, i) => (
            <li
              key={opt}
              id={`opt-${opt}`}
              role="option"
              aria-selected={selected === opt}
              style={{ background: i === focusedIdx ? '#eee' : 'transparent' }}
              onClick={() => { setSelected(opt); setOpen(false) }}
            >
              {opt}
            </li>
          ))}
        </ul>
      )}
    </div>
  )
}

describe('keyboard accessibility', () => {
  it('opens menu with Enter and selects option', async () => {
    const user = userEvent.setup()
    render(<SimpleSelect />)

    // 初始状态：菜单关闭
    const trigger = screen.getByRole('button', { name: /choose fruit/i })
    expect(trigger).toHaveAttribute('aria-expanded', 'false')

    // Enter 打开菜单
    await user.click(trigger)
    expect(screen.getByRole('listbox')).toBeInTheDocument()

    // 箭头键导航到第一项后 Enter 选中
    await user.keyboard('{ArrowDown}')
    await user.keyboard('{Enter}')
    expect(trigger).toHaveTextContent('Apple')
    expect(screen.queryByRole('listbox')).not.toBeInTheDocument()
  })

  it('closes menu with Escape without selecting', async () => {
    const user = userEvent.setup()
    render(<SimpleSelect />)

    await user.click(screen.getByRole('button'))
    expect(screen.getByRole('listbox')).toBeInTheDocument()

    // Escape 关闭
    await user.keyboard('{Escape}')
    expect(screen.queryByRole('listbox')).not.toBeInTheDocument()
  })
})
```

> **自我验证说明：** 键盘交互测试验证的是：用户无需鼠标即可完成操作。`aria-haspopup`、`aria-expanded`、`aria-selected` 是 WAI-ARIA 规范定义的属性，userEvent 不直接验证这些属性的值，但你的测试通过 `toHaveAttribute` 进行断言。`keyboard()` 的 `{ArrowDown}` 和 `{ArrowUp}` 触发的是 `keydown` 事件——组件需要监听这些事件来实现导航。

### tab

```typescript
// 方法签名
interface UserEventInstance {
  tab(options?: TabOptions): Promise<void>
}

interface TabOptions {
  shift?: boolean  // true = Shift+Tab（向后导航）
}
```

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

function FocusableForm() {
  return (
    <form>
      <label>
        Name:
        <input type="text" id="name" />
      </label>
      <label>
        Email:
        <input type="email" id="email" />
      </label>
      <button type="submit">Submit</button>
    </form>
  )
}

describe('tab navigation', () => {
  it('navigates through focusable elements in order', async () => {
    const user = userEvent.setup()
    render(<FocusableForm />)

    const nameInput = screen.getByLabelText(/name/i)
    const emailInput = screen.getByLabelText(/email/i)
    const submitBtn = screen.getByRole('button')

    // 初始焦点在 Name
    await user.click(nameInput)
    expect(nameInput).toHaveFocus()

    // Tab → Email
    await user.tab()
    expect(emailInput).toHaveFocus()

    // Tab → Submit
    await user.tab()
    expect(submitBtn).toHaveFocus()

    // Shift+Tab → 回到 Email
    await user.tab({ shift: true })
    expect(emailInput).toHaveFocus()
  })
})
```

> **自我验证说明**：`tab()` 默认按 DOM 树中的 `tabindex` 顺序导航可聚焦元素（`input`、`button`、`select`、`textarea`、`a[href]`、`[tabindex]` 等）。`shift: true` 模拟 Shift+Tab 向后导航。注意 `<input type="hidden">` 不会被聚焦。参考 [userEvent 文档 - tab](https://testing-library.com/docs/user-event/tab)。

### selectOptions / deselectOptions

```typescript
// 方法签名
interface UserEventInstance {
  selectOptions(
    target: Element,
    values: HTMLElement | HTMLElement[] | string | string[],
    options?: PointerOptions
  ): Promise<void>

  deselectOptions(
    target: Element,
    values: HTMLElement | HTMLElement[] | string | string[],
    options?: PointerOptions
  ): Promise<void>
}
```

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

function CitySelect() {
  const [selected, setSelected] = useState<string[]>([])
  return (
    <div>
      <label htmlFor="cities">Cities</label>
      <select id="cities" multiple value={selected}
        onChange={(e) => {
          const values = Array.from(e.target.selectedOptions, (o) => o.value)
          setSelected(values)
        }}
      >
        <option value="nyc">New York</option>
        <option value="london">London</option>
        <option value="tokyo">Tokyo</option>
      </select>
      <p>Selected: {selected.join(', ')}</p>
    </div>
  )
}

describe('selectOptions', () => {
  it('selects multiple options from a multi-select', async () => {
    const user = userEvent.setup()
    render(<CitySelect />)

    const select = screen.getByLabelText(/cities/i)

    // 按 value 选择
    await user.selectOptions(select, ['nyc', 'tokyo'])
    expect(screen.getByText(/nyc, tokyo/i)).toBeInTheDocument()

    // 取消选择
    await user.deselectOptions(select, 'nyc')
    expect(screen.getByText(/tokyo/i)).toBeInTheDocument()
    expect(screen.queryByText(/nyc/i)).not.toBeInTheDocument()
  })
})
```

> **自我验证说明**：`values` 参数接受 `<option>` 元素的 `value` 字符串、`<option>` 元素本身，或二者的数组。`selectOptions` 触发的完整事件序列：`mousedown → mouseup → click → input → change`。`multiple` 属性为 true 时，按住 Ctrl 点击是选择/取消切换；为 false 时，选择新选项自动取消旧选项。

### upload

```typescript
// 方法签名
interface UserEventInstance {

    target: Element,

```

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

function FileUploader() {
  const [files, setFiles] = useState<FileList | null>(null)
  return (
    <div>
      <label htmlFor="avatar">Avatar</label>
      <input id="avatar" type="file" multiple
        onChange={(e) => setFiles(e.target.files)}
      />
      {files && (
        <ul>
          {Array.from(files).map((f) => (
            <li key={f.name}>{f.name} ({f.size} bytes)</li>
          ))}
        </ul>
      )}
    </div>
  )
}

describe('file upload', () => {
  it('shows uploaded file names', async () => {
```

## 5.4 交互的异步本质

### 为什么所有方法都返回 Promise

userEvent v14 中的所有方法都返回 `Promise<void>`。这是因为大部分用户交互涉及：

1. **事件序列的异步调度**：`click` 涉及多个事件，中间可能有定时器
2. **延迟模拟**：`type({ delay: 100 })` 在每字符之间有延迟
3. **React 状态更新的批处理**：事件触发后 React 的异步渲染

```typescript
// ❌ 错误：没有 await
it('fails silently without await', () => {
  const user = userEvent.setup()
  render(<Counter />)

  user.click(screen.getByRole('button', { name: '+1' }))

  // 此时 click 事件还没触发完，React state 还没更新
  expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
  // 断言可能在点击完成前执行，导致失败
})
```

```typescript
// ✅ 正确：await 所有 userEvent 调用
it('works correctly with await', async () => {
  const user = userEvent.setup()
  render(<Counter />)

  await user.click(screen.getByRole('button', { name: '+1' }))

  expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
})
```

### React 状态更新与 userEvent 的时序

userEvent 内部通过 `act()` 包裹事件触发，确保 React 的状态更新在 Promise resolve 之前已完成：

```
await user.click(button)
         │
         ├→ act(() => {
         │    元素.focus()
         │    pointerdown → mousedown
         │    pointerup → mouseup
         │    click
         │   })
         │   ← React 状态更新已应用
         │
         ↓ Promise resolved → 可以安全断言
```

这意味着 `await user.click(...)` 之后，组件已经完成了重渲染。你不需要额外调用 `waitFor` 或 `act()`。

> **自我验证说明**：userEvent v14 始终返回 Promise，即使 `delay: 0`（默认）也是如此。这是因为内部状态机使用了微任务（microtask）来推进事件序列。这个设计保证了事件处理的完整性和可预测性。参考 [userEvent v14 迁移指南](https://testing-library.com/docs/user-event/v14-migration/)。

> **Jest 对比：** Jest 中 RTL 的 `act()` 警告是开发者最常见的痛点。当你用 `fireEvent` 触发事件时，如果组件的状态更新没有被 `act()` 包裹，Jest 会打印"an update was not wrapped in act()" 警告。userEvent 在 Vitest 和 Jest 中都会自动用 `act()` 包裹事件触发，但 Jest 的异步执行模型有时仍然会触发此警告，因为 Jest 的微任务调度与 React 的 flush 机制在边界情况下不完全同步。Vitest 基于 Vite 的 ESM 执行模型在这个问题上表现更好——因为 esbuild 的编译输出更接近浏览器原生 ESM 行为，React 的调度器能更准确地追踪状态更新边界。

> **为什么：** 用户操作本质上是异步的。用户点击按钮后，浏览器需要经过事件派发 → React 合成事件处理 → 状态更新 → 组件重渲染 → DOM 更新这一完整流程——所有这些都不是同步的。userEvent 的每个方法返回 Promise，就是为了确保在断言执行之前，上述"事件 → 状态 → 渲染"流水线已经完成。如果 userEvent 是同步的，你就必须手动 `act()` 包裹或手动 `waitFor`——这不仅增加了样板代码，还增加了遗漏包裹的风险。

### 场景 3：拖拽（pointer API）

userEvent 本身没有 `drag()` 方法，但可以通过 `pointer()` API 模拟拖拽：

```typescript
function DraggableList() {
  const [items, setItems] = useState(['Item A', 'Item B', 'Item C'])
  const [dragIndex, setDragIndex] = useState<number | null>(null)

  return (
    <ul>
      {items.map((item, i) => (
        <li
          key={item}
          draggable
          onDragStart={() => setDragIndex(i)}
          onDragOver={(e) => e.preventDefault()}
          onDrop={() => {
            if (dragIndex === null) return
            const newItems = [...items]
            const [removed] = newItems.splice(dragIndex, 1)
            newItems.splice(i, 0, removed)
            setItems(newItems)
            setDragIndex(null)
          }}
        >
          {item}
        </li>
      ))}
    </ul>
  )
}

describe('drag and drop', () => {
  it('reorders items on drag', async () => {
    const user = userEvent.setup()

    const items = screen.getAllByRole('listitem')
    expect(items[0]).toHaveTextContent('Item A')

    // 拖拽 Item A 到第三个位置
    await user.pointer([
      { target: items[0], coords: { x: 0, y: 0 } },  // pointerdown
      { target: items[2], coords: { x: 0, y: 50 } },  // pointermove → drop
    ])

    // 注意：jsdom 对 HTML5 Drag and Drop API 支持有限
    // 实际测试可能需要 fireEvent 辅助
  })
})
```

> 1. 使用 `fireEvent.dragStart` / `fireEvent.drop` 等 API 手动模拟拖拽事件
> 2. 使用 `@testing-library/user-event` 的 `pointer()` API（仅适用于 pointer 事件驱动的拖拽）

> 在撰写真实测试时，建议先验证 jsdom 对拖拽 API 的支持程度。参考 [jsdom 文档 - drag and drop](https://github.com/jsdom/jsdom#unimplemented-parts-of-the-web-platform)。

### 场景 5：搜索框防抖 + 异步建议列表

真实世界的搜索框通常结合了防抖（debounce）和异步请求——避免每次击键都触发网络请求，同时为用户提供即时建议：

```typescript
function SearchWithDebounce() {
  const [query, setQuery] = useState('')
  const [suggestions, setSuggestions] = useState<string[]>([])
  const [loading, setLoading] = useState(false)

  useEffect(() => {
    if (!query.trim()) {
      setSuggestions([])
      return
    }

    const timer = setTimeout(async () => {
      setLoading(true)
      try {
        const res = await fetch(`/api/search?q=${query}`)
        const data = await res.json()
        setSuggestions(data)
      } finally {
        setLoading(false)
      }
    }, 300)

    return () => clearTimeout(timer)
  }, [query])

  return (
    <div>
      <label htmlFor="search">Search</label>
      <input
        id="search"
        type="text"
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        aria-autocomplete="list"
        role="combobox"
        aria-expanded={suggestions.length > 0}
      />
      {loading && <p role="status">Searching...</p>}
      {suggestions.length > 0 && (
        <ul role="listbox">
          {suggestions.map((s) => (
            <li key={s} role="option">{s}</li>
          ))}
        </ul>
      )}
    </div>
  )
}

describe('SearchWithDebounce', () => {
  beforeEach(() => { vi.useFakeTimers() })
  afterEach(() => { vi.useRealTimers() })

  it('shows loading state after debounce delay', async () => {
    const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })
    render(<SearchWithDebounce />)

    const input = screen.getByRole('combobox')
    await user.type(input, 'rea')
    expect(screen.queryByRole('status')).not.toBeInTheDocument()

    await vi.advanceTimersByTimeAsync(300)
    await vi.waitFor(() => {
      expect(screen.getByRole('status')).toHaveTextContent('Searching...')
    })
  })

  it('clears suggestions when input is cleared', async () => {
    const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })
    render(<SearchWithDebounce />)

    const input = screen.getByRole('combobox')
    await user.type(input, 'react')

    // 等防抖触发，假设 API 返回了建议
    await vi.advanceTimersByTimeAsync(300)
    await vi.waitFor(() => {
      expect(screen.queryByRole('status')).not.toBeInTheDocument()
    })

    // 清空输入 → 建议列表消失
    await user.clear(input)
    expect(screen.queryByRole('listbox')).not.toBeInTheDocument()
  })

  it('cancels previous request on new input', async () => {
    const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })
    render(<SearchWithDebounce />)

    const input = screen.getByRole('combobox')
    await user.type(input, 're')
    await vi.advanceTimersByTimeAsync(100)  // 防抖还没到就继续输入
    await user.type(input, 'act')           // 延长 query → 重置防抖

    // 此时防抖应当被重置，总防抖时间从最后一次输入算
    await vi.advanceTimersByTimeAsync(299)
    expect(screen.queryByRole('status')).not.toBeInTheDocument()

    await vi.advanceTimersByTimeAsync(1)
    await vi.waitFor(() => {
      expect(screen.getByRole('status')).toHaveTextContent('Searching...')
    })
  })
})
```

> **自我验证说明：** 这个场景将[第3章](03-Vitest基础.md)的假定时器与第 5 章的 userEvent 结合为一个完整的防抖搜索测试。关键模式是 `userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })`——它让 userEvent 的内部逐字符输入延迟和组件的防抖 `setTimeout` 都由同一个假定时器驱动。`type('re')` 后快进 100ms 再 `type('act')` 的测试验证了 useEffect 的 cleanup 函数正确取消了旧的防抖——如果组件没有正确 `clearTimeout(timer)`，第一个防抖会在 300ms 时触发请求，而不是等待最后一次输入。测试中的"快进 299ms → 断言无请求 → 再快进 1ms → 断言有请求"模式精确地验证了防抖时序边界。注意这里必须用 `vi.advanceTimersByTimeAsync`（异步版本），因为防抖回调内部有 `await fetch(...)`。

### 渐进式示例：UserEvent 从入门到精通

```typescript
// Step 1：单次点击——最简单的交互
it('clicks a button', async () => {
  const user = userEvent.setup()
  render(<Counter />)

  await user.click(screen.getByRole('button', { name: '+1' }))
  expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
})

// Step 2：多次点击——状态累积验证
it('accumulates count over multiple clicks', async () => {
  const user = userEvent.setup()
  render(<Counter />)

  await user.click(screen.getByRole('button', { name: '+1' }))
  await user.click(screen.getByRole('button', { name: '+1' }))
  await user.click(screen.getByRole('button', { name: '+1' }))

  expect(screen.getByRole('status')).toHaveTextContent('Count: 3')
})

// Step 3：不同类型输入组合 + Tab 导航
it('fills a multi-field form step by step', async () => {
  const user = userEvent.setup()
  const onSubmit = vi.fn()
  render(<RegistrationForm onSubmit={onSubmit} />)

  await user.type(screen.getByLabelText(/name/i), 'Alice')
  await user.tab()  // 切换到下一个字段

  await user.type(screen.getByLabelText(/email/i), 'alice@test.com')
  await user.tab()

  await user.selectOptions(screen.getByLabelText(/role/i), 'admin')

  // 按 Enter 提交
  await user.keyboard('{Enter}')

  expect(onSubmit).toHaveBeenCalledWith({
    name: 'Alice',
    email: 'alice@test.com',
    role: 'admin',
  })
})

// Step 4：键盘快捷键 + 鼠标交互混合
it('uses Ctrl+A to select all and Delete to clear', async () => {
  const user = userEvent.setup()
  const handleChange = vi.fn()

  render(<textarea defaultValue="Hello World" onChange={handleChange} />)
  const editor = screen.getByRole('textbox')

  await user.click(editor)
  // Ctrl+A 全选
  await user.keyboard('{Control>}a{/Control}')
  // Delete 删除选中内容
  await user.keyboard('{Delete}')

  expect(handleChange).toHaveBeenCalled()
  expect(editor).toHaveValue('')
})
```

> **自我验证说明：** 这四个步骤从最简单的单次点击逐步演进到键盘+鼠标混合交互。关键模式是每个步骤都建立在之前步骤的基础上：Step 2 复用 Step 1 的点击逻辑但增加了状态验证，Step 3 引入 Tab 和键盘导航，Step 4 结合键盘快捷键修饰键。`user.tab()` 在填充表单时尤其有用——它模拟了真实用户填写表单后按 Tab 切换到下一个输入框的行为。

> **Jest 对比：** userEvent v14 的异步 API 设计在 Jest 和 Vitest 中行为一致——毕竟 userEvent 是与测试运行器无关的独立库。但在实际使用中，一个关键差异体现在 `setup({ advanceTimers })` 与假定时器的配合深度。Vitest 提供了 `vi.advanceTimersByTimeAsync`，它返回 `Promise` 并正确地推进微任务链，与 userEvent 内部的 `Promise` 调度完全兼容。Jest 的 `jest.advanceTimersByTime` 是同步方法——传递给 `advanceTimers` 后，userEvent 在等待内部延迟时会调用它，但同步的定时器推进无法处理嵌套的 async 回调，会导致 userEvent 的微任务调度出现竞态。具体表现是：在 Jest 中，`setup({ delay: 50, advanceTimers: jest.advanceTimersByTime })` 配合 `jest.useFakeTimers()` 时，`user.type(input, 'abc')` 可能在中间字符的 `keydown` 事件处卡住，因为同步的定时器推进无法处理事件回调中产生的微任务。Vitest 的 `advanceTimersByTimeAsync` 从设计上就解决了这个问题——它返回 `Promise`，让 userEvent 可以在每个定时器回调完成后 await 微任务消化，再推进下一个定时器。

## 5.6 反模式

### 反模式 1：忘记 await userEvent 调用

```typescript
// ❌ 错误：所有 userEvent 方法都返回 Promise
it('silently skips assertions', () => {
  const user = userEvent.setup()
  render(<Counter />)

  user.click(screen.getByRole('button', { name: '+1' }))

  // 这个断言在 click 完成前执行，可能通过也可能不通过
  expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
  // 测试不会报错，但也不可靠
})

// ✅ 正确：始终 await
it('reliably asserts after click', async () => {
  const user = userEvent.setup()
  render(<Counter />)

  await user.click(screen.getByRole('button', { name: '+1' }))

  expect(screen.getByRole('status')).toHaveTextContent('Count: 1')
})
```

### 反模式 2：setup() 放在 beforeEach 中但不传给测试

```typescript
// ❌ 错误：beforeEach 中的 user 变量没有被测试用例访问到
describe('bad pattern', () => {
  beforeEach(() => {
    const user = userEvent.setup()  // 局部变量，每个测试拿不到
  })

  it('cannot use user', async () => {
    // ReferenceError: user is not defined
    await user.click(screen.getByRole('button'))
  })
})

// ✅ 正确：在 describe 作用域声明，beforeEach 中赋值
describe('good pattern', () => {
  let user: ReturnType<typeof userEvent.setup>

  beforeEach(() => {
    user = userEvent.setup()
  })

  it('can use user', async () => {
    await user.click(screen.getByRole('button'))
  })
})

// ✅ 更简洁：每个测试独立 setup
it('setup per test is fine too', async () => {
  const user = userEvent.setup()
  await user.click(screen.getByRole('button'))
})
```

### 反模式 3：在同一 userEvent 实例上并发调用

```typescript
  await user.click(screen.getByRole('button', { name: 'Button 1' }))
```

### 反模式 4：userEvent 与 fireEvent 混用导致状态不一致

## 5.8 本章总结

- `userEvent` 模拟真实用户操作的完整事件序列，`fireEvent` 只触发单一事件
- userEvent v14 使用 `setup()` 创建独立实例，避免状态污染
- 所有核心方法：`click`、`dblClick`、`tripleClick`、`type`、`clear`、`keyboard`、`tab`、`selectOptions`、`deselectOptions`、`upload`、`hover`、`unhover`、`copy`、`cut`、`paste`
- 所有方法都是异步的，**必须 `await`**
- `keyboard()` 支持修饰键组合和特殊键，是键盘导航测试的核心
- `tab()` 模拟 Tab/Shift+Tab 焦点导航
- 复杂交互场景：表单填写+提交、键盘导航流程、拖拽
- setup() 不应该放在 beforeEach 局部作用域中而无法访问
- 始终使用 userEvent 替代 fireEvent，除非测试窗口 resize 或 scroll 等浏览器自动事件
- 不要在同一 userEvent 实例上并发调用方法
