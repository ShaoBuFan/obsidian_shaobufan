# 第十章：持续集成中的测试实践

## 学习目标

- 配置 GitHub Actions 运行 Vitest 测试套件
- 理解覆盖率配置与 ratcheting 策略
- 能诊断和提升测试执行性能
- 能定位和修复不稳定（flaky）测试

---

## 10.1 GitHub Actions 配置

```yaml
# .github/workflows/test.yml
name: Test

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  unit-and-integration:
    name: Unit & Integration Tests
    runs-on: ubuntu-latest
    timeout-minutes: 15

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npx tsc --noEmit

      - name: Run tests
        run: npx vitest run --coverage

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v5
        with:
          files: ./coverage/lcov.info
          flags: unittests
```

### 分片加速

当测试套件非常庞大时，可以将测试文件分到多个 CI 机器上并行运行：

```yaml
jobs:
  test:
    strategy:
      matrix:
        shard: [1, 2, 3, 4]  # 分成 4 个分片

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npx vitest run --shard=${{ matrix.shard }}/${{ strategy.job-total }}
```

### 缓存策略

```yaml
- name: Cache MSW Service Worker
  uses: actions/cache@v4
  with:
    path: public/mockServiceWorker.js
    key: msw-${{ hashFiles('package-lock.json') }}
```

## 10.2 覆盖率策略

### 合理的阈值设置

```typescript
// vitest.config.ts
coverage: {
  provider: 'v8',
  thresholds: {
    lines: 70,
    functions: 70,
    branches: 65, // 分支覆盖率通常最难达标
    statements: 70,
  },
}
```

**不要一开始就设 90%+**。从 60-70% 开始，随着项目成熟逐步提升。

### 覆盖率 Ratcheting（自动抬升）

Ratcheting 的核心思想：**覆盖率只允许上升，不允许下降**。

可以使用工具如 `codecov` 的 `threshold` 功能，或自定义脚本来实现：

```bash
# 比较当前覆盖率与 base 分支
npx vitest run --coverage
# 如果覆盖率下降，CI 失败
```

> 不要追求 100% 覆盖率。最后 10% 可能占用 50% 的时间，且往往是在测试 getter/setter 或 trivial 代码，ROI 极低。

### coverage.exclude

```typescript
coverage: {
  exclude: [
    'node_modules/',
    'src/test/',
    '**/*.test.{ts,tsx}',
    '**/*.d.ts',
    'src/types/',         // 纯类型定义
    'src/**/index.ts',     // barrel export 文件
  ],
}
```

## 10.3 测试性能优化

### 找出慢测试

```bash
# Vitest 会显示每个测试的耗时
npx vitest run --reporter=verbose
```

或使用 `@vitest/ui` 查看可视化报告。

### 优化策略

```typescript
// 1. 减少不必要的 setup：用 beforeAll 代替 beforeEach 处理不变的初始化
beforeAll(() => {
  // 数据库连接、大型 fixture 加载等一次性操作放这里
})

// 2. 避免在测试中等待真实的延迟
// ❌
await delay(500) // 浪费时间
// ✅
vi.useFakeTimers()
vi.advanceTimersByTime(500)

// 3. 对不依赖 DOM 的纯逻辑测试使用 node 环境（更快）
/**
 * @vitest-environment node
 */
it('pure function test (no DOM needed)', () => {})

// 4. 使用 pool: 'threads' 或 'forks' 启用文件级并行
// vitest.config.ts
test: { pool: 'threads' }
```

## 10.4 测试稳定性

Flaky test（时过时不过的测试）比没有测试更糟糕——它让开发者对 CI 失去信任。

### 常见原因与解决方案

| 原因 | 解决 |
|---|---|
| **异步操作未等待** | 确保所有 userEvent/query 都有 await/findBy |
| **定时器相关** | 用 fake timers 代替真实 setTimeout |
| **测试顺序依赖** | 确保 `afterEach` 中完全清理（cleanup + resetHandlers） |
| **日期/时间** | Mock `Date.now` 或使用固定日期 |
| **随机数据** | 在测试中使用固定 seed 或用工厂函数传入明确参数 |
| **API 调用未 mock** | 确保 `onUnhandledRequest: 'error'` 暴露遗漏的 handler |
| **并发冲突** | 对共享状态敏感的测试使用 `server.boundary()` 或串行运行 |

### 鉴定 Flaky Test

```bash
# 重复运行 100 次，找出不稳定的测试
for i in $(seq 1 100); do
  npx vitest run --reporter=verbose 2>&1 | tee -a test-output.log
done
```

Vitest 3.x 也支持内置的重复模式：

```bash
npx vitest --repeat=100
```

### 时间依赖修复示例

```typescript
// ❌ 依赖真实时间 → flaky
it('token expires after 1 hour', () => {
  // ...
})

// ✅ 使用固定时间
it('token expires after 1 hour', () => {
  vi.setSystemTime(new Date('2026-05-22T12:00:00Z'))
  // ... 测试逻辑
  vi.advanceTimersByTime(60 * 60 * 1000) // 1 小时
  // ... 断言 token 过期
})
```

3. 如果发现了一个 flaky test，你如何系统地定位和修复它？写出诊断流程
4. 设计一个覆盖率 ratcheting 脚本：比较当前覆盖率与 main 分支，低于则 CI 失败

---

## 本章总结

- GitHub Actions 中运行 Vitest：`npm ci → vitest run --coverage`
- 覆盖率 ratcheting 让覆盖率只升不降
- 缓存 `node_modules` 和 MSW worker 脚本可加速 CI
- fake timers 是优化测试速度的关键手段
- Flaky test 的根本原因通常是异步未等待、顺序依赖或真实时间依赖
- `--repeat` 标志可帮助鉴定不稳定的测试
