# 第三章：组件测试的核心原则

## 学习目标

- 深刻理解「测试行为，不测试实现细节」的实际含义
- 掌握查询优先级并能正确选择查询方法
- 理解 getBy / queryBy / findBy 的不同使用时机
- 能识别和避免常见的测试反模式
- 能独立为一个 React 组件设计完整的测试用例

---

## 3.1 测试行为，不测试实现细节

这是 React Testing Library 最重要的原则。但什么叫「实现细节」？什么又是「行为」？我们通过一个具体的例子来理解。

以下是一个搜索组件：

```typescript
// SearchBar.tsx
import { useState } from 'react'

interface SearchBarProps {
  onSearch: (query: string) => void
}

export function SearchBar({ onSearch }: SearchBarProps) {
  const [query, setQuery] = useState('')
  const [error, setError] = useState('')

  const handleSubmit = () => {
    if (query.trim().length < 2) {
      setError('请输入至少 2 个字符')
      return
    }
    setError('')
    onSearch(query)
  }

  return (
    <div>
      <input
        type="text"
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="搜索..."
        aria-label="搜索关键词"
      />
      <button onClick={handleSubmit}>搜索</button>
      {error && <p role="alert">{error}</p>}
    </div>
  )
}
```

**差：测试实现细节**

```typescript
test('bad: tests implementation details', () => {
  const onSearch = vi.fn()
  const { container } = render(<SearchBar onSearch={onSearch} />)

  const input = container.querySelector('input')  // ❌ CSS 选择器
  input!.value = 'test'                            // ❌ 直接操作 DOM 属性
  input!.dispatchEvent(new Event('change'))        // ❌ 手动触发事件

  const button = container.querySelector('button') // ❌ CSS 选择器
  button!.click()                                  // ❌ 原生 click

  expect(onSearch).toHaveBeenCalledWith('test')    // 只验证了回调，没验证 UI
})
```

**好：测试用户行为**

```typescript
test('good: tests user behavior', async () => {
  const user = userEvent.setup()
  const onSearch = vi.fn()
  render(<SearchBar onSearch={onSearch} />)

  // 用户看到的：输入框，输入文字，点击按钮
  const input = screen.getByRole('textbox', { name: '搜索关键词' })
  await user.type(input, 'test')
  await user.click(screen.getByRole('button', { name: '搜索' }))

  // 用户看到的结果：没有错误提示，搜索回调被触发
  expect(screen.queryByRole('alert')).not.toBeInTheDocument()
  expect(onSearch).toHaveBeenCalledWith('test')
})
```

好版本的优势：
1. 如果我把 `<input>` 改成 `<textarea>` 或者把 `onChange` 重构为 `onBlur`，测试依然有效
2. 如果我把 `aria-label` 改了但功能不变，只需要改一处
3. 测试代码读起来像是用户操作手册

**状态 vs 渲染输出**：

```typescript
// ❌ 错误：测试内部状态
expect(component.state.query).toBe('test')

// ✅ 正确：测试用户看到的
expect(screen.getByRole('textbox')).toHaveValue('test')
```

**Props vs 渲染输出**：

```typescript
// ❌ 错误：直接调用 prop 函数
wrapper.find('Button').prop('onClick')()

// ✅ 正确：用户点击
await user.click(screen.getByRole('button'))
```

## 3.2 查询优先级

RTL 的查询方法有一个明确的优先级排序。你应该按顺序选择，上级无法使用时才使用下级。

### 第一优先级：getByRole

`getByRole` 模拟屏幕阅读器识别元素的方式。每个 HTML 元素都有**隐式 role**：

```html
<button>点击</button>           → role="button"
<a href="/">首页</a>             → role="link"
<input type="text" />            → role="textbox"
<input type="checkbox" />       → role="checkbox"
<h1>标题</h1>                     → role="heading", level: 1
<img alt="头像" src="..." />     → role="img"
<nav>...</nav>                   → role="navigation"
```

```typescript
// 按 role 查询
screen.getByRole('button')
screen.getByRole('button', { name: '提交' })  // name 匹配文本内容或 aria-label
screen.getByRole('heading', { level: 1 })
screen.getByRole('textbox', { name: /邮箱/i })

// 查询不到时，查看可用的 role
screen.getByRole('nonexistent') // 报错信息会列出所有可用的 role
```

### 第二优先级：getByLabelText

用于表单字段：

```typescript
// 关联方式一：htmlFor + id
<label htmlFor="email">邮箱</label>
<input id="email" />

// 关联方式二：包裹
<label>邮箱 <input /></label>

// 关联方式三：aria-label
<input aria-label="搜索关键词" />

// 查询
screen.getByLabelText('邮箱')
screen.getByLabelText(/邮箱/i)  // 正则：大小写不敏感
```

### 第三优先级：getByPlaceholderText

```typescript
<input placeholder="请输入用户名" />
screen.getByPlaceholderText('请输入用户名')
```

### 第四优先级：getByText

用于非交互文本（段落、标题、span）：

```typescript
screen.getByText('欢迎回来，John')
screen.getByText(/欢迎回来/i)
```

### 第五优先级：getByDisplayValue

用于已填充的表单字段：

```typescript
<input value="当前值" readOnly />
screen.getByDisplayValue('当前值')
```

### 最后手段：getByTestId

```typescript
<div data-testid="user-card-42">...</div>
screen.getByTestId('user-card-42')
```

**何时使用 data-testid**：

- 列表中的动态元素，角色无法唯一区分
- 与其他库冲突（如第三方图表组件没有语义化结构）
- 即便如此，也应该是最后选择

## 3.3 getBy vs queryBy vs findBy

| 方法 | 找不到时 | 使用时机 |
|---|---|---|
| `getBy*` | 抛出错误 | 断言元素**现在一定在**DOM 中 |
| `queryBy*` | 返回 null | 断言元素**现在一定不在**DOM 中 |
| `findBy*` | 返回 Promise，超时抛错 | 等待元素**异步出现** |

```typescript
describe('Query Types', () => {
  it('getBy — 同步，元素必须存在', () => {
    render(<div>Hello</div>)
    const element = screen.getByText('Hello') // 一定存在，找不到直接抛错
    expect(element).toBeInTheDocument()
  })

  it('queryBy — 同步，元素应该不存在', () => {
    render(<div>Hello</div>)
    const element = screen.queryByText('Goodbye')
    expect(element).not.toBeInTheDocument() // null，不抛错
  })

  it('findBy — 异步，等待元素出现', async () => {
    render(<AsyncComponent />) // 组件内部有异步数据加载
    const element = await screen.findByText('数据加载完成')
    //          ^^^^^ 注意：findBy 返回 Promise，必须 await
    expect(element).toBeInTheDocument()
  })
})
```

**findBy 的超时设置**：

```typescript
// 对单个查询设置超时（默认 1000ms）
await screen.findByText('慢数据', {}, { timeout: 3000 })

// 全局设置
import { configure } from '@testing-library/react'
configure({ asyncUtilTimeout: 5000 })
```

## 3.4 render 的返回值

```typescript
const result = render(<MyComponent />)

// 最常用的方式：直接使用 screen（推荐）
screen.getByRole('button')

// result 提供的工具方法
result.debug()                // 打印当前 DOM 树
result.debug(element)         // 打印特定元素的 DOM
result.rerender(<MyComponent prop="new" />) // 用新 props 重新渲染
result.unmount()              // 手动卸载（cleanup 会自动做）
result.container              // 渲染的容器 DOM 节点（尽量避免直接使用）
result.asFragment()           // 返回 DocumentFragment（用于快照测试）
```

## 3.5 常见反模式

### 反模式 1：直接操作 container

```typescript
// ❌
const { container } = render(<Button />)
expect(container.firstChild).toHaveClass('btn-primary')

// ✅
render(<Button />)
expect(screen.getByRole('button')).toHaveClass('btn-primary')
```

### 反模式 2：测试组件内部 state

```typescript
// ❌
const { result } = renderHook(() => useCounter())
expect(result.current.count).toBe(0)

// 对 Hook 本身用 renderHook 是 OK 的。
// 但这个反模式指的是：测试组件时去窥探它的内部 state
// ❌
const wrapper = shallow(<Counter />)
expect(wrapper.state('count')).toBe(0)
```

`renderHook` 用于测试自定义 Hook 是完全正确的。反模式是用 enzyme 的 `shallow` + `state()` 来窥探组件内部。

### 反模式 3：过度使用 data-testid

```typescript
// ❌ 到处都是 testid
<div data-testid="header">
  <h1 data-testid="title">{title}</h1>
  <button data-testid="close-btn">×</button>
  <input data-testid="search-input" />
</div>

// ✅ 用语义化的 role 和 accessibility 属性
<div>
  <h1>{title}</h1>
  <button aria-label="关闭">×</button>
  <input aria-label="搜索" placeholder="搜索..." />
</div>
```

### 反模式 4：在测试中调用组件方法

```typescript
// ❌ 通过 ref 调用组件内部方法
const ref = React.createRef<CounterRef>()
render(<Counter ref={ref} />)
act(() => ref.current!.reset())

// ✅ 用户点击重置按钮
render(<Counter />)
await user.click(screen.getByRole('button', { name: /重置/i }))
```

### 反模式 5：使用 fireEvent 而非 userEvent

```typescript
// ❌ fireEvent 只触发单个事件
fireEvent.change(input, { target: { value: 'test' } })

// ✅ userEvent 模拟完整的用户输入序列
await user.type(input, 'test')
// user.type 内部执行：focus → keyDown → input → keyUp（每字符各一次）
```

### 反模式 6：未 await 异步操作

```typescript
// ❌ userEvent 的操作是异步的，没有 await
test('submits form', () => {
  user.click(screen.getByRole('button', { name: '提交' }))
  // 这行在 React 重新渲染前就执行了！
  expect(screen.getByText('提交成功')).toBeInTheDocument()
})

// ✅
test('submits form', async () => {
  await user.click(screen.getByRole('button', { name: '提交' }))
  expect(await screen.findByText('提交成功')).toBeInTheDocument()
})
```

## 3.6 实战完整示例：Posts 列表

现在我们从头为一个帖子列表组件写完整测试。

### 组件代码

```typescript
// PostsList.tsx
import { useState, useEffect } from 'react'

interface Post {
  id: number
  title: string
  author: string
}

interface PostsListProps {
  fetchPosts: () => Promise<Post[]>
}

export function PostsList({ fetchPosts }: PostsListProps) {
  const [posts, setPosts] = useState<Post[]>([])
  const [status, setStatus] = useState<'loading' | 'loaded' | 'error'>('loading')
  const [errorMessage, setErrorMessage] = useState('')

  useEffect(() => {
    fetchPosts()
      .then((data) => {
        setPosts(data)
        setStatus('loaded')
      })
      .catch((err) => {
        setErrorMessage(err instanceof Error ? err.message : '发生未知错误')
        setStatus('error')
      })
  }, [fetchPosts])

  if (status === 'loading') {
    return <div role="status">加载中...</div>
  }

  if (status === 'error') {
    return (
      <div role="alert">
        <p>加载失败：{errorMessage}</p>
        <button onClick={() => window.location.reload()}>重试</button>
      </div>
    )
  }

  if (posts.length === 0) {
    return <p>暂无帖子</p>
  }

  return (
    <ul aria-label="帖子列表">
      {posts.map((post) => (
        <li key={post.id}>
          <h2>{post.title}</h2>
          <p>作者：{post.author}</p>
        </li>
      ))}
    </ul>
  )
}
```

### 测试代码

```typescript
// PostsList.test.tsx
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

describe('PostsList', () => {
  describe('加载状态', () => {
    it('fetch 完成前显示加载中', () => {
      // 永不 resolve 的 Promise 模拟加载态
      const fetchPosts = vi.fn().mockImplementation(
        () => new Promise(() => {})
      )
      render(<PostsList fetchPosts={fetchPosts} />)

      expect(screen.getByRole('status')).toHaveTextContent('加载中...')
    })
  })

  describe('数据加载成功', () => {
    it('渲染帖子列表', async () => {
      const mockPosts = [
        { id: 1, title: 'Vitest 入门', author: 'Alice' },
        { id: 2, title: 'MSW 实战', author: 'Bob' },
      ]
      const fetchPosts = vi.fn().mockResolvedValue(mockPosts)

      render(<PostsList fetchPosts={fetchPosts} />)

      // 等待加载完成，帖子出现
      expect(await screen.findByText('Vitest 入门')).toBeInTheDocument()
      expect(screen.getByText('MSW 实战')).toBeInTheDocument()
      expect(screen.getByText('作者：Alice')).toBeInTheDocument()

      // 验证列表的结构
      const list = screen.getByRole('list', { name: '帖子列表' })
      expect(list).toBeInTheDocument()
    })

    it('空帖子列表时显示暂无数据', async () => {
      const fetchPosts = vi.fn().mockResolvedValue([])

      render(<PostsList fetchPosts={fetchPosts} />)

      expect(await screen.findByText('暂无帖子')).toBeInTheDocument()
    })
  })

  describe('加载失败', () => {
    it('显示错误信息和重试按钮', async () => {
      const fetchPosts = vi.fn().mockRejectedValue(new Error('网络连接失败'))

      render(<PostsList fetchPosts={fetchPosts} />)

      // 等待错误信息出现
      const alert = await screen.findByRole('alert')
      expect(alert).toHaveTextContent('网络连接失败')
      expect(screen.getByRole('button', { name: '重试' })).toBeInTheDocument()
    })
  })
})
```

> **自我验证说明**：此示例中 `fetchPosts` 通过 props 注入而非内部调用，这是**依赖注入**模式在测试中的应用。实际项目中更多使用 MSW 在网络层拦截请求，让组件直接调用真实的 API 函数。本节为了聚焦 RTL 核心概念，采用 props 注入方式。集成 MSW 的完整模式见第六章。

## 3.7 测试的 3A 模式

好的测试遵循 **Arrange → Act → Assert** 结构：

```typescript
it('用户输入少于2个字符时显示错误', async () => {
  // Arrange：准备
  const user = userEvent.setup()
  render(<SearchBar onSearch={vi.fn()} />)

  // Act：行动
  const input = screen.getByRole('textbox', { name: '搜索关键词' })
  await user.type(input, 'a')
  await user.click(screen.getByRole('button', { name: '搜索' }))

  // Assert：断言
  expect(screen.getByRole('alert')).toHaveTextContent('请输入至少 2 个字符')
})
```

在复杂测试中，在 AAA 之间加空行可以帮助视觉分隔。保持测试文件的可读性比保持测试代码的「简洁」更重要——一段在 Slack 代码审查中能被快速理解的测试，就是一段好测试。

## 本章总结

- 测试行为而非实现细节：问自己「用户能看到/做什么？」
- 查询优先级：getByRole > getByLabelText > getByPlaceholderText > getByText > getByDisplayValue > getByTestId
- getBy 用于断言存在、queryBy 用于断言不存在、findBy 用于等待异步出现
- 避免的六大反模式：直接操作 container、测试内部 state、过度用 testid、调用组件方法、用 fireEvent、不 await 异步
- 每个测试遵循 3A 模式：Arrange → Act → Assert
