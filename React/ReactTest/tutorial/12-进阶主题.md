## 12.1 视觉回归测试

视觉回归测试自动检测 UI 的视觉变化——不是检测「功能是否正常」，而是检测「像素是否变了」。

### 工作流

```
1. 构建组件 → 截图（baseline）
2. 修改代码 → 重新截图（current）
3. 对比 baseline vs current → 高亮差异
4. 人工审核：差异是预期内的还是 bug？
```

### 工具：Storybook + Chromatic

```bash
npm install -D @storybook/react @storybook/addon-essentials
npx chromatic --project-token=<token>
```

在 Storybook 中有两种测试模式：

**快照测试**：为每个 Story 截图，对比像素差异。

**交互测试**（play 函数）：模拟用户操作后进行断言或截图：

```typescript
// Button.stories.tsx
import { within, userEvent } from '@storybook/test'

export const Clicked: Story = {
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement)
    const button = canvas.getByRole('button')
    await userEvent.click(button)
    // 点击后 Chromatic 截图，检测 hover/active 样式
  },
}
```

### 适用场景与局限

- **适用**：设计系统组件库、品牌页面、响应式布局
- **局限**：动态数据（时间、随机内容）需要 mock；像素级对比过于敏感需调整阈值
- **不在本书深入**：视觉回归测试与 Vitest + MSW 不是同一层测试，更多与 Storybook/Chromatic 生态绑定

## 12.2 契约测试

### 什么是契约测试

在微服务/前后端分离架构中，前后端之间有一个「契约」——接口的输入输出格式。契约测试确保：

- 前端发送的请求格式符合后端期望
- 后端返回的响应格式符合前端期望

### MSW handler 作为轻量契约

你的 MSW handler 实际上就是一份可执行的契约参考：

```typescript
// 这个 handler 定义了「后端应该做什么」
http.post('/api/users', async ({ request }) => {
  const body = await request.json()
  // 如果前端传了错误格式，这个 handler 会返回 400
  if (!body.email || !body.name) {
    return HttpResponse.json(
      { message: 'email 和 name 为必填项' },
      { status: 400 }
    )
  }
  return HttpResponse.json({ id: '3', ...body }, { status: 201 })
})
```

如果 API 实际行为与 MSW handler 不一致，需要有一方修改——这就是契约测试的价值。

### 正式契约测试工具

- **Pact**：Consumer-Driven Contract Testing 的事实标准
- **Spot**：轻量级替代方案

当项目有多支独立团队（前端/A 后端/B 后端）且 API 变更频繁时，正式的契约测试工具值得投资。

## 12.3 Storybook + MSW 集成

### 用同一套 handler 服务 Storybook

```typescript
// .storybook/preview.tsx
import { initialize, mswLoader } from 'msw-storybook-addon'
import { handlers } from '../src/test/mocks/handlers'

initialize({ onUnhandledRequest: 'bypass' })

export const loaders = [mswLoader]

export const parameters = {
  msw: { handlers },
}
```

现在你的 Storybook 和测试共享同一套 MSW handler——Storybook 中的每个 story 都使用 mock API 数据渲染。

### Story 级别的 handler 覆盖

```typescript
// UserList.stories.tsx
export const EmptyState: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get('/api/users', () => HttpResponse.json([])),
      ],
    },
  },
}

export const ErrorState: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get('/api/users', () =>
          new HttpResponse(null, { status: 500 })
        ),
      ],
    },
  },
}
```

> 这意味着你只需要维护一套 handler，就能驱动**开发模式、Vitest 测试、Storybook 故事和 E2E 测试**——四个场景，一处定义，零代码改动。

## 12.4 无障碍测试

Testing Library 的查询优先级天然鼓励无障碍友好的 DOM 结构。但还可以更进一步。

### jest-axe / vitest-axe

```bash
npm install -D vitest-axe
```

```typescript
import { axe } from 'vitest-axe'

it('has no accessibility violations', async () => {
  const { container } = render(<LoginForm />)
  const results = await axe(container)
  expect(results).toHaveNoViolations()
})
```

在 CI 中运行无障碍测试：

```yaml
- name: Accessibility audit
  run: npx vitest run --reporter=verbose
```

**适用性**：自动无障碍测试能发现约 30% 的问题（缺失 alt、低对比度、标签关联等）。人工测试（屏幕阅读器实测）仍然不可或缺。

## 12.5 组件性能测试

Vitest 支持简单的基准测试（benchmark）：

```typescript
import { bench, describe } from 'vitest'

describe('List Rendering Performance', () => {
  bench('render 1000 items', () => {
    render(<LargeList itemCount={1000} />)
  })
})
```

```bash
npx vitest bench
```

这不是替代 React DevTools Profiler 或 Lighthouse 的完整性能测试，但可以在 CI 中快速标记显著的性能退化。

## 全书总结

我们从一个问题出发——「为什么写测试」——最终构建了一套完整的 React + TypeScript 测试体系。

### 核心原则

1. **测试行为，不测试实现细节**。如果你重构了内部逻辑但行为不变，测试不应该需要修改。
2. **在网络层模拟**。MSW 让你的应用程序在测试中运行「真实」的 HTTP 请求-响应周期，零代码改动。
3. **从用户视角断言**。用户看到什么、能做什么、发生了什么——这才是测试应该关心的。
4. **集成测试是最有价值的投资层**。渲染真实组件 + MSW 拦截网络 + userEvent 模拟操作 → 每行测试覆盖最多的生产代码。
5. **保持测试可维护**。测试代码也需要工程纪律——命名、组织、DRY 的边界。
6. **用 TypeScript 保护一切**。类型安全让 mock 数据、handler 参数、组件 props 保持一致，减少运行时错误。

### 技术栈速查

| 任务 | 工具 |
|---|---|
| 测试运行器 | Vitest |
| 组件渲染 + 查询 | @testing-library/react |
| 用户交互模拟 | @testing-library/user-event |
| DOM 断言 | @testing-library/jest-dom/vitest |
| API Mock | MSW v2 (http, HttpResponse) |
| 内存数据库（可选） | @mswjs/data |
| 类型检查 | TypeScript strict mode |
| 覆盖率 | @vitest/coverage-v8 |
| CI | GitHub Actions |

### 下一步

- 在本项目中为你最关键的组件写第一个集成测试
- 将 MSW handler 也接入 Storybook，让开发和测试使用同一套数据
- 配置 CI，让每次 PR 自动运行测试和覆盖率检查
- 定期审视测试套件的质量：有没有 flaky test？有没有测试「重复劳动」？

测试不是一次性工程，而是持续投资的代码资产。每增加一个有价值的测试，你就在代码库的未来可维护性和重构安全性上存入了一笔本金。
