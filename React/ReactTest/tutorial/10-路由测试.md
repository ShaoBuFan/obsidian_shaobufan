## 10.1 路由测试的挑战

React Router 的测试难点不在于技术复杂，而在于路由上下文是隐式的。

```typescript
// 一个使用了路由的组件
function UserProfile() {
  const { id } = useParams<{ id: string }>()
  const navigate = useNavigate()

  return (
    <div>
      <h1>User {id}</h1>
      <button onClick={() => navigate('/users')}>Back to list</button>
    </div>
  )
}

// 直接渲染会报错：useParams() requires a router context
render(<UserProfile />)
// ❌ Error: useParams() may be used only in the context of a <Router> component.
```

解决方案不是 mock `useParams`——而是提供真实的路由上下文。`MemoryRouter` 是专为测试设计的路由环境，它不依赖浏览器的 URL 地址栏，所有路由状态在内存中维护。

> **为什么：** 为什么用 `MemoryRouter` 而非 mock `useParams` 或 `useNavigate`？
>
> 根本原因在于：**mock 路由 API 跳过了真正的路由逻辑，只验证了"调用了函数"，而非"发生了导航"**。
>
> 当你 mock `useParams` 时，你告诉测试"路由参数的值是 X"，但实际的路径匹配——`:id` 是否从 URL 中正确提取、URL 变化时组件是否重新渲染——这些路由的核心行为都被跳过了。你的测试在验证一个"假设参数正确"的理想世界，而非真实的路由行为。
>
> 当你 mock `useNavigate` 时，你验证的是"`navigate('/users')` 被调用了"，但**真正的导航结果**——目标页面是否渲染、URL 是否变化、浏览器历史栈是否更新——全部丢失。一个在 `navigate` 调用后立即 `return null` 的组件，你的 mock 测试仍然通过（因为 `navigate` 被调用了），但用户看到的是白屏。
>
> `MemoryRouter` 的优势在于它提供了**真实的路由上下文**——`useParams` 从真实的 URL 路径中提取参数、`useNavigate` 触发真实的导航并更新路由状态。你的测试验证的是完整的"URL 进入 → 路由匹配 → 参数提取 → 组件渲染 → 用户交互 → 导航到新路由 → 新路由渲染"流程。
>
> 用 `MemoryRouter` 测试路由组件，你不是在测试"被 mock 的路由 API"，而是测试"真实的路由行为在内存中的副本"。这是你能达到的最高保真度的路由测试方式。
>
> 一个简单的判断标准：如果你的路由测试需要 mock `useParams`、`useNavigate`、`useLocation` 中的任何一个，你就做错了。正确的做法是用 `MemoryRouter` + `initialEntries` 提供路由上下文，用 `Routes` + `Route` 定义路由表。

## 10.2 MemoryRouter 模式

### 基本用法

```typescript
import { MemoryRouter } from 'react-router-dom'
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

// MemoryRouter 作为 wrapper 提供路由上下文
function renderWithRouter(ui: React.ReactElement, { initialEntries = ['/'] } = {}) {
  return render(ui, {
    wrapper: ({ children }) => (
      <MemoryRouter initialEntries={initialEntries}>
        {children}
      </MemoryRouter>
    ),
  })
}
```

### 渐进式示例：路由测试的三种进化阶段

用一个导航按钮的测试，展示三种测试策略的差异。

**阶段 1：mock useNavigate（最差）**

```typescript
// 只验证了 navigate 被调用，没验证导航是否真的发生
const mockNavigate = vi.fn()
vi.mock('react-router-dom', async (importOriginal) => {
  return { ...(await importOriginal()), useNavigate: () => mockNavigate }
})

it('navigates to settings', async () => {
  render(<Navigation />)
  await user.click(screen.getByRole('button', { name: /settings/i }))
  expect(mockNavigate).toHaveBeenCalledWith('/settings')
  // 但 Settings 页面真的渲染了吗？URL 真的变了吗？
  // 如果组件在 navigate() 之后 return null，测试仍然通过
})
```

**阶段 2：MemoryRouter + 路径断言（更好）**

```typescript
// 验证了 URL 真的变了
it('navigates to /settings', async () => {
  render(
    <MemoryRouter initialEntries={['/']}>
      <Navigation />
      <Route path="*" element={<PathDisplay />} />
    </MemoryRouter>,
  )
  await user.click(screen.getByRole('button', { name: /settings/i }))
  expect(screen.getByTestId('location-pathname')).toHaveTextContent('/settings')
})
// 验证了导航触发了 URL 变化，但仍未验证目标页面是否渲染
```

**阶段 3：MemoryRouter + 内容断言（最佳）**

```typescript
// 验证了导航的最终结果——目标页面已经渲染
it('renders settings page after navigation', async () => {
  render(
    <MemoryRouter initialEntries={['/']}>
      <Navigation />
      <Routes>
        <Route path="/settings" element={<p>Settings Page</p>} />
      </Routes>
    </MemoryRouter>,
  )
  await user.click(screen.getByRole('button', { name: /settings/i }))
  expect(screen.getByText('Settings Page')).toBeInTheDocument()
  // 完整验证了：点击 → navigate('/settings') → 路由匹配 → 目标组件渲染
})
```

> **三个阶段的本质差异**：阶段 1 验证"函数被调用"，阶段 2 验证"URL 变化"，阶段 3 验证"目标页面渲染"。只有阶段 3 覆盖了用户关心的完整导航链路。在路由测试中，**断言目标页面的内容**始终是最可靠的验证方式。

### 测试初始路由

```typescript
it('renders the correct page for initial route', () => {
  render(
    <MemoryRouter initialEntries={['/users']}>

  render(

      <Routes>
        <Route path="/" element={<p>Home</p>} />

```

> **自我验证说明**：
> - `MemoryRouter` 的 `initialEntries` 接受一个 URL 字符串数组，用于设置初始历史栈。测试可以通过这个选项模拟用户从任意路径进入应用。
> - 导航断言的最佳方式是在 `MemoryRouter` 内渲染对应的 `Route` 组件，通过目标路由渲染的内容来验证导航是否发生。这比检查 URL 字符串更接近用户视角。
> - `useLocation` 的 `pathname` 属性在 `MemoryRouter` 中是正确的，可以用于精确断言当前路径。

> **Jest 对比：MemoryRouter 在 Jest 和 Vitest 中的行为完全一致**
>
> `MemoryRouter` 是 React Router 的内置组件，与测试框架无关。无论是 Jest 还是 Vitest，`MemoryRouter` 的行为、API、渲染结果完全相同。这意味着一套路由测试代码可以无缝在两个框架间迁移。
>
> 但有一个工具链差异需要注意：**`React Router` 的版本选择**。
>
> ```typescript
> // React Router v5（旧版）：使用 MemoryRouter 的方式不同
> import { MemoryRouter, Route } from 'react-router-dom'
>
> // React Router v6（当前版本）：推荐本教程的写法
> import { MemoryRouter, Routes, Route } from 'react-router-dom'
> ```
>
> 如果你的项目从 React Router v5 迁移到 v6，路由测试的变化比 Jest → Vitest 的迁移大得多。v5 使用 `<Switch>` 和 `component` prop，v6 使用 `<Routes>` 和 `element` prop。测试代码需要相应重写。
>
> **迁移建议**：先完成 React Router v5 → v6 的迁移，再进行 Jest → Vitest 的迁移。因为路由 API 的变化影响测试逻辑，而测试框架的变化只影响 `vi.fn()` 等工具函数的命名。
>
> ```typescript
> // React Router v5 的测试写法
> render(
>   <MemoryRouter initialEntries={['/users/42']}>
>     <Route path="/users/:id" component={UserDetail} />
>   </MemoryRouter>,
> )
>
> // React Router v6 的测试写法
> render(
>   <MemoryRouter initialEntries={['/users/42']}>
>     <Routes>
>       <Route path="/users/:id" element={<UserDetail />} />
>     </Routes>
>   </MemoryRouter>,
> )
> ```

## 10.3 路由参数测试

### useParams

```typescript
// src/components/UserDetail.tsx
import { useParams, useNavigate } from 'react-router-dom'

interface User {
  id: number
  name: string
}

export function UserDetail() {
  const { id } = useParams<{ id: string }>()
  const navigate = useNavigate()
  const [user, setUser] = useState<User | null>(null)
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    setLoading(true)
    fetch(`/api/users/${id}`)
      .then((r) => r.json())
      .then((data: User) => {
        setUser(data)
        setLoading(false)
      })
  }, [id])

  if (loading) return <p>Loading user...</p>
  if (!user) return <p>User not found</p>

  return (
    <div>
      <h1>{user.name}</h1>
      <p>ID: {user.id}</p>
      <button onClick={() => navigate('/users')}>Back to users</button>
    </div>
  )
}
```

### useSearchParams

```typescript
// src/components/SearchPage.tsx
import { useSearchParams } from 'react-router-dom'

export function SearchPage() {
  const [searchParams, setSearchParams] = useSearchParams()
  const query = searchParams.get('q') ?? ''
  const page = Number(searchParams.get('page')) || 1

  const updateQuery = (newQuery: string) => {
    setSearchParams({ q: newQuery, page: '1' })
  }

  const nextPage = () => {
    setSearchParams({ q: query, page: String(page + 1) })
  }

  return (
    <div>
      <input
        aria-label="Search"
        value={query}
        onChange={(e) => updateQuery(e.target.value)}
      />
      <p>Showing results for: "{query}"</p>
      <p>Page: {page}</p>
      <button onClick={nextPage}>Next Page</button>
    </div>
  )
}
```

```typescript
// src/components/SearchPage.test.tsx
import { MemoryRouter } from 'react-router-dom'
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { SearchPage } from './SearchPage'

describe('SearchPage', () => {
  const renderSearch = (initialEntries = ['/search']) => {
    const user = userEvent.setup()

    render(
      <MemoryRouter initialEntries={initialEntries}>
        <SearchPage />
      </MemoryRouter>,
    )

    return { user }
  }

  it('reads initial search params from URL', () => {
    renderSearch(['/search?q=react&page=2'])

    expect(screen.getByLabelText(/search/i)).toHaveValue('react')
    expect(screen.getByText(/showing results for/i)).toHaveTextContent('"react"')
    expect(screen.getByText(/page:/i)).toHaveTextContent('Page: 2')
  })

  it('updates search params when query changes', async () => {
    const { user } = renderSearch()

    const input = screen.getByLabelText(/search/i)
    await user.type(input, 'testing')

    // 每输入一个字符都会调用 setSearchParams
    // 最终 query 应该是 'testing'，page 重置为 1
    expect(screen.getByText(/showing results for/i)).toHaveTextContent('"testing"')
    expect(screen.getByText(/page:/i)).toHaveTextContent('Page: 1')
  })

  it('increments page number', async () => {
    const { user } = renderSearch(['/search?q=react'])

    await user.click(screen.getByRole('button', { name: /next page/i }))

    expect(screen.getByText(/page:/i)).toHaveTextContent('Page: 2')
  })
})
```

> **自我验证说明**：
> - `useParams` 的 `id` 参数值是通过 URL 路径段传入的。`MemoryRouter` 的 `initialEntries` 设置初始 URL 路径，`Route` 的 `path` 模式中的 `:id` 段会自动匹配并提取参数。
> - `useSearchParams` 的读写测试不需要 mock 任何东西——`MemoryRouter` 完整支持 URL 查询参数的解析和设置。
> - `setSearchParams` 会合并当前所有 search params 并更新 URL。测试中可以通过渲染的文本内容间接验证 URL 参数的变化。

## 10.4 路由守卫测试

### 认证守卫

```typescript
// src/components/AuthGuard.tsx
import { Navigate, useLocation } from 'react-router-dom'

interface AuthGuardProps {
  isAuthenticated: boolean
  children: React.ReactNode
}

export function AuthGuard({ isAuthenticated, children }: AuthGuardProps) {
  const location = useLocation()

  if (!isAuthenticated) {
    // 将用户尝试访问的路径存入 state，登录后可以重定向回来
    return <Navigate to="/login" state={{ from: location }} replace />
  }

  return <>{children}</>
}
```

```typescript
// src/components/AuthGuard.test.tsx
import { MemoryRouter, Routes, Route } from 'react-router-dom'
import { render, screen } from '@testing-library/react'
import { AuthGuard } from './AuthGuard'

describe('AuthGuard', () => {
  const renderGuard = (isAuthenticated: boolean, initialEntries = ['/dashboard']) => {
    render(
      <MemoryRouter initialEntries={initialEntries}>
        <Routes>
          <Route
            path="/dashboard"
            element={
              <AuthGuard isAuthenticated={isAuthenticated}>
                <p>Dashboard Content</p>
              </AuthGuard>
            }
          />
          <Route path="/login" element={<p>Login Page</p>} />
        </Routes>
      </MemoryRouter>,
    )
  }

  it('renders children when authenticated', () => {
    renderGuard(true)

    expect(screen.getByText('Dashboard Content')).toBeInTheDocument()
    expect(screen.queryByText('Login Page')).not.toBeInTheDocument()
  })

  it('redirects to login when not authenticated', () => {
    renderGuard(false)

    expect(screen.getByText('Login Page')).toBeInTheDocument()
    expect(screen.queryByText('Dashboard Content')).not.toBeInTheDocument()
  })

  it('preserves the intended location in state', () => {
    render(
      <MemoryRouter initialEntries={['/dashboard']}>
        <Routes>
          <Route
            path="/dashboard"
            element={
              <AuthGuard isAuthenticated={false}>
                <p>Dashboard</p>
              </AuthGuard>
            }
          />
          <Route
            path="/login"
            element={<LocationStateDisplay />}
          />
        </Routes>
      </MemoryRouter>,
    )

    expect(screen.getByTestId('from-path')).toHaveTextContent('/dashboard')
  })
})

// 辅助组件：显示 location state
function LocationStateDisplay() {
  const location = useLocation()
  const from = (location.state as { from?: Location })?.from

  return <span data-testid="from-path">{from?.pathname}</span>
}
```

### 角色守卫

```typescript
// src/components/RoleGuard.tsx
import { Navigate } from 'react-router-dom'

type Role = 'admin' | 'user' | 'guest'

interface RoleGuardProps {
  userRole: Role
  allowedRoles: Role[]
  children: React.ReactNode
}

export function RoleGuard({ userRole, allowedRoles, children }: RoleGuardProps) {
  if (!allowedRoles.includes(userRole)) {
    return <Navigate to="/unauthorized" replace />
  }

  return <>{children}</>
}
```

```typescript
    expect(screen.getByText('Admin Panel Content')).toBeInTheDocument()
  })
})
```

> **自我验证说明**：
> - 路由守卫测试的核心是验证条件渲染和重定向。`isAuthenticated` 状态通过 props 直接传入——在测试中不需要 mock 认证服务。
> - `Navigate` 组件在渲染时立即执行重定向。因此断言应该检查**目标页面的内容**是否出现，而不是检查守卫的内容是否不出现。
> - `location.state` 的传递是 React Router 的一个重要特性。测试中通过目标路由中的组件读取 `location.state` 来验证 redirect state 是否正确。

## 10.5 延迟加载路由测试

### Lazy Route 组件

```typescript
// src/App.tsx
import { lazy, Suspense } from 'react'
import { Routes, Route } from 'react-router-dom'

const Dashboard = lazy(() => import('./pages/Dashboard'))
const UsersPage = lazy(() => import('./pages/UsersPage'))
const SettingsPage = lazy(() => import('./pages/SettingsPage'))

export function App() {
  return (
    <Suspense fallback={<div data-testid="loading">Loading page...</div>}>
      <Routes>
        <Route path="/dashboard" element={<Dashboard />} />
        <Route path="/users" element={<UsersPage />} />
        <Route path="/settings" element={<SettingsPage />} />
      </Routes>
    </Suspense>
  )
}
```

### 测试 Lazy Routes

```typescript
// src/App.test.tsx
import { MemoryRouter } from 'react-router-dom'
import { render, screen, waitFor } from '@testing-library/react'
import { App } from './App'

// 注意：lazy 加载的模块需要在测试环境中可解析
// 方法 1：确保 lazy 的模块路径正确，jsdom 可以解析
// 方法 2：如果模块有副作用（如 API 调用），用 MSW 覆盖

describe('App with lazy routes', () => {
  it('shows loading fallback while Dashboard is loading', () => {
    render(
      <MemoryRouter initialEntries={['/dashboard']}>
        <App />
      </MemoryRouter>,
    )

    // Suspense fallback 立即显示
    expect(screen.getByTestId('loading')).toBeInTheDocument()
  })

  it('eventually renders the lazy loaded Dashboard', async () => {
    render(
      <MemoryRouter initialEntries={['/dashboard']}>
        <App />
      </MemoryRouter>,
    )

    // await 等待 lazy 组件加载完成
    await waitFor(() => {
      expect(screen.queryByTestId('loading')).not.toBeInTheDocument()
    })
  })
})
```

### 模拟 lazy 模块的加载延迟

```typescript
// src/__mocks__/pages/Dashboard.tsx
// 不需要——Vitest 不会自动 mock lazy 加载的模块

// 但如果需要测试加载过程中的具体行为，可以创建一个延迟加载的组件：

it('displays fallback during component loading', async () => {
  // 创建一个故意延迟的 lazy 组件
  const LazyComponent = lazy(() => new Promise((resolve) => {
    setTimeout(() => {
      resolve({ default: () => <p>Delayed Content</p> } as any)
    }, 2000)
  }))

  vi.useFakeTimers()

  render(
    <MemoryRouter initialEntries={['/test']}>
      <Suspense fallback={<p>Loading...</p>}>
        <Routes>
          <Route path="/test" element={<LazyComponent />} />
        </Routes>
      </Suspense>
    </MemoryRouter>,
  )

  expect(screen.getByText('Loading...')).toBeInTheDocument()

  // 推进时间使 lazy 模块 resolve
  await act(async () => {
    vi.advanceTimersByTime(2000)
  })

  await waitFor(() => {
    expect(screen.getByText('Delayed Content')).toBeInTheDocument()
  })

  vi.useRealTimers()
})
```

### 更好的方式：用 MSW 模拟 lazy 模块的数据

```typescript
it('renders lazy-loaded user detail page with data', async () => {
  server.use(
    http.get('/api/users/1', () => {
      return HttpResponse.json({ id: 1, name: 'Alice' })
    }),
  )

  render(
    <MemoryRouter initialEntries={['/users/1']}>
      <Suspense fallback={<p>Loading page...</p>}>
        <Routes>
          <Route path="/users/:id" element={<UserDetail />} />
        </Routes>
      </Suspense>
    </MemoryRouter>,
  )

  // 等待 lazy 模块加载 + 数据获取
  await waitFor(() => {
    expect(screen.queryByText('Loading page...')).not.toBeInTheDocument()
  })

  // 数据渲染
  expect(await screen.findByText('Alice')).toBeInTheDocument()
})
```

> **自我验证说明**：
> - `React.lazy()` 使用动态 `import()`，在 Vitest 的 jsdom 环境中，这些 import 会被解析（取决于模块路径是否正确）。如果 lazy 加载的组件内部有 API 调用，需要用 MSW 覆盖。
> - Suspense 的 `fallback` 在 lazy 组件加载期间显示。测试中可以通过验证 fallback 出现和消失来确认 lazy 加载的完成。
> - 如果 lazy 加载的模块导入失败（路径不对或文件不存在），Vitest 会抛出模块解析错误，而不是优雅降级为显示 fallback。这通常意味着模块路径配置有问题。
> - `act()` + `vi.advanceTimersByTime()` 的组合适用于测试加载过程中的 fallback UI 行为，但在大多数实际测试中，`waitFor` 等待 fallback 消失就足够了。

> **Jest 对比：动态 import() 在 Jest 和 Vitest 中的解析差异**
>
> `React.lazy(() => import('./pages/Dashboard'))` 依赖 JavaScript 的动态 `import()`。在 Jest 和 Vitest 中，动态 import 的解析机制不同：
>
> | 方面 | Jest | Vitest |
> |------|------|--------|
> | 模块解析 | 通过 `jest.transform` 管道 | 通过 Vite 的 `resolve` 和 `esbuild` |
> | 支持的文件 | 默认 CommonJS，需额外配置 ESM | 原生支持 ESM 和 TypeScript |
> | Mock 动态导入 | 需要 `jest.mock()` 配合 `require()` | 使用 `vi.mock()` + `import()` |
>
> **实际影响**：如果你的 lazy 加载的组件使用 TypeScript 编写，在 Jest 中需要使用 `@babel/preset-typescript` 或 `ts-jest` 转译，而在 Vitest 中 esbuild 默认支持 TypeScript——**零配置**。
>
> ```typescript
> // Jest 中的懒加载测试需要配置
> // jest.config.js: transform: { '^.+\\.tsx?$': 'ts-jest' }
>
> // Vitest 中的懒加载测试 —— 不需要额外配置
> // vitest.config.ts: esbuild 默认支持 TypeScript
> const Dashboard = lazy(() => import('./pages/Dashboard'))
> ```
>
> 这意味着从 Jest 迁移到 Vitest 后，如果你的项目有 `React.lazy()` 路由，**移除了 Babel/ts-jest 配置后，lazy 加载仍然工作**——Vite 的模块解析负责动态 import 处理。

## 10.6 反模式

### 反模式 1：Mock useParams

```typescript
// ❌ mock useParams
vi.mock('react-router-dom', async (importOriginal) => {
  const mod = await importOriginal()
  return {
    ...mod,
    useParams: () => ({ id: '42' }),
  }
})

render(<UserDetail />)

// 问题：mock 跳过了真正的路由匹配逻辑
// - URL 没有变化，测试不验证路由是否匹配
// - useNavigate 也被 mock 了，导航测试不可用
// - 重构路由结构不影响测试——测试失去了回归保护

// ✅ 使用 MemoryRouter
render(
  <MemoryRouter initialEntries={['/users/42']}>
    <Routes>
      <Route path="/users/:id" element={<UserDetail />} />
    </Routes>
  </MemoryRouter>,
)

// 测试验证了完整的路由匹配 + 参数提取 + 组件渲染流程
```

### 反模式 2：Mock useNavigate

```typescript
// ❌ mock useNavigate
const mockNavigate = vi.fn()
vi.mock('react-router-dom', async (importOriginal) => {
  const mod = await importOriginal()
  return { ...mod, useNavigate: () => mockNavigate }
})

render(<UserDetail />)
await user.click(screen.getByRole('button'))
expect(mockNavigate).toHaveBeenCalledWith('/users')

// 问题：只验证了 navigate 被调用，没有验证真正的导航结果
// 目标页面是否渲染？URL 是否变化？

// ✅ 在 MemoryRouter 内渲染目标路由
render(
  <MemoryRouter initialEntries={['/users/42']}>
    <Routes>
      <Route path="/users/:id" element={<UserDetail />} />
      <Route path="/users" element={<p>Users List</p>} />
    </Routes>
  </MemoryRouter>,
)

await user.click(screen.getByRole('button'))
expect(screen.getByText('Users List')).toBeInTheDocument()
// 验证了真正的导航发生了——目标页面已经渲染
```

## 10.8 本章总结

- `MemoryRouter` 是 React Router 测试的核心工具——提供内存中的路由上下文，支持 `initialEntries` 设置初始历史栈
- 导航断言通过渲染目标路由组件并验证其内容来实现，比 mock navigate 更可靠
- 路由参数（`useParams`、`useSearchParams`）通过 `MemoryRouter` 的 `initialEntries` 注入 URL，不需要 mock
- 路由守卫测试关注条件渲染和重定向目标，通过断言目标页面内容来验证重定向
- Lazy route 测试需要等待 Suspense fallback 消失，用 MSW 覆盖 lazy 模块中的 API 调用
- 反模式：mock 路由 API（`useParams`、`useNavigate`）跳过了真正的路由逻辑，失去了测试的回归保护价值
