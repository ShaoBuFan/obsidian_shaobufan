## 9.1 表单测试的核心挑战

表单是前端最复杂的交互场景之一，测试需要覆盖：

```
用户视角的流程：

  填写字段 → 实时验证 → 提交 → 服务端验证 → 成功/错误
  │                         │               │
  │ 每个字段的              │ 验证全部       │ 成功：跳转
  │ 值变化、焦点、          │ 通过后将       │ 失败：显示
  │ 失焦、依赖关系           │ payload 发送    │ 服务端错误
                             │ 到服务端
```

测试要关注的问题：

- **交互序列**：填写顺序、失焦触发验证、字段间依赖
- **验证时机**：`onChange` 时立即验证 vs `onBlur` 时验证 vs `onSubmit` 时一次验证
- **异步提交**：提交中按钮禁用、提交后恢复、服务端错误回显
- **可访问性**：错误消息与输入框的关联、键盘导航、焦点管理

> **自我验证说明**：表单测试中的大部分交互应使用 `userEvent`（而非 `fireEvent`），因为 `userEvent` 触发的完整事件序列更接近真实用户操作。`userEvent.clear()` 和 `userEvent.type()` 模拟了逐字符输入和完整的焦点管理。

> **为什么：** 为什么表单测试应该通过用户交互验证，而非直接检查状态？
>
> 表单的核心行为是"用户操作→系统响应"的双向通道。如果你直接检查 `formState` 或组件内部状态，你测的是**状态容器的正确性**，而非**用户交互的正确性**。这两种测试不是等价的：
>
> - 直接检查状态：`expect(form.name).toBe('Alice')` — 这是实现细节测试
> - 通过用户交互验证：`await user.type(input, 'Alice'); expect(input).toHaveValue('Alice')` — 这是行为测试
>
> 差异在于：直接检查状态会放过"输入框绑定了错误的 onChange handler"这类 bug。如果组件中 `<input>` 的 `onChange` 不小心绑定到了另一个字段的更新函数，直接检查状态可能仍然看到正确的值（因为状态确实被更新了），但用户在页面上输入一个字段时，另一个字段的值改变了——这才是用户能感受到的 bug。
>
> 此外，`userEvent` 模拟的完整事件序列（focus → keydown → keypress → input → keyup → blur）比直接设置值更能暴露边缘情况。例如，某些格式化输入库只在 `keypress` 事件中处理字符，如果测试用 `fireEvent.change`（只触发 input 事件），格式化逻辑永远不会执行。

> **Jest 对比：userEvent 在 Jest 和 Vitest 中的一致性**
>
> `@testing-library/user-event` 是一个与测试框架无关的库。无论你使用 Jest 还是 Vitest，`userEvent.setup()` 的行为完全一致。这意味着从 Jest 迁移到 Vitest 时，所有 userEvent 相关的测试代码**不需要任何改动**。
>
> 但有一个配置差异需要注意：**Jest 的 timers 和 userEvent 的交互**。
>
> ```typescript
> // Jest + userEvent：使用 fake timers 时需要额外配置
> jest.useFakeTimers()
> const user = userEvent.setup({ advanceTimers: jest.advanceTimersByTime })
>
> // Vitest + userEvent：同样需要
> vi.useFakeTimers()
> const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTime })
> ```
>
> 这个 `advanceTimers` 选项告诉 `userEvent` 在每执行一步交互后推进假定时器。这在测试包含 `setTimeout` 或 `debounce` 逻辑的表单时很关键。如果忘记设置，`userEvent.type()` 可能永远不会完成（因为内部的定时器永远不会触发）。
>
> ```typescript
> // 测试 debounce 搜索表单
> it('searches after user stops typing', async () => {
>   vi.useFakeTimers()
>   const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTime })
>
>   render(<SearchForm />)
>
>   await user.type(screen.getByLabelText(/search/i), 'react')
>
>   // user.type 结束后，debounce 定时器已经被推进了
>   // 可以直接断言搜索结果
>   await waitFor(() => {
>     expect(screen.getByText(/results for react/i)).toBeInTheDocument()
>   })
>
>   vi.useRealTimers()
> })
> ```

## 9.2 受控表单测试

### 完整受控表单组件

```typescript
// src/components/ProfileForm.tsx
import { useState, ChangeEvent, FormEvent } from 'react'

interface ProfileFormData {
  name: string
  age: number
  country: string
  subscribe: boolean
  gender: string
  bio: string
}

interface ProfileFormProps {
  onSubmit: (data: ProfileFormData) => Promise<void>
  initialData?: Partial<ProfileFormData>
}

export function ProfileForm({ onSubmit, initialData }: ProfileFormProps) {
  const [form, setForm] = useState<ProfileFormData>({
    name: initialData?.name ?? '',
    age: initialData?.age ?? 18,
    country: initialData?.country ?? '',
    subscribe: initialData?.subscribe ?? false,
    gender: initialData?.gender ?? '',
    bio: initialData?.bio ?? '',
  })
  const [submitting, setSubmitting] = useState(false)

  const update = (field: keyof ProfileFormData, value: string | number | boolean) => {
    setForm((prev) => ({ ...prev, [field]: value }))
  }

  const handleSubmit = async (e: FormEvent) => {
    e.preventDefault()
    setSubmitting(true)
    try {
      await onSubmit(form)
    } finally {
      setSubmitting(false)
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <label htmlFor="name">Name</label>
      <input
        id="name"
        type="text"
        value={form.name}
        onChange={(e: ChangeEvent<HTMLInputElement>) => update('name', e.target.value)}
      />

      <label htmlFor="age">Age</label>
      <input
        id="age"
        type="number"
        value={form.age}
        onChange={(e: ChangeEvent<HTMLInputElement>) => update('age', Number(e.target.value))}
      />

      <label htmlFor="country">Country</label>
      <select
        id="country"
        value={form.country}
        onChange={(e: ChangeEvent<HTMLSelectElement>) => update('country', e.target.value)}
      >
        <option value="">Select a country</option>
        <option value="cn">China</option>
        <option value="us">United States</option>
        <option value="jp">Japan</option>
      </select>

      <label htmlFor="subscribe">
        <input
          id="subscribe"
          type="checkbox"
          checked={form.subscribe}
          onChange={(e: ChangeEvent<HTMLInputElement>) => update('subscribe', e.target.checked)}
        />
        Subscribe to newsletter
      </label>

      <fieldset>
        <legend>Gender</legend>
        <label htmlFor="gender-male">
          <input
            id="gender-male"
            type="radio"
            name="gender"
            value="male"
            checked={form.gender === 'male'}
            onChange={(e: ChangeEvent<HTMLInputElement>) => update('gender', e.target.value)}
          />
          Male
        </label>
        <label htmlFor="gender-female">
          <input
            id="gender-female"
            type="radio"
            name="gender"
            value="female"
            checked={form.gender === 'female'}
            onChange={(e: ChangeEvent<HTMLInputElement>) => update('gender', e.target.value)}
          />
          Female
        </label>
      </fieldset>

      <label htmlFor="bio">Bio</label>
      <textarea
        id="bio"
        value={form.bio}
        onChange={(e: ChangeEvent<HTMLTextAreaElement>) => update('bio', e.target.value)}
      />

      <button type="submit" disabled={submitting}>
        {submitting ? 'Saving...' : 'Save Profile'}
      </button>
    </form>
  )
}
```

```typescript
      subscribe: true,
      gender: 'male',

  it('disables submit button while submitting', async () => {
    // onSubmit 返回一个不会 resolve 的 Promise

      resolvePromise = resolve

    const user = userEvent.setup()
    render(<ProfileForm onSubmit={onSubmit} />)

    await user.click(screen.getByRole('button', { name: /save profile/i }))

    // 提交中
    expect(screen.getByRole('button', { name: /saving/i })).toBeDisabled()

    // 完成提交
    await act(async () => { resolvePromise(undefined) })

    expect(screen.getByRole('button', { name: /save profile/i })).toBeEnabled()
  })
})
```

> **自我验证说明**：
> - `getByLabelText` 是表单测试的首选查询——它通过 `<label htmlFor="id">` 关联输入框，与真实用户的使用方式一致。
> - `userEvent.clear()` 在输入新值前清空输入框（触发 focus → select → delete 事件序列），这对数字输入框尤其重要，因为 `type('25')` 会在现有值后面追加。
> - 提交中状态的测试需要手动控制 Promise 的 resolve 时机。返回一个永不 resolve 的 Promise 来锁定提交状态，然后手动 resolve 来恢复。

## 9.3 表单验证测试

### 带验证的表单组件

```typescript
// src/components/ValidatedForm.tsx
import { useState } from 'react'

interface ValidationErrors {
  name?: string
  email?: string
  username?: string
}

interface ValidatedFormProps {
  onSubmit: (data: { name: string; email: string; username: string }) => Promise<void>
  checkUsernameAvailability: (username: string) => Promise<boolean>
}

export function ValidatedForm({ onSubmit, checkUsernameAvailability }: ValidatedFormProps) {
  const [name, setName] = useState('')
  const [email, setEmail] = useState('')
  const [username, setUsername] = useState('')
  const [errors, setErrors] = useState<ValidationErrors>({})
  const [checkingUsername, setCheckingUsername] = useState(false)

  const validate = (): ValidationErrors => {
    const errs: ValidationErrors = {}
    if (!name.trim()) errs.name = 'Name is required'
    if (!email.trim()) {
      errs.email = 'Email is required'
    } else if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
      errs.email = 'Invalid email format'
    }
    if (!username.trim()) errs.username = 'Username is required'
    return errs
  }

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault()
    const validationErrors = validate()
    setErrors(validationErrors)
    if (Object.keys(validationErrors).length > 0) return

    try {
      await onSubmit({ name, email, username })
    } catch {
      // 服务端错误处理
    }
  }

  const handleUsernameBlur = async () => {
    if (!username.trim()) return
    setCheckingUsername(true)
    try {
      const available = await checkUsernameAvailability(username)
      if (!available) {
        setErrors((prev) => ({ ...prev, username: 'Username is already taken' }))
      }
    } finally {
      setCheckingUsername(false)
    }
  }

  return (
    <form onSubmit={handleSubmit} noValidate>
      <div>
        <label htmlFor="name">Name</label>
        <input
          id="name"
          type="text"
          value={name}
          onChange={(e) => setName(e.target.value)}
          aria-invalid={!!errors.name}
          aria-describedby={errors.name ? 'name-error' : undefined}
        />
        {errors.name && <span id="name-error" role="alert">{errors.name}</span>}
      </div>

      <div>
        <label htmlFor="email">Email</label>
        <input
          id="email"
          type="email"
          value={email}
          onChange={(e) => setEmail(e.target.value)}
          aria-invalid={!!errors.email}
          aria-describedby={errors.email ? 'email-error' : undefined}
        />
        {errors.email && <span id="email-error" role="alert">{errors.email}</span>}
      </div>

      <div>
        <label htmlFor="username">Username</label>
        <input
          id="username"
          type="text"
          value={username}
          onChange={(e) => setUsername(e.target.value)}
          onBlur={handleUsernameBlur}
          aria-invalid={!!errors.username}
          aria-describedby={errors.username ? 'username-error' : undefined}
        />
        {checkingUsername && <span>Checking availability...</span>}
        {errors.username && <span id="username-error" role="alert">{errors.username}</span>}
      </div>

      <button type="submit">Register</button>
    </form>
  )
}
```

```typescript
      await user.click(screen.getByRole('button', { name: /register/i }))

      expect(screen.queryByText('Invalid email format')).not.toBeInTheDocument()
    })
  })

  describe('async validation', () => {
    it('shows error when username is taken', async () => {
      const { user, checkUsername } = setup()
      checkUsername.mockResolvedValue(false)

      await user.type(screen.getByLabelText(/username/i), 'taken_user')
      // 失焦触发异步验证
      await user.click(screen.getByLabelText(/name/i))

      await waitFor(() => {
        expect(screen.getByText('Username is already taken')).toBeInTheDocument()
      })
    })

    it('clears username error when username is available', async () => {
      const { user, checkUsername } = setup()
      checkUsername.mockResolvedValue(true)

      await user.type(screen.getByLabelText(/username/i), 'new_user')
      await user.click(screen.getByLabelText(/name/i))

      await waitFor(() => {
        expect(screen.queryByText('Username is already taken')).not.toBeInTheDocument()
      })
    })
  })

  describe('form submission', () => {
    it('submits when all fields are valid', async () => {
      const { user, onSubmit } = setup()

      await user.type(screen.getByLabelText(/name/i), 'Alice')
      await user.type(screen.getByLabelText(/email/i), 'alice@example.com')
      await user.type(screen.getByLabelText(/username/i), 'alice123')

      await user.click(screen.getByRole('button', { name: /register/i }))

      expect(onSubmit).toHaveBeenCalledWith({
        name: 'Alice',
        email: 'alice@example.com',
        username: 'alice123',
      })
    })

    it('does not submit when validation fails', async () => {
      const { user, onSubmit } = setup()

      await user.click(screen.getByRole('button', { name: /register/i }))

      expect(onSubmit).not.toHaveBeenCalled()
    })
  })
})
```

> **自我验证说明**：
> - `aria-invalid` 属性让屏幕阅读器知道输入值不合法，测试中通过 `toHaveAttribute('aria-invalid', 'true')` 验证可访问性。这对于使用辅助技术的用户至关重要。
> - 异步验证通常用 `onBlur` 触发，所以测试中需要在输入后点击其他元素失焦，然后用 `waitFor` 等待异步验证结果。
> - `noValidate` 在 `<form>` 上禁用浏览器的原生验证，让自定义验证逻辑完全控制错误显示。

```typescript
// src/components/RHFRegisterForm.tsx
import { useForm } from 'react-hook-form'

interface RegisterFormData {
  name: string
  email: string
  password: string
}

interface RHFRegisterFormProps {
  onSubmit: (data: RegisterFormData) => Promise<void>
}

export function RHFRegisterForm({ onSubmit }: RHFRegisterFormProps) {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
    setError,
  } = useForm<RegisterFormData>()

  const onFormSubmit = async (data: RegisterFormData) => {
    try {
      await onSubmit(data)
    } catch (err) {
      // 服务端验证错误映射到字段
      if (err instanceof ApiError && err.status === 409) {
        setError('email', {
          type: 'manual',
          message: 'This email is already registered',
        })
      }
    }
  }

  return (
    <form onSubmit={handleSubmit(onFormSubmit)}>
      <div>
        <label htmlFor="rhf-name">Name</label>
        <input
          id="rhf-name"
          {...register('name', { required: 'Name is required' })}
          aria-invalid={!!errors.name}
        />
        {errors.name && (
          <span role="alert">{errors.name.message}</span>
        )}
      </div>

      <div>
        <label htmlFor="rhf-email">Email</label>
        <input
          id="rhf-email"
          type="email"
          {...register('email', {
            required: 'Email is required',
            pattern: {
              value: /^[^\s@]+@[^\s@]+\.[^\s@]+$/,
              message: 'Invalid email format',
            },
          })}
          aria-invalid={!!errors.email}
        />
        {errors.email && (
          <span role="alert">{errors.email.message}</span>
        )}
      </div>

      <div>
        <label htmlFor="rhf-password">Password</label>
        <input
          id="rhf-password"
          type="password"
          {...register('password', {
            required: 'Password is required',
            minLength: { value: 8, message: 'Password must be at least 8 characters' },
          })}
          aria-invalid={!!errors.password}
        />
        {errors.password && (
          <span role="alert">{errors.password.message}</span>
        )}
      </div>

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Registering...' : 'Register'}
      </button>
    </form>
  )
}
```

> **自我验证说明**：
> - react-hook-form 的 `register()` 方法返回 `{ onChange, onBlur, ref, name }`，RTL 的 `getByLabelText` 通过 `htmlFor` 关联 label 和 input，所以测试中可以直接用 `getByLabelText` 定位输入框。
> - `formState.isSubmitting` 状态通过按钮的 disabled 属性和文字间接断言——这是正确的测试方式。不直接检查 `isSubmitting` 的值，而是检查用户看到的效果。
> - 服务端错误映射（`setError` 是 react-hook-form 的 API，用于在提交后设置特定字段的错误消息）测试需要 `onSubmit` 抛出一个可以被 catch 的错误。

> **Jest 对比：react-hook-form 测试在 Jest 和 Vitest 中的兼容性**
>
> `react-hook-form` 本身是一个与测试框架无关的库。在 Jest 和 Vitest 中测试 react-hook-form 组件的方式几乎没有区别——因为它完全通过 `@testing-library/react` 的 `render` 和 `userEvent` 来驱动。
>
> 但有一个常见的迁移陷阱：**`@testing-library/react` v14+ 对 `act()` 警告的处理**。
>
> ```typescript
> // 在 Jest 中可能不会触发的 act 警告
> it('submits form data', async () => {
>   render(<RHFRegisterForm onSubmit={vi.fn()} />)
>
>   // 在 React Testing Library v14 + React 18 中，
>   // await user.click() 内部的 state 更新可能会触发 act() 警告
>   await user.click(screen.getByRole('button'))
>
>   // 如果测试在 Jest 中通过但在 Vitest 中报 act() 警告，
>   // 通常是因为 Vitest 的微任务调度时机与 Jest 不同
> })
> ```
>
> 解决方案：将断言包裹在 `waitFor` 中，让 React 有时间处理所有 state 更新：
>
> ```typescript
> // 更健壮的方式
> await user.click(screen.getByRole('button'))
> await waitFor(() => {
>   expect(onSubmit).toHaveBeenCalled()
> })
> ```
>
> 另一个差异：**Jest 的 `jest.fn()` 默认返回 `undefined`，而 `vi.fn()` 行为相同**。所以如果你的测试中 `onSubmit` 被调用时返回 `Promise<undefined>`，两者行为一致。但如果你的测试依赖于 `jest.fn().mockReturnValue(Promise.resolve(...))`，迁移到 Vitest 时需要改为 `vi.fn().mockResolvedValue(...)`——API 名称不同但语义等价。

## 9.5 表单可访问性测试

### 核心可访问性断言

```typescript
describe('form accessibility', () => {
  it('associates labels with inputs', () => {
    render(<ProfileForm onSubmit={vi.fn()} />)

    // getByLabelText 验证了 label 和 input 的关联
    const nameInput = screen.getByLabelText(/name/i)
    expect(nameInput).toHaveAttribute('id', 'name')

    // label 的 htmlFor 属性与 input 的 id 匹配
    const label = screen.getByText(/name/i)
    expect(label).toHaveAttribute('for', 'name')
  })

  it('marks invalid fields with aria-invalid', async () => {
    const user = userEvent.setup()
    render(<ValidatedForm onSubmit={vi.fn()} checkUsernameAvailability={vi.fn()} />)

    await user.click(screen.getByRole('button', { name: /register/i }))

    const nameInput = screen.getByLabelText(/name/i)
    expect(nameInput).toHaveAttribute('aria-invalid', 'true')
  })

  it('associates error messages with inputs via aria-describedby', async () => {
    const user = userEvent.setup()
    render(<ValidatedForm onSubmit={vi.fn()} checkUsernameAvailability={vi.fn()} />)

    await user.click(screen.getByRole('button', { name: /register/i }))

    const nameInput = screen.getByLabelText(/name/i)
    const errorId = nameInput.getAttribute('aria-describedby')

    // aria-describedby 指向了错误消息的 id
    expect(errorId).toBe('name-error')
    if (errorId) {
      expect(screen.getByText('Name is required')).toHaveAttribute('id', errorId)
    }
  })

  it('uses role="alert" for error messages', async () => {
    const user = userEvent.setup()
    render(<ValidatedForm onSubmit={vi.fn()} checkUsernameAvailability={vi.fn()} />)

    await user.click(screen.getByRole('button', { name: /register/i }))

    const alerts = screen.getAllByRole('alert')
    expect(alerts).toHaveLength(3) // name, email, username errors
  })

  it('supports keyboard navigation through fields', async () => {
    const user = userEvent.setup()
    render(<ProfileForm onSubmit={vi.fn()} />)

    // 初始焦点应该在页面上第一个可聚焦元素
    await user.tab()
    expect(screen.getByLabelText(/name/i)).toHaveFocus()

    await user.tab()
    expect(screen.getByLabelText(/age/i)).toHaveFocus()

    await user.tab()
    expect(screen.getByLabelText(/country/i)).toHaveFocus()

    // checkbox 是一个可聚焦元素
    await user.tab()
    expect(screen.getByLabelText(/subscribe/i)).toHaveFocus()

    // fieldset 内的 radio 按钮
    await user.tab()
    expect(screen.getByLabelText(/male/i)).toHaveFocus()

    await user.tab()
    expect(screen.getByLabelText(/female/i)).toHaveFocus()

    await user.tab()
    expect(screen.getByLabelText(/bio/i)).toHaveFocus()

    // 最后是提交按钮
    await user.tab()
    expect(screen.getByRole('button', { name: /save profile/i })).toHaveFocus()
  })

  it('submits form with keyboard', async () => {
    const onSubmit = vi.fn().mockResolvedValue(undefined)
    const user = userEvent.setup()
    render(<ProfileForm onSubmit={onSubmit} />)

    // 填写所有字段（用键盘）
    await user.type(screen.getByLabelText(/name/i), 'Alice')
    await user.type(screen.getByLabelText(/bio/i), 'Keyboard user')

    // Tab 到提交按钮并按 Enter
    await user.tab()
    await user.keyboard('{Enter}')

    expect(onSubmit).toHaveBeenCalled()
  })
})
```

> **自我验证说明**：
> - `userEvent.tab()` 模拟 Tab 键在可聚焦元素间导航。测试 Tab 顺序可以发现焦点管理问题——这些问题在屏幕阅读器用户使用中会直接变成功能性 bug。
> - `aria-invalid` + `aria-describedby` + `role="alert"` 三个属性构成了表单错误的完整可访问性方案：`aria-invalid` 标记字段状态，`aria-describedby` 将错误与字段关联，`role="alert"` 让屏幕阅读器在错误出现时朗读。
> - `user.keyboard('{Enter}')` 在聚焦提交按钮时按回车，等价于 `user.click()`，但更接近键盘用户的交互方式。

## 9.6 反模式

### 反模式 1：使用 fireEvent.change 替代 userEvent.type

```typescript
// ❌ fireEvent.change 不触发完整事件序列
fireEvent.change(input, { target: { value: 'hello' } })
// 没有 focus 事件、没有 keydown/keypress/input/keyup 序列
// 依赖这些事件的逻辑（格式化的输入、字符级验证）不会被触发

// ✅ userEvent.type 触发完整事件序列
await user.type(input, 'hello')
// focus → focusin → keydown → keypress → input → keyup
// 每个字符都是独立的事件序列
```

### 反模式 2：用 getByDisplayValue 替代 getByLabelText

```typescript
// ❌ 通过显示值查找输入框
const nameInput = screen.getByDisplayValue('Alice')

// 问题：如果两个字段有相同的值，测试无法区分
// 如果值变化了，测试就找不到元素

// ✅ 通过 label 查找
const nameInput = screen.getByLabelText(/name/i)
```

### 反模式 3：测试验证逻辑时检查内部状态

```typescript
// ❌ 直接检查 formState
const { result } = renderHook(() => useForm())
// 在测试中通过 ref 获取 formState 并检查 errors

// ✅ 检查用户看到的效果
// 错误消息是否显示、aria-invalid 是否设置、按钮是否禁用
expect(screen.getByText('Name is required')).toBeInTheDocument()
expect(screen.getByLabelText(/name/i)).toHaveAttribute('aria-invalid', 'true')
```

### 反模式 4：跳过 userEvent.setup()

```typescript
// ❌ 每次测试都创建一个新的 userEvent 实例
it('test 1', async () => {
  const user = userEvent.setup()
  // ...
})

it('test 2', async () => {
  const user = userEvent.setup()
  // ...
})

// 在 describe 级别 setup 一次更高效
// ✅
describe('Form', () => {
  let user: ReturnType<typeof userEvent.setup>

  beforeEach(() => {
    user = userEvent.setup()
  })

  it('test 1', async () => {
    await user.click(screen.getByRole('button'))
  })
})
```

## 9.8 本章总结

- 表单测试覆盖所有字段类型：text、number、select、checkbox、radio、textarea，每种类型有不同的交互模式和断言方法
- 验证测试分为三个层级：必填验证（同步）、格式验证（同步）、可用性检查（异步）
- 异步验证通过 `onBlur` 触发，测试需要先失焦再用 `waitFor` 等待结果
- react-hook-form 测试关注点：`register` 正确绑定、`formState` 的 `errors` 和 `isSubmitting`、`setError` 处理服务端错误
- 可访问性测试验证：`aria-invalid`、`aria-describedby`、`role="alert"`、焦点管理、键盘导航
- ant-design 和 MUI 等 UI 库的组件可能渲染非标准 DOM 结构，但可访问性属性和 role 仍然可以通过标准查询定位
