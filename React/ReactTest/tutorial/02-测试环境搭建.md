npm create vite@latest my-app -- --template react-ts

## 2.2 vitest.config.ts 详解

Vitest 配置可以写在 `vitest.config.ts` 中，也可以合入 `vite.config.ts` 的 `test` 字段。推荐**合并到 `vite.config.ts`** 以减少文件数量：

```typescript
// vite.config.ts
/// <reference types="vitest/config" />
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  test: {
    // === 核心配置 ===
    globals: true,                    // 使 describe/it/expect/vi 全局可用
    environment: 'jsdom',             // 模拟浏览器 DOM 环境
    setupFiles: ['./src/test/setup.ts'], // 每个测试文件前执行的 setup

    // === 文件匹配 ===
    include: ['src/**/*.{test,spec}.{ts,tsx}'],
    exclude: ['node_modules', 'dist'],

    // === CSS ===
    css: true,                        // 处理 CSS 导入（不报错）

    // === 覆盖率 ===
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
      include: ['src/**/*.{ts,tsx}'],
      exclude: [
        'src/**/*.test.*',
        'src/**/*.spec.*',
        'src/test/**',
        'src/**/index.ts',
      ],
    },

    // === 环境变量 ===
    env: {},
  },
})
```

### globals: true 的代价与收益

```typescript
// globals: true — 无需导入即可使用全局 API
describe('counter', () => {
  it('increments', () => {
    expect(1 + 1).toBe(2)
  })
})

// globals: false — 需要显式导入
import { describe, it, expect } from 'vitest'

describe('counter', () => {
  it('increments', () => {
    expect(1 + 1).toBe(2)
  })
})
```

收益：少写 import，测试更简洁（尤其是迁移自 Jest 的项目）
代价：编辑器可能报"找不到 describe"的 TS 错误——需要配置类型（见 2.3）

> **Jest 对比：** Jest 的配置与 Vitest 有一个根本差异：Jest 的 `jest.config.js` 与构建配置（Webpack/Vite）完全独立，路径别名需要在两边同步维护。如果你用 Webpack 构建、Jest 测试，`webpack.resolve.alias` 和 `jest.moduleNameMapper` 必须指向同一个映射表——它们是完全不同的 API，但描述的是同一件事。Vitest 继承 Vite 的 `resolve.alias`，Alias 只需配置一次。这种"一次配置，处处使用"的模式是 Vitest 架构的核心优势。

> **为什么：** `globals: true` 的核心矛盾在于运行效率和类型安全的权衡。运行时层面，全局 API 没有性能损失——`describe` 就是 `describe`，无论你 import 与否。但类型层面，TypeScript 需要知道全局 API 的类型签名，这通过 `"types": ["vitest/globals"]` 实现。如果你的项目有其他需要全局类型的库（如 Cypress 的 `cy`），全局类型声明可能冲突。大型项目建议在每个测试文件中显式 import，避免全局类型污染。小型项目或 Jest 迁移项目推荐 `globals: true` + `types: ["vitest/globals"]`，迁移成本最低。

### environment 选择：jsdom vs happy-dom

| 特性 | jsdom | happy-dom |
|------|-------|-----------|
| **Web 标准完整度** | 高（更接近真实浏览器） | 中（部分 API 缺失） |
| **速度** | 较慢 | 快 2-3x |
| **安装大小** | 大 | 小 |
| **layout/style** | 部分支持 | 基本不支持 |
| **推荐场景** | 组件测试（需要完整的 DOM API） | 纯数据/Hook 测试 |

```typescript
// 项目级默认 jsdom
test: { environment: 'jsdom' }

// 特定文件覆盖为 node（纯逻辑测试，不需要 DOM）
// @vitest-environment node
// （写在测试文件顶部注释）
```

> **自我验证说明**：`@vitest-environment` 注释是 Vitest 的内置功能，在文件顶部添加 `// @vitest-environment <name>` 即可覆盖项目级配置。参考 [Vitest 文档 - Environment](https://vitest.dev/guide/environment.html)。

> **为什么：** jsdom 比 happy-dom 更慢但更完整，是因为 jsdom 对 Web 标准 API 的实现更接近真实浏览器。happy-dom 的优化策略是"实现常用 API，未实现的部分抛错或返回 undefined"——这在你的组件使用 `getBoundingClientRect()`、`MutationObserver`、`IntersectionObserver` 等 API 时会造成诡异的难以调试的错误。推荐在组件测试中使用 jsdom：每次测试慢几十毫秒，远不如"测试通过了但实际功能坏了"的调试成本高。对于纯逻辑测试或 Hook 测试（不依赖 DOM），可以使用 `// @vitest-environment node` 跳过 jsdom 的额外开销。

### setupFiles 执行时机

`setupFiles` 在每个测试**文件**执行前运行（注意不是每个测试用例前）：

```
vitest 启动
  └→ 执行 setupFiles（跑一次）
  └→ 对每个测试文件：
       ├→ 执行 globalSetup（如果配置了）
       ├→ 加载测试文件
       └→ 按顺序执行：beforeAll → beforeEach → it → afterEach → afterAll
```

`setupFiles` 适合放：MSW server 生命周期注册、jest-dom matcher 扩展、全局 mock（`window.matchMedia` 等）。

## 2.3 TypeScript 配置

### tsconfig.json

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "types": ["vitest/globals"]
  },
  "include": ["src"],
  "references": [{ "path": "./tsconfig.node.json" }]
}
```

关键字段说明：

- **`"types": ["vitest/globals"]`**：让 TypeScript 识别 `describe`、`it`、`expect`、`vi` 等全局 API。仅当 `globals: true` 时需要
- **`"moduleResolution": "bundler"`**：现代 Vite 项目推荐，允许不带扩展名的 import 和 package.json exports 解析
- **`"skipLibCheck": true`**：跳过 `.d.ts` 的类型检查（加速编译，风险低）

### jest-dom matcher 类型扩展

`@testing-library/jest-dom` 提供如 `toBeInTheDocument()`、`toHaveTextContent()` 等额外的 matcher。这些 matcher 的类型需要被 TypeScript 识别：

```typescript
// src/test/setup.ts — 同时也解决类型问题
import '@testing-library/jest-dom/vitest'
//                       ^^^^^^^^
// 注意：是 /vitest 而非默认路径
```

`@testing-library/jest-dom/vitest` 会：
1. 导入所有 jest-dom matcher（`toBeInTheDocument`、`toHaveTextContent` 等）
2. 通过 `expect.extend()` 将它们注册到 Vitest 的 `expect` 对象
3. 扩展 Vitest 的 `Assertion` 和 `AsymmetricMatchers` 类型

> **自我验证说明**：`@testing-library/jest-dom` 从 v6.6 开始提供 `/vitest` 入口，直接兼容 Vitest 的断言类型系统。如果使用旧版本，需要手动写类型扩展声明。参考 [jest-dom 文档](https://github.com/testing-library/jest-dom#vitest)。

### （可选）vitest.d.ts 类型声明文件

如果不想在 `tsconfig.json` 中添加 `"types": ["vitest/globals"]`（因为它会影响全局类型作用域），可以创建一个独立的类型声明文件：

```typescript
// src/test/vitest.d.ts
/// <reference types="vitest/globals" />
/// <reference types="@testing-library/jest-dom/vitest" />
```

然后在 `tsconfig.json` 中通过 `include` 包含它（而非全局 `types` 字段）。

## 2.4 Setup 文件体系

### 完整 setup.ts

```typescript
// src/test/setup.ts
import '@testing-library/jest-dom/vitest'
import { cleanup } from '@testing-library/react'
import { afterEach } from 'vitest'

// 每个测试后自动清理 DOM（RTL 默认行为，显式声明更清晰）
afterEach(() => {
  cleanup()
})
```

RTL 的 `cleanup()` 会自动在每次测试后卸载渲染的组件。Vitest 环境下这个行为默认启用，但显式调用可以防止意外。

> **自我验证说明**：Vitest 从 v1.0 开始内置了对 RTL cleanup 的自动调用。但如果使用了 `globals: false` 或某些自定义配置，自动 cleanup 可能不会触发。显式调用 `cleanup()` 是无害的保险。参考 [RTL setup 文档](https://testing-library.com/docs/react-testing-library/setup/)。

> **Jest 对比：** Jest 的 `setupFiles` 和 `setupFilesAfterFramework` 两套机制经常让人困惑。前者在测试框架初始化前执行，此时 jest-dom 等扩展尚未注册；后者在框架初始化后执行，但名字冗长且易于混淆。如果放错了文件，jest-dom matcher 可能还没注册就被使用，导致诡异的"matcher not found"错误。Vitest 只有 `setupFiles`，语义等价于 Jest 的 `setupFilesAfterFramework`——在测试框架初始化完成后、每个测试文件执行前运行。这个简化不仅减少了配置错误，也让 setup 的执行时机更容易理解。

> **为什么：** 每次测试后调用 `cleanup()` 可以防止跨测试的状态泄漏。如果一个测试渲染了组件但没有卸载，下一个测试的 `screen.getByRole()` 可能找到上一个测试遗留的 DOM 元素，导致假阳性——测试通过了但断言的是旧状态。在大型测试套件中（500+ 测试），这种泄漏会逐渐积累，造成看似随机的间歇性失败。显式 `cleanup()` 确保每个测试在干净的 DOM 环境中运行，是测试隔离的基础保障。

### MSW 集成到 Setup（预留）

```typescript
// src/test/setup.ts — 含 MSW 的完整版本
import '@testing-library/jest-dom/vitest'
import { cleanup } from '@testing-library/react'
import { afterAll, afterEach, beforeAll } from 'vitest'
import { server } from '../mocks/server'

// 启动 MSW 服务
beforeAll(() => server.listen({ onUnhandledRequest: 'warn' }))

// 每个测试后重置 handler 覆盖
afterEach(() => server.resetHandlers())

// 所有测试结束后关闭 MSW
afterAll(() => server.close())

// 清理 DOM
afterEach(() => {
  cleanup()
})
```

### 延伸：自定义 render 函数的封装模式

当项目中引入多个 Provider（Router、QueryClient、Theme 等），每个测试都手动包裹组件会导致大量重复代码。推荐创建一个统一的测试工具文件：

```typescript
// src/test/test-utils.tsx
import { render, RenderOptions } from '@testing-library/react'
import { MemoryRouter } from 'react-router-dom'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

const queryClient = new QueryClient({
  defaultOptions: {
    queries: { retry: false },  // 测试中禁用 retry，避免超时
  },
})

function AllProviders({ children }: { children: React.ReactNode }) {
  return (
    <QueryClientProvider client={queryClient}>
      <MemoryRouter initialEntries={['/']}>
        {children}
      </MemoryRouter>
    </QueryClientProvider>
  )
}

function renderWithProviders(
  ui: React.ReactElement,
  options?: Omit<RenderOptions, 'wrapper'>
) {
  return render(ui, { wrapper: AllProviders, ...options })
}

// 重新导出所有需要的内容
export * from '@testing-library/react'
export { renderWithProviders }
```

然后在测试中替代原始的 `render` 使用：

```typescript
// 使用自定义 render 替代原始 render
import { renderWithProviders, screen } from '../../test/test-utils'

it('renders within provider context', async () => {
  renderWithProviders(<UserProfile userId={1} />)
  expect(await screen.findByText('Alice')).toBeInTheDocument()
})
```

> **自我验证说明：** `wrapper` 是一个接收 `{ children }` props 的 React 组件。每次 `render` 都会创建新的 wrapper 实例，因此不同测试间的 Provider 状态不会互相污染。`queryClient` 的 `retry: false` 在测试中至关重要：如果组件查询失败，React Query 默认会重试 3 次，导致测试超时。禁用 retry 让错误可以直接抛出，便于测试错误状态。

## 2.5 验证环境就绪

### 第一个"冒烟测试"

创建 `src/App.test.tsx`：

```typescript
import { render, screen } from '@testing-library/react'
import { describe, expect, it } from 'vitest'

function Hello({ name }: { name: string }) {
  return <h1>Hello, {name}!</h1>
}

describe('测试环境验证', () => {
  it('renders a heading', () => {
    render(<Hello name="World" />)
    expect(
      screen.getByRole('heading', { name: /hello, world/i })
    ).toBeInTheDocument()
  })
})
```

> **自我验证说明**：
> - `getByRole('heading', { name: /hello, world/i })` — `heading` 是 `<h1>` 的 ARIA role，`name` 选项匹配 accessible name（正则 `/i` 表示忽略大小写）
> - `toBeInTheDocument()` — jest-dom 提供的 matcher，非 Jest/Vitest 内置，需要 `import '@testing-library/jest-dom/vitest'`
> - 如果此测试通过，说明 `render`、`screen`、`getByRole`、`toBeInTheDocument` 均正常工作

### 运行测试

```bash
# 单次运行
npx vitest run

# watch 模式（开发时使用）
npx vitest

# 带覆盖率报告
npx vitest run --coverage

# Vitest UI（交互式图形界面）
npx vitest --ui
```

Vitest UI 模式下，你可以在浏览器中查看测试结果、过滤测试、查看每个测试的控制台输出、检查测试文件的依赖关系图。这在调试复杂测试时非常有用——特别是当某个测试在 CI 中失败但在本地无法复现时，UI 的筛选和日志浏览功能能加速排查。

### 测试过滤与选择执行

在大型项目中，你可能需要只运行特定范围的测试：

```bash
# 按文件名模式过滤
npx vitest --testNamePattern="useCounter"

# 或者简写
npx vitest -t "useCounter"

# 指定特定测试文件
npx vitest run src/components/Button.test.tsx

# 排除某些文件
npx vitest --exclude "src/legacy/**"
```

这些过滤方式不会加速测试执行本身，但它们能显著减少 TDD 循环中的等待时间——你不需要等待所有测试通过，只需要关心当前正在开发的模块相关的测试。

### 集成测试中的环境变量管理

测试中经常需要模拟不同的环境配置。Vitest 提供了两种方式设置环境变量：

```typescript
// 方式 1：在 vitest.config.ts 中统一设置
export default defineConfig({
  test: {
    env: {
      API_BASE_URL: 'http://localhost:4000',
      FEATURE_FLAG_NEW_UI: 'true',
    },
  },
})

// 方式 2：在测试文件中动态设置（注意：需要 beforeEach/afterEach 清理）
beforeEach(() => {
  vi.stubEnv('API_BASE_URL', 'http://test-api.local')
})

afterEach(() => {
  vi.unstubAllEnvs()
})

it('uses test API URL', () => {
  expect(import.meta.env.API_BASE_URL).toBe('http://test-api.local')
})
```

> **自我验证说明：** `vi.stubEnv()` 修改的是 `import.meta.env` 的值，不会污染 `process.env`。`vi.unstubAllEnvs()` 在 `afterEach` 中调用可以确保每个测试的环境变量互不干扰。参考 [Vitest 文档 - Environment Variables](https://vitest.dev/guide/environment.html#environment-variables)。

> **Jest 对比：** Jest 中设置环境变量需要手动管理 `process.env`，常见的模式是 `beforeEach(() => { process.env.API_KEY = 'test' })` 并在 `afterEach` 中删除。但 `process.env` 是全局单例，一旦修改会影响同进程中的其他测试文件。Vitest 的 `vi.stubEnv()` 在 worker 级别隔离环境变量，不同 worker 之间的环境变量互不影响。

### 常见环境错误排查

| 错误 | 原因 | 解决 |
|------|------|------|
| `ReferenceError: document is not defined` | 未配置 `environment: 'jsdom'` | 在 `vitest.config.ts` 中添加 `environment: 'jsdom'` |
| `describe is not defined` | `globals: false` 且未 `import { describe } from 'vitest'` | 要么打开 `globals`，要么显式 import |
| `Property 'toBeInTheDocument' does not exist` | 未导入 jest-dom matcher | `import '@testing-library/jest-dom/vitest'` 加入 setup |
| `TS2304: Cannot find name 'describe'` | TypeScript 不认识 vitest 全局 API | 在 `tsconfig.json` 的 `types` 加 `"vitest/globals"` |
| `Error: [vitest] There was an error when evaluating setup file` | setup 文件中有导入错误 | 检查 setup file 路径和其中的导入是否正确 |
| `Failed to parse source for import analysis` | 某个依赖的 CJS 格式不兼容 | 在 `deps.optimizer.ssr.include` 中添加该包名 |

## 2.7 反模式

### 反模式 1：跳过 setup 文件，每个测试文件重复 MSW 初始化

```typescript
// ❌ 每个测试文件都写一遍
beforeAll(() => server.listen())
afterEach(() => server.resetHandlers())
afterAll(() => server.close())
```

应该放在 `setupFiles` 中全局执行一次。

### 反模式 2：在 setup 中引入太重的逻辑

```typescript
// ❌ setup.ts 不应包含业务数据或特定组件逻辑
beforeAll(() => {
  localStorage.setItem('user', JSON.stringify({ id: 1, role: 'admin' }))
  // 不同测试可能需要不同的 localStorage 状态
})
```

setup 只放全局基础设施。每个测试自己管理所需的初始状态。

### 反模式 3：配置了 globals 却不配置 types

```typescript
// vitest.config.ts 中设置了 globals: true
// 但 tsconfig.json 中没有 "types": ["vitest/globals"]
// 结果：编辑器报红，但实际上测试能跑
```

类型配置和运行时配置必须同步。

## 2.8 本章练习

1. 从零搭建一个 React + TypeScript + Vite 项目，按本章步骤配置测试环境，确保冒烟测试通过
2. 尝试在 `globals: false` 和 `globals: true` 之间切换，体会差异
3. 故意制造三个错误：去掉 `environment: 'jsdom'`、去掉 jest-dom import、去掉 types 配置。观察报错信息，学会快速诊断

## 2.9 本章总结

- `vitest.config.ts` 的核心五件套：`globals`、`environment`、`setupFiles`、`include`、`coverage`
- TypeScript 整合的关键：`types: ["vitest/globals"]` + `import '@testing-library/jest-dom/vitest'`
- setup 文件放全局基础设施，不放业务初始化
- 冒烟测试是验证环境配置的最快方式
- 遇到环境错误时，从 `environment` → `globals` → `types` → `setupFiles` 顺序排查
