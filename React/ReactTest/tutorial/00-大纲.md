## 第1章：测试架构思维

### 1.1 为什么要写测试
- 测试的本质：自动化验证行为预期的工具
- 不写测试的真实成本：回归恐惧 → 重构停滞 → 代码腐化
- 测试覆盖率不等于测试质量
- 测试作为活文档：比 README 更可信的行为契约

### 1.2 测试金字塔到测试奖杯
- 经典测试金字塔（Unit → Integration → E2E）及其局限
- Kent C. Dodds 的 Testing Trophy：静态分析为底座，集成测试为重心
- 为什么前端测试中集成测试的投资回报率最高
- 各层工具的对应：TypeScript/ESLint → Vitest+RTL → Playwright/Cypress

### 1.3 React Testing Library 哲学
- "你的测试越接近用户使用方式，就越能给你信心"
- 测试行为而非实现细节
- 不测试 state、props、内部方法、生命周期
- 反模式清单：`container.querySelector`、`instance()`、`state` 断言

### 1.4 技术栈选型：为什么是 Vitest + MSW + RTL
- Vitest vs Jest：性能（原生 ESM、vite 驱动的 transforms）、API 兼容性、配置复杂度对比
- MSW vs vi.mock/fetch-mock：网络层拦截 vs 模块级 mock 的哲学差异
- RTL vs Enzyme：设计哲学的根本对立
- TypeScript 的角色：类型安全作为测试的静态前置防线

### 1.5 本章练习
- 思考题：你当前的测试覆盖了哪些层？缺失了什么？
- 动手：用 `npx create vite` 创建一个 React+TS 项目骨架

## 第5章：用户事件模拟

### 5.1 userEvent vs fireEvent
- `fireEvent`：直接触发单个 DOM 事件（`fireEvent.click(el)`）
- `userEvent`：模拟真实用户交互序列（`user.click(el)` 触发了 focus → pointerdown → mousedown → click → pointerup → mouseup）
- 为什么 userEvent 提供更高可信度
- 唯一仍应使用 fireEvent 的场景

### 5.2 userEvent.setup() v14 API
- `userEvent.setup()` 返回 `UserEvent` 实例
- 全局配置：`advanceTimers`、`delay`、`skipHover`、`pointerEventsCheck`
- 为什么必须 `setup()`（v14 的行为变更）

### 5.3 核心交互方法
- `click()` / `dblClick()` / `tripleClick()`
- `type()` / `clear()` / `paste()`
- `keyboard()`：单键、组合键、特殊键
- `tab()`：焦点导航
- `selectOptions()` / `deselectOptions()`
- `upload()`：文件上传模拟
- `hover()` / `unhover()`
- `copy()` / `cut()` / `paste()`

### 5.4 交互的异步本质
- 所有 `userEvent.*` 方法返回 Promise
- `await user.click()` 的必要性
- React 状态更新 + userEvent 的时序理解

### 5.5 复杂交互场景
- 表单完整填写 + 提交
- 拖拽模拟
- 键盘导航（TAB + Enter + Escape 流程）
- 剪切板操作

### 5.6 反模式
- 忘记 `await` userEvent 调用
- `userEvent.setup()` 放在 `beforeEach` 中但不传给测试
- 在非 jsdom 环境下使用 `userEvent`

## 第6章：MSW 哲学与实践

### 6.1 为什么要做网络层 Mock
- 模块级 mock（`vi.mock('axios')`）的局限：测试与实现耦合，换 HTTP 库测试全灭
- 网络层拦截：Mock 协议而非模块，换 fetch/axios/ky 测试都不变
- MSW 的核心洞察：Service Worker API 是浏览器原生网络拦截标准

### 6.2 MSW 架构模型
- `setupServer`（Node/测试）vs `setupWorker`（浏览器/开发）
- Handler → Server → 请求拦截 三层模型
- 请求匹配流程：URL 模式 → 方法 → 条件（path params / search params / headers / body）

### 6.3 Handler 定义
- `http.get` / `http.post` / `http.put` / `http.patch` / `http.delete`
- 路径参数提取（`:id`）和 TypeScript 泛型
- 请求体解析（`await request.json()`）
- 响应构造：`HttpResponse.json()`、`HttpResponse.text()`、`HttpResponse.xml()`、`HttpResponse.error()`
- 条件响应：基于请求参数返回不同 mock 数据

### 6.4 响应策略
- 正常响应：`HttpResponse.json(data, { status: 200 })`
- 创建响应：`HttpResponse.json(data, { status: 201 })`
- 客户端错误：`new HttpResponse(null, { status: 400/401/403/404 })`
- 服务端错误：`new HttpResponse(null, { status: 500 })`
- 网络错误：`HttpResponse.error()`（TypeError，非 HTTP 响应）

### 6.5 Server 生命周期
- `server.listen({ onUnhandledRequest })`：启动拦截，配置未处理策略
- `server.resetHandlers()`：重置到初始 handler 列表
- `server.use()`：运行时追加 handler（LIFO 覆盖）
- `server.restoreHandlers()`：仅回退 `use()` 的覆盖
- `server.close()`：停止拦截释放资源

### 6.6 Per-test Handler 覆盖模式
- 在单个测试中覆盖 handler 响应 404/500/网络错误
- 在 `describe` 块中覆盖默认 handler
- `server.use()` 与 `server.resetHandlers()` 的配合

### 6.7 网络策略
- `onUnhandledRequest: 'warn'`：未匹配请求警告（开发）
- `onUnhandledRequest: 'error'`：未匹配请求报错（严格模式）
- `onUnhandledRequest: 'bypass'`：静默放行
- 自定义 bypass 逻辑：排除 analytics、health check 等非测试请求

### 6.8 GraphQL Mock
- `graphql.query()` 和 `graphql.mutation()`
- 按 operation name 匹配
- 在 MSW v2 中使用 `HttpResponse.json()` 构造 GraphQL 响应

### 6.9 MSW v1 → v2 迁移要点
- `res(ctx.json(data))` → `HttpResponse.json(data)`
- `rest.get(path, (req, res, ctx) => ...)` → `http.get(path, ({ request }) => ...)`
- `req.body` → `await request.json()`
- `ctx.delay(ms)` → `await delay(ms)`

### 6.10 本章练习
- 搭建完整的 MSW 配置（handlers/server/setup），编写覆盖 GET/POST/404/500/网络错误的测试

## 第14章：从 Jest 迁移到 Vitest

### 14.1 迁移动机与风险评估
- Vitest 的优势在实际项目中的体现
- 兼容性矩阵：哪些 Jest 特性无缝迁移，哪些需要调整
- 迁移的渐进式策略（不要一次全部重写）

### 14.2 全局 API 对照
- `jest.fn()` → `vi.fn()`（签名 1:1 兼容）
- `jest.mock()` → `vi.mock()`（hoist 行为一致）
- `jest.spyOn()` → `vi.spyOn()`（完全兼容）
- `jest.setTimeout()` → `test.timeout` 或 `vi.setConfig()`
- 定时器：`jest.useFakeTimers()` → `vi.useFakeTimers()`（API 类似但有细微差异）
- 环境：`jest.setupFiles` → `vitest setupFiles`
- Snapshot：格式完全兼容

### 14.3 配置迁移对照表
- `jest.config.js/ts` → `vitest.config.ts`
- `transform` → Vite plugins 自动处理
- `moduleNameMapper` → `resolve.alias`
- `testEnvironment: 'jsdom'` → `environment: 'jsdom'`
- `setupFilesAfterFramework` → `setupFiles`

### 14.4 需要调整的模式
- `jest.requireActual()` → `vi.importActual()`（异步 vs 同步）
- `jest.mock` 工厂函数内的变量引用 → `vi.hoisted()`
- `jest-environment` 文件级注释 → `@vitest-environment` 注释
- `@jest-environment` → `@vitest-environment`

### 14.5 模块 mock 的差异
- Vitest 对 ESM/CJS 混合模块的处理
- `__mocks__` 目录自动 mock 的行为差异
- `importOriginal` 的异步特性

### 14.6 迁移检查清单
- 步骤 1：安装 vitest 并创建 vitest.config.ts
- 步骤 2：全局替换 `jest` → `vi`（大部分 1:1）
- 步骤 3：调整 `jest.fn()` 的 `mockImplementation` 等链式调用
- 步骤 4：处理 `jest.requireActual` → `vi.importActual`
- 步骤 5：处理 `jest.mock` 工厂中引用的外部变量
- 步骤 6：验证 snapshot 兼容性
- 步骤 7：运行全部测试，修复剩余的差异

### 14.7 本章练习
- 将一个使用 Jest 的项目迁移到 Vitest（至少包含 mock、snapshot、定时器）
