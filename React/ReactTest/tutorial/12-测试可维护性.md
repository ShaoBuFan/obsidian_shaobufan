## 12.1 测试代码也是代码

### 为什么可维护性对测试同样重要

测试代码在项目生命周期中会被频繁阅读和修改——每次功能变更、重构、bug 修复都需要更新测试。如果测试代码难以理解、充满重复、组织混乱，开发者最终会放弃维护测试，导致测试逐渐腐烂。

测试可维护性的三条原则：

```
1. 可读性 > 简洁性：测试的首要读者是人，不是机器
2. 重复有代价，抽象也有代价：不是所有重复都应该被抽取
3. 测试是文档的最好形式：好的测试结构本身就是活文档
```

### DRY 原则的适用范围和例外

```typescript
// ❌ 过度抽象：一个函数隐藏了太多 setup 细节
function setupFullPage() {
  const user = userEvent.setup()
  const queryClient = new QueryClient({ defaultOptions: { queries: { retry: false } } })
  const utils = render(<FullPage />, {
    wrapper: ({ children }) => (
      <MemoryRouter>
        <QueryClientProvider client={queryClient}>
          <ThemeProvider>{children}</ThemeProvider>
        </QueryClientProvider>
      </MemoryRouter>
    ),
  })
  return { user, ...utils }
}

// 使用时的困惑：这个 setup 到底做了什么？哪些状态是默认的？
const { user } = setupFullPage()

// ✅ 适度抽象：封装 provider 但暴露可配置选项
function renderWithProviders(
  ui: ReactElement,
  {
    initialEntries = ['/'],
    queryClient = new QueryClient({ defaultOptions: { queries: { retry: false } } }),
  }: RenderOptions = {}
) {
  return render(ui, {
    wrapper: ({ children }) => (
      <MemoryRouter initialEntries={initialEntries}>
        <QueryClientProvider client={queryClient}>
          {children}
        </QueryClientProvider>
      </MemoryRouter>
    ),
  })
}

// 使用清晰：知道有 router 和 query，可以指定初始路由
renderWithProviders(<UserPage />, { initialEntries: ['/users/1'] })
```

什么时候该抽取重复代码：

| 场景 | 抽 | 不抽 |
|------|----|------|
| Provider 嵌套 | 封装为 `renderWithProviders` | 不多于 2 层时内联也可 |
| 测试数据创建 | 工厂函数（`buildUser()`） | 只有一个测试用到的小对象 |
| MSW handler 覆盖 | 工厂函数（`createUserHandler()`） | 每个测试覆盖自己的 handler |
| beforeEach 逻辑 | 如果 80%+ 测试共享 | 差异化太大时分开写 |

> **为什么：DRY 有例外——抽象的代价**
>
> "Don't Repeat Yourself" 是编程的黄金法则，但在测试代码中需要谨慎应用。每一次抽象（抽取函数、合并 setup）都有三个隐藏成本：
>
> 1. **可读性损失**：读者需要跳转到函数定义才能理解 setup 做了什么。`setupFullPage()` 这个调用本身没有传达任何信息——setup 了什么？默认值是什么？被测试组件需要哪些 Provider？这些关键信息被抽象屏障遮挡了。
> 2. **耦合增加**：共享的 setup 函数调用者越多，修改它就越困难——每个调用者都隐式依赖 setup 的全部副作用，修改一处可能破坏多处测试。
> 3. **调试难度**：测试失败时，你需要在 shared setup 和测试用例之间来回跳转，定位问题来源的成本远高于复制粘贴的维护成本。
>
> Sandi Metz 的名言 **"Duplication is far cheaper than the wrong abstraction"**（重复远便宜于错误的抽象）在测试中尤其成立。当你忍不住想抽取重复代码时，先问三个问题：
> - 这个重复是"信息重复"还是"偶然重复"？如果修改了一个，另一个也必须改，就是信息重复——值得抽取。如果只是看起来结构相似但语义不同——不要抽取。
> - 抽取后的抽象是否让测试更易读？更短的测试代码 ≠ 更易读的测试代码。
> - 这个 setup 被多少测试共享？少于 3 处时，宁可重复也不要抽象。

### ARRANGE-ACT-ASSERT 三段式布局

每段之间用空行分隔，这是测试代码最重要的排版约定：

```typescript
it('updates user name successfully', async () => {
  // ARRANGE：准备测试数据、渲染组件
  const user = userEvent.setup()
  renderWithProviders(<UserEditor userId={1} />)
  await screen.findByRole('heading', { name: /alice/i })

  // ACT：执行用户操作
  await user.clear(screen.getByLabelText(/name/i))
  await user.type(screen.getByLabelText(/name/i), 'Bob')
  await user.click(screen.getByRole('button', { name: /save/i }))

  // ASSERT：断言结果
  await screen.findByRole('heading', { name: /bob/i })
  expect(screen.getByRole('alert')).toHaveTextContent(/saved/i)
})
```

> **自我验证说明**：AAA 模式不是强制语法规则，而是一种排版纪律。每段只做一件事：ARRANGE 放所有前置准备，ACT 放触发动作，ASSERT 放断言。三段之间用空行分隔。这种模式让测试的"前提→操作→结果"结构一目了然。

## 12.2 自定义 Render

### 从简单到完善的 renderWithProviders

```typescript
import { render, type RenderOptions, type RenderResult } from '@testing-library/react'
import {
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'
import { MemoryRouter } from 'react-router-dom'
import { type ReactElement, type ReactNode } from 'react'

// ---- 支持：Router + QueryClient ----

interface CustomRenderOptions extends Omit<RenderOptions, 'wrapper'> {
  initialEntries?: string[]
  queryClient?: QueryClient
}

function renderWithProviders(
  ui: ReactElement,
  options: CustomRenderOptions = {}
): RenderResult {
  const {
    initialEntries = ['/'],
    queryClient = new QueryClient({
      defaultOptions: {
        queries: { retry: false, gcTime: 0 },
      },
    }),
    ...renderOptions
  } = options

  function Wrapper({ children }: { children: ReactNode }) {
    return (
      <MemoryRouter initialEntries={initialEntries}>
        <QueryClientProvider client={queryClient}>
          {children}
        </QueryClientProvider>
      </MemoryRouter>
    )
  }

  return render(ui, { wrapper: Wrapper, ...renderOptions })
}

// ---- 返回增强的 utilities ----

interface RenderResultEnhanced extends RenderResult {
  user: ReturnType<typeof userEvent.setup>
}

function renderWithProvidersEnhanced(
  ui: ReactElement,
  options: CustomRenderOptions = {}
): RenderResultEnhanced {
  const user = userEvent.setup()
  const result = renderWithProviders(ui, options)
  return { ...result, user }
}
```

### 使用

```typescript
// 基础用法：所有 Provider 自动注入
it('renders with defaults', () => {
  renderWithProviders(<HomePage />)
  expect(screen.getByRole('heading')).toBeInTheDocument()
})

// 指定初始路由
it('renders at specific route', () => {
  renderWithProviders(<ProductPage />, { initialEntries: ['/products/42'] })
  expect(screen.getByText('Product 42')).toBeInTheDocument()
})

// 带 userEvent 的增强版
it('submits form', async () => {
  const { user } = renderWithProvidersEnhanced(<ContactForm />)
  await user.type(screen.getByLabelText(/email/i), 'test@example.com')
  await user.click(screen.getByRole('button', { name: /submit/i }))
  // ...
})
```

> **自我验证说明**：`renderWithProviders` 的返回值类型是 `RenderResult`，与 RTL 的 `render` 完全一致。这意味着你可以把它当作 `render` 的直接替换，零学习成本。增强版返回 `user` 实例，避免了在每个测试中重复调用 `userEvent.setup()`。

> **Jest 对比：测试工具的组织方式**
>
> 在 Jest 中，测试工具的自定义 render 函数组织方式与 Vitest 几乎相同：
>
> ```typescript
> // Jest 同样可以封装 renderWithProviders
> import { render, RenderOptions } from '@testing-library/react'
> import { MemoryRouter } from 'react-router-dom'
>
> function renderWithProviders(
>   ui: React.ReactElement,
>   options: RenderOptions = {}
> ) {
>   return render(ui, {
>     wrapper: ({ children }) => (
>       <MemoryRouter>{children}</MemoryRouter>
>     ),
>     ...options,
>   })
> }
> ```
>
> 核心差异不在工具本身的写法，而在配置方式：
>
> | 维度 | Jest | Vitest |
> |------|------|--------|
> | 工具文件位置 | `src/test/test-utils.tsx` | 相同 |
> | 模块别名 | 需在 `moduleNameMapper` 中额外配置 | 继承 Vite `resolve.alias`，无需重复配置 |
> | TypeScript 处理 | 需 `ts-jest` 或 `@babel/preset-typescript` | esbuild 原生处理，零配置 |
> | import 路径 | `import { renderWithProviders } from '../test/test-utils'` | 完全相同 |
>
> 无论使用哪个框架，工具文件组织原则一致：在 `src/test/` 下集中管理，测试文件通过相对路径导入。迁移到 Vitest 时，test-utils 的代码本身几乎不需要改动——改动的是配置文件。

## 12.3 测试数据工厂

### 为什么需要数据工厂

真实项目的 API 响应通常包含大量字段。在每个测试中手动构造完整对象会导致：

1. 测试代码膨胀：20 行数据 + 3 行逻辑
2. 维护困难：API 字段变化需要改所有测试
3. 无效数据：很多字段测试根本不关心

解决方案：**用工厂函数生成带合理默认值的测试数据**。

### 基础工厂函数

```typescript
// src/test/factories.ts
import { faker } from '@faker-js/faker'

// ---- User 工厂 ----

interface User {
  id: number
  name: string
  email: string
  role: 'admin' | 'user'
  avatar: string | null
  createdAt: string
}

function buildUser(overrides: Partial<User> = {}): User {
  return {
    id: faker.number.int({ min: 1, max: 10000 }),
    name: faker.person.fullName(),
    email: faker.internet.email().toLowerCase(),
    role: faker.helpers.arrayElement(['admin', 'user'] as const),
    avatar: faker.helpers.maybe(() => faker.image.avatar()),
    createdAt: faker.date.past().toISOString(),
    ...overrides,
  }
}

// ---- Product 工厂 ----

interface Product {
  id: number
  title: string
  price: number
  description: string
  category: string
  inStock: boolean
}

function buildProduct(overrides: Partial<Product> = {}): Product {
  return {
    id: faker.number.int({ min: 1, max: 10000 }),
    title: faker.commerce.productName(),
    price: parseFloat(faker.commerce.price()),
    description: faker.commerce.productDescription(),
    category: faker.commerce.department(),
    inStock: faker.datatype.boolean(),
    ...overrides,
  }
}

export { buildUser, buildProduct }
export type { User, Product }
```

### 使用 Overrides 模式

```typescript
import { buildUser, buildProduct } from '../test/factories'

describe('UserProfile', () => {
  it('displays admin badge for admin users', () => {
    // 只覆盖关心的字段，其他字段使用 faker 生成
    const admin = buildUser({ role: 'admin' })

    renderWithProviders(<UserProfile user={admin} />)

    expect(screen.getByText(/admin/i)).toBeInTheDocument()
  })

  it('does not show admin badge for regular users', () => {
    const regularUser = buildUser({ role: 'user' })

    renderWithProviders(<UserProfile user={regularUser} />)

    expect(screen.queryByText(/admin/i)).not.toBeInTheDocument()
  })

  it('shows alt text for users without avatar', () => {
    const userWithoutAvatar = buildUser({ avatar: null })

    renderWithProviders(<UserProfile user={userWithoutAvatar} />)

    const img = screen.getByRole('img')
    expect(img).toHaveAttribute('alt', expect.stringContaining(userWithoutAvatar.name))
  })
})
```

### 工厂函数 + 批量生成

```typescript
describe('ProductList', () => {
  it('renders all products', () => {
    // 生成 10 个产品，每条测试独立
    const products = Array.from({ length: 10 }, () => buildProduct())

    renderWithProviders(<ProductList products={products} />)

    const items = screen.getAllByRole('listitem')
    expect(items).toHaveLength(10)
  })

  it('shows empty state when no products in stock', () => {
    // 所有产品都 out of stock
    const products = Array.from({ length: 3 }, () => buildProduct({ inStock: false }))

    renderWithProviders(<ProductList products={products} />)

    expect(screen.getByText(/no products available/i)).toBeInTheDocument()
  })
})
```

### 数据工厂与 MSW Handler 的整合

```typescript
import { http, HttpResponse } from 'msw'
import { buildUser } from '../test/factories'

// 用工厂函数生成 handler 的响应数据
export const handlers = [
  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json(
      buildUser({ id: Number(params.id) })
    )
  }),

  http.get('/api/users', () => {
    return HttpResponse.json(
      Array.from({ length: 5 }, () => buildUser())
    )
  }),
]
```

> **自我验证说明**：`@faker-js/faker` 的每个数据方法在每次调用时生成不同的随机值，所以 `buildUser()` 每次返回不同的用户数据。这在测试中有两个好处：（1）防止测试间的数据耦合；（2）增加了测试数据的变化性，可能发现意料之外的边界问题。如果需要固定数据，通过 `overrides` 传入即可。`faker` 的 TypeScript 类型全部内置，`faker.person.fullName()` 等方法都有完整的类型签名。

> **渐进式示例：测试数据的三个进化阶段**
>
> 测试数据的组织方式随项目复杂度逐步演进：
>
> **阶段 1：手动硬编码（适合原型阶段，1-2 个字段）**
> ```typescript
> const user = { id: 1, name: 'Alice', email: 'alice@test.com', role: 'admin' as const }
> ```
> 优点：直接、0 依赖；痛点：每个测试重复写、API 字段变更时要改所有测试。
>
> **阶段 2：工厂函数（适合中等规模项目）**
> ```typescript
> function buildUser(overrides: Partial<User> = {}): User {
>   return {
>     id: faker.number.int(),
>     name: faker.person.fullName(),
>     email: faker.internet.email(),
>     role: 'user',
>     ...overrides,
>   }
> }
> ```
> 优点：只写关心的字段，其他自动生成；痛点：每个测试仍然需要手动配置 MSW handler 的响应数据。
>
> **阶段 3：工厂 + MSW 内存数据库（适合大型项目）**
> ```typescript
> // @mswjs/data 定义模型，工厂函数生成数据，MSW handler 从 DB 读取
> db.user.create(buildUser({ role: 'admin' }))
> // 测试直接通过 DB API 插入/查询数据，handler 自动响应
> ```
> 优点：数据创建、API mock、测试验证三者统一，CRUD 测试无需手写 handler；痛点：需要配置 `@mswjs/data` 模型定义。
>
> 建议在项目有 5+ 个 API 端点需要 mock 时升级到阶段 3。在此之前阶段 2 已经够用。

## 12.4 共享 Handler 与 Mock 数据库

### @mswjs/data：内存数据库

对于 CRUD 场景（增删改查），手动维护 mock 数据的状态非常繁琐。`@mswjs/data` 为 MSW handler 提供了一个轻量级的内存数据库。

```bash
npm install -D @mswjs/data
```

```typescript
// src/mocks/db.ts
import { factory, primaryKey, type PersistedModel } from '@mswjs/data'
import { faker } from '@faker-js/faker'

// 定义数据模型（类似 Prisma schema）
export const db = factory({
  user: {
    id: primaryKey(Number),
    name: String,
    email: String,
    role: String,
    createdAt: () => faker.date.past().toISOString(),
  },
  product: {
    id: primaryKey(Number),
    title: String,
    price: Number,
    inStock: Boolean,
  },
})

// 初始化种子数据
function seed() {
  // 每条测试前清空数据库
  db.user.deleteMany({})
  db.product.deleteMany({})

  // 插入种子数据
  Array.from({ length: 3 }, (_, i) =>
    db.user.create({
      id: i + 1,
      name: faker.person.fullName(),
      email: faker.internet.email().toLowerCase(),
      role: i === 0 ? 'admin' : 'user',
    })
  )

  Array.from({ length: 5 }, (_, i) =>
    db.product.create({
      id: i + 1,
      title: faker.commerce.productName(),
      price: parseFloat(faker.commerce.price()),
      inStock: faker.datatype.boolean(),
    })
  )
}

export { seed }
```

### Handler 工厂函数

```typescript
// src/mocks/handlers.ts
import { http, HttpResponse } from 'msw'
import { db, seed } from './db'

// 工厂函数：接受配置返回 handler 数组
export function createHandlers(options: { seed?: boolean } = {}) {
  if (options.seed !== false) {
    seed()
  }

  return [
    // GET /api/users
    http.get('/api/users', () => {
      const users = db.user.getAll()
      return HttpResponse.json(users)
    }),

    // GET /api/users/:id
    http.get('/api/users/:id', ({ params }) => {
      const user = db.user.findFirst({
        where: { id: { equals: Number(params.id) } },
      })

      if (!user) {
        return new HttpResponse(null, { status: 404 })
      }

      return HttpResponse.json(user)
    }),

    // POST /api/users
    http.post('/api/users', async ({ request }) => {
      const body = (await request.json()) as { name: string; email: string }
      const user = db.user.create({
        id: db.user.getAll().length + 1,
        ...body,
        role: 'user',
      })
      return HttpResponse.json(user, { status: 201 })
    }),

    // DELETE /api/users/:id
    http.delete('/api/users/:id', ({ params }) => {
      const deleted = db.user.delete({
        where: { id: { equals: Number(params.id) } },
      })

      if (!deleted) {
        return new HttpResponse(null, { status: 404 })
      }

      return new HttpResponse(null, { status: 204 })
    }),
  ]
}
```

### 在测试中使用

```typescript
// src/mocks/server.ts
import { setupServer } from 'msw/node'
import { createHandlers } from './handlers'

export const server = setupServer(...createHandlers())

// ---- 在测试中操作数据库 ----

import { db, seed } from '../mocks/db'

describe('UserManagement', () => {
  beforeEach(() => {
    seed() // 重置数据库到种子状态
  })

  it('creates and lists users', async () => {
    const user = userEvent.setup()
    renderWithProviders(<UserManagement />)

    // 验证种子数据渲染
    const rows = await screen.findAllByRole('row')
    // 种子数据插入了 3 个 user，加上 header 行 → 4 行
    expect(rows).toHaveLength(4)

    // 创建新用户
    await user.click(screen.getByRole('button', { name: /add user/i }))
    await user.type(screen.getByLabelText(/name/i), 'Charlie')
    await user.type(screen.getByLabelText(/email/i), 'charlie@test.com')
    await user.click(screen.getByRole('button', { name: /save/i }))

    // 验证新用户出现在列表中
    await screen.findByText('charlie@test.com')
    const rowsAfter = screen.getAllByRole('row')
    expect(rowsAfter).toHaveLength(5)
  })

  it('deletes a user directly via DB for setup', async () => {
    // 直接操作数据库删除用户
    db.user.delete({ where: { id: { equals: 1 } } })

    renderWithProviders(<UserManagement />)

    const rows = await screen.findAllByRole('row')
    expect(rows).toHaveLength(3) // header + 2 users（删除了 id=1）
  })
})
```

> **自我验证说明**：`@mswjs/data` 的查询 API 类似于 Prisma：`findFirst({ where: { field: { operator: value } } })`、`delete({ where: {...} })`、`getAll()`。工厂函数 `createHandlers()` 返回的是 MSW handler 数组，可以直接传给 `setupServer`。`seed` 函数在每次 `createHandlers()` 调用时（默认）重新初始化数据，确保测试隔离。

## 12.5 测试文件组织

### Co-location vs Tests 目录

两种主流策略：

| 维度 | Co-location（旁置） | 集中 Tests 目录 |
|------|-------------------|---------------|
| **文件路径** | `src/components/UserList.test.tsx` | `src/__tests__/UserList.test.tsx` |
| **发现速度** | 源文件和测试文件在同一目录，方便关联 | 全部测试在一个目录树，方便概览 |
| **重构友好** | 移动组件时测试跟着走 | 移动组件后需要手动更新测试路径 |
| **导入路径** | 相对路径短：`../api` | 需要跳到 `../../src/components/UserList` |
| **构建排除** | 需要在 Vite/tsconfig 中排除 `*.test.*` | 天然在 `src/` 之外 |

**本教程推荐：Co-location**。理由是：

1. 重构友好：拖拽组件到新目录，测试文件一起移动
2. import 路径短：`import { buildUser } from '../test/factories'` 而非 '../../test/factories'
3. 可见性强：打开组件目录就能看到是否有测试

## 12.6 测试命名规范

### 文件命名

```
# 推荐
UserProfile.test.tsx          # 组件测试
useUser.test.ts               # Hook 测试
format.test.ts                # 纯函数测试
checkout-flow.test.tsx        # 端到端流程测试

# Vitest 也支持
UserProfile.spec.tsx
UserProfile.integration.test.tsx  # 显式标记集成测试
```

### describe 命名

`describe` 描述被测试的单元名称：

```typescript
describe('UserProfile', () => {          // 组件名
describe('useUser', () => {              // Hook 名
describe('formatCurrency', () => {       // 函数名
describe('UserManagement', () => {       // 功能模块
describe('GET /api/users', () => {       // API 端点
```

### it/test 命名

两种主流风格：

```typescript
// 风格 A："should [behavior] when [condition]"
// 优点：统一、清晰、适合非英语母语者
it('should show error when email is invalid', () => { /* ... */ })
it('should navigate to login when unauthenticated', () => { /* ... */ })

// 风格 B：BDD-style，直接描述行为
// 优点：更简洁，不需要 "should" 的噪音
it('displays error for invalid email', () => { /* ... */ })
it('redirects unauthenticated users to /login', () => { /* ... */ })

// 项目内保持一致即可。本教程使用风格 B
```

### 嵌套 describe 的组织原则

```typescript
describe('UserProfile', () => {
  describe('when user is logged in', () => {
    it('displays user name', () => { /* ... */ })
    it('shows logout button', () => { /* ... */ })
  })

  describe('when user is not logged in', () => {
    it('shows login prompt', () => { /* ... */ })
    it('does not show user data', () => { /* ... */ })
  })

  describe('error states', () => {
    it('shows error message on fetch failure', () => { /* ... */ })
    it('shows retry button on network error', () => { /* ... */ })
  })
})
```

> **自我验证说明**：`describe` 嵌套 2-3 层为宜。超过 3 层会降低可读性。`describe` 的字符串应该构造出完整的句子：`describe('UserProfile') + describe('when not logged in') + it('shows login prompt')` → "UserProfile when not logged in shows login prompt"。

## 12.7 反模式

### 反模式 1：在测试中写重复的 Provider 嵌套

```typescript
// ❌ 每个测试文件手动嵌套 Provider
it('test 1', () => {
  render(<Component />, {
    wrapper: ({ children }) => (
      <MemoryRouter>
        <QueryClientProvider client={queryClient}>
          {children}
        </QueryClientProvider>
      </MemoryRouter>
    ),
  })
})

// ✅ 封装为 renderWithProviders
it('test 1', () => {
  renderWithProviders(<Component />)
})
```

### 反模式 2：硬编码测试数据

```typescript
// ❌ 手写完整对象，大部分字段与测试无关
const user = {
  id: 1,
  name: 'Alice',
  email: 'alice@test.com',
  role: 'admin' as const,
  avatar: 'https://example.com/avatar.jpg',
  createdAt: '2025-01-01T00:00:00.000Z',
}

// ✅ 工厂函数 + overrides，只写关心的字段
const user = buildUser({ role: 'admin' })
```

### 反模式 3：测试文件散落在无法预期的位置

```typescript
// ❌ 没有固定规则
src/UserProfile.test.tsx          // 在 src 根目录
src/components/test/UserList.test.tsx  // 在 test 子目录
src/UserProfile/__tests__/UserProfile.test.tsx  // 在 __tests__

// ✅ 统一 co-location
src/components/UserProfile/UserProfile.test.tsx
src/components/UserList/UserList.test.tsx
```

### 反模式 4：it 描述不清晰

```typescript
// ❌
it('works', () => { /* ... */ })
it('test 2', () => { /* ... */ })

// ✅
it('returns users sorted by name in ascending order', () => { /* ... */ })
it('returns empty array when no users match the filter', () => { /* ... */ })
```

### 反模式 5：在多个测试间共享同一个 MSW handler 实例的副作用

```typescript
// ❌ 测试间共享 handler 状态
const handler = http.get('/api/users', () => {
  return HttpResponse.json(allUsers)  // allUsers 可能被测试 A 修改了
})

// ✅ 每个测试独立初始化
beforeEach(() => {
  seed()
})
```

## 12.8 本章练习

1. **封装 renderWithProviders**
   查看当前项目的组件测试，识别重复的 Provider 嵌套模式。封装 `renderWithProviders`，包含至少 Router + QueryClient 两个 Provider。

2. **构建数据工厂**
   选择项目中的一个数据模型，用 `@faker-js/faker` 创建工厂函数：
   - 所有字段使用 faker 生成
   - 支持 `Partial<Model>` overrides
   - 工厂函数有完整的 TypeScript 类型

3. **使用 @mswjs/data**
   - 用 `@mswjs/data` 定义与步骤 2 相同的数据模型
   - 创建 handler 工厂函数，基于内存数据库提供 CRUD
   - 编写一个测试，直接操作数据库插入/删除数据后验证组件渲染

4. **重构测试命名**
   找到项目中命名不佳的测试（如 `it('test 1')`、`it('works')`），全部改为描述性命名。

## 12.9 本章总结

- 测试可维护性的三条原则：可读性 > 简洁性、重复有代价抽象也有代价、测试是活文档
- AAA 三段式布局（空行分隔）是测试排版的标准
- `renderWithProviders` 封装 Provider 嵌套，返回类型与 `render` 一致
- `@faker-js/faker` + overrides 模式：用工厂函数生成真实感测试数据
- `@mswjs/data` 提供内存数据库，handler 工厂函数支持跨环境复用
- Co-location（文件旁置 `.test.tsx`）优于集中 tests 目录
- `describe` 命名被测试单元，`it` 描述行为，嵌套 2-3 层为宜
- 每次重构测试代码时，先考虑可读性，再考虑 DRY
