# 第1章：测试架构思维

## 学习目标

- 理解自动化测试的本质和不同层级的价值
- 掌握从测试金字塔到测试奖杯（Testing Trophy）的思维转变
- 内化 React Testing Library 的核心哲学
- 能独立论证 Vitest + MSW + RTL 技术栈的选型理由

---

## 1.1 为什么要写测试

### 测试的本质

测试是一段代码的"行为契约"。你写下一个断言，就是在声明：**给定这些条件，这段代码应该产生那个结果**。这个契约有三个价值：

1. **保护已有行为**：任何破坏契约的改动都会被立即发现
2. **作为活文档**：测试比 README 更可信——README 可能过期，但失败的测试不会说谎
3. **降低重构恐惧**：有了安全网，你才敢大胆地改善代码结构

### 不写测试的真实成本

没有测试的项目并非"省了时间"。不写测试的代价是递延的：

- **回归恐惧**：每次改代码都要手动验证所有可能受影响的功能 → 实际结果是根本不验证
- **重构停滞**：代码质量在没有安全网的情况下持续恶化，形成"代码烂 → 不敢改 → 更烂"的恶性循环
- **知识流失**：新人不敢改旧代码，老人离职带走隐性知识，没有人知道某段逻辑为什么写成这样

### 覆盖率不等于质量

```typescript
// 高覆盖率低质量的典型例子
function add(a: number, b: number): number {
  return a + b
}

it('should be covered', () => {
  expect(add(1, 2)).toBe(3) // 100% 行覆盖率，但几乎没有价值
})
```

覆盖率告诉你**哪些代码被执行了**，但不告诉你**行为是否正确**。100% 覆盖率的代码仍然可以充满 bug。把覆盖率当作下限安全网，而非质量目标。

## 1.2 从测试金字塔到测试奖杯

### 经典测试金字塔

```
       ┌─────┐
       │ E2E │  少量、慢、脆弱
      ┌┴─────┴┐
      │ 集成测试 │  中等数量
     ┌┴─────────┴┐
     │  单元测试   │  大量、快、稳定
    └──────────────┘
```

金字塔模型的核心主张：**底层多，顶层少**。但这个模型针对的是后端服务。在前端，纯单元测试（测试一个函数）往往价值不高——真正复杂的是组件如何协同工作、数据如何流转、用户如何交互。

### Testing Trophy（测试奖杯）

Kent C. Dodds 在 2018 年提出了 Testing Trophy 模型，重新定义了前端测试的投资分配：

```
         ┌─────────────┐
         │  E2E (少许)   │  Playwright / Cypress
        ┌┴─────────────┴┐
        │  集成测试 (多数) │  Vitest + RTL + MSW
       ┌┴───────────────┴┐
       │  单元测试 (适量)   │  Vitest (纯函数/工具)
      ┌┴─────────────────┴┐
      │  静态分析 (全覆盖)   │  TypeScript + ESLint
     └──────────────────────┘
```

关键变化：

- **静态分析成为底座**：TypeScript 的类型检查是最廉价、最快速的"测试"。类型错了，编译不过，连测试都不用跑
- **集成测试成为重心**：组件 + 数据流 + 用户交互 的组合才是前端 bug 的高发区。渲染一个组件，mock 网络请求，模拟用户操作，断言渲染结果——这是投资回报率最高的测试
- **E2E 只覆盖关键路径**：用 E2E 验证"用户能完成核心任务"（登录 → 下单 → 支付），而非穷举所有边界条件

### 为什么集成测试的 ROI 最高

```typescript
// 单元测试：测试一个纯函数
it('formats currency correctly', () => {
  expect(formatCurrency(1234.5)).toBe('$1,234.50')
})
// 价值：确保格式化逻辑正确 ✓
// 缺失：不知道这个函数在实际组件中是否被正确调用

// 集成测试：测试组件 + 数据 + 交互
it('displays product price formatted correctly', async () => {
  render(<ProductCard productId={1} />)
  const price = await screen.findByText('$1,234.50')
  expect(price).toBeInTheDocument()
})
// 价值：确保格式化逻辑在组件中被正确使用 ✓
// 价值：确保数据获取后组件正确渲染 ✓
// 价值：确保用户最终看到了正确的价格 ✓
```

集成测试的一个断言覆盖了：网络请求 → 数据解析 → 状态更新 → 组件渲染 → 格式化 → DOM 输出。而纯单元测试只覆盖了格式化这一个环节。

> **Jest 对比：** 在 Jest 中实践 Testing Trophy 模型时，你需要额外配置 TypeScript（通过 ts-jest 或 @babel/preset-typescript）和 ES Modules（需要 --experimental-vm-modules 标志），而且 Jest 的配置与构建配置（Webpack/Vite）完全独立，alias 和 plugin 必须在两边同步。Vitest 通过继承 Vite 的统一配置管道将配置成本降到几乎为零——TypeScript、JSX、CSS Modules、路径别名全部从 `vite.config.ts` 自动继承，无需重复配置。

> **为什么：** 集成测试的 ROI 高于单元测试，不是因为单元测试"没有价值"，而是因为前端应用中真正的复杂性不在独立的纯函数中，而在组件之间的协作中。一个"提交"按钮是否正常工作，取决于它在正确的时机被注入了正确的回调、接收了正确的数据、并在网络请求完成后更新了状态——这些跨越组件边界和层级的问题只有集成测试能可靠捕获。单元测试验证的是"每个零件合格"，集成测试验证的是"总装完成后的整机工作正常"——而用户购买并使用的是整机。

## 1.3 React Testing Library 哲学

### 核心原则

> 你的测试越接近用户使用软件的方式，就越能给你信心。

这个原则有三层含义：

1. **用户不关心你的 state 叫什么**：他们只关心按钮是否可点击、文本是否可见
2. **用户不关心你的 props 怎么传**：他们只关心中间步骤产生正确结果了吗
3. **用户不关心你的组件怎么拆分的**：他们只关心整个流程走通了没有

### 测试行为，而非实现细节

```typescript
// ❌ 测试实现细节：断言内部 state
const { result } = renderHook(() => useCounter())
act(() => { result.current.increment() })
expect(result.current.count).toBe(1)
// 问题：如果 count 改名为 value，测试就断了；但用户行为完全没变

// ✅ 测试行为：断言用户看到的东西
render(<Counter />)
await user.click(screen.getByRole('button', { name: '+' }))
expect(screen.getByRole('status')).toHaveTextContent('1')
// 无论内部 state 叫什么、实现怎么改，用户行为不变 → 测试不变
```

### 实现细节清单

以下内容**不应该在测试中直接断言**（即它们是实现细节）：

| 实现细节 | 为什么不该测 | 应该测什么 |
|----------|------------|----------|
| 组件内部 state (`useState`) | 重构 state 结构不应该破坏测试 | 用户看到的渲染结果 |
| props 传递 | 测试应该验证最终渲染效果 | 子组件最终渲染的内容 |
| 组件名称/实例 | Enzyme 的 `instance()` / `name()` | 不需要知道组件的名字 |
| 生命周期方法调用 | `componentDidMount` 被调了几次 | 渲染结果和副作用的效果 |
| 内部方法调用 | `handleClick` 被调了吗 | 点击后发生了什么变化 |
| CSS class 名 | CSS Modules 的 hash 会变 | 元素是否可见、有无某属性 |

### 反模式：container.querySelector

```typescript
// ❌ 最常用的反模式
const { container } = render(<LoginForm />)
const submitBtn = container.querySelector('.submit-btn')
expect(submitBtn).toBeDisabled()

// ✅ 用户查找按钮的方式
render(<LoginForm />)
const submitBtn = screen.getByRole('button', { name: /submit/i })
expect(submitBtn).toBeDisabled()
```

用户从来不在脑子里跑 `document.querySelector('.submit-btn')` 来找按钮。他们看到文字说"提交"，就点那个按钮。你的测试也应该这么做。

> **Jest 对比：** React Testing Library 在 Jest 和 Vitest 中的查询 API 完全相同——RTL 本身不依赖测试运行器。但在 TDD 循环中，性能差异会被放大：Jest 的 watch 模式在 500+ 测试文件的项目中越来越慢，因为每次文件变更都需要重新执行整个 transform 管道；Vitest 的 watch 基于 Vite 的 HMR，只重新编译变更模块及其依赖，响应速度比 Jest 快 5-10 倍。"改代码 → 等测试 → 再改"的循环时间直接决定了你的心流状态能否保持。

## 1.4 技术栈选型：Vitest + MSW + RTL

### Vitest vs Jest：为什么换

| 维度 | Vitest | Jest |
|------|--------|------|
| **启动速度** | 原生 ESM，Vite 驱动的 transform | 需要 Jest 自己的 transform 管道 |
| **配置成本** | 零配置起步，复用 Vite 配置 | 独立配置，需要额外的 transform 设置 |
| **HMR 测试** | 支持 watch mode 下的 HMR | watch mode 基于文件变更 |
| **TypeScript** | 原生支持，通过 Vite | 需要 `ts-jest` 或 `@babel/preset-typescript` |
| **API 兼容** | 与 Jest 高度兼容（`vi` 替换 `jest`） | 事实标准 |
| **ESM 支持** | 一流支持 | 实验性支持，配置复杂 |
| **性能** | 快 2-10x（取决于项目大小） | 稳定但较慢 |

选择 Vitest 的核心原因：**如果你已经用 Vite 构建项目，Vitest 几乎零成本启动**。配置文件就是 `vite.config.ts` 的扩展，transform 和 alias 设置天然一致。Jest 需要维护一套平行的构建配置。

### MSW vs vi.mock：拦截层级的选择

这是本教程最重要的架构决策之一：

```typescript
// 方案 A：模块级 mock（vi.mock）
vi.mock('./api', () => ({
  fetchUser: vi.fn().mockResolvedValue({ id: 1, name: 'Alice' }),
}))

// 方案 B：网络层拦截（MSW）
server.use(
  http.get('/api/users/:id', () =>
    HttpResponse.json({ id: 1, name: 'Alice' })
  )
)
```

方案 A 的问题：

- **与实现耦合**：如果你把 `./api` 里的 `fetchUser` 改名为 `getUser`，mock 也要改
- **不测试真实请求流程**：`fetchUser` 内部的数据转换、错误处理、`fetch` 调用都被跳过了
- **切换 HTTP 库测试全灭**：从 `fetch` 换成 `axios`，所有 mock 都要重写

方案 B 的优势：

- **协议层 mock**：不管你的组件用 `fetch`、`axios`、`ky` 还是 `react-query`，只要最终发 HTTP 请求，MSW 就能拦截
- **测试完整的请求→响应流程**：从组件调用 API 函数 → fetch 发出 → MSW 拦截 → 返回响应 → 数据解析 → 状态更新 → 渲染，全链路覆盖
- **handler 可复用**：测试和浏览器开发环境可以共享同一套 MSW handler

### RTL vs Enzyme：哲学的根本对立

| 维度 | React Testing Library | Enzyme |
|------|----------------------|--------|
| **查询方式** | 按角色/标签/文本（用户视角） | 按 component name / CSS selector / props |
| **测试内容** | 渲染结果（用户看到的） | 内部 state、props、生命周期 |
| **与 React 版本** | 紧跟 React 更新 | 适配滞后 |
| **shallow render** | 不支持（刻意为之） | 核心特性 |
| **设计哲学** | 测试行为 | 测试实现 |

RTL 不支持 shallow render 是**刻意设计，不是功能缺失**。shallow render 测试的是组件树结构——这是实现细节。你的用户不知道也不关心组件树长什么样。

### TypeScript 的角色

TypeScript 不是你测试工具链的一部分——它**就是你的第一条防线**。在你写任何测试之前，类型系统已经在工作了：

```typescript
// TypeScript 已经捕获了这些错误，不需要测试：
function getUserName(user: User): string {
  return user.namme // Property 'namme' does not exist on type 'User'. Did you mean 'name'?
}

// 你的测试只需要验证类型系统无法证明的东西：
it('formats user name correctly', () => {
  expect(getUserName({ name: ' Alice ' })).toBe('Alice') // trim 逻辑
})
```

> **为什么：** TypeScript 能在你写任何测试之前就捕获整类错误，是因为它工作在编译阶段而非运行时。测试需要你手动构造输入然后断言输出——如果你漏写了某个边界条件的测试，对应的 bug 就不会被发现。但 TypeScript 静态分析整个代码库，不依赖你写了多少测试。只要通过类型检查，类型错配、属性名拼写错误、空值访问——这些"不可能的错误"就被排除了。测试应专注于验证类型系统无法证明的行为逻辑（如 trim、边界条件、UI 渲染），而非重复类型系统已保证的事。

### 自我验证说明

本节为概念性内容，不涉及可执行的代码。所有框架对比基于官方文档和社区实践。观点性内容（"为什么选 Vitest"）基于可验证的技术差异（原生 ESM vs transform 管道），而非主观偏好。

### 渐进式示例：测试奖杯在实践中的体现

下面的示例展示同一个功能以三种不同的测试粒度实现，对应 Testing Trophy 的三个层级：

```typescript
// 功能：显示用户订单总价
// 假设存在 calcTotal(items) 纯函数、OrderSummary 组件和 OrderPage 页面组件

// 级别 1：纯函数单元测试（Testing Trophy 的"适量"层）
// 验证：计算逻辑本身是否正确
describe('Level 1: calcTotal 纯函数', () => {
  it('sums item prices correctly', () => {
    const items = [
      { price: 10, quantity: 2 },
      { price: 15, quantity: 1 },
    ]
    expect(calcTotal(items)).toBe(35)
  })

  it('handles empty items array', () => {
    expect(calcTotal([])).toBe(0)
  })

  it('handles decimal prices', () => {
    const items = [{ price: 9.99, quantity: 3 }]
    expect(calcTotal(items)).toBeCloseTo(29.97)
  })
})

// 级别 2：组件集成测试（Testing Trophy 的"多数"层）
// 验证：组件正确接收 props 并格式化显示
describe('Level 2: OrderSummary 组件', () => {
  it('displays formatted total from props', () => {
    render(<OrderSummary items={sampleItems} />)
    expect(screen.getByText('$35.00')).toBeInTheDocument()
  })

  it('displays empty state when no items', () => {
    render(<OrderSummary items={[]} />)
    expect(screen.getByText(/no items/i)).toBeInTheDocument()
  })
})

// 级别 3：完整集成测试（Testing Trophy 的"多数"层——最高 ROI）
// 验证：网络请求 → 数据解析 → 状态更新 → 组件渲染 全链路
describe('Level 3: OrderPage 完整流程', () => {
  it('loads and displays order total', async () => {
    render(<OrderPage />)
    expect(screen.getByText(/loading/i)).toBeInTheDocument()
    expect(await screen.findByText('$35.00')).toBeInTheDocument()
  })

  it('shows error when fetch fails', async () => {
    server.use(
      http.get('/api/orders/1', () => HttpResponse.error())
    )
    render(<OrderPage />)
    expect(await screen.findByRole('alert')).toHaveTextContent(/failed/i)
  })
})
```

> **自我验证说明：** 三个级别都验证了"订单总价"这一功能，但测试的目标不同。级别 1 验证纯计算逻辑（单元测试的最佳对象），级别 3 验证从网络请求到最终渲染的完整用户路径——这才是前端 bug 的真正高发区。如果时间有限，优先级为：级别 3 > 级别 2 > 级别 1。本示例中的 MSW server 生命周期管理参考[第6章](06-MSW哲学与实践.md)。

## 1.5 反模式：新手的常见误区

### 误区 1：测试一切

```typescript
// ❌ 测试 React 本身的行为
it('renders a div', () => {
  render(<MyComponent />)
  expect(screen.getByRole('button')).toBeInTheDocument() // 这不是测试你的代码，是测试 React
})
```

测试框架能正常工作的东西（"React 能渲染吗"）没有价值。测试你**自己写的逻辑**。

### 误区 2：mock 一切外部依赖

```typescript
// ❌ mock 了 React Router、mock 了 API、mock 了 localStorage……
// 结果：你的测试只验证了 mock 的行为，不是真实代码的行为
```

Mock 越少，测试越有价值。MSW 是唯一的 Mock 层——其他依赖应该用真实的。

### 误区 3：一个测试验证了太多东西

```typescript
// ❌ "上帝测试"：一次验证加载→成功→错误→空状态→重试五个流程
// 结果：任何一个失败你都不知道到底什么坏了
```

一个测试只验证一个行为路径。

## 1.6 本章练习

1. 打开你当前的项目（或任意一个 React 项目），统计：
   - 有多少单元测试？
   - 有多少集成测试？
   - 有多少 E2E 测试？
   - 你的测试分布符合 Testing Trophy 模型吗？

2. 找一个现有的测试，判断它测试的是"行为"还是"实现细节"。如果是后者，试着用 RTL 哲学重写它的查询方式。

3. 思考题：如果你的项目从 Jest 切换到 Vitest，哪些地方可能会出问题？列出你最担心的 3 个点——我们会在[第14章](14-Jest迁移指南.md)回答这些问题。

## 1.7 本章总结

- 测试本质是行为契约，不是覆盖率竞赛
- Testing Trophy：静态分析为底座，集成测试为重心，E2E 为关键路径验证
- RTL 哲学：测试行为，不测试实现细节。用 `getByRole`，不用 `querySelector`
- Vitest + MSW 的组合在速度和架构正确性上都优于传统方案
- TypeScript 是你的第一条防线——在写测试之前，让类型系统先工作
