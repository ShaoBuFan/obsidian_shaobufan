## 15.1 WebSocket 测试

### MSW v2 的 WebSocket 拦截

MSW v2 提供了 `ws.link()` 用于在测试中拦截 WebSocket 连接。这是测试实时通信功能的标准方式——不需要修改应用代码，MSW 在网络层拦截连接并模拟服务端行为。

```typescript
// src/mocks/handlers/websocket.ts
import { ws, HttpResponse } from 'msw'

// 创建一个 WebSocket 链接处理器
const chatWs = ws.link('wss://api.example.com/chat')

export const wsHandlers = [
  chatWs.addEventListener('connection', ({ client }) => {
    // 客户端连接成功，发送欢迎消息
    client.send(JSON.stringify({ type: 'welcome', payload: { userId: client.id } }))

    // 监听客户端发来的消息
    client.addEventListener('message', (event) => {
      const data = JSON.parse(event.data)

      switch (data.type) {
        case 'send_message': {
          // 模拟服务端回声
          client.send(JSON.stringify({
            type: 'message_broadcast',
            payload: {
              id: Date.now().toString(),
              user: data.payload.user,
              text: data.payload.text,
              timestamp: new Date().toISOString(),
            },
          }))
          break
        }
        case 'ping': {
          client.send(JSON.stringify({ type: 'pong' }))
          break
        }
      }
    })

    // 处理断开连接
    client.addEventListener('close', () => {
      console.log(`Client ${client.id} disconnected`)
    })
  }),
]
```

### 断线重连测试

```typescript
describe('reconnection', () => {
  it('reconnects after connection loss', async () => {
    const { result } = renderHook(() => useChat())

    // 等待初始连接
    await waitFor(() => {

    await waitFor(() => {

      result.current.disconnect()

    act(() => {
      result.current.sendMessage({ user: 'Alice', text: 'offline msg' })

    // 重连后消息应该被发送
    await waitFor(() => {
      // 假设服务端会广播重连后的消息
      expect(result.current.messages).toContainEqual(

      )
    })
```

### 假定时器 + WebSocket 事件序列

```typescript
describe('WebSocket with fake timers', () => {

      expect(result.current.connected).toBe(true)
    })
  })
})
```

> **自我验证说明**：MSW v2 的 `ws.link()` 在测试中拦截 `WebSocket` 构造函数。`addEventListener('connection')` 模拟服务端行为，`client.send()` 从服务端向客户端推送消息，`client.addEventListener('message')` 接收客户端消息。参考 [MSW WebSocket 文档](https://mswjs.io/docs/recipes/websocket)。

> **Jest 对比：WebSocket 测试**
>
> 在 Jest 中测试 WebSocket 同样需要 mock，但方式不同。Jest 没有 MSW 的 `ws.link()` 等价物，通常需要手动 mock `WebSocket` 构造函数：
>
> ```typescript
> // Jest — 手动 mock WebSocket
> class MockWebSocket {
>   readyState: number = WebSocket.OPEN
>   onopen: ((event: Event) => void) | null = null
>   onmessage: ((event: MessageEvent) => void) | null = null
>   onclose: ((event: CloseEvent) => void) | null = null
>   send: jest.Mock = jest.fn()
>   close: jest.Mock = jest.fn()
> }
>
> beforeAll(() => {
>   global.WebSocket = MockWebSocket as any
> })
>
> afterAll(() => {
>   delete (global as any).WebSocket
> })
> ```
>
> MSW v2 的 `ws.link()` 方案的两个核心优势：
> 1. **不修改原生 API**——不需要替换 `global.WebSocket`，MSW 在网络层拦截
> 2. **服务端行为真实模拟**——`client.send()` 模拟服务端推送，不是简单的方法调用
>
> 手动 mock 方案最大的问题是：你 mock 的是"你认为 WebSocket 应该怎么工作"，MSW mock 的是"WebSocket 实际上怎么工作"。后者更接近真实。迁移到 Vitest 时不会改变这一点——两者都使用 MSW，差异在于 Vitest 环境下 `msw/node` 的配置更简单。

## 15.2 Server-Sent Events (SSE) 测试

SSE 测试的核心挑战是模拟 `EventSource` —— 浏览器原生的 SSE API。由于 jsdom 不提供 `EventSource` 实现，我们需要 mock 它。

### EventSource Mock

```typescript
// src/test/mocks/eventSource.ts

type EventSourceCallback = (event: MessageEvent) => void

interface MockEventSourceInstance {
  close: ReturnType<typeof vi.fn>
  simulateMessage: (data: string) => void
  simulateError: () => void
  connect: () => void
}

// 创建可控制的 Mock EventSource
export function createMockEventSource(): {
  EventSourceMock: typeof EventSource
  instances: MockEventSourceInstance[]
} {
  const instances: MockEventSourceInstance[] = []

  class EventSourceMock {
    static CONNECTING = 0
    static OPEN = 1
    static CLOSED = 2

    readyState: number = EventSourceMock.CONNECTING
    onopen: ((event: Event) => void) | null = null
    onmessage: ((event: MessageEvent) => void) | null = null
    onerror: ((event: Event) => void) | null = null
    url: string
    withCredentials: boolean = false

    private listeners: Map<string, Set<EventSourceCallback>> = new Map()
    private _close = vi.fn()

    constructor(url: string) {
      this.url = url

      // 模拟异步连接
      setTimeout(() => {
        this.readyState = EventSourceMock.OPEN
        this.onopen?.(new Event('open'))
      }, 0)
    }

    addEventListener(type: string, callback: EventSourceCallback): void {
      if (!this.listeners.has(type)) {
        this.listeners.set(type, new Set())
      }
      this.listeners.get(type)!.add(callback)
    }

    removeEventListener(type: string, callback: EventSourceCallback): void {
      this.listeners.get(type)?.delete(callback)
    }

    close(): void {
      this._close()
      this.readyState = EventSourceMock.CLOSED
    }

    // 模拟方法（测试使用）
    simulateMessage(data: string): void {
      const event = new MessageEvent('message', { data })
      this.onmessage?.(event)
      this.listeners.get('message')?.forEach((cb) => cb(event))
    }

    simulateError(): void {
      this.onerror?.(new Event('error'))
      this.listeners.get('error')?.forEach((cb) => cb(new Event('error')))
    }

    get closeMock() {
      return this._close
    }
  }

  return { EventSourceMock, instances }
}
```

### SSE 测试示例

```typescript
// src/hooks/useNotifications.test.ts
import { renderHook, waitFor } from '@testing-library/react'

describe('useNotifications SSE', () => {
  let eventSourceMock: ReturnType<typeof createMockEventSource>
  let mockInstance: MockEventSourceInstance

  beforeEach(() => {
    eventSourceMock = createMockEventSource()
    vi.stubGlobal('EventSource', eventSourceMock.EventSourceMock)
  })

  afterEach(() => {
    vi.unstubGlobal('EventSource')
  })

  it('receives streaming notifications', async () => {
    const { result } = renderHook(() => useNotifications('/api/notifications/stream'))

    // 初始状态：未连接
    expect(result.current.status).toBe('connecting')

    // 等待连接建立
    await waitFor(() => {
      expect(result.current.status).toBe('connected')
    })

    // 模拟服务端推送消息
    const instance = eventSourceMock.instances[0]
    instance.simulateMessage(JSON.stringify({ type: 'order_update', orderId: 123 }))

    await waitFor(() => {
      expect(result.current.notifications).toHaveLength(1)
      expect(result.current.notifications[0]).toEqual({
        type: 'order_update',
        orderId: 123,
      })
    })

    // 第二条消息
    instance.simulateMessage(JSON.stringify({ type: 'system_alert', level: 'info' }))

    await waitFor(() => {
      expect(result.current.notifications).toHaveLength(2)
    })
  })

  it('handles connection error and reconnects', async () => {
    const onReconnect = vi.fn()
    const { result } = renderHook(() =>
      useNotifications('/api/notifications/stream', { onReconnect })
    )

    await waitFor(() => {
      expect(result.current.status).toBe('connected')
    })

    // 模拟错误
    eventSourceMock.instances[0].simulateError()

    await waitFor(() => {
      expect(onReconnect).toHaveBeenCalled()
    })
  })

  it('closes connection on unmount', () => {
    const { unmount } = renderHook(() => useNotifications('/api/notifications/stream'))

    const instance = eventSourceMock.instances[0]
    unmount()

    expect(instance.closeMock).toHaveBeenCalled()
  })
})
```

> **自我验证说明**：`EventSource` 是浏览器 API，jsdom 不提供原生实现。常见的 SSE 库（如 `@microsoft/fetch-event-source`）基于 `fetch`，可以用 MSW 拦截。原生 `EventSource` 必须手动 mock。`vi.stubGlobal` 是 Vitest 的全局变量注入方法，在 `afterEach` 中用 `vi.unstubGlobal` 恢复。

## 15.3 React 18 Concurrent Features

### Suspense 边界测试

React 18 的 Suspense 允许组件在数据加载时"暂停"渲染，显示 fallback UI。

```typescript
// 使用 Suspense 的组件
function Dashboard() {
  return (
    <Suspense fallback={<div>Loading dashboard...</div>}>
      <DashboardContent />
    </Suspense>
  )
}

function DashboardContent() {
  const data = use(fetchDashboardData()) // 使用 React 19 use() 或 Suspense 数据获取
  return <div>{data.title}</div>
}
```

Suspense 测试的关键：验证 fallback → 内容 的过渡：

```typescript
describe('Suspense loading', () => {
  it('shows fallback while data loads', async () => {
    render(<Dashboard />)

    // Suspense fallback 应该立即显示
    expect(screen.getByText(/loading dashboard/i)).toBeInTheDocument()

    // 数据加载后，内容出现
    const title = await screen.findByText('Dashboard')
    expect(title).toBeInTheDocument()
  })

  it('handles Suspense with multiple boundaries', async () => {
    render(
      <Suspense fallback={<div>Loading app...</div>}>
        <Header />
        <Suspense fallback={<div>Loading profile...</div>}>
          <UserProfile />
        </Suspense>
        <Suspense fallback={<div>Loading feed...</div>}>
          <Feed />
        </Suspense>
      </Suspense>
    )

    // 外层 fallback
    expect(screen.getByText(/loading app/i)).toBeInTheDocument()

    // 所有 Suspense 边界都解决后，外层 fallback 消失
    await waitForElementToBeRemoved(() => screen.queryByText(/loading app/i))
  })
})
```

### startTransition 测试

`startTransition` 将状态更新标记为非紧急，允许 React 延迟更新 UI 以保持响应性：

```typescript
function SearchPage() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState<string[]>([])
  const [isPending, startTransition] = useTransition()

  function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
    const value = e.target.value
    setQuery(value)

    // 标记为 transition（非紧急更新）
    startTransition(() => {
      setResults(search(value))
    })
  }

  return (
    <div>
      <input aria-label="Search" value={query} onChange={handleChange} />
      {isPending && <div>Updating...</div>}
      <ul>
        {results.map((r) => <li key={r}>{r}</li>)}
      </ul>
    </div>
  )
}
```

```typescript
describe('startTransition', () => {
  it('shows pending state during transition', async () => {
    const user = userEvent.setup()
    render(<SearchPage />)

    // 初始状态：无 pending 指示器
    expect(screen.queryByText(/updating/i)).not.toBeInTheDocument()

    // 输入触发 transition
    const input = screen.getByRole('textbox', { name: /search/i })
    await user.type(input, 'hello')

    // transition 可能显示 pending
    // 注意：transition 在测试环境中可能立即完成
    // 推荐做法：验证最终结果而非 pending 状态
    await vi.waitFor(() => {
      expect(screen.getAllByRole('listitem').length).toBeGreaterThan(0)
    })
  })
})
```

> **自我验证说明**：在测试中，`startTransition` 的 pending 状态可能不会出现（因为测试环境中的状态更新通常是同步的）。更可靠的测试策略是：直接断言 transition 完成后的结果，而非 pending UI 本身。`useTransition` 的行为在 React 18 的测试环境中是确定的——`startTransition` 的回调会同步执行。

### useDeferredValue 测试

`useDeferredValue` 允许延迟更新 UI 的某一部分，让紧急更新先渲染：

```typescript
function DeferredList() {

      <SlowList text={deferredText} />

}
```

## 15.4 错误边界测试

### 测试错误捕获

Error Boundary 是一个类组件，使用 `componentDidCatch` 或 `static getDerivedStateFromError`：

```typescript
// ErrorBoundary.tsx
class ErrorBoundary extends React.Component<
  { children: React.ReactNode; fallback: React.ReactNode; onError?: (error: Error) => void },
  { hasError: boolean }
> {
  constructor(props: { children: React.ReactNode; fallback: React.ReactNode; onError?: (error: Error) => void }) {
    super(props)
    this.state = { hasError: false }
  }

  static getDerivedStateFromError() {
    return { hasError: true }
  }

  componentDidCatch(error: Error) {
    this.props.onError?.(error)
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback
    }
    return this.props.children
  }
}
```

```typescript
// ErrorBoundary.test.tsx
import { render, screen } from '@testing-library/react'

describe('ErrorBoundary', () => {
  // 子组件：故意抛错的组件
  function BrokenComponent({ shouldThrow }: { shouldThrow: boolean }) {
    if (shouldThrow) {
      throw new Error('Broken!')
    }
    return <div>Working</div>
  }

  it('renders fallback when child throws', () => {
    // 避免控制台被错误日志刷屏
    const consoleSpy = vi.spyOn(console, 'error').mockImplementation(() => {})

    render(
      <ErrorBoundary fallback={<div>Something went wrong</div>}>
        <BrokenComponent shouldThrow={true} />
      </ErrorBoundary>
    )

    expect(screen.getByText(/something went wrong/i)).toBeInTheDocument()
    expect(screen.queryByText('Working')).not.toBeInTheDocument()

    consoleSpy.mockRestore()
  })

  it('calls onError callback when child throws', () => {
    const onError = vi.fn()
    const consoleSpy = vi.spyOn(console, 'error').mockImplementation(() => {})

    render(
      <ErrorBoundary fallback={<div>Error</div>} onError={onError}>
        <BrokenComponent shouldThrow={true} />
      </ErrorBoundary>
    )

    expect(onError).toHaveBeenCalledWith(expect.any(Error))
    expect(onError.mock.calls[0][0].message).toBe('Broken!')

    consoleSpy.mockRestore()
  })

  it('renders children normally when no error', () => {
    render(
      <ErrorBoundary fallback={<div>Error</div>}>
        <BrokenComponent shouldThrow={false} />
      </ErrorBoundary>
    )

    expect(screen.getByText('Working')).toBeInTheDocument()
    expect(screen.queryByText(/error/i)).not.toBeInTheDocument()
  })
})
```

### 错误恢复测试

```typescript
// ErrorBoundaryWithReset.tsx
class ErrorBoundaryWithReset extends React.Component<
  {
    children: React.ReactNode
    fallback: (props: { resetError: () => void }) => React.ReactNode
    onError?: (error: Error) => void
  },
  { hasError: boolean }
> {
  constructor(props: any) {
    super(props)
    this.state = { hasError: false }
  }

  static getDerivedStateFromError() {
    return { hasError: true }
  }

  componentDidCatch(error: Error) {
    this.props.onError?.(error)
  }

  handleReset = () => {
    this.setState({ hasError: false })
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback({ resetError: this.handleReset })
    }
    return this.props.children
  }
}
```

```typescript
describe('error recovery', () => {
```

### 错误上报 callback 验证

```typescript
describe('error reporting', () => {
  it('reports error to monitoring service', async () => {
    const reportError = vi.fn()
    const consoleSpy = vi.spyOn(console, 'error').mockImplementation(() => {})

    render(
      <ErrorBoundary
        fallback={<div>Error</div>}
        onError={(error) => reportError(error.message)}
      >
        <BrokenComponent shouldThrow={true} />
      </ErrorBoundary>
    )

    expect(reportError).toHaveBeenCalledWith('Broken!')

    consoleSpy.mockRestore()
  })

  it('includes component stack in error report', () => {
    const reportError = vi.fn()
    const consoleSpy = vi.spyOn(console, 'error').mockImplementation(() => {})

    render(
      <ErrorBoundary fallback={<div>Error</div>} onError={reportError}>
        <BrokenComponent shouldThrow={true} />
      </ErrorBoundary>
    )

    // React 的 componentDidCatch 接收 error 和 info 两个参数
    expect(reportError).toHaveBeenCalledWith(
      expect.any(Error),
      expect.objectContaining({
        componentStack: expect.any(String),
      })
    )

    consoleSpy.mockRestore()
  })
})
```

> **自我验证说明**：测试 Error Boundary 时，必须用 `vi.spyOn(console, 'error').mockImplementation(() => {})` 抑制 React 的报错日志——React 在开发模式下会 console.error 所有 catch 到的错误，这些日志会污染测试输出。`getDerivedStateFromError` 在渲染阶段被调用，因此不需要 `act()` 包裹——它发生在 render 过程中。

> **Jest 对比：Error Boundary 测试**
>
> Jest 中 Error Boundary 的测试方式几乎相同——差异主要在 mock 函数和类型的写法上：
>
> ```typescript
> // Jest
> const onError = jest.fn()
> const consoleSpy = jest.spyOn(console, 'error').mockImplementation(() => {})
>
> render(
>   <ErrorBoundary fallback={<div>Error</div>} onError={onError}>
>     <BrokenComponent shouldThrow={true} />
>   </ErrorBoundary>
> )
>
> expect(onError).toHaveBeenCalled()
> expect(screen.getByText(/error/i)).toBeInTheDocument()
> consoleSpy.mockRestore()
>
> // Vitest（仅替换 jest → vi，其余完全一致）
> const onError = vi.fn()
> const consoleSpy = vi.spyOn(console, 'error').mockImplementation(() => {})
> ```
>
> 这是迁移友好的典型案例：`jest.spyOn` → `vi.spyOn` 的函数签名完全一致，`vi.fn()` 的 Mock API 也与 Jest 对齐。Error Boundary 测试在迁移时通常不需要任何逻辑调整——只需文本替换。

> **渐进式示例：Error Boundary 测试的三种粒度**
>
> Error Boundary 的测试可以根据功能拆分为三个独立层面：
>
> **层面 1：渲染行为（fallback 是否正确显示）**
> ```typescript
> it('renders fallback on error', () => {
>   vi.spyOn(console, 'error').mockImplementation(() => {})
>
>   render(
>     <ErrorBoundary fallback={<div>Error</div>}>
>       <BrokenComponent shouldThrow={true} />
>     </ErrorBoundary>
>   )
>
>   expect(screen.getByText(/error/i)).toBeInTheDocument()
> })
> ```
> 这是 Error Boundary 的最小可行测试——验证"错误发生时显示 fallback"。层面 1 是**必须**的。
>
> **层面 2：回调通知（onError 是否正确调用）**
> ```typescript
> it('calls onError with the thrown error', () => {
>   const onError = vi.fn()
>   vi.spyOn(console, 'error').mockImplementation(() => {})
>
>   render(
>     <ErrorBoundary fallback={<div>Error</div>} onError={onError}>
>       <BrokenComponent shouldThrow={true} />
>     </ErrorBoundary>
>   )
>
>   expect(onError).toHaveBeenCalledWith(expect.any(Error))
> })
> ```
> 如果 Error Boundary 关联了错误上报服务（Sentry、Datadog 等），这个测试验证上报逻辑是否被触发。层面 2 是**推荐**的（有 onError 时）。
>
> **层面 3：状态恢复（reset 后能否正常渲染）**
> ```typescript
> it('recovers after error reset', async () => {
>   // 通过 fallback 的 resetError 函数重置边界
>   // 验证子组件重新渲染（参考 15.4 完整示例）
> })
> ```
> 这是最完整的测试——验证从"错误→fallback→reset→正常渲染"的完整生命周期。层面 3 是**可选**的（仅当 Error Boundary 支持 reset 时）。
>
> 三个层面是递增的依赖关系：层面 1 不依赖任何其他测试，层面 2 需要层面 1 的结构，层面 3 需要层面 1+2 的机制。

## 15.5 性能测试

### 渲染性能基准

```typescript
// src/test/perf/renderBenchmark.ts
import { render } from '@testing-library/react'

/**
 * 测量组件的渲染时间
 * 注意：这些是粗略的基准，不是精确的性能测量
 */
function measureRenderTime(ui: React.ReactElement, iterations = 100): {
  average: number
  median: number
  max: number
  min: number
} {
  const durations: number[] = []

  for (let i = 0; i < iterations; i++) {
    const start = performance.now()
    const { unmount } = render(ui)
    unmount()
    const end = performance.now()
    durations.push(end - start)
  }

  durations.sort((a, b) => a - b)

  return {
    average: durations.reduce((a, b) => a + b, 0) / durations.length,
    median: durations[Math.floor(durations.length / 2)],
    max: durations[durations.length - 1],
    min: durations[0],
  }
}
```

```typescript
// BigList.perf.test.ts
describe('BigList performance', () => {
  it('renders 1000 items under 500ms', () => {
```

## 15.6 Vitest Browser Mode

### 何时用 Browser Mode

jsdom 模拟了浏览器的 DOM API，但有一些根本限制：

| 场景 | jsdom | Browser Mode |
|------|-------|-------------|
| DOM API 基本操作 | 支持 | 支持 |
| CSS 计算样式 | 部分支持 | 完全支持 |
| 布局测量（offsetHeight, getBoundingClientRect） | 返回 0 | 真实值 |
| Canvas / WebGL | 不支持 | 支持 |
| 动画帧 requestAnimationFrame | 需要 mock | 真实 |
| 视觉断言（截图对比） | 不支持 | 支持 |
| 真实事件冒泡 | 部分 | 完全 |

```typescript
// vitest.config.ts — browser mode 配置
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  test: {
    browser: {
      enabled: true,
      provider: 'playwright',         // 或 'webdriverio'
      headless: true,
      instances: [
        { browser: 'chromium' },      // Chrome
        { browser: 'firefox' },       // Firefox
        { browser: 'webkit' },        // Safari
      ],
    },
    setupFiles: ['./src/test/setup.ts'],
  },
})
```

### 与 Playwright 的关系

```
Vitest Browser Mode vs Playwright:

Vitest Browser Mode:
  - 单元/集成测试在真实浏览器中运行
  - 同一个测试框架（Vitest）
  - 适用于需要真实浏览器 API 的组件测试
  - 可复用现有 jsdom 测试

Playwright:
  - E2E 测试——从用户视角操作完整应用
  - 独立的测试框架
  - 跨页面、跨域、多标签页操作
  - 视觉回归、截图对比

推荐分工：
  - Vitest Browser Mode: 组件级别的浏览器 API 测试
  - Playwright: 用户流程级别的 E2E 测试
  - 两者互补，不是替代关系
```

### Browser Mode 的基本测试

```typescript

// src/test/utils.ts —— 测试和 Storybook 共享的工具

// 共用的 renderWithProviders
import { render, RenderOptions } from '@testing-library/react'
import { MemoryRouter } from 'react-router-dom'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

function createTestQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: { retry: false },
    },
  })
}

interface CustomRenderOptions extends RenderOptions {
  initialRoute?: string
}

export function renderWithProviders(
  ui: React.ReactElement,
  options: CustomRenderOptions = {}
) {
  const { initialRoute = '/', ...renderOptions } = options
  const queryClient = createTestQueryClient()

  function Wrapper({ children }: { children: React.ReactNode }) {
    return (
      <QueryClientProvider client={queryClient}>
        <MemoryRouter initialEntries={[initialRoute]}>
          {children}
        </MemoryRouter>
      </QueryClientProvider>
    )
  }

  return {
    ...render(ui, { wrapper: Wrapper, ...renderOptions }),
    queryClient,
  }
}
```

## 15.8 本章练习

1. **WebSocket 测试**：为实时聊天应用编写测试，覆盖连接、消息收发、断线重连三个场景。使用 MSW 的 `ws.link()`。

   - 重置后恢复渲染

## 15.9 本章总结

- WebSocket 测试：MSW v2 的 `ws.link()` 拦截 `WebSocket` 构造函数，模拟服务端双向通信
- SSE 测试：需手动 mock `EventSource`（jsdom 不支持），通过 `vi.stubGlobal` 注入
- Suspense 边界测试：验证 fallback → 内容 的过渡，`findBy*` 等待异步渲染完成
- `startTransition` / `useDeferredValue`：测试环境中可能同步执行，断言最终结果而非中间状态
- Error Boundary：用 `componentDidCatch` 捕获错误，用 `getDerivedStateFromError` 控制渲染
- 性能测试：`performance.now()` 测量渲染时间，设置阈值做回归检测
- Browser Mode：需要真实浏览器 API 时使用，与 Playwright 互补而非替代
- Storybook 集成：共享 MSW handlers、render utils、Provider 装饰器
