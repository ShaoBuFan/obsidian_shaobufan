# 第四章：用户事件模拟

## 学习目标

- 理解 `userEvent` 与 `fireEvent` 的本质区别
- 掌握 userEvent v14 的异步 API 和 setup 模式
- 能够模拟键盘、鼠标、剪贴板等常见交互
- 理解 fake timers 下 userEvent 的特殊配置
- 能够解决 `act()` 警告

---

## 4.1 userEvent vs fireEvent

这两个 API 的名字很容易让人以为它们是可互换的。但它们的实现差异巨大：

```typescript
import { fireEvent, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'

// fireEvent：只触发一个底层 DOM 事件
fireEvent.click(screen.getByRole('button'))

// userEvent：模拟完整的用户交互序列
const user = userEvent.setup()
await user.click(screen.getByRole('button'))
```

**userEvent.click()** 内部实际触发的事件序列：

```
pointerover → pointerenter → mouseover → mouseenter
→ pointermove → mousedown → pointerdown
→ focus → pointerup → mouseup → click
```

当用户在真实浏览器中点击一个按钮时，这些事件**全部会发生**。fireEvent.click 只触发了最后一步 `click`——如果你的组件在 `mousedown` 中阻止了默认行为，fireEvent 根本测不出来，但 userEvent 能。

**何时可以使用 fireEvent**：几乎永远不应该用。唯一合理的场景是：你确定只需要触发一个底层事件（如 `scroll`、`resize`），且该事件没有 userEvent 的等效方法。

## 4.2 userEvent.setup() 详解

从 v14 开始，userEvent 要求先创建 `user` 实例：

```typescript
const user = userEvent.setup(options?)
```

所有交互方法都挂载在这个实例上，且**全部返回 Promise**：

```typescript
await user.click(element)
await user.dblClick(element)
await user.type(element, 'text')
await user.keyboard('{Enter}')
await user.clear(element)
await user.selectOptions(element, 'option')
await user.deselectOptions(element, 'option')
await user.upload(element, file)
await user.hover(element)
await user.unhover(element)
await user.paste(element, 'text')
await user.tab()
```

### setup 配置选项

```typescript
const user = userEvent.setup({
  // 将内部计时器配置为使用 vi.advanceTimersByTime
  advanceTimers: vi.advanceTimersByTime,

  // 跳过 hover 检查（当元素被遮挡但用户仍可点击时）
  skipHover: false,

  // 跳过 pointer-events CSS 检查
  pointerEventsCheck: 1, // 0: 不检查, 1: 警告, 2: 抛错

  // 跳过自动清理（不推荐，可能导致测试污染）
  skipAutoCleanup: false,

  // 写入到 DOM 之前将数据写入剪贴板
  writeToClipboard: false,
})
```

> **自我验证说明**：上述选项来自 `@testing-library/user-event` v14.x 的 `setup()` 函数类型定义。`advanceTimers` 选项在与 Vitest fake timers 配合时特别有用。

## 4.3 键盘交互

### user.type：逐字符键入

```typescript
const user = userEvent.setup()

it('键入文字', async () => {
  render(<input aria-label="用户名" />)
  const input = screen.getByRole('textbox', { name: '用户名' })

  await user.type(input, 'alice')

  expect(input).toHaveValue('alice')
})

it('键入时触发 onChange', async () => {
  const onChange = vi.fn()
  render(<input aria-label="搜索" onChange={(e) => onChange(e.target.value)} />)

  await user.type(screen.getByRole('textbox'), 'react')

  expect(onChange).toHaveBeenCalledTimes(5) // r, e, a, c, t 分别触发
  expect(onChange).toHaveBeenLastCalledWith('react')
})
```

`user.type` 对每个字符执行：`focus → keydown: {key} → keypress: {key} → input → keyup: {key}`。这意味着表单验证、防抖和输入格式化函数都能被正确测试。

### user.keyboard：快捷键与特殊键

```typescript
it('按 Escape 关闭弹窗', async () => {
  const onClose = vi.fn()
  render(<Dialog onClose={onClose} />)

  await user.keyboard('{Escape}')

  expect(onClose).toHaveBeenCalled()
})

it('Shift+Enter 提交表单', async () => {
  const onSubmit = vi.fn()
  render(<textarea aria-label="评论" onKeyDown={(e) => {
    if (e.key === 'Enter' && e.shiftKey) onSubmit()
  }} />)

  const textarea = screen.getByRole('textbox')
  await user.click(textarea)
  await user.keyboard('{Shift>}{Enter}{/Shift}')

  expect(onSubmit).toHaveBeenCalled()
})

it('Tab 键在表单字段间切换', async () => {
  render(
    <form>
      <input aria-label="姓名" />
      <input aria-label="邮箱" />
      <button type="submit">提交</button>
    </form>
  )

  // 聚焦第一个输入
  await user.click(screen.getByRole('textbox', { name: '姓名' }))
  expect(screen.getByRole('textbox', { name: '姓名' })).toHaveFocus()

  // Tab 到第二个输入
  await user.tab()
  expect(screen.getByRole('textbox', { name: '邮箱' })).toHaveFocus()

  // Tab 到按钮
  await user.tab()
  expect(screen.getByRole('button', { name: '提交' })).toHaveFocus()
})
```

`user.keyboard` 支持的语法：

```
{key}       - 单个按键：{Enter}, {Escape}, {Backspace}, {Delete}, {ArrowLeft}...
{Shift>}    - 按住 Shift（开始）
{/Shift}    - 释放 Shift（结束）
{Shift>}A{/Shift}  - 输入大写 A
```

### user.clear：清空输入框

```typescript
it('清空已有文本', async () => {
  render(<input aria-label="搜索" defaultValue="old text" />)
  const input = screen.getByRole('textbox')

  await user.clear(input)

  expect(input).toHaveValue('')
})
```

## 4.4 鼠标交互

### user.click

```typescript
it('点击按钮触发动作', async () => {
  const onClick = vi.fn()
  render(<button onClick={onClick}>提交</button>)

  await user.click(screen.getByRole('button', { name: '提交' }))

  expect(onClick).toHaveBeenCalledTimes(1)
})

it('点击被禁用的按钮不触发动作', async () => {
  const onClick = vi.fn()
  render(<button onClick={onClick} disabled>提交</button>)

  await user.click(screen.getByRole('button', { name: '提交' }))

  expect(onClick).not.toHaveBeenCalled()
  // userEvent 会先检查按钮是否 disabled，是的话不触发 click
})
```

### user.hover / user.unhover

```typescript
it('hover 显示提示文字', async () => {
  render(<Tooltip text="这是提示"><button>悬停</button></Tooltip>)

  // 初始状态：提示不显示
  expect(screen.queryByText('这是提示')).not.toBeInTheDocument()

  // 悬停
  await user.hover(screen.getByRole('button', { name: '悬停' }))
  expect(screen.getByText('这是提示')).toBeInTheDocument()

  // 移开
  await user.unhover(screen.getByRole('button', { name: '悬停' }))
  expect(screen.queryByText('这是提示')).not.toBeInTheDocument()
})
```

## 4.5 剪贴板操作

```typescript
it('粘贴文本到输入框', async () => {
  render(<textarea aria-label="编辑器" />)

  // user.paste 内部先 focus 元素，再触发 paste 事件
  await user.click(screen.getByRole('textbox'))
  await user.paste('粘贴的内容')

  expect(screen.getByRole('textbox')).toHaveValue('粘贴的内容')
})
```

`user.paste` 的 `writeToClipboard` 选项（在 setup 中设置）决定了粘贴前是否将内容写入系统剪贴板。

it('上传文件', async () => {
  const onUpload = vi.fn()

  // 验证文件对象属性
  const uploadedFile = onUpload.mock.calls[0][0] as File
  expect(uploadedFile.name).toBe('test.txt')
})

## 4.7 表单交互实战

```typescript
describe('LoginForm', () => {
  it('完整登录流程', async () => {
    const user = userEvent.setup()
    const onLogin = vi.fn()
    render(<LoginForm onLogin={onLogin} />)

    // 1. 填写邮箱
    await user.type(
      screen.getByRole('textbox', { name: /邮箱/i }),
      'alice@example.com'
    )

    // 2. 填写密码
    await user.type(
      screen.getByLabelText(/密码/i),
      'securePassword123'
    )

    // 3. 点击登录按钮
    await user.click(screen.getByRole('button', { name: /登录/i }))

    // 4. 验证回调
    await waitFor(() => {
      expect(onLogin).toHaveBeenCalledWith({
        email: 'alice@example.com',
        password: 'securePassword123',
      })
    })
  })

  it('未填写邮箱时显示验证错误', async () => {
    const user = userEvent.setup()
    render(<LoginForm onLogin={vi.fn()} />)

    // 不填写邮箱，直接点击登录
    await user.click(screen.getByRole('button', { name: /登录/i }))

    // 等待验证错误出现
    expect(await screen.findByText(/请输入邮箱/i)).toBeInTheDocument()
  })
})
```

## 4.8 常见问题

### act(...) 警告

```
Warning: An update to Component inside a test was not wrapped in act(...).
```

这个警告的根本原因是：**React 的状态更新在测试结束后才完成**。解决方案不是手动加 `act()`，而是：

1. **await 所有 userEvent 操作**
2. **用 `findBy*` 或 `waitFor` 等待状态更新反映到 DOM**

```typescript
// ❌ 会产生 act 警告
test('updates after click', () => {
  user.click(screen.getByRole('button')) // 没有 await
  expect(screen.getByText('已更新')).toBeInTheDocument()
})

// ✅ 不会产生 act 警告
test('updates after click', async () => {
  await user.click(screen.getByRole('button'))
  expect(await screen.findByText('已更新')).toBeInTheDocument()
})
```

### fake timers 下的 userEvent

当使用 `vi.useFakeTimers()` 时，userEvent 的内部延迟也会被虚拟化。如果不配置 `advanceTimers`，userEvent 的异步操作将永远不 resolve：

```typescript
it('fake timers with userEvent', async () => {
  vi.useFakeTimers()
  // 关键：让 userEvent 使用 vitest 的虚拟计时器
  const user = userEvent.setup({
    advanceTimers: vi.advanceTimersByTime,
  })

  render(<button onClick={vi.fn()}>按钮</button>)
  await user.click(screen.getByRole('button'))
  // 不会卡住

  vi.useRealTimers()
})
```

> **自我验证说明**：`userEvent.setup({ advanceTimers: vi.advanceTimersByTime })` 的用法来自 `@testing-library/user-event` v14.x 官方文档的 "Testing with Fake Timers" 章节。

## 本章总结

- `userEvent` 模拟完整的用户交互序列，`fireEvent` 只触发单个 DOM 事件——**永远优先用 userEvent**
- userEvent v14 所有方法返回 Promise，**必须 await**
- `user.setup({ advanceTimers })` 是 fake timers 下的正确配置
- `user.type` 逐字符触发完整键盘事件序列；`user.keyboard` 用于组合键和特殊键
- `user.click` 自动检查元素是否 disabled
- act() 警告的根因是未等待异步操作完成——加 `await` 和用 `findBy*` 即可解决
