# 第六章：数据层集成测试

## 学习目标

- 理解集成测试在测试奖杯中的核心地位
- 掌握 `renderWithProviders` 自定义 render 模式的编写
- 能测试 React Query / TanStack Query 的数据加载全流程
- 掌握完整的状态覆盖：loading、data、empty、error、network error
- 掌握测试数据工厂与 Builder 模式

---

## 6.1 集成测试的定位

在测试奖杯模型中，集成测试是前端测试的**最大投资层**。它的边界是：

- **渲染真实组件** + **真实的数据请求**（通过 MSW 拦截）+ **真实的用户交互**
- 与单元测试的区别：不 mock 组件内部的模块，让数据流自然流动
- 与 E2E 的区别：不启动浏览器，在 jsdom 中运行，速度快几十倍

一个典型的集成测试架构：

```
组件调用 API 函数
  → API 函数使用 fetch 发出请求
    → MSW 拦截请求，返回 mock 响应
      → API 函数解析响应，返回数据
        → 组件根据数据渲染 UI
          → 测试用 screen 断言用户看到的内容
```

## 6.2 自定义 render wrapper

当组件依赖 Context Provider（Router、QueryClient、Theme、Store）时，每个测试文件都写一遍 Provider 包装既不 DRY、也容易遗漏。自定义 render wrapper 是解决方案。

### 基础版：Router + QueryClient

```typescript
// src/test/test-utils.tsx
import { render, RenderOptions } from '@testing-library/react'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'
import { MemoryRouter } from 'react-router-dom'
import { ReactElement } from 'react'

// 创建的测试用 QueryClient，关键设置：不重试
export function createTestQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        retry: false,   // 测试中不重试，避免超时
        gcTime: 0,      // 即刻垃圾回收，避免跨测试泄漏
      },
      mutations: {
        retry: false,
      },
    },
  })
}

interface ExtendedRenderOptions extends Omit<RenderOptions, 'wrapper'> {
  route?: string
  queryClient?: QueryClient
}

export function renderWithProviders(
  ui: ReactElement,
  {
    route = '/',
    queryClient = createTestQueryClient(),
    ...renderOptions
  }: ExtendedRenderOptions = {}
) {
  window.history.pushState({}, 'Test page', route)

  function Wrapper({ children }: { children: React.ReactNode }) {
    return (
      <QueryClientProvider client={queryClient}>
        <MemoryRouter initialEntries={[route]}>
          {children}
        </MemoryRouter>
      </QueryClientProvider>
    )
  }

  return {
    queryClient,
    ...render(ui, { wrapper: Wrapper, ...renderOptions }),
  }
}

// 重新导出所有 RTL 工具，让测试文件统一从 test-utils 导入
export * from '@testing-library/react'
export { userEvent } from '@testing-library/user-event'
```

> **自我验证说明**：`MemoryRouter` 来自 `react-router-dom` v6，`initialEntries` 接受路由历史数组。`QueryClient` 来自 `@tanstack/react-query`，`gcTime`（原名 `cacheTime`）设为 0 确保测试间缓存不泄漏。此模式与 bulletproof-react 和 TkDodo 的 testing-react-query 示例一致。

### 使用

```typescript
// 测试文件中
import { renderWithProviders, screen, userEvent } from '@/test/test-utils'

it('renders on specific route', () => {
  renderWithProviders(<UserProfile />, { route: '/users/1' })
  expect(screen.getByText('用户详情')).toBeInTheDocument()
})
```

## 6.3 React Query 集成测试

一个使用 React Query 的组件，完整的数据流测试：

```typescript
// UserList.tsx — 使用 React Query 的组件
import { useQuery } from '@tanstack/react-query'

interface User {
  id: number
  name: string
  email: string
}

function UserList() {
  const { data: users, isLoading, isError, error } = useQuery<User[]>({
    queryKey: ['users'],
    queryFn: async () => {
      const res = await fetch('/api/users')
      if (!res.ok) throw new Error(`Error: ${res.status}`)
      return res.json()
    },
  })

  if (isLoading) return <div role="status">加载中...</div>
  if (isError) return <div role="alert">错误：{(error as Error).message}</div>
  if (!users || users.length === 0) return <p>暂无用户</p>

  return (
    <ul aria-label="用户列表">
      {users.map((user) => (
        <li key={user.id}>
          <span>{user.name}</span>
          <span>{user.email}</span>
        </li>
      ))}
    </ul>
  )
}
```

### 完整测试套件

```typescript
// UserList.test.tsx
import { http, HttpResponse } from 'msw'
import { server } from '@/test/mocks/server'
import { renderWithProviders, screen } from '@/test/test-utils'

describe('UserList', () => {
  describe('加载状态', () => {
    it('请求过程中显示 loading', () => {
      // 使用一个永不 resolve 的 Promise 卡住请求
      server.use(
        http.get('/api/users', () => new Promise(() => {}))
      )
      renderWithProviders(<UserList />)
      expect(screen.getByRole('status')).toHaveTextContent('加载中...')
    })
  })

  describe('正常数据', () => {
    it('渲染用户列表', async () => {
      renderWithProviders(<UserList />)
      // 默认 handler 返回正常数据
      expect(await screen.findByText('Alice')).toBeInTheDocument()
      expect(screen.getByText('Bob')).toBeInTheDocument()
    })
  })

  describe('空数据', () => {
    it('显示暂无用户', async () => {
      server.use(
        http.get('/api/users', () => HttpResponse.json([]))
      )
      renderWithProviders(<UserList />)
      expect(await screen.findByText('暂无用户')).toBeInTheDocument()
    })
  })

  describe('服务器错误', () => {
    it('显示 500 错误信息', async () => {
      server.use(
        http.get('/api/users', () =>
          new HttpResponse(null, { status: 500 })
        )
      )
      renderWithProviders(<UserList />)
      expect(await screen.findByRole('alert')).toHaveTextContent('Error: 500')
    })
  })

  describe('网络错误', () => {
    it('显示网络错误信息', async () => {
      server.use(
        http.get('/api/users', () => HttpResponse.error())
      )
      renderWithProviders(<UserList />)
      expect(await screen.findByRole('alert')).toBeInTheDocument()
    })
  })
})
```

> **自我验证说明**：该示例中 UserList 使用 React Query 的 `useQuery`，MSW 在 `fetch` 层拦截。`retry: false` 确保错误测试不会因重试而超时。加载状态测试使用不 resolve 的 Promise 卡住请求——这是一种常用的技巧。

## 6.4 Mutation 测试（增删改）

```typescript
// CreateUserForm.tsx
function CreateUserForm() {
  const queryClient = useQueryClient()
  const mutation = useMutation({
    mutationFn: async (newUser: { name: string; email: string }) => {
      const res = await fetch('/api/users', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(newUser),
      })
      if (!res.ok) {
        const err = await res.json()
        throw new Error(err.message)
      }
      return res.json()
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['users'] })
    },
  })

  const handleSubmit = async (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault()
    const form = e.currentTarget
    const formData = new FormData(form)
    mutation.mutate({
      name: formData.get('name') as string,
      email: formData.get('email') as string,
    })
  }

  return (
    <form onSubmit={handleSubmit}>
      <input name="name" aria-label="姓名" required />
      <input name="email" aria-label="邮箱" type="email" required />
      <button type="submit" disabled={mutation.isPending}>
        {mutation.isPending ? '创建中...' : '创建用户'}
      </button>
      {mutation.isError && (
        <p role="alert">{(mutation.error as Error).message}</p>
      )}
      {mutation.isSuccess && <p role="status">创建成功！</p>}
    </form>
  )
}
```

```typescript
// CreateUserForm.test.tsx
describe('CreateUserForm', () => {
  it('成功创建用户并显示成功消息', async () => {
    const user = userEvent.setup()
    renderWithProviders(<CreateUserForm />)

    await user.type(screen.getByRole('textbox', { name: '姓名' }), 'Charlie')
    await user.type(screen.getByRole('textbox', { name: '邮箱' }), 'charlie@example.com')
    await user.click(screen.getByRole('button', { name: '创建用户' }))

    expect(await screen.findByRole('status')).toHaveTextContent('创建成功！')
  })

  it('邮箱重复时显示错误信息', async () => {
    const user = userEvent.setup()
    // MSW handler 返回 409 冲突
    server.use(
      http.post('/api/users', () =>
        HttpResponse.json({ message: '邮箱已被注册' }, { status: 409 })
      )
    )

    renderWithProviders(<CreateUserForm />)
    await user.type(screen.getByRole('textbox', { name: '姓名' }), 'Charlie')
    await user.type(screen.getByRole('textbox', { name: '邮箱' }), 'taken@example.com')
    await user.click(screen.getByRole('button', { name: '创建用户' }))

    expect(await screen.findByRole('alert')).toHaveTextContent('邮箱已被注册')
  })

  it('提交过程中显示 loading 并禁用按钮', async () => {
    const user = userEvent.setup()
    // 延迟响应，留出时间检查 loading 状态
    server.use(
      http.post('/api/users', async () => {
        await delay(500)
        return HttpResponse.json({ id: 3 }, { status: 201 })
      })
    )

    renderWithProviders(<CreateUserForm />)
    await user.type(screen.getByRole('textbox', { name: '姓名' }), 'Charlie')
    await user.type(screen.getByRole('textbox', { name: '邮箱' }), 'charlie@example.com')
    await user.click(screen.getByRole('button'))

    // 立即检查：按钮显示 loading 状态且被禁用
    expect(screen.getByRole('button')).toHaveTextContent('创建中...')
    expect(screen.getByRole('button')).toBeDisabled()

    // 等待完成
    expect(await screen.findByRole('status')).toHaveTextContent('创建成功！')
  })
})
```

## 6.5 测试数据工厂

随着测试增多，每个测试中手写用户/产品/订单数据会变得繁琐且不一致。测试数据工厂解决了这个问题。

### 简单工厂函数

```typescript
// src/test/factories.ts
interface User {
  id: string
  name: string
  email: string
  role: 'admin' | 'user'
  createdAt: number
}

export function createTestUser(overrides?: Partial<User>): User {
  const id = overrides?.id ?? String(Math.random())
  return {
    id,
    name: `User ${id}`,
    email: `user${id}@example.com`,
    role: 'user',
    createdAt: Date.now(),
    ...overrides,
  }
}

// 使用
const adminUser = createTestUser({ role: 'admin' })
const user42 = createTestUser({ id: '42', name: 'Alice' })
```

### Builder 模式

当对象有复杂的可选字段时，Builder 模式提供更可读的构造：

```typescript
class UserBuilder {
  private user: User = {
    id: '1',
    name: 'Default User',
    email: 'default@example.com',
    role: 'user',
    createdAt: Date.now(),
  }

  withId(id: string): this {
    this.user.id = id
    return this
  }

  withName(name: string): this {
    this.user.name = name
    return this
  }

  withEmail(email: string): this {
    this.user.email = email
    return this
  }

  withRole(role: 'admin' | 'user'): this {
    this.user.role = role
    return this
  }

  build(): User {
    return { ...this.user }
  }
}

// 使用 — 可读性极好
const admin = new UserBuilder().withName('Admin').withRole('admin').build()
const suspendedUser = new UserBuilder().withRole('user').build()
```

## 本章总结

- 集成测试是前端测试的最大投资层：渲染真实组件 + MSW 拦截网络 + 断言 UI
- `renderWithProviders` 封装 Router、QueryClient 等全局 Provider，避免重复代码
- React Query 测试的关键配置：`retry: false`、`gcTime: 0`
- 覆盖所有数据状态：loading → data → empty → server error → network error
- 测试数据工厂（简单函数 / Builder / falso）提升测试可维护性
