## 7.1 测试策略分层

### 三层架构

数据层在前端应用中通常分为三个层级，每层有独立的测试关注点：

```
API Client          Data Hook           UI Component
(fetch/axios)  →   (状态管理)      →   (渲染与交互)

测试重点：          测试重点：            测试重点：
请求/响应正确性      状态转换正确性          用户看到的最终结果
HTTP 状态码处理      Loading/Error 覆盖     数据正确渲染
参数序列化           依赖变更后重新请求       交互反馈
```

MSW 的核心价值在于：**三层可以共享同一套 handler 定义**。API Client 测试验证请求是否发对，Data Hook 测试验证状态是否正确转换，UI Component 测试验证用户是否看到正确结果——它们都基于相同的 MSW mock 网络层，不需要为每一层写不同的 mock。

```typescript
// 共享的 handlers 同时服务于三层测试
export const handlers = [
  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({ id: Number(params.id), name: 'Alice' })
  }),
]

// API Client 测试：验证 fetch 调用和响应解析
// Data Hook 测试：验证 loading → success 状态切换
// UI Component 测试：验证用户看到 "Alice"
```

> **为什么：** 为什么要测试三个层，而非只测 UI 组件？
>
> 每一层测试覆盖不同的错误类别，三层互为补充而非冗余：
>
> | 错误类别 | 举例 | 仅在 UI 测试中发现？ |
> |----------|------|---------------------|
> | URL 拼写错误 | `/api/users` 写成 `/api/user` | 可能，但如果 UI 恰好不显示 URL |
> | 状态码处理 | 204 被当成错误而非成功 | 不会——UI 只关心 data 是否为 null |
> | 参数序列化 | query params 拼错 | 不会——Hook 已经返回了数据 |
> | 状态转换 | loading → error → 又变回 loading | 不会——UI 只看到了最终状态 |
> | 数据格式 | `user.name` 在组件里显示 | 这是 UI 测试擅长的 |
>
> 简单规则：**API Client 测试验证"请求发了、响应收了"；Data Hook 测试验证"状态变了、错误处理了"；UI 测试验证"用户看到了、操作了"**。三层的交集是同一个 MSW handler 定义的数据，但每层关注点正交。这也是为什么 MSW 的"一套 handler 服务所有层"是高效模式——它保证了数据源一致，但每层的测试意图是独立的。

> **Jest 对比：分层测试策略在 Jest 中的实现差异**
>
> 如果你在 Jest 中实现同样的三层测试，最直接的方式是在各个测试文件中分别 mock 对应的模块：
>
> ```typescript
> // Jest 方式：每层单独 mock
> // API Client 测试：mock fetch
> global.fetch = jest.fn()
>
> // Data Hook 测试：mock apiClient
> jest.mock('../api/client', () => ({ get: jest.fn() }))
>
> // UI 测试：mock hook
> jest.mock('../hooks/useUser', () => ({ useUser: () => ({ user: mockUser, loading: false }) }))
> ```
>
> 问题很明显：**每层的 mock 策略不同**，没有统一的数据源。当 API 响应结构变化时，三个 mock 都需要分别更新，而且很容易漏掉。
>
> Vitest + MSW 的优势在于**统一的 mock 层**：
>
> ```typescript
> // MSW 方式：所有三层测试共享同一套 handler
> // 无论测 API Client、Data Hook 还是 UI，都使用：
> http.get('/api/users/:id', () => HttpResponse.json({ id: 1, name: 'Alice' }))
> ```
>
> 这个差异的根源在于：Jest 鼓励的模块级 mock 是**水平分裂**的（每层 mock 不同的模块），而 MSW 的 handler 是**垂直统一**的（所有层共享同一网络层）。垂直统一的架构意味着当数据模型变化时，你只需要更新 handler 和类型定义，所有层的测试都能受益。

### 渐进式示例：同一数据场景的三层测试

用一个具体的数据获取流程，展示三层测试各自验证什么。场景：获取用户 ID 为 1 的数据。

**数据流路径**：
```
UserProfile 组件 → useUser Hook → ApiClient.get → fetch → MSW 拦截 → mock 响应
   │                │              │
UI 测试            Hook 测试      API Client 测试
```

**Layer 1：API Client 测试 — 验证请求/响应正确性**

```typescript
// 关注点：URL 拼接、响应格式、HTTP 状态码
it('fetches user from correct endpoint', async () => {
  const user = await api.get<User>('/users/1')

  expect(user).toEqual({
    id: 1, name: 'Alice', email: 'alice@example.com',
  })
  // 如果 URL 拼成了 /api/users//1（多了一个斜杠），MSW handler 不会匹配
  // 如果响应格式变了，这里的断言会失败
})
```

**Layer 2：Data Hook 测试 — 验证状态转换**

```typescript
// 关注点：loading → 数据/错误 → 完成的状态转换
it('provides user data through hook', async () => {
  const { result } = renderHook(() => useUser(1))

  expect(result.current.loading).toBe(true)         // 初始加载态

  await waitFor(() => expect(result.current.loading).toBe(false))

  expect(result.current.user?.name).toBe('Alice')   // 数据到达
  expect(result.current.error).toBeNull()            // 无错误
})
```

**Layer 3：UI 组件测试 — 验证用户看到的结果**

```typescript
// 关注点：用户看到的最终内容、交互反馈
it('displays user name in the UI', async () => {
  render(<UserProfile userId={1} />)

  expect(await screen.findByText('Alice')).toBeInTheDocument()
  // 不关心内部状态管理，只关心用户看到的最终结果
})
```

> **三层测试的协同价值**：这三层测试覆盖了从网络请求到 UI 渲染的完整路径，但它们共享同一个 MSW handler。当 API 响应结构变化时（比如 `name` 改为 `fullName`），只需更新 handler 和类型定义，三层的测试断言会同时暴露需要调整的地方。这正是 MSW "一套 handler 服务所有测试"的核心理念。

> **自我验证说明**：这种分层策略的收益在于——当 API 响应结构变化时，只需更新 handler 和数据模型，三层测试中的断言可能都需要调整，但每一层的测试逻辑不变。如果使用模块级 mock（`vi.mock('./api')`），API Client 的变化不会传播到上层测试，反而会制造"测试全部通过但页面坏了"的假象。这在 MSW 官方最佳实践中称为"network layer mocking"。

## 7.2 API Client 单元测试

### ApiClient 实现

```typescript
// src/api/client.ts
export class ApiError extends Error {
  constructor(
    public status: number,
    public body: string,
  ) {
    super(`API Error: ${status}`)
    this.name = 'ApiError'
  }
}

export class ApiClient {
  private baseUrl: string

  constructor(baseUrl: string) {
    this.baseUrl = baseUrl
  }

  private async request<T>(path: string, options?: RequestInit): Promise<T> {
    const res = await fetch(`${this.baseUrl}${path}`, {
      headers: { 'Content-Type': 'application/json' },
      ...options,
    })
    if (!res.ok) {
      const body = await res.text()
      throw new ApiError(res.status, body)
    }
    return res.json() as Promise<T>
  }

  async get<T>(path: string): Promise<T> {
    return this.request<T>(path)
  }

  async post<T>(path: string, body: unknown): Promise<T> {
    return this.request<T>(path, {
      method: 'POST',
      body: JSON.stringify(body),
    })
  }

  async put<T>(path: string, body: unknown): Promise<T> {
    return this.request<T>(path, {
      method: 'PUT',
      body: JSON.stringify(body),
    })
  }

  async delete<T>(path: string): Promise<T> {
    return this.request<T>(path, { method: 'DELETE' })
  }
}
```

### MSW Handlers 定义

```typescript
// src/mocks/handlers.ts
import { http, HttpResponse } from 'msw'

interface User {
  id: number
  name: string
  email: string
}

export const handlers = [
  http.get<{ id: string }>('/api/users/:id', ({ params }) => {
    return HttpResponse.json<User>({
      id: Number(params.id),
      name: 'Alice',
      email: 'alice@example.com',
    })
  }),

  http.post<{}, Omit<User, 'id'>>('/api/users', async ({ request }) => {
    const body = await request.json() as Omit<User, 'id'>
    return HttpResponse.json<User>(
      { id: Date.now(), ...body },
      { status: 201 },
    )
  }),

  http.put<{ id: string }, Partial<User>>('/api/users/:id', async ({ request, params }) => {
    const body = await request.json() as Partial<User>
    return HttpResponse.json<User>({
      id: Number(params.id),
      name: body.name ?? 'Alice',
      email: body.email ?? 'alice@example.com',
    })
  }),

  http.delete<{ id: string }>('/api/users/:id', () => {
    return new HttpResponse(null, { status: 204 })
  }),
]
```

### 测试 API Client

```typescript
// src/api/client.test.ts
import { describe, expect, it, beforeEach } from 'vitest'
import { http, HttpResponse } from 'msw'
import { server } from '../mocks/server'
import { ApiClient, ApiError } from './client'

const api = new ApiClient('https://api.example.com')

describe('ApiClient', () => {
  describe('GET', () => {
    it('fetches a user by id', async () => {
      const user = await api.get<User>('/users/1')

      expect(user).toEqual({
        id: 1,
        name: 'Alice',
        email: 'alice@example.com',
      })
    })

    it('throws ApiError on 404', async () => {
      server.use(
        http.get('/api/users/:id', () => {
          return new HttpResponse(null, { status: 404 })
        }),
      )

      await expect(api.get<User>('/users/999')).rejects.toThrow(ApiError)
      await expect(api.get<User>('/users/999')).rejects.toMatchObject({
        status: 404,
      })
    })

    it('throws ApiError on 500', async () => {
      server.use(
        http.get('/api/users/:id', () => {
          return HttpResponse.json(
            { message: 'Internal error' },
            { status: 500 },
          )
        }),
      )

      await expect(api.get<User>('/users/1')).rejects.toMatchObject({
        status: 500,
      })
    })

    it('handles network failure', async () => {
      server.use(
        http.get('/api/users/:id', () => HttpResponse.error()),
      )

      await expect(api.get<User>('/users/1')).rejects.toThrow(TypeError)
    })
  })

  describe('POST', () => {
    it('creates a user and returns 201', async () => {
      const newUser = await api.post<User>('/users', {
        name: 'Bob',
        email: 'bob@example.com',
      })

      expect(newUser).toMatchObject({
        name: 'Bob',
        email: 'bob@example.com',
      })
      expect(newUser.id).toEqual(expect.any(Number))
    })
  })

  describe('PUT', () => {
    it('updates a user', async () => {
      const updated = await api.put<User>('/users/1', { name: 'Alice Updated' })

      expect(updated.name).toBe('Alice Updated')
    })
  })

  describe('DELETE', () => {
    it('deletes a user and returns 204', async () => {
      const result = await api.delete<null>('/users/1')

      expect(result).toBeNull()
    })
  })
})
```

> **自我验证说明**：
> - `ApiClient` 的泛型方法签名确保请求/响应的类型安全。测试中 `api.get<User>('/users/1')` 返回 `Promise<User>`，TypeScript 会在编译阶段检查断言中的属性名是否正确。
> - `HttpResponse.error()` 在 fetch 层面抛 `TypeError: Failed to fetch`，不同于 5xx 的 HTTP 错误。前者是网络传输级错误，后者是 HTTP 协议级错误。测试中必须区分这两种错误场景。
> - 每个测试用 `server.use()` 覆盖 handler，`afterEach` 中 `server.resetHandlers()` 确保测试间隔离。

## 7.3 Data Hook 集成测试

### 自定义 Data Hook

```typescript
// src/hooks/useUser.ts
import { useState, useEffect } from 'react'
import { api } from '../api/client'

interface User {
  id: number
  name: string
  email: string
}

interface UseUserState {
  user: User | null
  loading: boolean
  error: Error | null
}

export function useUser(id: number | null): UseUserState {
  const [state, setState] = useState<UseUserState>({
    user: null,
    loading: false,
    error: null,
  })

  useEffect(() => {
    if (id === null) return

    setState({ user: null, loading: true, error: null })

    api.get<User>(`/users/${id}`)
      .then((user) => setState({ user, loading: false, error: null }))
      .catch((error) => setState({ user: null, loading: false, error }))
  }, [id])

  return state
}
```

### 使用 renderHook 测试

```typescript
// src/hooks/useUser.test.ts
import { renderHook, waitFor } from '@testing-library/react'
import { http, HttpResponse } from 'msw'
import { server } from '../mocks/server'
import { useUser } from './useUser'

describe('useUser', () => {
  it('starts with loading state', () => {
    const { result } = renderHook(() => useUser(1))

    expect(result.current.loading).toBe(true)
    expect(result.current.user).toBeNull()
    expect(result.current.error).toBeNull()
  })

  it('transitions to success state with user data', async () => {
    const { result } = renderHook(() => useUser(1))

    await waitFor(() => expect(result.current.loading).toBe(false))

    expect(result.current.user).toEqual({
      id: 1,
      name: 'Alice',
      email: 'alice@example.com',
    })
    expect(result.current.error).toBeNull()
  })

  it('transitions to error state on 404', async () => {
    server.use(
      http.get('/api/users/:id', () => {
        return new HttpResponse(null, { status: 404 })
      }),
    )

    const { result } = renderHook(() => useUser(999))

    await waitFor(() => expect(result.current.loading).toBe(false))

    expect(result.current.user).toBeNull()
    expect(result.current.error).not.toBeNull()
    expect(result.current.error).toBeInstanceOf(Error)
  })

  it('re-fetches when id changes', async () => {
    const { result, rerender } = renderHook(
      ({ id }) => useUser(id),
      { initialProps: { id: 1 } },
    )

    await waitFor(() => expect(result.current.loading).toBe(false))
    expect(result.current.user?.name).toBe('Alice')

    // 切换到 user/2
    rerender({ id: 2 })

    expect(result.current.loading).toBe(true)
    await waitFor(() => expect(result.current.loading).toBe(false))
    // handler 返回固定数据，所以 name 还是 Alice
    expect(result.current.user?.id).toBe(2)
  })

  it('does not fetch when id is null', () => {
    const { result } = renderHook(() => useUser(null))

    expect(result.current.loading).toBe(false)
    expect(result.current.user).toBeNull()
  })
})
```

> **自我验证说明**：
> - `renderHook` 的第一个参数是回调函数，返回 `{ result, rerender, unmount }`。`result.current` 在每次 Hook 更新后反映最新值。
> - 需要等待异步操作完成时使用 `waitFor` 轮询 `result.current.loading` 直到变为 `false`。`findBy*` 查询不可用——`renderHook` 不渲染 DOM。
> - `rerender` 接受新 props，触发 Hook 重新执行。TypeScript 泛型确保初始 props 和 rerender props 类型一致。

### 工厂函数模式

当不同的测试需要不同的 mock 数据时，工厂函数比在测试中用 `server.use()` 覆盖更方便：

```typescript
// src/mocks/domains/users.ts
import { http, HttpResponse } from 'msw'

export function createUserHandler(overrides?: Partial<User>) {
  return http.get('/api/users/:id', () => {
    return HttpResponse.json<User>({
      id: 1,
      name: 'Alice',
      email: 'alice@example.com',
      ...overrides,
    })
  })
}

// 测试中使用
it('displays custom user name', async () => {
  server.use(createUserHandler({ name: 'Custom Name' }))
  // ...测试逻辑
})

it('handles missing email gracefully', async () => {
  server.use(createUserHandler({ email: '' }))
  // ...测试逻辑
})
```

```typescript
// 更完整的工厂：支持完全自定义响应
interface UserHandlerConfig {
  status?: number
  data?: Partial<User>
  delay?: number
}

export function createUserHandler(config: UserHandlerConfig = {}) {
  const { status = 200, data = {}, delay: delayMs = 0 } = config

  return http.get('/api/users/:id', async () => {
    if (delayMs) await delay(delayMs)

    return HttpResponse.json<User>(
      {
        id: 1,
        name: 'Alice',
        email: 'alice@example.com',
        ...data,
      },
      { status },
    )
  })
}

// 覆盖错误场景
server.use(createUserHandler({ status: 500 }))
server.use(createUserHandler({ status: 404 }))

// 覆盖超时场景
server.use(createUserHandler({ delay: 10000 }))
```

> **自我验证说明**：工厂函数的核心优势是让测试意图明确——`createUserHandler({ status: 500 })` 比 `http.get('/api/users/:id', () => HttpResponse.json(null, { status: 500 }))` 更清晰地表达了测试场景。同时，用户数据的默认值集中管理，不会散落在各个测试中。

### Handler 内请求验证：警察 + 演员

工厂模式解决了"返回什么响应"的问题，但数据层测试还有一个同等重要的维度：**你的代码发出了正确的请求吗？**

MSW 的 resolver 函数是请求验证和响应模拟的**统一入口**。你不需要两个独立的机制——一个 resolver 同时完成两件事：

```
你的项目代码发出 fetch('/api/users', { method: 'POST', body: {...} })
        │
        ▼
    MSW 拦截请求
        │
        ▼
    Handler resolver({ request })
        ├── await request.json()           ← 拿到请求体（真实内容，不是 mock）
        ├── 检查 request.method           ← 验证请求方法
        ├── 检查 request.headers          ← 验证 Content-Type、Authorization
        ├── 检查 URL searchParams         ← 验证查询参数
        └── return HttpResponse.json(...)  ← 控制返回的响应
        │
        ▼
你的代码收到响应，继续执行（状态更新、渲染……）
        │
        ▼
    expect(screen.findByText(...))         ← 断言最终 UI 结果
```

**核心洞察**：你的项目代码不知道自己在被测试——它正常发 HTTP 请求。MSW 在中间层替你完成了"检查发出去了什么"和"控制返回了什么"两项工作。

#### 基础请求验证

```typescript
// src/api/users.ts — 你的项目代码
async function createUser(data: CreateUserInput): Promise<User> {
  const res = await fetch('/api/users', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer token' },
    body: JSON.stringify(data),
  })
  if (!res.ok) throw new Error(`HTTP ${res.status}`)
  return res.json()
}

// src/api/users.test.ts — 验证请求本身
it('sends the correct request', async () => {
  const captured = vi.fn()

  server.use(
    http.post('/api/users', async ({ request }) => {
      const body = await request.json()
      // 捕获所有请求细节供后续断言
      captured({
        method: request.method,
        headers: Object.fromEntries(request.headers),
        body,
      })
      return HttpResponse.json({ id: 1, name: body.name, email: body.email }, { status: 201 })
    })
  )

  await createUser({ name: 'Alice', email: 'alice@test.com' })

  // 断言代码发出了正确的请求
  expect(captured).toHaveBeenCalledWith(
    expect.objectContaining({
      method: 'POST',
      headers: expect.objectContaining({
        'content-type': 'application/json',
        'authorization': 'Bearer token',
      }),
      body: { name: 'Alice', email: 'alice@test.com' },
    })
  )
})
```

> **自我验证说明**：`vi.fn()` 在这里充当"请求证据袋"——resolver 在收到请求时记录关键信息（method / headers / body），测试用例随后断言这些记录是否符合预期。`vi.fn()` 比手动 push 到数组更好，因为它自带 `toHaveBeenCalledWith` 等断言方法、自动追踪调用次数、失败时打印实际值。`Object.fromEntries(request.headers)` 将 Headers 对象转为普通对象以便 `expect.objectContaining` 做部分匹配。

#### 直接在 resolver 内断言

对于关键请求约束，可以在 resolver 内部直接断言——让错误的请求在到达响应逻辑之前就暴露：

```typescript
http.post('/api/users', async ({ request }) => {
  const body = await request.json()

  // 直接断言关键约束（失败时堆栈指向这里，而非测试末尾）
  if (!body.name || typeof body.name !== 'string') {
    throw new Error(`Invalid name: ${body.name}`)
  }
  if (!body.email?.includes('@')) {
    throw new Error(`Invalid email: ${body.email}`)
  }

  return HttpResponse.json({ id: 1, ...body }, { status: 201 })
})
```

> **哪种方式更好？** `captured` + 末尾断言更适合"验证常规行为"（大多数测试场景），失败信息会出现在测试输出中，语义清晰。resolver 内断言更适合"防御非法状态"——如果你确信某些参数组合在正常业务逻辑中永远不会出现，让它们直接报错更快暴露问题。

### 增强工厂模式：场景控制 + 请求验证不丢失

工厂模式的问题是：当你为"成功"和"验证错误"两个场景各写一个工厂函数时，**请求验证逻辑要么被重复，要么被遗漏**。

解决方案：把请求验证作为工厂的**内置能力**，通过回调暴露给测试：

```typescript
// src/mocks/domains/users.ts
import { http, HttpResponse } from 'msw'

type CreateUserScenario = 'success' | 'duplicateEmail' | 'serverError' | 'networkError'

export function createUserHandler(scenario: CreateUserScenario = 'success') {
  // 请求验证回调 —— 每个测试可以注册自己的验证逻辑
  let onRequest: ((body: unknown, request: Request) => void) | undefined

  const handler = http.post('/api/users', async ({ request }) => {
    const body = await request.json()

    // 触发测试注册的验证逻辑
    onRequest?.(body, request)

    // 根据场景返回对应响应
    switch (scenario) {
      case 'success':
        return HttpResponse.json({ id: 1, ...(body as object) }, { status: 201 })
      case 'duplicateEmail':
        return HttpResponse.json(
          { errors: { email: 'Already registered' } },
          { status: 409 }
        )
      case 'serverError':
        return new HttpResponse(null, { status: 500 })
      case 'networkError':
        return HttpResponse.error()
    }
  })

  return {
    handler,
    // 链式注册验证逻辑
    verify(fn: (body: unknown, request: Request) => void) {
      onRequest = fn
      return this
    },
  } as const
}
```

使用方式——工厂负责场景，`verify()` 负责请求验证：

```typescript
// 场景1：不需要验证请求细节，只关心响应 → 直接用
it('shows server error message', async () => {
  server.use(createUserHandler('serverError').handler)
  render(<RegistrationForm />)
  await user.click(screen.getByRole('button', { name: /register/i }))
  expect(await screen.findByText(/server error/i)).toBeInTheDocument()
})

// 场景2：需要验证发出了什么 → 链式调用 .verify()
it('sends trimmed user input', async () => {
  const { handler } = createUserHandler('success')
    .verify((body) => {
      // 在请求发出的瞬间验证
      expect(body).toHaveProperty('name', 'Alice')
      expect(body).toHaveProperty('email', 'alice@test.com')
    })

  server.use(handler)

  await user.type(screen.getByLabelText(/name/i), '  Alice  ')
  await user.type(screen.getByLabelText(/email/i), 'alice@test.com')
  await user.click(screen.getByRole('button', { name: /register/i }))

  // verify 回调在 handler 被触发时执行，此处只需断言 UI 结果
  expect(await screen.findByText(/welcome, alice/i)).toBeInTheDocument()
})

// 场景3：验证请求头（如 Authorization token）
it('includes auth token in request', async () => {
  const { handler } = createUserHandler('success')
    .verify((_body, request) => {
      expect(request.headers.get('Authorization')).toBe('Bearer mock-token')
    })

  server.use(handler)

  await createUser({ name: 'A', email: 'a@t.com' })
  // verify 中的断言在 handler 触发时自动执行
})
```

> **为什么：** `onRequest` 回调比在 handler 内部硬编码断言更灵活——同一个"成功"场景，测试 A 关心请求 body 的字段完整性，测试 B 关心 Authorization header 是否存在，测试 C 不关心请求细节。三种需求共享同一个 handler 工厂，各自通过 `verify()` 注册自己的验证逻辑。这遵循了"工厂管理共享逻辑，回调暴露差异化需求"的原则。

> **自我验证说明**：`verify()` 返回 `this` 使链式调用成为可能。`as const` 确保 TypeScript 推导出字面量类型而非宽泛的 `{ handler: HttpHandler; verify: ... }`。`(body as object)` 展开是因为 TypeScript 在 `switch` 的收窄上下文中无法确定泛型约束。

### 工厂模式的完整使用图谱

```
测试需要什么？              用什么？
─────────────────────────────────────────────
只验证 UI 结果               createUserHandler('success').handler
验证请求 body               .verify((body) => { ... })
验证请求 headers             .verify((_body, req) => { ... })
验证错误响应                 createUserHandler('serverError').handler
验证网络失败                 createUserHandler('networkError').handler
同场景多测试各有不同验证       .verify(fn1) / .verify(fn2) / .verify(fn3)
不需要验证请求               直接用 .handler（跳过 verify）
```

### 同一 API 的多次调用验证

实际项目中，同一个 API 端点经常被多次调用，且每次的请求和响应可能不同：

- **重试**：第一次 500，第二次 200
- **轮询**：每次返回不同状态（pending → processing → done）
- **分页**：`?page=1` 和 `?page=2` 请求参数不同
- **竞态**：快速连续调用，需验证最后一次请求的参数
- **防抖搜索**：打字"a"→"ab"→"abc"，每次触发不同请求

核心挑战：**同一个 URL 模式被多次匹配，handler 如何区分"第几次调用"，测试又如何独立验证每次？**

#### 方案 1：调用计数器 + 序列响应

最直观的方案——handler 内部维护计数器，按调用顺序返回不同响应：

```typescript
it('retries on first failure then succeeds', async () => {
  let callCount = 0

  server.use(
    http.get('/api/status', () => {
      callCount++
      if (callCount === 1) {
        return new HttpResponse(null, { status: 500 })  // 第一次：失败
      }
      return HttpResponse.json({ status: 'ready' })       // 第二次：成功
    })
  )

  // 被测代码内部实现了"失败则重试一次"
  const result = await fetchWithRetry('/api/status', { retries: 1 })

  expect(result.status).toBe('ready')
  expect(callCount).toBe(2)  // 确认确实调了两次
})
```

> **自我验证说明**：`let callCount` 是闭包变量，在 handler 外部声明但在内部读写。每次请求进入 handler 时 `callCount++`，允许你基于调用序号做分支。这个模式简单直接，适合 2-3 次调用的场景。超过 5 次时分支逻辑会变复杂，此时用方案 2（序列数组）更好。

#### 方案 2：序列响应数组

预定义响应序列，每次调用取下一个：

```typescript
it('polls until status is ready', async () => {
  const responses = [
    HttpResponse.json({ status: 'pending' }, { status: 200 }),
    HttpResponse.json({ status: 'pending' }, { status: 200 }),
    HttpResponse.json({ status: 'ready' }, { status: 200 }),
  ]

  server.use(
    http.get('/api/status', () => {
      return responses.shift() ?? HttpResponse.json({}, { status: 200 })
    })
  )

  const result = await pollUntilReady('/api/status', { maxAttempts: 5 })

  expect(result.status).toBe('ready')
  expect(responses).toHaveLength(0)  // 所有预定义响应都被消费
})
```

> **为什么：** `responses.shift()` 每次取走数组第一个元素，天然按序推进。如果实际调用次数超过数组长度，`shift()` 返回 `undefined`，`??` 提供兜底。`toHaveLength(0)` 断言所有预期响应都被消费——如果代码少调一次，最后一个 `'ready'` 没被 shift，数组长度 > 0，测试失败。

#### 方案 3：vi.fn() 作为 handler——白盒验证每次调用

当你需要**精确验证每次调用的参数**时，直接用 `vi.fn()` 实现 handler：

```typescript
it('paginates with correct offsets', async () => {
  const handler = vi.fn()

  // vi.fn 内部的实现返回正常响应
  handler.mockImplementation(({ request }) => {
    const url = new URL(request.url)
    const page = url.searchParams.get('page')
    return HttpResponse.json({ page, items: [] })
  })

  server.use(
    http.get('/api/items', handler)
  )

  // 被测代码：一次性加载 3 页
  await fetchAllPages('/api/items', { pages: 3 })

  // 验证总共调了 3 次
  expect(handler).toHaveBeenCalledTimes(3)

  // 验证每次调用时的 page 参数
  const calls = handler.mock.calls
  const page1Url = new URL(calls[0][0].request.url)
  const page2Url = new URL(calls[1][0].request.url)
  const page3Url = new URL(calls[2][0].request.url)

  expect(page1Url.searchParams.get('page')).toBe('1')
  expect(page2Url.searchParams.get('page')).toBe('2')
  expect(page3Url.searchParams.get('page')).toBe('3')
})
```

> **自我验证说明**：`vi.fn()` 直接作为 handler 可以工作，因为 MSW 的 resolver 签名为 `(args: ResolverArgs) => HttpResponse | Promise<HttpResponse>`，而 `vi.fn()` 的类型是 `(...args: any[]) => any`，二者兼容。但需要显式设置 `mockImplementation`，否则默认返回 `undefined`（MSW 收到 undefined 的响应行为未定义）。`handler.mock.calls` 是一个二维数组：`calls[0]` = 第一次调用的参数数组，`calls[0][0]` = 第一次调用的第一个参数（即 `{ request, params, cookies, requestId }`）。

#### 方案 4：`{ once: true }` 多次注册

MSW v2 支持 `{ once: true }` 选项——handler 在匹配一次后自动移除。可以多次 `server.use()` 注册同一路径的不同 handler，靠 LIFO + once 实现序列：

```typescript
it('fails twice then succeeds on third attempt', async () => {
  // 新 handler 被 prepend 到链的最前面
  // 每次匹配后自动移除，露出下一个

  // 第 3 次调用 → 最先匹配（LIFO 头部）→ 成功后移除
  server.use(
    http.get('/api/data', () => HttpResponse.json({ ok: true }), { once: true })
  )
  // 第 2 次调用
  server.use(
    http.get('/api/data', () => new HttpResponse(null, { status: 500 }), { once: true })
  )
  // 第 1 次调用 → 最先注册的排在最前面，但 LIFO 使其变为最后匹配
  server.use(
    http.get('/api/data', () => HttpResponse.error(), { once: true })
  )

  // 第 1 次：HttpResponse.error() 匹配（LIFO：最后 use 的最先匹配）
  // 第 2 次：error handler 已移除 → 500 handler 匹配
  // 第 3 次：500 handler 已移除 → success handler 匹配
  const result = await retryWithBackoff('/api/data', { maxRetries: 3 })
  expect(result.ok).toBe(true)
})
```

> **为什么：** LIFO（Last In First Out）+ `{ once: true }` 的组合意味着你要**反向注册 handler**——最后一个 `server.use()` 的 handler 最先被匹配。示例中注册顺序是 `success → 500 → error`，但匹配顺序是 `error → 500 → success`。如果顺序让你困惑，用方案 2（序列数组）更直观。

#### 方案选择速查

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 2-3 次调用，简单分支 | 方案 1：计数器 | 最简单，3 行代码 |
| 3+ 次调用，递增行为 | 方案 2：序列数组 | 清晰的"响应队列"语义 |
| 需要精确验证每次的参数 | 方案 3：vi.fn() | `mock.calls` 提供完整调用历史 |
| 每次行为完全不同 | 方案 4：`{ once: true }` | 每个 handler 独立，关注点分离 |
| 竞态条件 | 方案 1 或 2 | 需要在响应中加 `delay` 模拟时序 |

#### 综合示例：防抖搜索——参数验证 + 取消验证

这是最复杂的真实场景：打字过程中多次调用同一 API，且前面的调用应该被取消：

```typescript
it('cancels stale searches and only uses the latest results', async () => {
  vi.useFakeTimers()
  const calls: string[] = []           // 记录所有请求的查询参数
  let resolveSecond: () => void

  server.use(
    http.get('/api/search', async ({ request }) => {
      const query = new URL(request.url).searchParams.get('q') ?? ''
      calls.push(query)

      if (query === 'ab') {
        // 第二次请求：延迟返回（模拟慢网络）
        await new Promise<void>((r) => { resolveSecond = r })
      }
      // 第三次请求：立即返回
      return HttpResponse.json({ results: [`result for ${query}`] })
    })
  )

  const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })
  render(<SearchBox debounceMs={200} />)
  const input = screen.getByRole('searchbox')

  // 打字："a" → 200ms → "ab" → 200ms → "abc"
  await user.type(input, 'a')
  await vi.advanceTimersByTimeAsync(201)
  await user.type(input, 'b')
  await vi.advanceTimersByTimeAsync(201)
  await user.type(input, 'c')

  // 此时前两次调用已发出，第三次调用在防抖延迟中
  expect(calls).toEqual(['a', 'ab'])
  // "abc" 还没发出（防抖 200ms 未到）

  await vi.advanceTimersByTimeAsync(201)

  // 第三次调用发出了
  expect(calls).toEqual(['a', 'ab', 'abc'])

  // 释放第二次调用的延迟响应
  resolveSecond!()

  // 关键断言：只有最后一次查询的结果被展示
  // 第二次调用的响应虽然到了，但因为是过时的，被丢弃
  await vi.advanceTimersByTimeAsync(100)  // React 处理更新
  expect(screen.queryByText('result for ab')).not.toBeInTheDocument()
  expect(await screen.findByText('result for abc')).toBeInTheDocument()

  vi.useRealTimers()
})
```

> **自我验证说明**：这个测试同时验证了三个维度的正确性：（1）`calls` 数组记录了每次请求的 URL 参数——确认代码对每个输入都发出了正确的请求；（2）`resolveSecond!()` 手动控制第二次请求的响应时机——模拟"慢网络导致响应乱序到达"的真实场景；（3）最终 UI 断言只展示最新的结果——验证了竞态取消逻辑。`advanceTimersByTimeAsync` 配合 userEvent 的 `advanceTimers` 选项确保打字内部延迟不被真实等待。

> **洞察：`const calls: string[] = []` 是方案 3（vi.fn）的轻量替代。** vi.fn 给你 `mock.calls`、`toHaveBeenCalledTimes`、`toHaveBeenCalledWith` 等完整的断言 API——代价是访问调用参数时需要 `calls[0][0].request.url` 这种深层路径。而 `calls: string[]` 直接把每次调用中最关键的几个值推入数组，后续用 `toEqual` 一次性断言顺序和内容——`expect(calls).toEqual(['a', 'ab', 'abc'])` 一行验证了"调了几次、每次的参数是什么、顺序对不对"三个问题。你不需要在"vi.fn 的完整能力"和"简单数组的记录能力"之间二选一——两个变量可以并存于同一个 handler 中，各司其职。

### vi.fn 工厂：把请求验证封装为可复用模块

上面的 `vi.fn()` + `handler.mock.calls[n]` 模式有一个明显的问题：**每次测试都要在 handler 里写一遍请求解析和响应逻辑**。这些代码和 `/api/auth` 的业务语义强相关，应该在工厂中集中定义，测试只做断言。

API 的"请求验证能力"和"响应组装能力"应该和 API 端点本身放在一起——这是 handler 的内建能力，不是测试的临时脚本。

```typescript
// src/mocks/domains/auth.ts
import { http, HttpResponse } from 'msw'
import { vi } from 'vitest'

type AuthCall =
  | { type: 'check'; method: 'GET'; mode: string }
  | { type: 'login'; method: 'POST'; username: string; password: string }
  | { type: 'unknown'; method: string }

export function createAuthHandler() {
  // 每次调用工厂都创建新的 vi.fn，测试间不会互相污染
  const handler = vi.fn(({ request }: { request: Request }) => {
    const url = new URL(request.url)

    if (request.method === 'GET') {
      const mode = url.searchParams.get('mode')
      recordCall({ type: 'check', method: 'GET', mode: mode ?? '' })
      return HttpResponse.json({ isLoggedIn: mode === 'check' })
    }

    if (request.method === 'POST') {
      // ⚠️ 注意：request.body 是 ReadableStream，只能读一次
      //    handler 里读完后用 .clone() 保留原始 body 给断言用
      const body = await request.clone().json() as { mode: string; username: string; password: string }
      recordCall({
        type: 'login',
        method: 'POST',
        username: body.username,
        password: body.password,
      })
      return HttpResponse.json({ isLoggedIn: body.mode === 'login' })
    }

    recordCall({ type: 'unknown', method: request.method })
    return new HttpResponse(null, { status: 405 })
  })

  // ─── 内部记录数组（比 mock.calls 更语义化）───
  const calls: AuthCall[] = []
  function recordCall(call: AuthCall) {
    calls.push(call)
  }

  return {
    // 直接丢给 server.use()
    handler,

    // ─── 便捷查询 API ───
    // 总调用次数（等价于 handler.mock.calls.length，但不需要 import vi）
    get callCount() { return calls.length },

    // 按类型过滤调用
    checks()  { return calls.filter(c => c.type === 'check') },
    logins()  { return calls.filter(c => c.type === 'login') },

    // 原始记录数组（简洁断言用）
    calls,
  }
}
```

测试文件现在只做断言，不再包含请求解析逻辑：

```typescript
import { createAuthHandler } from '@/mocks/domains/auth'
import { server } from '@/mocks/server'

it('sends check on mount, then login on click', async () => {
  // ─── 一行创建，一行注册 ───
  const auth = createAuthHandler()
  server.use(auth.handler)

  render(<LoginPage />)
  await screen.findByRole('status')

  // ─── 断言初始化请求 ───
  expect(auth.callCount).toBe(1)
  expect(auth.checks()).toEqual([
    expect.objectContaining({ type: 'check', method: 'GET', mode: 'check' }),
  ])

  // ─── 点击登录 ───
  await user.click(screen.getByRole('button', { name: 'Login' }))

  // ─── 断言第二次请求 ───
  expect(auth.callCount).toBe(2)
  expect(auth.logins()).toEqual([
    expect.objectContaining({
      type: 'login',
      method: 'POST',
      username: 'alice',
      password: '123',
    }),
  ])
})

it('handles failed login check', async () => {
  // 另一个测试，独立的工厂实例，不污染前一个测试的记录
  const auth = createAuthHandler()
  server.use(auth.handler)

  render(<LoginPage />)
  await screen.findByRole('status')

  expect(auth.checks()).toHaveLength(1)
  expect(screen.getByRole('status')).toHaveTextContent('loggedOut')
})
```

> **为什么用 `request.clone()` 而不是直接读 `request.json()`？** `request.body` 是 ReadableStream，只能消费一次。handler 里读完 body 后，如果你想在测试中用 `handler.mock.calls[0][0].request.json()` 再读一次来断言——body 已经空了，读到的是 `{}`。`request.clone()` 创建一个独立的 body 副本给 handler 用，原始 body 留给 `mock.calls` 中的断言。不过在这个工厂设计中，`recordCall` 已经把关键字段提取到 `AuthCall` 对象里了，测试不再需要直接读 `request.body`，所以 `clone()` 更多是防御性保险。

> **自我验证说明**：工厂函数返回的 `handler` 就是一个标准的 MSW HttpHandler（vi.fn 实现），可以直接传给 `server.use()`。`AuthCall` 是工厂内部定义的判别联合类型，让 `auth.checks()` 和 `auth.logins()` 的返回值有精确的 TypeScript 类型。每个测试调用一次 `createAuthHandler()` 创建独立实例，`calls` 数组不会跨测试污染。

### 工厂封装等级选择

同一类 API 的请求验证，根据复用程度有三层封装可选：

```
等级 1：测试内直接写 vi.fn —— 一次性测试，不值得抽
等级 2：项目级 domain 工厂（如上面的 createAuthHandler）—— 同一 API 被多个测试使用
等级 3：通用工具函数 —— 跨 API 共享的"创建带记录的 handler"模式
```

等级 3 的例子——把"带调用记录的 handler 创建"抽象为通用工具：

```typescript
// src/mocks/utils.ts
import { vi } from 'vitest'
import { http, HttpResponse } from 'msw'

// 通用工厂工厂：输入 endpoint + 响应逻辑，输出带记录能力的 handler
export function createTrackedHandler<TCall>(
  resolver: (args: { request: Request; params: Record<string, string> }) => {
    response: HttpResponse
    record: TCall
  },
) {
  const calls: TCall[] = []
  const handler = vi.fn((args: { request: Request; params: Record<string, string> }) => {
    const { response, record } = resolver(args)
    calls.push(record)
    return response
  })

  return { handler, calls }
}
```

等级 2（领域工厂）是你最常写的。等级 3 在 5+ 个 API 端点有相似验证需求时才值得引入。不要跳到等级 3 之前先确认等级 2 真的不够用。

### 超时模拟

```typescript
// 组件：带超时的数据获取
async function fetchWithTimeout(url: string, timeoutMs: number): Promise<Response> {
  const controller = new AbortController()
  const timer = setTimeout(() => controller.abort(), timeoutMs)

  try {
    const res = await fetch(url, { signal: controller.signal })
    return res
  } finally {
    clearTimeout(timer)
  }
}

// 测试
it('aborts request after timeout', async () => {
  vi.useFakeTimers()

  server.use(
    http.get('/api/slow', async () => {
      await delay(10000)
      return HttpResponse.json({ data: 'too late' })
    }),
  )

  const fetchPromise = fetchWithTimeout('/api/slow', 3000)

  await vi.advanceTimersByTimeAsync(3000)

  await expect(fetchPromise).rejects.toThrow(/aborted/i)

  vi.useRealTimers()
})
```

> **Jest 对比：假定时器 API 的命名差异**
>
> 从 Jest 迁移的开发者会发现 `vi.useFakeTimers()` 和 `jest.useFakeTimers()` 在基本功能上完全一致，但 Vitest 在以下方面有扩展：
>
> | 功能 | Jest | Vitest |
> |------|------|--------|
> | 启用假定时器 | `jest.useFakeTimers()` | `vi.useFakeTimers()` |
> | 恢复真实定时器 | `jest.useRealTimers()` | `vi.useRealTimers()` |
> | 推进时间 | `jest.advanceTimersByTime(ms)` | `vi.advanceTimersByTime(ms)` |
> | 运行所有待处理定时器 | `jest.runAllTimers()` | `vi.runAllTimers()` |
> | 只运行待处理的微任务 | 无原生 API | `vi.runOnlyPendingTimersAsync()` |
>
> **关键区别**：Vitest 的 `vi.advanceTimersByTimeAsync(ms)` 和 `vi.runAllTimersAsync()` 支持异步定时器，这在处理 `setTimeout` + `fetch` 组合时很有用。Jest 27+ 也有类似的支持（`jest.advanceTimersByTimeAsync`），但 API 名称不同。
>
> 另一个实际区别：**Vitest 默认使用 `@sinonjs/fake-timers` 作为底层实现**，而 Jest 使用自己的实现。这意味着在复杂场景下（多个定时器嵌套、Date 模拟），两者的行为可能有细微差异。如果你的团队从 Jest 迁移，建议在迁移后重新运行所有定时器相关的测试，验证行为一致。
>
> ```typescript
> // Jest 写法
> jest.useFakeTimers()
> jest.advanceTimersByTime(3000)
>
> // Vitest 写法（基本一致）
> vi.useFakeTimers()
> vi.advanceTimersByTime(3000)
> ```

### 重试逻辑

```typescript
// 组件：带重试的数据获取
async function fetchWithRetry<T>(
  fetchFn: () => Promise<T>,
  retries: number = 3,
): Promise<T> {
  let lastError: unknown
  for (let attempt = 0; attempt < retries; attempt++) {
    try {
      return await fetchFn()
    } catch (err) {
      lastError = err
      if (attempt === retries - 1) throw err
    }
  }
  throw lastError
}

// 测试
it('retries on network error and succeeds on third attempt', async () => {
  let attempt = 0

  server.use(
    http.get('/api/unstable', () => {
      attempt++
      if (attempt < 3) return HttpResponse.error()
      return HttpResponse.json({ status: 'ok' })
    }),
  )

  const result = await fetchWithRetry(
    () => fetch('/api/unstable').then((r) => r.json()),
    3,
  )

  expect(result).toEqual({ status: 'ok' })
  expect(attempt).toBe(3)
})

it('throws after exhausting retries', async () => {
  server.use(
    http.get('/api/failing', () => HttpResponse.error()),
  )

  await expect(
    fetchWithRetry(() => fetch('/api/failing'), 2),
  ).rejects.toThrow()
})
```

### 竞态条件

```typescript
// Hook：搜索请求，确保只使用最新的响应
function useSearch(query: string) {
  const [results, setResults] = useState<SearchResult[]>([])

  useEffect(() => {
    let ignore = false

    fetch(`/api/search?q=${encodeURIComponent(query)}`)
      .then((r) => r.json())
      .then((data: SearchResult[]) => {
        if (!ignore) setResults(data)
      })

    return () => { ignore = true }
  }, [query])

  return results
}

// 测试
it('only applies the latest search result', async () => {
  server.use(
    http.get('/api/search', ({ request }) => {
      const url = new URL(request.url)
      const q = url.searchParams.get('q')
      // 根据 query 返回不同数据
      return HttpResponse.json([
        { id: 1, title: `Result for ${q}` },
      ])
    }),
  )

  const { result, rerender } = renderHook(
    ({ query }) => useSearch(query),
    { initialProps: { query: 'first' } },
  )

  // 快速切换查询，第二次响应的数据应该覆盖第一次
  rerender({ query: 'second' })

  await waitFor(() => {
    expect(result.current[0].title).toBe('Result for second')
  })
})
```

### 乐观更新

```typescript
// Hook：乐观更新点赞状态
function useLike(postId: number) {
  const [liked, setLiked] = useState(false)
  const [error, setError] = useState<string | null>(null)

  const toggleLike = async () => {
    const previous = liked
    setLiked(!liked) // 先更新 UI
    setError(null)

    try {
      const res = await fetch(`/api/posts/${postId}/like`, {
        method: 'POST',
      })
      if (!res.ok) throw new Error('Failed to like')
    } catch {
      setLiked(previous) // 失败后回滚
      setError('Failed to update. Reverted.')
    }
  }

  return { liked, toggleLike, error }
}

// 测试
it('reverts liked state when request fails', async () => {
  server.use(
    http.post('/api/posts/:id/like', () => {
      return new HttpResponse(null, { status: 500 })
    }),
  )

  const { result } = renderHook(() => useLike(1))

  // 初始状态
  expect(result.current.liked).toBe(false)

  // 发起乐观更新
  await act(async () => {
    await result.current.toggleLike()
  })

  // 请求失败后回滚
  expect(result.current.liked).toBe(false)
  expect(result.current.error).toBe('Failed to update. Reverted.')
})

it('keeps liked state when request succeeds', async () => {
  server.use(
    http.post('/api/posts/:id/like', () => {
      return new HttpResponse(null, { status: 201 })
    }),
  )

  const { result } = renderHook(() => useLike(1))

  await act(async () => {
    await result.current.toggleLike()
  })

  expect(result.current.liked).toBe(true)
  expect(result.current.error).toBeNull()
})
```

> **自我验证说明**：
> - 竞态条件测试的关键是验证 `ignore` 闭包变量确实阻断了过时的响应。在 `rerender` 后，第一次请求的响应即使晚到也不应该覆盖第二次的结果。
> - 乐观更新测试需要 `act()` 包裹 `toggleLike()` 调用，因为该方法同时触发了 state 变更和异步请求。`act()` 确保 React 处理完所有 state 更新后再执行断言。
> - `HttpResponse.error()` 模拟网络错误（`TypeError`），而 `{ status: 500 }` 是 HTTP 错误。两者在错误处理中的路径不同，需要分别测试。

http.get('/api/users/:id', async ({ params }) => {
  const user = await database.findUser(Number(params.id))

## 7.8 本章总结

- 数据层测试分为三层：API Client（请求/响应正确性）、Data Hook（状态转换）、UI Component（用户看到的最终结果）
- MSW 让三层可以共享同一套 handler 定义，避免每层重复 mock
- API Client 测试覆盖四种场景：正常响应、空数据、HTTP 错误（4xx/5xx）、网络错误（TypeError）
- `renderHook` + `waitFor` 是测试 Data Hook 的核心组合
- MSW 的 resolver 函数是**请求验证和响应模拟的统一入口**：在 handler 内部同时完成"检查发出了什么"和"控制返回了什么"
- Handler 按 domain 组织，用工厂函数支持参数化响应，避免在测试中重复定义 handler
- 增强工厂模式（`createXxxHandler(scenario).verify(fn)`）将场景控制与请求验证解耦——工厂管"返回什么"，回调管"验证请求"
- 同一 API 多次调用的验证有四种方案：计数器（2-3次简单分支）、序列数组（3+次递增）、`vi.fn()`（精确验证每次参数）、`{ once: true }`（每次行为完全不同）
- 网络边界场景需要单独编写测试：超时用假定时器、重试用计数器、竞态用闭包变量、乐观更新用 `act()` 包裹
- 网络错误（`HttpResponse.error()` 触发 TypeError）和 HTTP 错误（4xx/5xx 状态码）是两种不同的错误类型，测试中必须区分
