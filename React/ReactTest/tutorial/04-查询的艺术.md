## 4.1 查询优先级详解

### 核心原则

> 你的查询越接近用户和辅助技术在页面中定位元素的方式，你的测试就越可信。

用户不会在脑子里跑 CSS 选择器。用户通过"那个写着'提交'的按钮"、"那个标签叫'邮箱'的输入框"来与页面交互。屏幕阅读器通过 ARIA role（按钮、标题、链接）和 accessible name（标签文本）来向视障用户描述页面。好的查询就是好的无障碍实践。

### 优先级表

| 优先级 | 查询 | 用户/辅助技术如何使用 | 何时降级 |
|--------|------|---------------------|---------|
| 1 | `getByRole` | 屏幕阅读器通过 role 和 accessible name 定位元素 | 元素缺乏语义 role（纯 div）或 name 无法区分多个同 role 元素 |
| 2 | `getByLabelText` | 用户通过 `<label>` 文本找到对应输入框 | 输入框无关联 label |
| 3 | `getByPlaceholderText` | placeholder 是视觉辅助提示 | label 不可用时的次选（placeholder 对屏幕阅读器支持不稳定） |
| 4 | `getByText` | 用户阅读页面上的文字来定位内容 | 交互元素优先用 role + name |
| 5 | `getByDisplayValue` | 用户看输入框中当前填的值 | 验证已填充表单的值 |
| 6 | `getByAltText` | 用户通过 alt 文本理解图像内容 | 仅适用于 `<img>`、`<area>` 等支持 alt 的元素 |
| 7 | `getByTitle` | title 属性展现为 tooltip | 无障碍支持差，最后一根稻草前的选择 |
| 8 | `getByTestId` | 用户无法感知 `data-testid` | 仅当语义查询完全不适用时（动态内容、非语义组件） |

### 为什么 getByRole 是第一选择

```typescript
// getByRole 同时验证了「类型」和「标签」
screen.getByRole('button', { name: 'Submit' })
//                ^^^^^^          ^^^^^^^^^^^^
//                元素类型          accessible name

// 相比之下，getByText 只验证了文本内容
screen.getByText('Submit') // 是个 button 还是 span？不确定

// getByTestId 什么都没有验证
screen.getByTestId('submit-btn') // 类型？标签？全不知道
```

`getByRole` 失败时的报错信息是第一优先级的调试工具：

```
Unable to find an accessible element with the role "button" and name "Submit"

Here are the accessible roles:

  heading "My Form" (name matched from content):
    <h1 />

  button "Cancel" (name matched from content):
    <button>Cancel</button>

  textbox "Email" (name matched from associated label):
    <input type="email" id="email" />
```

它不仅告诉你找不到什么，还列出了页面上所有可访问的元素——相当于自动生成的无障碍审计报告。

> **自我验证说明**：`getByRole` 的 `name` 选项匹配的是浏览器计算的 [accessible name](https://www.w3.org/TR/accname-1.2/)，计算规则包括：`aria-label` > `aria-labelledby` > 元素内容文本 > `title`。大多数情况下它就是你期望的文本。`name` 接受字符串或正则表达式。

> **Jest 对比：** React Testing Library 的查询 API 不依赖任何测试运行器——在 Jest 和 Vitest 中调用方式完全相同。然而，Vitest 的 watch 模式在查询失败的调试循环中优势明显：修改组件代码 → `getByRole` 失败 → 无障碍树打印在控制台 → 快速修复 → vitest HMR 只重新执行当前文件 → 几秒内闭环。Jest 的 watch 模式需要重新 transform 所有依赖——相同场景下可能需要 10-20 秒。

> **为什么：** 查询优先级不是主观的编码风格偏好——它源自 W3C 的 [Accessible Name and Description Computation](https://www.w3.org/TR/accname-1.2/) 标准。`getByRole` 查询的 `name` 选项匹配的是浏览器为屏幕阅读器计算的 accessible name。当一个查询用 `getByRole('button', { name: /submit/i })` 通过时，你不仅验证了按钮存在于 DOM 中，还验证了屏幕阅读器用户能正确地找到并理解这个按钮。一个难以用 `getByRole` 定位的组件，往往也意味着它对残障用户不友好——这是一个双向验证。

## 4.2 getBy、queryBy、findBy 的选择逻辑

### 三族 API 对照

```
          元素应该存在   元素应该不存在   异步等待元素出现
单元素    getBy*        queryBy*         findBy*
多元素    getAllBy*     queryAllBy*      findAllBy*
```

### 决策流程

```
你的测试需要验证：
│
├─ 元素应该存在，且现在是同步的？
│   → getBy* / getAllBy*
│
├─ 元素应该不存在？
│   → queryBy* / queryAllBy*
│
├─ 元素会在异步操作后出现？
│   → findBy* / findAllBy*
│
└─ 你不知道用哪个？
    → 先用 getByRole，失败了看报错信息
```

### 实战示例

```typescript
import { render, screen, within } from '@testing-library/react'

describe('UserList', () => {
  it('renders users when data loads', async () => {
    render(<UserList />)

    // 加载中：loading 指示器应该存在
    expect(screen.getByText(/loading/i)).toBeInTheDocument()

    // 加载完：loading 应该消失
    await waitForElementToBeRemoved(() => screen.queryByText(/loading/i))

    // 数据列表应该出现
    const users = await screen.findAllByRole('listitem')
    expect(users).toHaveLength(3)
  })

  it('shows empty message when no users', async () => {
    // 覆盖 MSW handler 返回空数组
    render(<UserList />)

    // 异步等待空状态出现
    const emptyMsg = await screen.findByText(/no users found/i)
    expect(emptyMsg).toBeInTheDocument()

    // 确认列表不存在
    expect(screen.queryByRole('list')).not.toBeInTheDocument()
  })
})
```

### 关键规则

1. **`findBy*` = `waitFor` + `getBy*` 的语法糖**——用 `findBy*` 替代 `waitFor(() => getBy*())`
2. **用 `queryBy*` 断言元素不存在**——`getBy*` 在找不到时就抛错，无法用于"应该不存在"的断言
3. **用 `findBy*` 等待异步元素**——比 `waitFor` 更简洁
4. **`findByRole` 是最安全的异步查询**——失败时同样会输出无障碍树

> **自我验证说明**：RTL 的 `findBy*` 超时默认 1000ms，可通过 `{ timeout: 3000 }` 选项调整。`waitFor` 超时默认也是 1000ms。这两个值在集成测试中通常需要调大（推荐 3000-5000ms），因为网络请求需要时间。参考 [RTL async docs](https://testing-library.com/docs/dom-testing-library/api-async/)。

## 4.3 render 方法精讲

### 完整签名

```typescript
import { render, RenderOptions, RenderResult } from '@testing-library/react'

function render(
  ui: React.ReactElement,
  options?: RenderOptions
): RenderResult
```

### RenderOptions

```typescript
interface RenderOptions {
  // 包裹组件的 Provider（最常用）
  wrapper?: React.JSXElementConstructor<{ children: React.ReactNode }>

  // 自定义查询（高级用法，通常不需要）
  queries?: Queries

  // hydrate 模式（SSR 测试，通常不需要）
  hydrate?: boolean

  // 渲染容器（默认 document.body 中的 div）
  container?: HTMLElement

  // React DOM 渲染目标（默认 container）
  baseElement?: HTMLElement
}
```

### RenderResult

```typescript
interface RenderResult {
  // 所有 query 方法（已绑定 container）
  ...queries  // getByRole, queryByText, findByLabelText, ...

  // DOM 节点
  container: HTMLElement
  baseElement: HTMLElement

  // 工具方法
  debug(element?: HTMLElement): void       // 打印当前 DOM（调试神器）
  rerender(ui: React.ReactElement): void    // 用新 props 重新渲染
  unmount(): void                            // 卸载组件
  asFragment(): DocumentFragment             // 获取 DOM 片段
}
```

### wrapper —— 最关键的选项

```typescript
import { render } from '@testing-library/react'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

// 场景：你的组件依赖 QueryClientProvider
const queryClient = new QueryClient({
  defaultOptions: { queries: { retry: false } },
})

function renderWithProviders(ui: React.ReactElement) {
  return render(ui, {
    wrapper: ({ children }) => (
      <QueryClientProvider client={queryClient}>
        {children}
      </QueryClientProvider>
    ),
  })
}

// 使用
it('fetches and displays data', async () => {
  renderWithProviders(<UserProfile />)
  expect(await screen.findByText('Alice')).toBeInTheDocument()
})
```

> **自我验证说明**：`wrapper` 是一个 React 组件，接收 `{ children }` 作为 props。每次 `render` 都会创建新的 wrapper 实例，因此不同测试之间的 Provider 状态不会互相污染。

> **为什么：** 使用 `wrapper` 而非在每个测试中手动包裹组件的核心原因是维护性。当项目引入一个新的 Provider（如 ThemeProvider、AuthProvider），使用 `wrapper` 集中管理意味着只需修改一个文件。如果 Provider 之间存在依赖关系（如 React Query 依赖 Router 提供 location），`wrapper` 中的嵌套顺序也只需在一处维护。没有 `wrapper` 时，每个新增 Provider 都需要修改所有涉及组件渲染的测试——这在 100+ 测试的项目中是不可接受的维护成本。

### debug() —— 你的调试助手

```typescript
it('debugging example', () => {
  render(<ComplexForm />)

  // 打印当前 DOM（等同于 console.log(container.innerHTML)）
  screen.debug()

  // 只打印特定元素
  const form = screen.getByRole('form')
  screen.debug(form)

  // 限制打印长度
  screen.debug(undefined, 200) // 最多 20000 字符（默认 7000）
})
```

### 延伸：自定义 queries（高级场景）

RTL 的 8 级查询优先级覆盖了 95% 的场景。但在某些特殊场景下——例如检测一个元素是否包含特定 data 属性、或对第三方组件的渲染结果做结构验证——你可能需要自定义查询：

```typescript
import { buildQueries, within } from '@testing-library/react'

// 创建一个自定义 query：查找包含特定 data-test-category 的元素
const queryAllByCategory = (container: HTMLElement, category: string) =>
  Array.from(container.querySelectorAll(`[data-test-category="${category}"]`))

const [
  queryByCategory,
  getAllByCategory,
  getByCategory,
  findAllByCategory,
  findByCategory,
] = buildQueries(
  queryAllByCategory,
  (c) => `Found multiple elements with category`,
  (c) => `Could not find element with category`
)

// 在测试中使用
it('finds elements by custom category', () => {
  render(<Dashboard />)
  const charts = getByCategory(document.body, 'charts')
  expect(charts).toBeInTheDocument()
})
```

> **自我验证说明：** `buildQueries` 返回标准的 `getBy`/`getAllBy`/`queryBy`/`findBy`/`findAllBy` 五个方法——与 RTL 内置查询的方法签名一致。自定义查询应谨慎使用：大多数场景下，添加 `aria-label` 或 `role` 后使用 `getByRole` 比自定义查询更符合 RTL 的哲学。自定义查询的真正价值在于：处理第三方组件库的内部结构（如 MUI 的 Select 渲染出的 DOM 层级）或多语言文本的动态查找。参考 [RTL 文档 - Custom Queries](https://testing-library.com/docs/react-testing-library/custom-queries/)。

## 4.4 screen 与 within

### screen —— 全局视角

```typescript
import { screen } from '@testing-library/react'

// screen 拥有所有 query 方法，且自动绑定 document.body
screen.getByRole('button', { name: /submit/i })
screen.queryByText(/no results/i)
screen.findByLabelText(/email/i)
screen.getAllByRole('listitem')
```

`screen` 的优势：
- 不需要从 `render()` 的解构中取出 query 方法
- 测试中新增查询时不需要回去修改解构语句
- `screen.debug()` 比 `const { debug } = render()` 更方便

### within —— 限定范围

```typescript
import { within } from '@testing-library/react'

it('checks items within a specific section', () => {
  render(<Dashboard />)

  // 限定查询范围到侧边栏
  const sidebar = screen.getByRole('complementary')
  expect(within(sidebar).getByText('Menu')).toBeInTheDocument()

  // 限定查询范围到表格
  const table = screen.getByRole('table')
  const rows = within(table).getAllByRole('row')
  expect(rows).toHaveLength(4) // header + 3 data rows
})
```

`within` 的典型场景：
- 表单分组（fieldset 内的字段）
- 列表项内的子元素
- 对话框内的按钮（区分对话框内外的同名按钮）
- 表格行内的单元格

## 4.5 常用断言模式速查

```typescript
// === 存在性 ===
expect(element).toBeInTheDocument()
expect(element).toBeVisible()
expect(element).not.toBeVisible()

// === 文本内容 ===
expect(element).toHaveTextContent('Hello World')
expect(element).toHaveTextContent(/hello/i)
expect(input).toHaveValue('user input')

// === 属性 ===
expect(link).toHaveAttribute('href', '/home')
expect(element).toHaveClass('active')
expect(element).toHaveStyle('color: rgb(255, 0, 0)')

// === 交互状态 ===
expect(button).toBeDisabled()
expect(button).toBeEnabled()
expect(checkbox).toBeChecked()
expect(checkbox).not.toBeChecked()
expect(input).toBeRequired()
expect(element).toHaveFocus()

// === 结构 ===
expect(list).toContainElement(specificItem)
expect(list).toContainHTML('<span>hello</span>') // 慎用：实现细节
expect(container).toBeEmptyDOMElement()
```

> **自我验证说明**：`toHaveStyle` 接收的是计算机样式（computed style）而非 CSS 中写的值。`rgb(255, 0, 0)` 而非 `red`、`#ff0000`。如果样式不匹配，检查是否写了 CSS 简写形式（浏览器计算后会展开）。

## 4.6 反模式警示

### 反模式 1：container.querySelector

```typescript
// ❌
const { container } = render(<LoginForm />)
const btn = container.querySelector('.submit-btn')

// ✅
const btn = screen.getByRole('button', { name: /submit/i })
```

### 反模式 2：getByTestId 作为默认选择

```typescript
// ❌ 偷懒：全部用 testid
screen.getByTestId('header')
screen.getByTestId('header-title')
screen.getByTestId('header-title-text')

// ✅ 语义优先
screen.getByRole('banner')                      // <header>
within(screen.getByRole('banner')).getByRole('heading', { name: /title/i })
```

### 反模式 3：getByText 查找交互元素

```typescript
// ❌ getByText 不验证交互语义
screen.getByText('Delete')

// ✅ getByRole 验证了它是一个按钮，并且叫 "Delete"
screen.getByRole('button', { name: /delete/i })
```

### 反模式 4：测试中操作 container 做断言

```typescript
// ❌ 手动比较 innerHTML
const { container } = render(<ItemList items={[]} />)
expect(container.innerHTML).toBe('')

// ✅ 语义断言
expect(screen.queryByRole('listitem')).not.toBeInTheDocument()
```

---

> **Jest 对比：** 在 Enzyme 时代，最常见的做法是 `component.find('[data-testid="submit"]')`——通过 data-testid 直接定位 DOM 节点。迁移到 RTL 后，`data-testid` 从"默认选择"降级为"最后手段"。这种变化不是工具层面的，而是哲学层面的：你不再把测试标识符注入 DOM，而是利用浏览器本身的无障碍结构（ARIA role、label、文本内容）来定位元素。从 Enzyme + Jest 迁移到 RTL + Vitest 时，最大的变化不是测试运行器，而是查询哲学——所有以 `data-testid` 开头的查询都需要被语义查询替换。

## 4.7 渐进式示例：从初学者到工匠级

### 示例 1：基础按钮

```typescript
// 组件
function Greeting() {
  return <button onClick={() => alert('Hello!')}>Say Hello</button>
}

// 测试
it('renders greeting button', () => {
  render(<Greeting />)
  const button = screen.getByRole('button', { name: /say hello/i })
  expect(button).toBeInTheDocument()
})
```

> 一个查询就够。`button` role + `name` 匹配。

### 示例 2：同名多按钮

```typescript
// 组件
function TodoList() {
  return (
    <ul>
      <li>Buy milk <button>Delete</button></li>
      <li>Walk dog <button>Delete</button></li>
      <li>Read book <button>Delete</button></li>
    </ul>
  )
}

// 测试
it('deletes a specific todo item', () => {
  render(<TodoList />)

  // 找第二个 Delete 按钮
  const items = screen.getAllByRole('listitem')
  const deleteBtn = within(items[1]).getByRole('button', { name: /delete/i })

  // 或者：直接用 accessible name（aria-label）
  // <button aria-label="Delete Walk dog">Delete</button>
  // screen.getByRole('button', { name: /delete walk dog/i })
})
```

> `within` 限定范围是区分同名元素的首选方案。添加 `aria-label` 是更彻底的无障碍优化。

### 示例 3：异步列表加载

```typescript
// 组件（使用 useEffect + fetch 加载数据）
function UserList() {
  const [users, setUsers] = useState<User[]>([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    fetch('/api/users')
      .then((r) => r.json())
      .then((data) => {
        setUsers(data)
        setLoading(false)
      })
  }, [])

  if (loading) return <p>Loading...</p>
  if (users.length === 0) return <p>No users found</p>
  return (
    <ul>
      {users.map((u) => (
        <li key={u.id}>{u.name}</li>
      ))}
    </ul>
  )
}

// 测试
it('displays users after loading', async () => {
  render(<UserList />)

  // loading 状态同步检查
  expect(screen.getByText(/loading/i)).toBeInTheDocument()

  // 等待数据出现（隐式 waitFor）
  const userItems = await screen.findAllByRole('listitem')

  expect(userItems).toHaveLength(2)
  expect(userItems[0]).toHaveTextContent('Alice')
})
```

> `findAllByRole` 内部轮询 DOM，直到元素出现或超时。

### 示例 4：动态表单

```typescript
// 组件
function AddressForm() {
  return (
    <form>
      <fieldset>
        <legend>Shipping Address</legend>

        <label htmlFor="street">Street</label>
        <input id="street" type="text" />

        <label htmlFor="city">City</label>
        <input id="city" type="text" />

        <label htmlFor="zip">ZIP Code</label>
        <input id="zip" type="text" />
      </fieldset>

      <fieldset>
        <legend>Billing Address</legend>

        <label htmlFor="billing-street">Street</label>
        <input id="billing-street" type="text" />
      </fieldset>
    </form>
  )
}

// 测试
it('has isolated shipping and billing sections', () => {
  render(<AddressForm />)

  // 用 fieldset 的 legend 做分组隔离
  const shipping = screen.getByRole('group', { name: /shipping address/i })
  const billing = screen.getByRole('group', { name: /billing address/i })

  // shipping 里有三个输入框
  expect(within(shipping).getAllByRole('textbox')).toHaveLength(3)

  // billing 里有一个
  expect(within(billing).getAllByRole('textbox')).toHaveLength(1)

  // 精确查找：通过关联的 label 文本
  expect(
    within(shipping).getByLabelText(/street/i)
  ).toBeInTheDocument()
  expect(
    within(billing).getByLabelText(/street/i)
  ).toBeInTheDocument()
})
```

> `<fieldset>` 创建了 `group` role，`<legend>` 提供了 accessible name。`within` + `getByLabelText` 的组合让你可以精确区分同名字段。

### 示例 5：复杂界面——综合运用

```typescript
// 组件
function ProductDashboard() {
  return (
    <main>
      <header>
        <h1>Products</h1>
        <button aria-label="Add product">+</button>
      </header>

      <table>
        <thead>
          <tr>
            <th>Name</th>
            <th>Price</th>
            <th>Actions</th>
          </tr>
        </thead>
        <tbody>
          <tr>
            <td>Widget</td>
            <td>$9.99</td>
            <td>
              <button aria-label="Edit Widget">Edit</button>
              <button aria-label="Delete Widget">Delete</button>
            </td>
          </tr>
          <tr>
            <td>Gadget</td>
            <td>$19.99</td>
            <td>
              <button aria-label="Edit Gadget">Edit</button>
              <button aria-label="Delete Gadget">Delete</button>
            </td>
          </tr>
        </tbody>
      </table>
    </main>
  )
}

// 测试
it('provides accessible product actions', () => {
  render(<ProductDashboard />)

  const table = screen.getByRole('table')
  const rows = within(table).getAllByRole('row')

  // 第一行数据（跳过 header）
  const widgetRow = rows[1]
  expect(within(widgetRow).getByText('Widget')).toBeInTheDocument()
  expect(within(widgetRow).getByText('$9.99')).toBeInTheDocument()

  // 通过 aria-label 找到精确的操作按钮
  expect(
    within(widgetRow).getByRole('button', { name: /edit widget/i })
  ).toBeInTheDocument()
  expect(
    within(widgetRow).getByRole('button', { name: /delete widget/i })
  ).toBeInTheDocument()
})
```

> 这个示例综合了 `within`、`role`、`aria-label`、`getByText`。注意这里没有使用任何 `data-testid`——每个元素都通过语义信息被定位。

### 自我验证说明

以上 5 个示例基于 RTL 官方查询优先级和 accessible name 计算规范。关键点：
- 示例 2（同名按钮）：`within` + `getByRole` 是正确的区分方式。不推荐用 `getAllByRole('button')[1]`——索引不稳定，DOM 结构一变就坏
- 示例 3（异步加载）：`findAllByRole` 的 timeout 默认 1000ms。MSW 的 handler 是同步模拟的，所以数据几乎瞬间返回。真实测试中如果超时，检查 MSW handler 是否匹配了请求
- 示例 4（表单）：`fieldset` 的 `role` 是 `'group'`，`legend` 作为其 accessible name
- 示例 5（复杂界面）：`aria-label` 为图标按钮提供了 accessible name。如果没有 `aria-label`，这些按钮将只有空 name

### 示例 6：表单验证（getByRole + queryByRole 组合）

```typescript
function ValidationForm() {
  const [errors, setErrors] = useState<Record<string, string>>({})

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault()
    const formData = new FormData(e.currentTarget as HTMLFormElement)
    const newErrors: Record<string, string> = {}
    if (!formData.get('username')) newErrors.username = 'Username is required'
    if (!formData.get('password')) newErrors.password = 'Password is required'
    setErrors(newErrors)
  }

  return (
    <form onSubmit={handleSubmit}>
      <label htmlFor="username">Username</label>
      <input id="username" aria-invalid={!!errors.username} />
      {errors.username && <p role="alert">{errors.username}</p>}

      <label htmlFor="password">Password</label>
      <input id="password" type="password" aria-invalid={!!errors.password} />
      {errors.password && <p role="alert">{errors.password}</p>}

      <button type="submit">Register</button>
    </form>
  )
}

it('shows validation errors on empty submit', () => {
  render(<ValidationForm />)

  // queryByRole 确认错误不存在
  expect(screen.queryByRole('alert')).not.toBeInTheDocument()

  fireEvent.click(screen.getByRole('button', { name: /register/i }))

  // getAllByRole 批量验证多个错误
  const alerts = screen.getAllByRole('alert')
  expect(alerts).toHaveLength(2)
  expect(alerts[0]).toHaveTextContent(/username is required/i)
  expect(alerts[1]).toHaveTextContent(/password is required/i)
})
```

> 这个示例展示了 `queryByRole`（验证元素不存在）和 `getAllByRole`（批量验证多个同 role 元素）的组合。每个错误使用 `role="alert"` 确保屏幕阅读器能及时播报错误。

### 示例 7：动态搜索建议（findByRole + within 组合）

```typescript
function SearchWithSuggestions() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState<string[]>([])

  useEffect(() => {
    if (query.length < 2) {
      setResults([])
      return
    }
    fetch(`/api/search?q=${query}`)
      .then(r => r.json())
      .then(data => setResults(data.suggestions))
  }, [query])

  return (
    <div>
      <label htmlFor="search-input">Search</label>
      <input id="search-input" onChange={e => setQuery(e.target.value)} value={query} />
      {results.length > 0 && (
        <ul role="listbox" aria-label="Suggestions">
          {results.map((r, i) => (
            <li key={i} role="option">{r}</li>
          ))}
        </ul>
      )}
    </div>
  )
}

it('shows suggestions after typing enough chars', async () => {
  const user = userEvent.setup()
  render(<SearchWithSuggestions />)

  const input = screen.getByLabelText(/search/i)

  // 不足 2 字符不应有建议
  await user.type(input, 'a')
  expect(screen.queryByRole('listbox')).not.toBeInTheDocument()

  // 继续输入触发异步查询
  await user.type(input, 'l')

  // findByRole 等待异步结果
  const listbox = await screen.findByRole('listbox', {}, { timeout: 2000 })
  expect(listbox).toBeInTheDocument()

  const options = within(listbox).getAllByRole('option')
  expect(options.length).toBeGreaterThan(0)
})
```

> 这个示例综合了 `getByLabelText`、`queryByRole`、`findByRole` 和 `within` 四种查询方法。`findByRole` 在异步数据场景下等效于 `waitFor(() => getByRole(...))` 但语法更简洁。注意 `findByRole` 的第三个参数可传入 `{ timeout }` 覆盖默认的 1000ms 超时。

## 4.9 本章总结

- 查询优先级：`getByRole` > `getByLabelText` > `getByPlaceholderText` > `getByText` > `getByDisplayValue` > `getByAltText` > `getByTitle` > `getByTestId`
- 存在 → `getBy*`，不存在 → `queryBy*`，异步 → `findBy*`
- `screen` 全局查询，`within` 限定范围
- `wrapper` 注入 Provider，`debug()` 打印 DOM
- 测试行为通过断言 DOM 状态，不通过 `container.querySelector`
- 好查询 = 好无障碍。如果你的组件难以用语义查询定位，它可能对屏幕阅读器用户也不友好——这是一个双赢的信号
