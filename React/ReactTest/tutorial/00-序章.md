# 序章：测试的思维模型

## 学习目标

- 理解测试在前端工程中的真正价值
- 掌握测试奖杯（Testing Trophy）模型及其与传统金字塔的区别
- 理解本书技术栈（Vitest + RTL + MSW + TypeScript）的设计理由
- 建立「测试是设计反馈系统」的思维模型

---

## 我们为什么写测试

如果你问十个开发者「为什么要写测试」，九个人会回答「为了找 bug」。这个答案不算错，但它遗漏了测试更重要的价值。

测试首先是一个**设计反馈系统**。

想象一下：你写了一段代码，如果不通过测试来「使用」它，你怎么知道它的接口设计是否合理？怎么知道调用方需要写多少样板代码才能用上你的函数？怎么知道边界条件是否被正确处理？

测试强迫你**以使用者的视角审视自己的代码**。当你发现为某个组件写测试很痛苦时——需要 mock 十几个依赖、需要设置复杂的状态、需要在测试中等待多层异步——那通常不是测试的问题，而是**组件设计的问题**。

> 测试难写 → 代码耦合度高 → 重构使其可测试 → 设计变好。测试不是目的，好设计才是。

测试的第二个价值是**让重构变得安全**。没有测试的重构就像在没有安全网的情况下走钢丝——你永远不知道哪一步会让你坠入 bug 的地狱。有了可靠的测试套件，重构变成了一种愉快的活动：改代码、跑测试、绿灯、提交。

测试的第三个价值才是**捕获回归 bug**。这发生在 CI 中——当某个 PR 不小心破坏了已有功能，测试的红灯会阻止它进入生产环境。

所以正确的顺序是：**设计反馈 → 重构安全 → 回归捕获**。

---

## 什么是「好的测试」

好的测试有一个黄金标准：**如果重构了实现细节但行为不变，测试应该不修改就仍然通过**。

换句话说，测试应该与**行为**耦合，而不是与**实现**耦合。这是一个很容易说但很难做的原则。

一个反面例子：

```typescript
// 差：测试实现细节
test('setLoading is called with true then false', () => {
  const setLoading = vi.fn()
  render(<UserList onLoadingChange={setLoading} />)
  expect(setLoading).toHaveBeenCalledWith(true)
  expect(setLoading).toHaveBeenCalledWith(false)
})
```

这个测试验证的不是「用户能看到什么」，而是「内部回调是否以特定顺序被调用」。如果你后来把 loading 状态改成内部管理，测试就失败了——但用户看到的东西完全没变。

一个好的版本：

```typescript
// 好：测试用户看到的
test('shows loading indicator while fetching', async () => {
  render(<UserList />)
  expect(screen.getByRole('progressbar')).toBeInTheDocument()
  expect(await screen.findByText('Ada Lovelace')).toBeInTheDocument()
  expect(screen.queryByRole('progressbar')).not.toBeInTheDocument()
})
```

好测试的特征：

1. **稳定**：不应该在无关的代码变动时失败
2. **快速**：应该在几毫秒到几秒内完成，否则开发者不会频繁运行
3. **可读**：一个陌生人应该能从测试名称和内容理解被测行为
4. **独立**：测试之间不应有依赖或执行顺序要求

## 测试奖杯：前端测试的新模型

传统的测试金字塔（单元测试多 → 集成测试中等 → E2E 测试少）对后端服务很适用，但对前端不够准确。前端框架的核心价值是**将数据转化为用户界面**——这是一个集成行为，而不是孤立的单元行为。

Kent C. Dodds 提出的**测试奖杯（Testing Trophy）**更适合前端：

```
        ┌───────────┐
        │    E2E     │  Playwright / Cypress
       ┌┴───────────┴┐    少量，关键用户流程
       │   E2E (少)   │
    ┌──┴───────────────┴──┐
    │   Integration (多)   │  Vitest + RTL + MSW
   ┌┴───────────────────────┴┐  最大投资层
   │      Unit (中等)         │  Vitest（hooks/utils）
  ┌┴───────────────────────────┴┐
  │  Static Analysis (全是)      │  TypeScript + ESLint
 └───────────────────────────────┘
```

各层的分工：

| 层 | 工具 | 测试什么 | 数量 |
|---|---|---|---|
| Static | TypeScript strict + ESLint | 类型错误、语法错误、常见 bug 模式 | 无处不在 |
| Unit | Vitest | 纯逻辑函数、工具函数、简单 Hook | 中等 |
| Integration | Vitest + RTL + MSW | 组件 + API 调用 + 用户交互 | **最多** |
| E2E | Playwright | 关键用户流程（登录、支付、核心功能） | 少量 |

**关键洞察**：在前端，集成测试的投入产出比最高。一个集成测试——渲染组件、触发 API 调用（MSW 拦截）、模拟用户操作、断言 UI 结果——可以覆盖几十行组件代码和多个模块交互。它是从用户视角验证「这玩意儿能正常工作」的最直接方式。

## 本书技术栈与选择理由

### 核心组合：Vitest + React Testing Library + MSW + TypeScript

**Vitest 为什么替代 Jest？**

Vite 已经成为 React 项目的事实标准构建工具，而 Vitest 是它的天然伴侣。具体数据：

- 冷启动快约 27 倍（0.3s vs 8s）
- 完整测试套件执行快约 10 倍
- 配置文件只需约 18 行（vs Jest 的 50+ 行 + 8-10 个依赖）
- 原生 TypeScript 支持，无需 ts-jest 或 babel-jest
- 原生 ESM 支持，无需复杂转译
- Watch 模式基于 Vite 的 HMR，精准重跑受影响的测试
- 与 vite.config.ts 共享配置，避免路径解析不一致

但 Vitest 的真正优势不在速度——在于**测试行为与生产行为的一致**。当 Vitest 和 Vite 共享同一个配置、同一个 esbuild 转译管道、同一个路径解析时，你在测试中看到的和你最终部署的不会有「Jest 下能过，生产上炸了」的谜之差异。

**React Testing Library 的设计哲学**

RTL 的核心原则简单得近乎粗暴：**从用户的角度测试你的应用**。用户不看你的 state、不看你的 props、不看你的组件内部方法。用户只看屏幕上有什么、能点击什么、能输入什么、点击/输入后发生了什么。

这导致了一个根本性的法则——查询优先级：

```
getByRole          ← 最高：模拟屏幕阅读器/无障碍体验
  getByLabelText   ← 表单字段
  getByPlaceholderText ← 输入框
  getByText        ← 非交互文本
  getByDisplayValue ← 表单当前值
  getByTestId      ← 最后手段（万不得已）
```

你会在第三章深入学习每个查询的用法。现在只需记住：**如果你用 `container.querySelector('.my-class')` 写了测试，停下来，想一想用户能不能「看到」这个类名。答案是不能。**

**MSW 的设计哲学**

传统的 API Mock 方式是替换模块——`vi.mock('./api')` 或 `vi.spyOn(axios, 'get')`。这有几个问题：

1. 你的 mock 绑定了**具体 HTTP 客户端**（换了 axios 就要重写所有 mock）
2. 你只 mock 了函数的返回值，没有 mock 真实的请求-响应周期
3. 无法测试请求体、请求头、状态码处理等真实 HTTP 行为

MSW 的解决方案是：**在网络层拦截请求**。你的 fetch/axios/ky/React Query 代码完全不知道自己在和 mock 打交道——它发出真实请求，MSW 在请求离开进程之前将其拦截并返回定义好的响应。

这意味着：

- 你写的测试代码与生产代码**完全一致**，没有替换任何模块
- 换了 HTTP 客户端？测试不需要改
- 可以测试真实的状态码处理、Header 解析、错误响应
- **同一套 handler 可以同时用于开发、测试、Storybook 和调试**

## 如何使用本书

本书按以下结构组织：

1. **序章**（你正在读的这个）：建立思维框架
2. **第一至四章**：环境搭建、Vitest 基础、组件测试原则、用户事件
3. **第五至六章**：MSW 深度使用、数据层集成测试
4. **第七至八章**：Hook 测试、表单/路由/状态管理
5. **第九至十章**：测试可维护性、CI 实践
6. **第十一至十二章**：Jest 迁移、进阶主题

每章包含：
- **学习目标**：阅读前就知道要掌握什么
- **核心概念**：理论讲解 + 代码示例
- **实战示例**：完整的、可运行的 TypeScript 代码
- **常见反模式**：该避开的坑
- **练习与思考**：巩固所学

**代码约定**：

- 所有代码默认在严格 TypeScript + Vitest + jsdom 环境中运行
- `vi` 全局可用（`globals: true`）
- 每个代码块后附「自我验证说明」，标注导入路径、函数签名来源和前提条件
- 示例使用 `@/` 路径别名（指向 `src/`）

> **自我验证说明规范**：由于本书编写环境无法实际运行测试，每个代码示例都经过了以下验证流程：(1) 交叉对照 Vitest / RTL / MSW 官方最新文档确认 API 签名；(2) 对照 bulletproof-react 等真实项目的测试代码确认导入路径和配置模式；(3) TypeScript 类型推演确认泛型参数和返回值类型正确。

## 本章总结

- 测试首先是**设计反馈系统**，其次才是 bug 发现工具
- 好的测试与行为耦合，不与实现耦合
- 前端测试应遵循**测试奖杯模型**：集成测试（RTL + MSW）是最大投资层
- Vitest 替代 Jest 的核心优势：速度、配置简化、与 Vite 的行为一致性
- RTL 的核心理念：像用户一样测试你的应用
- MSW 的核心理念：在网络层拦截，让测试运行在真实环境中
