# 3.1 · RAG 提示工程

检索拿到了 5 个 chunk，LLM 要用它们回答问题。同样的检索结果，不同的 prompt 设计，答案质量天差地别。

RAG prompt 的根本张力在信息量和注意力之间——塞的上下文越多，信息越丰富，但每个 chunk 获得的平均注意力越低。这个张力不可消除，只能管理。

标准 RAG prompt 的结构很自然：system 指令（"根据以下参考资料回答"）→ 检索结果（chunks，标注来源）→ 用户问题。但每个部分都有陷阱。

**指令措辞**。最关键的决策是"如果资料中没有答案该怎么办"。两个极端：(a) 强制只用资料回答→ 资料不完整时模型会硬编；(b) "如果找不到就说找不到"→ 可能过于保守，资料部分相关时也说"找不到"。最佳实践是三层指令："如果资料包含答案，直接回答并引用来源"→"如果资料只包含部分信息，回答已知部分并说明缺失"→"如果完全无法回答，明确说明"。

**Chunk 呈现**。每个 chunk 必须有清晰边界和来源标识。`[来源 N: 文件路径]` 有两个作用：显式作用是让 LLM 知道来源，隐式作用是让注意力机制聚焦——清晰的边界标记帮助模型区分不同来源和避免张冠李戴。

**Chunk 排序**。表面上看应该按相关性从高到低排。但 "Lost in the Middle"（Liu et al., 2023）表明 LLM 对上下文中间位置的信息处理能力显著低于开头和结尾。所以最优排列可能是 U 型——最相关的放开头和结尾，次要的放中间。

**不要塞原文，先给摘要**。如果 chunk 很长或数量多，先对每个 chunk 生成一句话摘要放在 prompt 开头作为"信息地图"，后面再放完整 chunk。这符合人类的阅读习惯——先看目录，再精读章节。

对你的向量记忆库，MCP `search_notes` 返回的每个结果已包含 `source` 和 `metadata`。Claude Code 在消费检索结果时，chunk 的呈现和排序由其自身的上下文处理机制决定——你能控制的是返回结果的**粒度**（chunk 不要太大）和**来源标注**（每个结果必须有明确的文件路径）。

## 思考题

1. "请根据参考资料回答"和"请参考以下资料，运用你的知识回答"——这两个 prompt 有什么本质区别？第二个什么时候是危险的？
2. 如果你的检索结果中有一段的观点和你笔记中的主流观点相反，把它放进 prompt 会发生什么？你怎么用 prompt 指令引导 LLM 正确处理这种矛盾？
