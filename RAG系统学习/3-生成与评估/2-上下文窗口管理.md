# 3.2 · 上下文窗口管理

检索结果多了用不完，少了不够用。上下文窗口 = LLM 的工作记忆——容量大但不均匀，注意力分布偏向前后两端。

塞多少（Top-K）不是一个"越多越好或越少越好"的问题。K 太小 → 可能漏掉关键信息。K 太大 → 信噪比下降。FiD 的实验提供了一个锚点：在 Natural Questions 上，段落数从 5 到 25 持续提升性能，25 之后边际收益递减。这意味着"K 大 = 噪声多"的直觉在实验上并不总是成立——cross-attention 有能力在多个文档中定位相关信息。经验法则：定义类问题 3-5 个 chunk，综合性问题至少 10-25 个。

按什么顺序排列受 "Lost in the Middle"（Liu et al., 2023）的约束：LLM 对上下文中间位置的注意力显著衰减。最优排列是 U 型——最相关放开头和结尾，次要的放中间。或按来源逻辑顺序——如果 chunks 来自同一文档的不同部分且存在叙事连贯性，尊重原文顺序可能比按相关性排列更好。

什么时候压缩而非塞原文？判断标准不是 chunks 的数量，而是它们占上下文窗口的比例。超过窗口的 50% 时考虑压缩。压缩是有损操作——摘要丢掉了措辞细节、精确数字、引用出处。精确事实查询 → 原文；概念理解查询 → 摘要。

对你的向量记忆库，MCP `search_notes` 的 `top_k` 默认值建议为 5，但应该允许 Claude Code 显式传递更大的值——对于需要跨笔记综合分析的问题，5 可能不够。

## 思考题

1. FiD 实验显示段落数在 25 之前持续提升——这和"K 太大 = 噪声淹没信号"的直觉矛盾吗？在什么条件下这个结论不成立？
2. U 型排列和"最重要的放最前面"的直觉——它们的矛盾可以被调和吗？什么时候应该放弃 U 型排列？
