# 3.4 · RAG 评估维度与指标

没有可靠的评估体系，所有优化都在黑箱里掷骰子——你换了切分策略，调了 prompt，改了 top_k，但不知道系统到底是变好了还是变坏了。

RAG 评估的根本挑战是没有标准的"正确答案"。对于"深模块的核心思想是什么"，多段文本都能合理回答。传统 IR 的精确率/召回率需要人工标注的相关性判断——成本太高。

[RAGAS](https://docs.ragas.io/) 用 LLM 作为评判者，在不需要人工标注的情况下评估 RAG。四个指标覆盖了 95% 以上的评估需求：

**Faithfulness（忠实度）**衡量答案中的每条陈述是否能在检索到的上下文中找到依据。LLM 将回答分解为原子声明，逐条验证。`faithfulness = 被验证声明数 / 总声明数`，目标 0.90-1.00。这是最重要的指标——直接衡量幻觉。

**Context Precision（上下文精确度）**衡量检索回来的 chunks 里有多少是真正有用的。LLM 评判每个 chunk 与问题的相关性，排名靠前的权重更高。目标 0.85-0.98。衡量信号/噪声比。

**Context Recall（上下文召回率）**衡量应该被检索到的信息是否都找到了。从参考答案中提取关键陈述，检查是否被检索上下文覆盖。目标 0.70-0.95。和 Precision 互补——Precision 衡量"找回来的东西有多干净"，Recall 衡量"该找的都找了吗"。

**Answer Relevancy（回答相关性）**衡量回答是否直接针对问题。LLM 根据回答生成反向问题，计算反向问题与原始查询的相似度。目标 0.88-0.99。

两个常见误诊：(1) 把 Context Recall 和 Answer Relevancy 混为一谈——前者衡量检索，后者衡量生成。(2) 以为 Faithfulness 高就万事大吉——完全可能忠实于检索内容但答非所问（Answer Relevancy 低）。

对你的向量记忆库，目前没有任何评估机制。在实现 MCP 服务器后，应该建立一个小型评估集（10-20 个典型查询 + 参考答案），每次调整（切分策略、嵌入模型、top_k）后跑一次 RAGAS。用数据而非直觉做决策。

## 思考题

1. RAGAS 用 LLM 评判 LLM——这种"LLM-as-judge"方式的潜在风险是什么？评估 LLM 的偏见会怎么影响你的优化方向？
2. 如果你的评估集只有 10 条查询，四个指标中有两个变好了、两个变差了——你怎么判断这次改动是"值得的"？
