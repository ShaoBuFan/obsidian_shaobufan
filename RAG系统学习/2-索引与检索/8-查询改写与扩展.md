# 2.8 · 查询改写与扩展

前面讨论的所有检索优化都假设用户的查询是合理的。但真实查询往往很短、很模糊、用的是和文档不同的措辞。"深模块是啥"和文档里的"深层模块的核心特征是接口远小于实现"之间隔着一个词汇-语义鸿沟——用户用日常语言提问，文档用专业术语书写。

查询改写就是在这个鸿沟上架桥。不改用户的意思，但把同一个意思用检索系统更容易匹配的形式表达。

**HyDE（Gao et al., 2023）** 的思路出人意料但极其有效：让 LLM 编一个假答案，用假答案的向量去检索。为什么这能工作？LLM 生成的假答案——即使可能包含事实错误——使用的词汇分布和句式结构接近真实文档。它把短查询展开成了和文档同分布的长文本，绕过了查询-文档之间的词汇鸿沟。

> HyDE 最反直觉的性质：假答案的事实正确性不重要。重要的是它的词汇分布。一个概念正确但措辞不像医学文献的假答案，检索效果不如一个措辞像医学文献但有部分事实错误的假答案。

代价：每次查询多一次 LLM 调用。对个人 RAG 系统，查询频率不高，可接受。

**查询分解** 把复杂问题拆成子问题分别检索。"深模块和浅模块的区别是什么，各有什么例子？"→ 查"深模块定义+例子"和"浅模块定义+例子"，各自检索后合并。

**Step-back prompting** 先问一个更抽象的"后退一步"的问题，用它的检索结果提供上下文骨架。"Java I/O 为什么是浅模块？"→ 先查"深模块和浅模块的评价标准"，再在找到的标准框架下回答具体问题。

对你的向量记忆库，最实用的第一步是 HyDE。实现简单（MCP `search_notes` 加一个可选参数），对定义类和解释类问题效果最好——这正是学习笔记检索的主要用例。精确命令查询（"CS61A 的作业截止日期"）不需要 HyDE——原始查询本身已经够精确。

## 思考题

1. HyDE 生成的假答案如果有严重的事实错误（比如把"深模块"解释成"代码行数很多的模块"），对检索的影响是什么？为什么 HyDE 的设计者认为这不是致命问题？
2. 查询分解和 step-back prompting 的区别是什么？什么类型的查询适合分解，什么类型适合后退？
