# 2.6 · 混合检索策略

稀疏检索管精确匹配，稠密检索管语义泛化。真实查询混合了两者——"ERR_CERT_DATE_INVALID 怎么修？"需要精确命中错误码，"怎么让代码更容易改？"需要语义理解"可维护性"。混合检索同时跑两套检索，合并结果。

为什么合并两个检索器比单独用任何一个好？不是因为他们各自信号互补——是因为"相关性"本身是多元的。精确字符串匹配和语义方向接近是两种不同的度量，没有一个单一度量能覆盖所有查询类型。

三种合并方式，从简单到复杂：

**RRF（Reciprocal Rank Fusion）** 只关心排名，不关心具体分数。`RRF_score = Σ 1/(k + rank_i)`，k 通常取 60。两个检索器的分数尺度完全不同（BM25 是 0-100，余弦是 0-1）——RRF 对此免疫，因为它只看排名。最稳健，不需要调参，不需要分数归一化。

**线性融合** 需要先把两个分数拉到同一尺度（min-max 或 z-score），再加权求和。`final = α·sparse_norm + (1-α)·dense_norm`。α 控制权重——偏向稀疏还是偏向稠密。需要为 α 调参。

**自适应融合** 根据查询类型动态调 α。短查询 + 特殊字符 → 偏向稀疏。长句子 + 自然语言 → 偏向稠密。核心思路是判断查询更像"关键词查询"还是"概念查询"。

一个重要但反直觉的提醒（来自 DPR 的实验）：如果稠密检索已经足够强，强行加入稀疏检索可能降低效果。混合检索不是"越多越好"——是两个检索器的质量差在起作用。如果你的稠密检索器在领域内很强、稀疏检索器很弱（反之亦然），融合可能不如单选。

对你的向量记忆库，如果迁移到 BGE-M3，一个模型同时输出稠密和稀疏向量——不需要额外的 BM25 引擎。这是 BGE-M3 最实际的优势：零额外基础设施成本获得混合检索能力。

## 思考题

1. RRF 完全不看分数只看排名——如果 BM25 把一篇完全不相关的文档排在第 1 位（查询词偶然出现在其中多次），RRF 能纠正吗？这暴露了 RRF 的什么隐含假设？
2. DPR 的实验表明 BM25+DPR 混合不如纯 DPR——在什么条件下会出现这种情况？你的向量记忆库符合这个条件吗？
