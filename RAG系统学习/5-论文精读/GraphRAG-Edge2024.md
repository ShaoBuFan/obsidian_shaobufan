# GraphRAG · From Local to Global: A Graph RAG Approach to Query-Focused Summarization

Edge et al. (Microsoft Research), 2024. 知识图谱 + RAG 的最具影响力方案。

## 核心问题

向量 RAG 擅长**局部检索**（找到包含某个具体事实的段落），但在**全局理解**（"这些文档讨论了哪些核心主题？有哪些对立的观点？"）上表现糟糕——因为全局问题的答案不可能存在于任何一个单独的 chunk 中。==能否构建一个知识图谱，然后用图上的社区结构来生成全局性的摘要，让 RAG 系统具备"看到整个数据集"的能力？==

## 方法

GraphRAG 的索引管道分五步：

### 1. 实体和关系提取

用 LLM 从每个 chunk 中提取：
- **实体**：人名、组织、地点、概念、事件等。每个实体附带一个简短描述
- **关系**：实体之间的关系（"领导"、"位于"、"反对"、"支持"）。每条关系附带权重（强度）

关键设计：用 ==self-reflection prompting==——LLM 先提取一次，再反思"我漏掉了什么"，修正后输出。这缓解了"长 chunk → 更少实体被提取"的问题。

### 2. 知识图谱构建

用 NetworkX 构建加权有向图。节点 = 实体，边 = 关系（权重 = 关系强度 × 出现次数）。

### 3. 社区检测

用 ==Leiden 算法==（Louvain 的改进版）递归检测图的社区结构。Leiden 在速度和社区质量上都优于 Louvain，且保证每个社区是连通的。

社区检测是递归的——从顶层（整个图的几个大社区）到底层（大量的细粒度小社区），形成一个多层级结构。

### 4. 社区摘要生成

对每个社区（每一层），用 LLM 生成结构化摘要。内容包括：社区成员、核心关系、主要主题、与其他社区的联系。摘要附带==评分（rating）==表示该社区在数据集中的重要性。

这些预生成的社区摘要就是 GraphRAG 的核心资产——它们使全局查询成为可能。

### 5. 查询

三种查询模式：
- **Global Search**：MapReduce 式处理所有社区摘要 → 合成全局答案
- **Local Search**：查询中的实体 → 图遍历找相关节点 → 混合图和文本信息 → 生成局部答案
- **DRIFT Search**：从全局到局部逐步细化（先确定大方向，再深入细节）

## 关键实验

微软报告的指标（需注意来源是微软自己的评估）：
- 事实准确率：比基线 RAG 提高 37%
- 全局问题覆盖率：92%（vs 基线 68%）
- 答案可追溯性：89%（vs 基线 45%）

**2025 年"GraphRAG 2.0"更新——动态社区选择：**
- 原始方案不管查询是什么，所有社区摘要都被处理
- 新版用 LLM 在每个层级评估社区与查询的相关性，不相关的社区被==剪枝==
- 结果：token 成本下降 77%

## 优势与局限

**优势：**
- 首次系统性解决了"全局问题"——传统 RAG 对此毫无办法
- 社区检测是自动的、递归的、语义驱动的——不需要人工定义分类体系
- 多层级社区结构使同一套索引支持从细粒度到全局的不同查询粒度

**局限：**
- **索引成本极高**：用户的报告显示 10 章小说约消耗 470 万 tokens（实体提取）+ 45 万 tokens（社区摘要）。扩大到完整知识库成本可能不可接受
- 依赖 LLM（GPT-4）做实体提取和摘要——质量受 LLM 本身能力影响，且成本随数据量线性增长
- 图是静态的——增量更新不友好
- 对短文档集可能得不偿失——需要一定的数据量才能形成有意义的社区结构

## 对本项目的启发

1. **GraphRAG 不是你的向量记忆库的替代品——它是一个完全不同量级的工具**。你的笔记数量（几十到几百篇）和主题集中度下，向量 RAG + 复合评分大概率已经够用。GraphRAG 的"全局理解"在个人知识库的规模下可能 overkill。

2. **动态社区选择的启示**：即使不实现完整的 GraphRAG，"先粗筛再精排"的思想可以应用到你的检索流程中——先用简单的相关性判断排除明显无关的 chunks，再对候选做精细排序。

3. **GraphRAG 真正适合你的是"回顾"场景**：当你积累了几百条对话记录和笔记后，偶尔想知道"我在过去半年里主要关注哪些话题？"——这种跨文档的元分析才是 GraphRAG 的用武之地。不着急，等到向量记忆库的三层架构（设计/08）全部实现后再考虑。

4. **验证了你在 [GraphRAG 笔记](../4-进阶专题/1-GraphRAG.md) 中的分析**：GraphRAG 是向量 RAG 的**补充**而非替代。两者的使用场景不同。
