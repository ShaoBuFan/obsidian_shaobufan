# FiD · Leveraging Passage Retrieval with Generative Models for Open Domain QA

Izacard & Grave (Facebook AI / Université PSL), ACL 2021. 多文档融合的经典。

## 核心问题

RAG 虽然能检索多段文本，但生成器实际上只能有效利用 Top-1 或 Top-2 的结果。当答案分散在多个文档中时（"列出 X 的三个主要原因"——第一个原因在文档 1，第二个在文档 3，第三个在文档 7），RAG 很难整合。==能否让模型同时处理几十个甚至上百个检索段落，把分散在多处的信息融合成一个连贯的答案？==

## 方法

FiD（Fusion-in-Decoder）的核心架构出奇简单：

1. **独立编码**：每个检索到的段落独立通过 Encoder（T5 的 encoder），生成各自的隐藏状态。段落之间在编码阶段**没有任何交互**。

2. **拼接解码**：将所有段落的隐藏状态沿序列维度拼接成一个巨大的上下文序列，然后喂给 Decoder。Decoder 通过 ==cross-attention 同时关注所有段落==——融合发生在解码阶段，不在编码阶段。

```
段落1 → [Encoder] → h1 ─┐
段落2 → [Encoder] → h2 ─┼─→ [拼接] → [Decoder] → 答案
段落3 → [Encoder] → h3 ─┘
... (最多 100 个段落)
```

> FiD 的美学：不设计复杂的段落间交互机制（那会让编码复杂度变成 O(N²)）。编码保持 O(N)，融合全交给 decoder 的 cross-attention——decoder 自己决定关注哪些段落的哪些 token。简单，可扩展，效果好。

## 关键实验

**扩展性**：当段落数从 5 增加到 25 时，性能持续提升。从 25 → 100 时，边际收益递减但仍有增长。这与 RAG 形成鲜明对比——RAG 在 Top-5 之后增加段落数几乎没有收益。

**Natural Questions**：FiD 用 100 个段落达到 51.4 EM（Exact Match），比当时的 SOTA 高出约 5 个点。

**TriviaQA**：首次达到 70+ EM。这个结果非常重要——TriviaQA 的问题答案往往需要跨文档推理（"哪个演员在电影 A 和电影 B 中都出现过？"），FiD 是第一个能可靠处理这类跨文档问题的架构。

**消融实验**：
- 段落数量从 5→25→100：持续提升但递减。这个曲线对工程实践很重要——它告诉你 25 之后多花的计算成本换来的收益很少。
- Decoder 大小：T5-large（770M）→ T5-3B，性能有大幅跳跃。生成器的容量是重要变量。

## 优势与局限

**优势：**
- 极简架构，极强效果。编码并行化也容易。
- 可扩展到 100 个段落——比 RAG 的实际有效利用数量高一个数量级。
- 零跨段落交互的编码设计使得推理时的段落数可动态调整。

**局限：**
- 100 个段落的 full cross-attention 在长文本上计算量很大——每个 decoder token 要关注所有 100 个段落的所有 token
- 独立编码意味着每个段落丢失了"全局上下文"——它不知道其他段落在说什么。如果段落间有隐含的逻辑关系（"如上一段所述..."），FiD 捕捉不到
- 需要 Encoder 的隐藏状态全部保存在内存中直到解码完成——对 GPU 内存不友好

## 对本项目的启发

1. **段落数比你想象的重要**：当前向量记忆库默认 top_k=5。FiD 的实验表明性能在 25 段之前持续提升。对于需要综合多篇笔记的问题（"我在哪些章节里讨论过模块深度的概念？"），5 段可能不够。建议 MCP `search_notes` 支持更大的 top_k（至少 20），让 Claude Code 根据问题类型自己决定用多少。

2. **编码阶段保持独立，融合交给下游**：FiD 的设计哲学对应了你 RAG 管道的一个关键原则——chunk 的嵌入应该独立完成（不依赖其他 chunk），检索后的融合（排序、压缩、拼接）在 Claude Code 侧完成。不要试图在嵌入阶段建立 chunk 间关系。

3. **Cross-attention 的工程代价**：个人 RAG 场景下，你的检索量级远小于 FiD（几十个 chunk vs 100 个段落），这个代价对你来说不存在。但如果你未来扩展到对话历史 + 笔记 + 外部文档的综合检索，段落数可能快速膨胀。
