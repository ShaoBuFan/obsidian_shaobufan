# RAG · Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks

Lewis et al. (Facebook AI), NeurIPS 2020. RAG 的命名论文。

## 核心问题

LLM 的参数记忆有两个根本限制：（1）知识是"冻住"的——训练截止日后发生的事模型不知道；（2）知识是"隐式"的——你不知道模型记住了一段信息还是碰巧生成了它。能否把知识从模型参数中解耦出来，放到一个外部检索库中，让模型在生成时**按需获取**？

## 方法

RAG 的架构由两部分组成：

- **Retriever**：DPR（Dense Passage Retriever）。把查询编码为向量，从 Wikipedia 中检索 Top-K 段落。
- **Generator**：BART（Seq2Seq Transformer）。把检索到的段落和查询拼接起来，生成答案。

两者**端到端训练**——检索器和生成器的参数同时在优化。这与先固定检索器再训练生成器的 pipeline 方案不同。

**两种解码策略：**

1. **RAG-Sequence**：整个回答序列使用**同一组**检索段落。适合"答案集中在某一段"的问题。
2. **==RAG-Token==**：每个 token 可以依赖**不同组**的检索段落。理论上更灵活——生成"巴黎"时参考一段，生成"是法国首都"时参考另一段。但实际上 RAG-Sequence 在大多数任务上表现更好。

> RAG-Token 比 RAG-Sequence 更优雅但不一定更好——这提醒我们：检索段落的频繁切换可能破坏生成的连贯性。对向量记忆库来说，这意味着每个问题的检索结果应该相对稳定，不要为了"每个 token 最相关"而牺牲上下文连续性。

## 关键实验

**开放域问答：** RAG 在所有四个 QA 数据集（Natural Questions、TriviaQA、WebQuestions、CuratedTREC）上达到当时 SOTA。NQ 上比纯 BART 提高了 15 个点的 EM 准确率。

**知识更新实验**：把 Wikipedia 检索库从 2016 年 12 月换成 2018 年 12 月——==只换数据，不重新训练模型==——在需要最新知识的问题上正确率显著提升。这是 RAG 最核心的论证：知识可以独立于模型参数更新。

**MS-MARCO 生成任务**：RAG 生成的内容比 BART 更具体、更有信息量，幻觉更少。人工评估中，RAG 在"事实正确性"和"信息完整性"两个维度上均显著优于纯参数模型。

**消融实验的发现：**
- 只检索不生成（直接返回检索到的段落）：效果远低于完整 RAG。生成器的作用不仅是"翻译"检索结果——它在整合、推理、填补空白。
- 不端到端训练（固定 DPR + 只训练 BART）：效果显著低于端到端联合训练。检索器和生成器需要互相适应。

## 优势与局限

**优势：**
- 第一次把"可更新的知识库 + LLM"完整论证清楚
- 端到端训练检索器和生成器，两者互相适应
- 实验覆盖四个 QA 数据集 + MS-MARCO + Jeopardy，论证充分

**局限：**
- 检索器是 Wikipedia-only——没有讨论多源、私域、非文本数据的检索
- RAG-Token 的 re-rank 开销大，实际部署少用
- 没有讨论"当检索结果为空或不相关时该怎么办"——这是生产环境最常见的问题
- 此后 4 年 RAG 领域的爆炸性发展（HyDE、Self-RAG、GraphRAG）证明这篇论文定义的"一次检索→一次生成"范式只是 RAG 的起点

## 对本项目的启发

1. **你的向量记忆库本质上就是 RAG 的 retriever 组件**——只是 generator 角色由 Claude Code 扮演。论文的一个核心结论直接适用于你：检索器质量 > 生成器质量对你的系统来说就是 ChromaDB 检索质量 > prompt 优化效果。

2. **端到端训练的启示**：你无法像论文那样联合训练 retriever 和 generator（Claude 的参数你改不了），但这意味着 generator 不会"适应"你的 retriever 的检索偏好。你需要让 retriever 的输出格式（chunk 大小、元数据、排序）==适配 Claude 的信息消费习惯==——清晰边界、明确来源、不过长。

3. **知识更新的论证**：你不重新训练嵌入模型，只重新索引新笔记 → 这就是论文的核心论点在个人知识管理场景的直接应用。
