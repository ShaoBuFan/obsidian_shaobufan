# 第四章：当表单长大以后

> [上一章](03-告诉用户他们填错了.md)我们学会了内置校验。但冷静想一想：如果一个表单有 30 个字段，每个字段 5 条规则——150 条校验散落在 JSX 里。代码会变成什么样？你还能快速回答"这个字段到底有哪些规则"吗？本章引入一种根本性的组织方式：==Schema 校验==。

---

## 4.1 内置校验的"天花板"

先看看 3 个字段时的代码：

```tsx
<input {...register('name', { required: '必填', minLength: { value: 2, message: '至少 2 个字符' } })} />
<input {...register('email', { required: '必填', pattern: { value: /^\S+@\S+\.\S+$/, message: '格式不对' } })} />
<input {...register('age', { required: '必填', valueAsNumber: true, min: { value: 1, message: '最小 1' }, max: { value: 150, message: '最大 150' } })} />
```

3 个字段，还算能看。但 15 个字段呢？三个问题会逐渐显现：

1. **校验规则和 UI 混在一起**——JSX 变成了校验配置的海洋。你想找"邮箱字段有哪些校验规则"，得在几百行 JSX 里肉眼搜索
2. **规则无法复用**——3 个表单各有邮箱字段，邮箱的校验规则写了 3 遍，改一处要改三处
3. **没有类型安全**——`data` 的类型不会从校验规则自动推导。你手动写了一个 `interface FormData`，和校验规则是两套独立的东西——改一个忘了改另一个，TypeScript 不会帮你发现

> 这三个问题的本质是同一个：**校验规则是"业务知识"，它不应该和"视图层"（JSX）绑定。** 换一个 UI 组件库，校验规则不该跟着变。规则和视图之间唯一的联系应该只是一个字段名——`'email'`。

## 4.2 Schema 校验：一份规则，处处使用

==Schema 校验==把"脑中的规则"直接写成可执行的代码，并且让 TypeScript 自动推导出每个字段的类型。

三个角色协作：

```
Zod Schema（规则定义）  →  @hookform/resolvers（桥接）  →  react-hook-form（执行）
───────────────────     ──────────────────────────      ─────────────────────
定义"什么是对的"         把 Zod 错误翻译成 RHF 格式       管理"什么时候校验"
独立于任何组件           薄薄一层，你基本感受不到它         收集数据、分发错误
```

### 安装

```bash
npm install @hookform/resolvers zod
```

## 4.3 从内置校验迁移到 Zod

先看内置写法的样子，再对比迁移后的效果：

```tsx
// 旧写法：规则混在 JSX 里——规则和视图绑定
<input {...register('name', {
  required: '必填',
  minLength: { value: 2, message: '至少 2 个字符' },
  maxLength: { value: 20, message: '最多 20 个字符' },
})} />
```

迁移到 Zod：

```tsx
import { z } from 'zod';

// 1. 规则独立于组件——可以放在任何文件里
const schema = z.object({
  name: z
    .string()
    .min(2, '至少 2 个字符')
    .max(20, '最多 20 个字符'),
  email: z
    .string()
    .email('邮箱格式不正确'),
  age: z
    .number({ invalid_type_error: '请输入数字' })
    .min(1)
    .max(150),
});

// 2. 类型自动推导——不需要手写 interface
type FormData = z.infer<typeof schema>;
// FormData = { name: string; email: string; age: number }
```

在组件中使用：

```tsx
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';

function SignupForm() {
  const {
    register,
    handleSubmit,
    formState: { errors },
  } = useForm<FormData>({
    resolver: zodResolver(schema), // zodResolver 做桥接
  });

  const onSubmit = (data: FormData) => {
    // data 的类型自动是 { name: string; email: string; age: number }
    console.log(data);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      {/* register 不再需要写校验规则！schema 已经定义了 */}
      <input {...register('name')} />
      {errors.name && <span>{errors.name.message}</span>}

      <input {...register('email')} />
      {errors.email && <span>{errors.email.message}</span>}

      <input type="number" {...register('age', { valueAsNumber: true })} />
      {errors.age && <span>{errors.age.message}</span>}
    </form>
  );
}
```

**关键的转变**：校验逻辑从 JSX 中**消失**了，跑到了 `schema` 对象里。组件的 `register` 变干净了——它只负责绑定 DOM，不再负担校验规则。

> 这种分离叫 ==关注点分离（Separation of Concerns）==。校验规则是"业务规则"，属于业务逻辑层；UI 组件是"视图层"。业务规则应该独立于视图存在——明天你把 Input 换成第三方组件，校验规则不应该跟着变。

## 4.4 跨字段校验：当校验需要看两个字段

单个字段的校验 Zod 内置了。但"两次密码必须一致"这类规则涉及两个字段，怎么写？

```tsx
const schema = z
  .object({
    password: z.string().min(8, '至少 8 位'),
    confirmPassword: z.string(),
  })
  .refine(
    (data) => data.password === data.confirmPassword,
    { message: '两次密码不一致', path: ['confirmPassword'] }
  );
```

`refine` 的意思是"再精炼一下"——在单个字段校验全部通过之后，用整个 data 对象再做一次判断。`path` 指定错误挂到哪个字段下面。

### 更复杂的场景：多个跨字段规则

```tsx
const schema = z
  .object({
    startDate: z.string(),
    endDate: z.string(),
  })
  .superRefine((data, ctx) => {
    if (new Date(data.endDate) <= new Date(data.startDate)) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: '结束日期必须大于开始日期',
        path: ['endDate'],
      });
    }

    if (data.startDate < '2020-01-01') {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: '起始日期不能早于 2020 年',
        path: ['startDate'],
      });
    }
  });
```

`superRefine` 是 `refine` 的加强版——在一个函数里可以添加多个错误，每个绑定到具体的 `path`。

> **refine vs superRefine 怎么选**：一条跨字段规则用 `refine`，简洁；多条用 `superRefine`，每个 `ctx.addIssue` 明确指定错误归属。

## 4.5 可选字段的处理

```tsx
// 字段可选——不填也行，填了则必须符合格式
website: z.string().url('请输入有效的 URL').optional().or(z.literal(''))

// 解读：
// .optional()  → 可以不填（此时值是 undefined）
// .or(z.literal(''))  → 填了但留空（空字符串）也算合法
// 否则 → 必须通过 url() 校验
```

可选字段如果不处理空字符串，用户清空输入框后反而会看到"请输入有效 URL"——这体验很差。

## 4.6 为什么选 Zod 而不是 Yup

如果你搜 react-hook-form 教程，经常看到配合 `yup` 使用。两者做同样的事，但：

| | Zod | Yup |
|---|---|---|
| TypeScript 支持 | 原生一等公民 | 需要额外安装 `@types/yup` |
| 类型推导 | `z.infer<typeof schema>`——自动，不需要写第二遍 | `yup.InferType<typeof schema>` |
| 学习曲线 | 链式调用，直觉友好 | 也链式，风格相近 |
| 社区 | 增长极快，新项目默认选择 | 历史悠久，存量项目多 |
| 包体积 | ~60KB | ~40KB |

**原则**：新项目用 Zod，已有项目用 Yup 就继续用，不必迁移。

## 本章要点

- 内置校验规则多了会让 JSX 臃肿——根本问题是**规则和视图耦合**
- Schema 校验 = ==规则集中定义 + 类型自动推导 + 跨组件复用==
- Zod 定义规则 → `zodResolver` 桥接 → react-hook-form 执行——三部分各司其职
- `refine` 处理简单跨字段校验，`superRefine` 处理多个跨字段规则
- `z.infer<typeof schema>` 从 schema 推导 TypeScript 类型——==一处定义，类型和校验永不分离==

## 思考题

1. "校验规则独立于 UI"听上去正确。但有没有场景把规则放在 JSX 里反而是更好的选择？比如一个只用一次、只有两个字段的搜索栏？
2. `z.infer<typeof schema>` 推导出的类型，和手动写的 `interface` 有任何区别吗？如果 schema 里有 `.transform()`，推导结果会是什么？
3. 你维护的项目里有 6 个表单，其中 4 个用了 Zod，2 个用了内置校验。团队新人问你"校验规则应该写在哪里"，你怎么回答？
