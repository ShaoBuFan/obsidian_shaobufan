# 第八章：高级模式与技巧

> 前七章你已经能写出 90% 的表单。剩下 10% 是一些精准场景——读取值但不触发渲染、回填已有数据、拆分巨型表单、处理服务端校验错误、让表单能被屏幕阅读器朗读。这些场景的 API 就是本章内容。

---

## 8.1 watch vs getValues：什么时候读值

两个函数都能拿到字段的当前值。区别只有一个——是否触发渲染：

| | watch | getValues |
|---|---|---|
| 触发组件重渲染 | 是 | 否 |
| 本质 | 订阅了某个字段的变化 | "偷看一眼"当前值 |
| 典型场景 | 根据一个字段控制另一个字段的显示 | 在事件回调里读一下值，不需要改变 UI |

### 判断标准：问自己一个问题

"值变了之后，页面需要跟着变吗？"

- **需要** → `watch`：选了"其他"才显示理由输入框
- **不需要** → `getValues`：提交前去日志服务打个点，不需要 UI 变化

```tsx
// watch：页面要跟着值变
const type = watch('type');
const showOther = type === 'other'; // UI 根据 type 的不同而不同

// getValues：不触发渲染，随时随地偷看一眼
const handleLog = () => {
  const currentEmail = getValues('email');
  analytics.track('form_field_blur', { field: 'email', value: currentEmail });
};
```

### watch 的三种写法

```tsx
const name = watch('name');                     // 单个字段 → 返回单个值
const [name, email] = watch(['name', 'email']);  // 多个字段 → 返回数组
const allValues = watch();                       // 全部字段 → 慎用：任何字段变化都渲染
```

> `watch()` 全量监听会让你的表单退化回受控组件——每次按键都重渲染。除非你真的需要实时追踪所有字段的值（极少见），否则**永远给 watch 传具体的字段名**。

## 8.2 setValue 与 reset：什么时候改值

有些时候值的来源不是用户的键盘——从服务器加载数据回填、一键填充默认地址。

### setValue：改个别字段

```tsx
const { setValue } = useForm();

// 基础：设值
setValue('name', '张三');

// 高级：设值并控制副作用
setValue('name', '张三', {
  shouldValidate: true, // 设值后立即校验这个字段
  shouldDirty: true,    // 标记表单为"被修改过"
  shouldTouch: true,    // 标记字段为"被触碰过"
});
```

### reset：批量处理

```tsx
const { reset } = useForm({
  defaultValues: { name: '', email: '' },
});

// 清空表单——回到 defaultValues
reset();

// 替换为特定值——编辑页面加载已有数据
reset({ name: '张三', email: 'zhangsan@mail.com' });
```

### 怎么选

| | reset | setValue |
|---|---|---|
| 一次设多个字段 | 方便 | 调多次 |
| 后续 `isDirty` | false（reset 把新值设为"没改过"的基准） | true（算你改的） |
| 典型场景 | 加载编辑数据、提交后清空 | 某个操作改个别字段 |

**编辑表单的标准套路**：

```tsx
const { register, handleSubmit, reset } = useForm();

useEffect(() => {
  fetch('/api/user/123')
    .then(r => r.json())
    .then(reset); // 直接传 reset——reset(formData) 会回填所有字段
}, [reset]);

const onSubmit = async (data) => {
  await save(data);
  reset(data); // 保存成功——把当前值设为"新基准"，isDirty 回到 false
};
```

## 8.3 FormProvider：拆分巨型表单

当一个表单大到需要拆成多个文件（多步骤表单、团队协作每人写一个步骤），怎么让所有子组件共享同一个 `useForm` 实例？

直接通过 props 一层层传会变成 ==prop drilling==——中间组件被迫接收并转发它不关心的东西。

react-hook-form 提供了 Context 方案：

```tsx
import { useForm, FormProvider, useFormContext } from 'react-hook-form';

// 外层：提供表单实例——像一个"共享大脑"
function WizardForm() {
  const methods = useForm();
  return (
    <FormProvider {...methods}>
      <StepOne />
      <StepTwo />
      <StepThree />
    </FormProvider>
  );
}

// 内层任意深处：通过 Context 拿到同一个表单实例
function StepTwo() {
  const { register, formState: { errors } } = useFormContext();
  return <input {...register('address')} />;
}
```

`FormProvider {...methods}` 把 useForm 的**全部返回值**（register、handleSubmit、control、formState...）注入 React Context。`useFormContext` 在任何深度的子组件里取用。

> Context 更新会让所有消费 `useFormContext` 的组件重渲染。不要滥用——如果表单不大，一个文件写完更简单。拆分的原则是：**拆分是因为文件太长难以理解，不是因为"每个字段都应该有自己的文件"。** 通常 100-150 行是一个合理单文件上限。

## 8.4 setError：把服务端错误注入表单

Zod 和内置校验处理的是**前端规则**——格式、长度、范围。但有些错误只有服务器才知道——"用户名已存在""优惠券已过期"。

用 `setError` 把服务端返回的错误注入到对应字段：

```tsx
const { setError } = useForm({
  resolver: zodResolver(schema),
});

const onSubmit = async (data: FormData) => {
  try {
    await api.submit(data);
  } catch (error) {
    if (error.code === 'EMAIL_EXISTS') {
      // 绑定到具体字段——错误会出现在 errors.email 里
      setError('email', {
        type: 'server',
        message: '该邮箱已被注册，请直接登录',
      });
    } else if (error.code === 'COUPON_INVALID') {
      setError('coupon', {
        type: 'server',
        message: '优惠券已过期',
      });
    } else {
      // 表单级别的错误（不属于任何字段）
      setError('root', {
        type: 'server',
        message: '提交失败，请稍后重试',
      });
    }
  }
};

// root 错误不绑定到任何字段，需要单独显示
{errors.root && <div role="alert" className="form-error">{errors.root.message}</div>}
```

`'root'` 用于不归任何一个字段管的错误——比如"服务器崩了，请稍后重试"。这种错误无法用 `errors.fieldName` 访问，只能通过 `errors.root`。

## 8.5 Schema 工厂：校验规则依赖外部状态时

大部分表单的校验规则是**静态的**——定义一次，永远不变。但有时规则依赖运行时状态：

```tsx
// 场景：诊所预约——是否允许约到过去的时间，取决于诊所的配置
const getBookingSchema = (allowPastTime: boolean) => {
  return z.object({
    date: z.date(),
    time: z.string(),
  }).superRefine((data, ctx) => {
    if (!allowPastTime && isInPast(data.date, data.time)) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: '预约时间不能是过去',
        path: ['time'],
      });
    }
  });
};

// 在组件中使用
type BookingFormData = z.infer<ReturnType<typeof getBookingSchema>>;

function BookingForm() {
  const { clinicConfig } = useClinicConfig();

  const form = useForm<BookingFormData>({
    resolver: zodResolver(getBookingSchema(clinicConfig.allowPastAppointments ?? false)),
  });
  // ...
}
```

==Schema 工厂模式==让 Zod schema 可以依赖 React state / Context / props——而不用把校验逻辑又写回 JSX 里。关键：`z.infer<ReturnType<typeof getBookingSchema>>` 保持了类型推断。

## 8.6 无障碍访问（Accessibility）

好的表单不只是"能用"——视障用户通过屏幕阅读器也应该能顺利填写。

### 错误消息的 ARIA 属性

```tsx
<input
  {...register('email')}
  id="email"
  aria-invalid={!!errors.email}
  aria-describedby={errors.email ? 'email-error' : undefined}
/>
{errors.email && (
  <p id="email-error" role="alert">
    {errors.email.message}
  </p>
)}
```

- `aria-invalid` → 屏幕阅读器会播报"此字段无效"
- `aria-describedby` → 把 input 和它的错误消息关联——屏幕阅读器焦点到 input 时会读出错误
- `role="alert"` → 错误出现时**主动播报**，不用等焦点移过去

### shouldUnregister：隐藏的字段还保留值吗

```tsx
// 默认行为：即使字段从 DOM 卸载了（条件渲染隐藏），提交时 data 仍然包含它
{showOther && <input {...register('otherReason')} />}

// 如果希望隐藏时清除值：
useForm({ shouldUnregister: true })
```

## 8.7 trigger：手动触发校验

有些场景你需要在不提交的情况下主动触发校验——比如一个"实时校验"开关被打开，或者当某个字段的值变化后需要校验另一个字段：

```tsx
const { trigger } = useForm();

// 校验单个字段——返回 boolean
const valid = await trigger('email');

// 校验多个字段
const results = await trigger(['email', 'password']);

// 校验整个表单
const allValid = await trigger();
```

典型用法：用户选了"银行卡支付"，需要对之前隐藏的银行卡号字段执行校验：

```tsx
const method = watch('method');

useEffect(() => {
  if (method === 'card') {
    trigger('cardNumber'); // 卡片字段出现后立即校验
  }
}, [method, trigger]);
```

## 8.8 dirtyFields：比 isDirty 更精细的追踪

`isDirty` 告诉你"表单有字段被修改过"——但不同字段有不同的脏状态。`dirtyFields` 精确到每个字段：

```tsx
const { formState: { dirtyFields } } = useForm();

// dirtyFields.email === true  → 只有 email 被修改过
// dirtyFields.name  === undefined → name 没被碰过

// 典型场景：只提交被修改的字段
const onSubmit = async (data) => {
  const changedData = Object.keys(dirtyFields).reduce((acc, key) => {
    acc[key] = data[key];
    return acc;
  }, {} as Record<string, any>);
  await api.patch(changedData); // PATCH 只发送改过的字段
};
```

结合 `reset(data)` 后 `dirtyFields` 也会清空——再次确认：==reset 把传入的值设为"新的未修改基准"==。

## 本章要点

- ==值变了页面要变 → `watch`==；==只要值不要渲染 → `getValues`==。`watch()` 全量监听等于退化为受控，慎用
- `setValue` 改个别字段；`reset` 批量回填/清空——编辑表单首选 reset
- `FormProvider` + `useFormContext` 拆分巨型表单，避免 prop drilling
- `setError` 把服务端错误注入表单——`'root'` 用于不属于任何字段的表单级错误
- Schema 工厂模式让动态校验规则保持类型安全
- 无障碍：`aria-invalid`、`aria-describedby`、`role="alert"` 让屏幕阅读器正确播报错误
- `trigger` 手动触发单字段或全表单校验——不依赖提交
- `dirtyFields` 提供==字段级脏追踪==——配合 PATCH 请求只发送变更字段

## 思考题

1. `watch('email')` 会导致包含这个调用的组件重渲染。如果你把 watch 放在一个独立的子组件里（只显示被 watch 的字段相关的 UI），父组件会受影响吗？这暗示了什么架构策略？
2. `reset(data)` 之后 `isDirty` 变回 `false`。但如果用户在 reset 之后又修改了一个字段，`isDirty` 变回 `true` 时，比较的基准是 reset 传入的 data 还是原始的 defaultValues？
3. Schema 工厂模式每次渲染都创建新的 schema 对象。这对 `zodResolver` 的性能有影响吗？如何用 `useMemo` 优化？
4. `trigger` 和 `handleSubmit` 都执行校验。它们在校验逻辑上有什么区别？什么时候只能用 `trigger`？
