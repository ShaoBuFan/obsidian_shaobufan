# 文本切分策略

文本切分（chunking）是 RAG 管道里最被低估但也最关键的一步。切得好不好，直接决定检索质量。

## 核心矛盾

```
切太短 → 一段话不完整，缺上下文，检索到了也看不懂
切太长 → 混入太多无关内容，稀释语义，向量指向不精准
```

## 三种策略

### 策略 1：固定长度切分

按字符/token 数硬切，相邻段之间重叠防止拦腰斩断。

```python
def chunk_by_size(text, chunk_size=500, overlap=100):
    chunks = []
    start = 0
    while start < len(text):
        chunks.append(text[start:start + chunk_size])
        start += chunk_size - overlap
    return chunks
```

> 问题：在 500 字处硬切，可能正好把一段论述拦腰斩断。

### 策略 2：自然边界切分（实际项目主流）

优先在标题、段落、句子边界处切。对结构良好的文档（如 Obsidian 笔记）效果最好。

优先级：`# 标题` > `空行（段落）` > `句号/换行` > `硬切`

```python
paragraphs = text.split('\n\n')
chunks = []
current = ""
for p in paragraphs:
    if len(current) + len(p) < max_chunk:
        current += '\n\n' + p if current else p
    else:
        chunks.append(current)
        current = p
```

### 策略 3：语义切分

对每句话生成向量，相邻句做余弦相似度。相似度突然下降 = 话题切换点，在此切断。

```
句子1 vec → [话题A]
句子2 vec → [话题A]  相似度 0.92 ✓ 继续
句子3 vec → [话题A]  相似度 0.88 ✓ 继续
句子4 vec → [话题B]  相似度 0.41 ✗ ← 在这切！
句子5 vec → [话题B]  相似度 0.90 ✓ 新段落开始
```

> 效果最好但也最慢 —— 每句话都要跑一遍 embedding。

## 成熟方案

LangChain 的 `SemanticChunker` 和 LlamaIndex 的 `SemanticSplitterNodeParser`，底层逻辑几乎一样：embedding 相似度找断点。

区别：
- **LlamaIndex** 更稳定（核心模块），产出结构化 Node 对象，直接接向量库
- **LangChain** 仍在 `experimental`，产出纯文本列表
- 两者都支持三种断点策略：百分位阈值（percentile）、标准差（standard_deviation）、四分位距（interquartile）

## 反直觉的研究结论（2025-2026）

> 语义分块并没有稳定优于简单的按句子递归切分。

- 2026.01 系统分析：在 ≤5000 token 下，按句子切和语义切分统计无差异
- 2026.04 Scadea 基准：递归 512-token 切分准确率 **69%**，语义分块 **54%**，差 15 个点
- Superlinked / Chroma 评估：**chunk_size 和 overlap 才是主导变量**，不是切分器类型

原因：句子本身就是天然的语义边界，作者已经在话题切换处换句了。embedding 的平均化效应反而可能引入噪声。

### 语义分块真正值得用的场景

1. 主题密度极不均匀的文档（叙事段落 + 密集表格交替）
2. 超小 chunk（<200 token），递归切分难以保持连贯
3. 已经在为每句话做 embedding —— 边际成本接近零

## Max-Min 方案（2025.12，Milvus）

> **先 embed，后 chunk** —— 颠覆了传统"先切再向量化"的流程。

| 步骤 | 传统 | Max-Min |
|------|------|---------|
| 1 | 切分 → | **Embed 所有句子** |
| 2 | Embed → | 相邻句余弦相似度 |
| 3 | 存向量 | 在语义真正变化处切断 |

优点：动态边界、3 个超参数、低额外开销（复用句子 embedding）。弱点：只做局部比较，跨段落的远期呼应检测不到。

## 自适应分块（2026.03，LREC）

对每个文档自动选最合适的策略，用五个指标评估：

- 引用完整性（References Completeness）
- 块内凝聚力（Intrachunk Cohesion）
- 文档上下文连贯性（Document Contextual Coherence）
- 块完整性（Block Integrity）
- 大小合规（Size Compliance）

效果：回答正确率从 62-64% → 72%，多回答 30% 的问题。

> 这个方向可能比"全局一刀切用语义分块"更有前途。

## 对本项目的适用性

本 vault 的笔记本身按标题、段落组织良好。大概率：

**按 `## 标题` 切 + 按段落合并到合适长度** 的效果不会比语义分块差，且零额外计算。

如果尝试语义分块，最合理切入点是 Max-Min 方案 —— embedding 之后顺手算相邻相似度，成本极低。效果不好就扔掉，回到简单方案。

## 一个容易被忽略的细节

切完之后，每段的**元数据**和检索质量强相关：

```python
{
    "text": "深层模块的核心思想是...",
    "metadata": {
        "source": "代码重构哲学/讲义/ch04.md",
        "chapter": "第4章",
        "section": "深模块的定义",
        "date": "2025-03-15"
    }
}
```

> 检索到一段后，用户可以点回去看上下文，AI 也能据此判断前后文范围。
