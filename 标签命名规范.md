# 标签命名规范

## 原则

1. **搜索优先创建**：新建标签前，检查 vault 中是否已有语义对等的标签（Obsidian 中 `tag:` 搜索匹配所有子标签）。
2. **单一层级归属**：同一概念只应该存在于一个命名空间下——不要同时有 `TypeScript` 和 `测试/TypeScript`。
3. **命名格式**：camelCase（`react-hook-form`）或 kebab-case（`discriminatedUnion`），用 `/` 建立层级。标签不能含空格。

---

## 命名空间

| 命名空间 | 用途 | 示例 |
|----------|------|------|
| `book/` | 读书笔记 | `book/软件设计哲学` |
| `测试/` | 测试工具与技术 | `测试/RTL`、`测试/Vitest`、`测试/MSW`、`测试/表单`、`测试/进阶` |
| `工具/` | 基础设施与开发工具 | `工具/Vitest`、`工具/MSW`、`工具/RTL` |
| `方法/` | 方法论与设计技术 | `方法/降低复杂性`、`方法/战略性编程` |
| `思想/` | 抽象概念与设计哲学 | `思想/可读性与可理解性` |
| `概念/` | 笔记中的具体概念 | `概念/深模块`、`概念/通用模块` |
| `部分/` | 书籍章节分组 | `部分/核心设计技术`、`部分/扩展话题` |

## 顶级独立标签

以下领域使用顶级标签（无命名空间前缀），因为它们本身就是知识区的根概念：

- `React`、`ReactHook`、`react-hook-form`
- `CS自学`、`TypeScript`、`Zod`、`Vite`
- `第一性原理`、`关注点分离`、`单一真相来源`

## 新建标签检查清单

1. `tag:<关键词>` 搜索 vault，确认没有同名或语义对等的标签
2. 确定归属命名空间（是否已有同类标签在该空间下？）
3. 选择 camelCase 或 kebab-case，与同空间已有标签风格一致
4. 避免单数/复数变体（`组件` vs `组件s`——统一用单数）
5. 标签必须包含至少一个非数字字符（`#1984` 无效，`#y1984` 有效）

## 维护

标签体系不是一成不变的。如果发现：
- 同一概念跨区使用不同标签 → 批量重命名统一
- 某标签下文件过多（>50）→ 考虑拆分子标签
- 某标签只有 1 个文件 → 考虑是否需要独立存在
