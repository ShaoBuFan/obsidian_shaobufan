# 3 · ANN 索引算法

有了向量和相似度度量，还需要最后一步：从几百万个向量里找到和查询最接近的 Top-K。暴力扫描是 O(N·D)——100 万个 768 维向量，每次搜索 7.68 亿次浮点运算。RAG 的实时检索不允许这个延迟。

低维空间（2D、3D）有精确的 O(log N) 搜索方案——KD-Tree 之类。但一旦维度超过 10-20，维数灾难让一切空间划分方法退化：高维空间中所有点之间的距离都差不多，任何划分边界附近都有大量点，查询点几乎总是落在多个边界之间。

所以精确搜索不可行。那就接受近似——用可控的精度损失换取数量级的加速。99.9% 的召回率和 100% 的召回率对下游 LLM 生成质量的影响不可区分，但搜索可能快 100 倍。

HNSW 是目前最主流的 ANN 算法，ChromaDB 的默认索引。它的名字从里向外拆：`(Hierarchical (Navigable (Small World)))`。每一层解决一个问题。

**最内层——Small World（小世界图）。**"六度分隔"现象：社交网络里任意两个人平均通过 6 步连接。这不是因为社交网络是随机的——是因为少数"长程连接"（你在国外认识一个人）剧烈缩短了图的直径，同时本地连接（你的朋友互相认识）保证了局部搜索的可靠性。这种结构恰好是向量搜索需要的：长程边让你快速跨越语义空间，本地边让你在目标附近精确导航。

**中间层——Navigable（可导航性）。**在小世界图上做贪心搜索：从任意点出发，走到离查询最近的邻居，重复直到无法更近。这个算法极其简单——复杂的是图的结构，不是搜索逻辑。贪心搜索在随机图上容易陷入局部最优——你需要靠图的构造来避免。

**最外层——Hierarchical（分层）。**这是 HNSW 解决局部最优的方法。构建一个多层跳表：顶层节点极少、边极长（洲际航班），底层包含所有节点、边极短（步行街道）。搜索从顶层开始贪心，找到该层的最优点后下沉一层继续，逐层到底。

为什么分层解决了局部最优？顶层的长程边让你在几步内进入"大致正确"的区域——你不可能卡在错误的社区里，因为顶层几乎没有"社区"结构，连通性极强。到了底层，你已经在大方向正确的区域内，贪心搜索的局部最优就是全局近似最优。

> HNSW 的天才不在搜索算法——贪心搜索极其简单。天才在数据结构：通过概率性的分层，让短程边和长程边各居其层，使得朴素贪心搜索自动获得 O(log N) 复杂度。

三个关键参数：`M`（每节点最大邻居数）、`efConstruction`（构建时的候选队列大小）、`efSearch`（查询时的候选队列大小）。只有 `efSearch` 可以在运行时调——查询时传更大的值能提升召回率但有延迟代价。`M` 和 `efConstruction` 在索引构建时确定，创建后不可更改。

ChromaDB ≥ 1.0 的 HNSW 配置在 collection 创建时通过 `configuration` 字典传入：

```python
collection = client.create_collection(
    name="memory",
    configuration={
        "hnsw": {
            "space": "cosine",
            "ef_construction": 200,
            "ef_search": 50,
            "max_neighbors": 32
        }
    }
)
```

你的向量记忆库的规模（几千到几万 chunk，384 维）下，ChromaDB 的默认参数（M=16, ef_construction=100）已经够用。不需要折腾 IVF+PQ 或其他方案——ANN 搜索在你的场景中不是瓶颈。

## 思考题

1. HNSW 用随机概率决定节点晋升到哪一层——不是启发式规则。为什么要随机？如果改为"连接数越多的节点晋升越高"会怎样？
2. 你的向量库从一万条涨到一百万条时，`efSearch` 需要调大吗？为什么调大能提高召回率，但同时也增加延迟？
