# 4 · 嵌入模型选型与对比

前几篇笔记推导了整个嵌入→相似度→索引的逻辑链。这个链条有一个隐性前提——嵌入模型本身的语义空间和你的文档的语义空间必须重合。如果模型没见过中文哲学笔记的语言分布，你的向量从一开始就投射错了位置。后面的检索和重排做的都是错的坐标上的对的运算。

你的向量记忆库当前用的是 `all-MiniLM-L6-v2`。22.7M 参数，384 维，~80MB，CPU 上每秒处理 ~780 字。这个选型在启动阶段是正确的——它足够小、足够快、能跑。但它中文语义理解很弱：模型以英文训练为主，256 token 的上下文窗口对中文长段落直接截断，在 C-MTEB 中文基准上的表现和专用中文模型差距显著。

推荐迁移目标是 `BAAI/bge-m3`。567M 参数，1024 维，~1.2GB，8192 token 上下文，100+ 语言，中文是母语级的。C-MTEB 得分约 64.6（vs all-MiniLM 的中文表现约在 40-50 区间）。迁移代价：更大的模型（CPU 推理变慢），1024 维向量（存储空间约 2.7 倍），必须重建 ChromaDB 索引。

BGE-M3 有一个独特优势：一个模型同时输出稠密向量（语义搜索）和稀疏向量（关键词匹配）。这意味着你不需要额外维护一个 Elasticsearch 做 BM25——同一模型出两种向量，在 ChromaDB 中统一存储，混合检索。

> 嵌入模型的选型判断标准只有一个：模型训练的语义空间和你的文档的语义空间是否重合。你的笔记是中文的，所以模型必须在中文上强。这不是"BGE-M3 更好"的问题——是 all-MiniLM 和你的文档之间存在一个语义空间不匹配的结构性问题。

迁移建议在 MCP 服务器实现**之前**做——避免后续换模型时所有已注册的 MCP 工具和已存储的向量元数据都要重新适配。如果 CPU 推理 BGE-M3 太慢，可以继续用 all-MiniLM 做原型，但要知道中文检索质量的天花板很低。

其他值得关注的模型：OpenAI `text-embedding-3`（多语言质量顶级但 API 依赖）、`bge-large-zh-v1.5`（纯中文优化，比 BGE-M3 轻但无稀疏输出）、Jina embeddings v3（8192 token，任务特定 LoRA）。这些对你当前的个人 RAG 场景来说都是过度方案——BGE-M3 是投入产出比最高的迁移目标。

---

## 思考题

1. 语义空间不匹配的问题在多语言场景中尤其严重。如果一篇笔记是中英混杂的（中文论述 + 英文术语），all-MiniLM 和 BGE-M3 各有什么优劣？
2. 迁移嵌入模型意味着所有向量都要重新计算。有没有一种方案能在不重建索引的情况下切换模型？
