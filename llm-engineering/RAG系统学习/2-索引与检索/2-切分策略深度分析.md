# 2.2 · 切分策略深度分析

嵌入模型需要输入一段文本，输出一个向量。但如果你的文档是 5000 字的章节，直接用一个向量表示整章——"深模块"、"浅模块"、"Java I/O"、"Unix I/O"全混在一起，向量指向一个模糊的平均值。搜"深模块"能搜到，搜"Java I/O 为什么不好"也能搜到——因为向量被稀释了，什么都能含糊地匹配，什么都不精准。

切分就是要解决这个问题。但切分面临一个不可消除的张力：

```
切太短 → 每个 chunk 语义精准，但缺少上下文。搜到一段"这导致了 Classitis"——前面没提是什么，后面没提怎么解决。
切太长 → 上下文完整，但语义被稀释。搜"模块设计原则"能匹配，搜"Java I/O 的具体问题"也能匹配——都不精准。
```

> 这不是一个可以被"优化解决"的问题。这是一个权衡——你只能在对精准度和对完整度之间选择，不能同时最大化两者。

三种策略对应三种权衡方式。

**固定长度切分**按 token 数硬切，重叠若干 token 防止拦腰斩断。最简单，最暴力。不管你切在哪——句子中间、代码块中间、表格中间都不管。重叠只缓解了拦腰斩断的问题，没解决语义单元被打散的问题。

**自然边界切分**在文档的自然边界处切：`## 标题` > 空行（段落边界） > 句号。等于让作者帮你做切分——作者已经用标题和段落标记了语义边界。对结构良好的文档（Obsidian 笔记），这是最佳方案：零额外计算，尊重作者意图。

**语义切分**对每句话嵌入，计算相邻句子的余弦相似度。相似度突然下降 = 话题切换，在此切断。理论上最优——边界由语义驱动而非规则。实际上并不总是优于简单方案：2026.01 的系统分析发现，在 ≤5000 token 下，按句子切和语义切分统计上无差异。2026.04 的 Scadea 基准中，递归 512-token 切分准确率 69%，语义分块 54%——差了 15 个点。

语义分块为什么没有稳定领先？句子本身就是天然的语义边界——作者已经在话题切换处换了句子。embedding 的平均化效应可能在边界处引入噪声。chunk_size 和 overlap 才是主导变量，不是切分器类型。

对你的向量记忆库，当前 `ingest_notes.py` 按段落切分，一个直接的改进是先按 `## 标题` 切，再在标题段内按段落合并到合适长度。Obsidian 笔记是高度结构化的——每个 `##` 下的内容大概率构成一个语义完整单元。

## 思考题

1. 如果你的笔记有长段落和短列表交替出现，自然边界切分会把列表切碎吗？切碎后每个碎片单独嵌入——对检索有什么影响？
2. Max-Min 方案"先 embed 后 chunk"颠覆了传统流程——如果嵌入的成本已经很低（便宜的嵌入 API），这会不会变成默认的切分方式？什么时候不应该用它？
