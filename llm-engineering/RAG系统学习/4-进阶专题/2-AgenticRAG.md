# 4.2 · Agentic RAG

传统 RAG 是"检索 → 拼接 → 生成"的固定管道。这个管道假设每个查询都需要检索——但实际上很多查询不需要（闲聊、常识、模型已知的知识）。它还假设检索结果总是有用的——但检索可能返回不相关的内容。

Agentic RAG 把检索从管道的一步变成 LLM 可以随时调用的工具。LLM 自己决定要不要检索、检索什么、检索结果好不好、是不是该重新搜。

Self-RAG（Asai et al., 2023）是这条线的里程碑。它在 LLM 生成过程中插入特殊 token：`<Retrieve>`（需要检索）和 `<NoRetrieve>`（不需要），`<Relevant>`（检索结果相关）和 `<Irrelevant>`（不相关，重新搜），`<Cite>`（这条需要引用）和 `<NoCite>`（不需要）。推理时 LLM 先生成决策 token，再根据决策去检索或直接回答。约 30-40% 的输入不需要检索——这意味着标准 RAG 在这些输入上做了不必要的检索。

对你的场景，当你把 MCP `search_notes` 注册给 Claude Code 后，Claude Code 本身就是 Agentic RAG 系统——它可以自主决定什么时候调用 `search_notes` 检索你的笔记。区别在于 Claude 的决策基于自身推理而非专门 fine-tune 的 reflection token。

一个缺失的环节：检索质量判断。Self-RAG 有 critic 步骤判断检索结果好不好。你的向量记忆库缺这个——如果检索到的 5 个 chunk 都不相关，目前没有自动检测机制。复合评分的四维公式可以充当轻量级的 critic——重排后分数最高的 chunk 仍低于阈值，可能意味着检索质量差，需要提醒用户或触发二次检索。

---

## 思考题

1. Self-RAG 的 30-40% `<NoRetrieve>` 比率意味着每三次查询就有一次不需要检索。在你的笔记场景中，什么类型的查询不需要检索就能回答？这个判断标准是"Claude 的训练数据里有没有"还是"这条信息是不是用户独有的"？
2. 当检索结果里的 chunk 都不相关时，Agentic RAG 应该怎么做——扩大检索范围？改写查询？直接告诉用户找不到？这三个选项各在什么情况下最优？
