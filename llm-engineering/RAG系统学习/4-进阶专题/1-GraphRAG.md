# 4.1 · GraphRAG

标准 RAG 擅长"什么是深模块"——检索到相关段落，生成答案。它不擅长"我的笔记里讨论了哪几个核心设计思想"——因为没有一个单独的 chunk 包含这个全局答案。

这不是向量检索"做得不好"的问题。向量检索的定义就是找语义最接近查询的若干片段——它是局部的。当查询需要跨文档的全局综合时，没有片段能单独覆盖答案。你需要从整体中合成答案，而这超出了纯向量检索的能力。

Microsoft GraphRAG（Edge et al., 2024）的解决方案是：先建知识图谱，再在图上的社区结构中生成全局摘要。

管道分五步：(1) LLM 从每个 chunk 提取实体和关系；(2) 构建加权有向图（NetworkX）；(3) Leiden 算法递归检测图的社区结构（每个社区是一个主题簇）；(4) LLM 对每个社区生成结构化摘要——该社区的核心实体、主要关系、主题概括；(5) 查询时，Global Search 做 MapReduce 处理所有社区摘要，Local Search 做图遍历+文本混合检索。

社区检测是递归的——从顶层的大社区到底层的小社区，形成一个多层级结构。这意味着同一套索引可以支持不同粒度的查询：顶层摘要回答"全局主题"，底层细节回答"某个具体实体的关系"。

代价：索引极贵。10 章小说约 470 万 tokens（实体提取）+ 45 万 tokens（社区摘要）。2025 年的动态社区选择（不相关的社区被剪枝）把查询时成本降了 77%，但索引的固定成本仍在。

GraphRAG 不是向量 RAG 的替代品——是补充。绝大多数查询（"XX 是什么意思？"）用向量 RAG 就够了。当你需要"我的笔记里有哪些主要思想流派？"这种全局综合时，GraphRAG 才有价值。对你的向量记忆库，在对话入库和三层架构实现之后再考虑——暂时不是优先级。

## 思考题

1. GraphRAG 的社区摘要和你的三层架构中的层2（离线摘要）有什么本质区别？它们是同一件事的两种说法吗？
