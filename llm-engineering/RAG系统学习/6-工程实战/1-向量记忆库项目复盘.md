# 1 · 向量记忆库项目复盘

对 [向量记忆库](../../向量记忆库/design/00-总览.md) 当前状态的完整盘点：已实现的部分、已设计但未实现的部分、架构设计中的取舍、以及改进方向。

> 目的是在动手补齐缺口之前，先建立对项目全貌的清晰认知——知道什么已经有了、什么还缺着、以及它们之间的依赖关系。

## 架构总览

```
┌──────────────────────────────────────────────────┐
│                  ✅ 已实现                         │
│                                                   │
│  from_scratch.py ──── TF-IDF + char n-grams       │
│  ingest_notes.py ──── ChromaDB + sentence-transformers │
│  search.py ────────── CLI 语义搜索                 │
│                                                   │
│                  ❌ 已设计，未实现                   │
│                                                   │
│  MCP Server ──────── search_notes / save / sync    │
│  复合评分 ────────── 四维 rerank                    │
│  对话入库 ────────── JSONL 提取 + stop hook        │
│  同步触发器 ──────── git hook / cron               │
│  层2 摘要生成 ────── LLM 定期批处理                 │
│  层3 用户画像 ────── 持久化 Markdown 文件           │
└──────────────────────────────────────────────────┘
```

## 已实现部分评估

### `from_scratch.py` (343 行)

纯 Python 标准库实现，零依赖。使用字符 bigram/trigram 分词 + TF-IDF 向量化 + 余弦相似度。

**优点：** 教学价值极高。理解这段代码就能完全理解"向量检索到底在做什么"——不需要 ChromaDB，不需要 sentence-transformers。词汇表上限 10,000 的工程取舍是有意识的设计选择。

**局限：** TF-IDF 基于词频，不是语义理解。"深模块"和"抽象屏障"在 TF-IDF 空间里可能完全不相关——尽管它们语义上高度相关。另外，字符 n-gram 对中文是 workaround，不是最优解。

**定位：** 教学参考，不应在生产中使用。

### `ingest_notes.py` (201 行)

ChromaDB + sentence-transformers 的生产级管道。变更检测（MD5 文件哈希）、按段落切分、元数据标注（源路径、标题、章节、标签）、写入 ChromaDB。

**优点：** 设计良好的管道。MD5 变更检测避免了全量重建，`--force` 标志处理了需要手动重建的情况。元数据结构合理，`type: note` 字段为未来的对话类型预留了空间。

**可改进点：**
- 当前只支持按段落边界切分，不支持按标题切分（对结构化的 Obsidian 笔记来说，按 `##` 标题切分可能更合适）
- 没有实现设计文档 [03](../../向量记忆库/design/03-切分策略.md) 中讨论的 Max-Min 切分实验
- `sync_notes` 逻辑在脚本里，但没有被 MCP 工具包装，Claude Code 无法调用

### `search.py` (79 行)

CLI 搜索工具。支持 `--top-k` 和 `--type` 过滤。

**优点：** 简洁、可用。输出格式包含源路径、标题、章节、日期——可读性好。

**可改进点：**
- 不支持复合评分重排（设计/07 的功能）
- 不支持查询改写（HyDE 等）
- 不区分 source_type（note/conversation），只有 `--type` 的简单过滤
- 没有任何缓存策略，每次搜索都重新加载模型

## 已设计但未实现部分分析

### 优先级评估

| 缺口 | 优先级 | 理由 |
|------|--------|------|
| MCP 服务器 | ==最高== | 这是入口。没有它，Claude Code 无法调用向量记忆库。后续所有功能都依赖它 |
| 复合评分 | 高 | 显著提升检索质量。当前只用语义相似度，缺少深度/最近性/重要性维度 |
| 对话入库 | 高 | 这是让"记忆库"真正有记忆的关键——不只是笔记，还有对话历史 |
| 库同步触发器 | 中 | 已有 MD5 变更检测，只是缺自动触发。git hook 和 cron 都不复杂 |
| 层2 摘要生成 | 中 | 需要 LLM 集成，但逻辑清晰 |
| 层3 用户画像 | 低 | 依赖层2 和对话入库先有足够的数据积累 |

### MCP 服务器（设计/04）

设计文档中已经包含了约 130 行的 MCP 服务器代码框架。三个工具：
- `search_notes(query, top_k, source_type)` — 搜索 + 重排
- `save_to_memory(content, tags, importance)` — 显式持久化
- `sync_notes(source_dir)` — 触发笔记同步

**实现时需要考虑的：** MCP Server 是独立进程，需要在 Claude Code 启动时自动拉起。这意味着要处理进程生命周期、错误恢复、以及 ChromaDB 的并发访问。

### 复合评分（设计/07）

设计中的四维评分公式：

```
final = semantic_sim * w1 + depth_score * w2 + recency * w3 + importance * w4
```

- **semantic_sim**：余弦相似度（已有）
- **depth_score**：重复提及计数（需要维护 mention_count 元数据）
- **recency**：指数衰减，半衰期 30 天（需要在元数据中加入最后访问时间）
- **importance**：显式标签 + 对话长度 + 持久化标记（需要 save_to_memory 先实现）

**实现时需要考虑的：** 权重 w1-w4 的默认值如何确定？设计文档没给具体值。需要实验调参。另外，四个维度的分数需要归一化到同一尺度。

### 对话入库（设计/05）

最复杂的未实现功能。三个关键子问题：
1. JSONL 解析：Claude Code 的对话日志格式是什么？需要先收集实际日志分析
2. 触发方式：精确 MCP 调用（用户主导）vs. Stop Hook 自动采集（全自动）vs. 混合
3. 隐私过滤：需要实现 `sanitize()` 函数，自动洗刷 API 密钥等敏感信息

**实现时需要考虑的：** Stop Hook 是 Claude Code 的功能，需要配置 `settings.json`。但前提是先把逻辑写成可调用的脚本。

## 架构设计值得保留的决策

1. **三层架构（设计/08）**：向量检索 + 定期摘要 + 用户画像的分层设计是合理的。即使暂时只实现层1，也为后续扩展留好了接口
2. **复合评分的思想**：单纯依赖语义相似度确实不够。记忆不是"语义最相关的就是最重要的"——频率、时间、显式标记都是维度
3. **ChromaDB 选型**：轻量、Python-native、不需要额外服务。对个人使用场景是正确的
4. **变更检测用 MD5**：简单、有效、够用。没有过度工程化

## 架构设计值得重新审视的决策

1. **`all-MiniLM-L6-v2` 对中文的支持**：这个模型主要基于英文训练。设计/02 中已经提到了"若中文效果不足则切至 BGE-M3"。建议在路径早期（MCP 服务器实现之前）就做这个评估，避免后续迁移成本
2. **按段落切分 vs. 按标题切分**：对 Obsidian 笔记，按 `##` 标题切分可能比按段落切分效果更好。设计/03 中推荐了基于标题的切分，但 `ingest_notes.py` 当前实现的是按段落
3. **ID 模式**：`{filepath}#{chunk_index}` 是脆弱的——文件重命名会导致所有 chunk ID 失效。但这对个人使用场景影响不大

## 改进优先级路线图

```
Phase 1（立即）: 评估嵌入模型 → 决定是否迁移到 BGE-M3
Phase 2（核心）: 实现 MCP 服务器 → 让 Claude Code 能调用向量记忆库
Phase 3（增强）: 实现复合评分 → 提升检索质量
Phase 4（扩展）: 实现对话入库 → 让记忆库有"记忆"
Phase 5（远期）: 层2 摘要 + 层3 画像 → 三层架构完整闭环
```

> MCP 服务器是整个项目的瓶颈——没有它，向量记忆库对 Claude Code 来说是隐形的。所以优先级最高。
