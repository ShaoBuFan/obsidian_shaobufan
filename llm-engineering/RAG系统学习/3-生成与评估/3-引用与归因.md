# 3.3 · 引用与归因

RAG 的输出如果没有引用来源，用户无法判断答案中哪句话来自检索内容、哪句是 LLM 编的。

引用的唯一目的是可验证性——让用户能回到原文确认"这真的是原文的意思吗"。做不到这一点的引用和幻觉没区别。

三种引用策略。内联引用（逐句标注来源编号 `[1]`）最精确但最容易被误读——LLM 可能在引用正确的来源但歪曲了来源的意思。块级引用（答案末尾列出所有来源）适合答案是从多个来源综合而来的总结类问题。来源链接 + 原文摘要让用户能快速判断"这个来源值得点回去看吗"。

两个最危险的陷阱：(1) LLM 编造引用——"来源 3: 数据结构与算法.pdf 第 45 页"，来源文件根本不存在。解决方法：prompt 中明确要求**只使用提供的来源编号**。(2) 来源对了但理解错了——引用了第 4 章但把"深模块"理解成了"代码写得很长的模块"。内联引用格式本身无法检测这种错误，需要 faithfulness 评估（RAGAS）做追加检验。

对你的向量记忆库，MCP `search_notes` 的返回格式如果显式包含一个 `citation` 字段（格式化的引用文本），可以让 Claude Code 更容易标注来源。当前返回的 `source` 和 `metadata` 字段已经包含了引用所需的信息。

## 思考题

1. LLM 编造引用和"理解错了但来源引用正确"——哪个在实际使用中更难被发现？对你的个人笔记场景，哪个危害更大？
2. 如果你的笔记里有两个 chunk 对同一概念给出了不同定义（来自不同时间的笔记），引用时该怎么处理？有没有可能让系统检测到这种矛盾并提示用户？
