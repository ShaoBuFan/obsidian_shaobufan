# HyDE · Precise Zero-Shot Dense Retrieval without Relevance Labels

Gao et al. (CMU / Stanford), ACL 2023. 查询改写方向最有影响力的论文之一。

## 核心问题

稠密检索依赖查询和文档在同一个向量空间中的相似度。但查询通常很短（几个词），文档通常很长（几百词）——它们的向量分布天然不同。能否在不重新训练嵌入模型的情况下，把短查询"拉"到长文档的语义空间中？

## 方法

思路出人意料但极其有效：==让 LLM 先编一个假答案，然后用这个假答案的向量去检索==。

```
用户查询: "心肌梗死有什么症状？"
    ↓
LLM 生成假答案 (HyDE document):
"心肌梗死，俗称心脏病发作，是由于冠状动脉阻塞导致心肌缺血坏死的急症。
常见症状包括胸痛（压榨感或灼烧感，持续超过几分钟）、左臂或左肩放射痛、
呼吸困难、出冷汗、恶心呕吐、头晕或晕厥。部分患者特别是糖尿病患者和
老年人可能表现为不典型的症状如上腹痛或单纯乏力..."
    ↓
假答案 → embedding → 向量检索 → 真实相关文档
```

**为什么这能工作？**

==查询和文档之间的"词汇-语义鸿沟"==是稠密检索的主要障碍。用户写"心肌梗死症状"，医学文献写"急性心肌梗死的临床表现包括胸骨后压榨性疼痛...（300 字）"。两者的向量可能指向不同的方向。

但 LLM 生成的假答案——即使可能包含事实错误——使用的是**和真实文档相似的词汇分布和句式结构**。它把短查询展开成了和文档同分布的长文本。

> HyDE 最反直觉的核心：假答案的**事实正确性不重要**。重要的是它的词汇和句式接近真实文档。一个概念上正确但措辞不像医学文献的答案，检索效果不如一个措辞像医学文献但部分事实有误的答案。

## 关键实验

- **无监督设置**（完全不用标注数据）：HyDE 在多个检索基准上超越当时 SOTA 的无监督稠密检索方法，包括 Contriever（无监督版）
- **与 BM25 对比**：HyDE + 稠密检索的组合在大多数任务上优于 BM25
- **语言模型的影响**：用更大的 LLM（InstructGPT 175B vs FLAN-T5 3B）生成 HyDE，检索效果持续提升。这暗示 HyDE 的上限受限于 LLM 的生成质量
- **对不同查询类型的适用性**：定义类、解释类问题效果好；精确事实查询（"某人的出生日期"）下 HyDE 可能引入误导性信息

## 优势与局限

**优势：**
- 完全零样本——不需要任何检索相关标注数据
- 实现极简单：一个 prompt + 一次 embedding + 一次向量检索
- 可叠加到任何稠密检索系统上，不需要修改嵌入模型或向量库

**局限：**
- 每次查询多一次 LLM 调用，增加延迟和成本
- LLM 生成的假答案如果有严重的领域知识错误，可能把查询方向带偏
- 对关键词式短查询（"ERR_CONN_REFUSED"）可能生成冗长但不精确的假答案

## 对本项目的启发

1. **你的向量记忆库最直接的检索质量提升手段**：在 MCP `search_notes` 中加一个可选的 HyDE 步骤。代价是每次搜索多一次 Claude 调用，但对你个人的使用频率来说完全可接受。

2. **HyDE 在你的笔记场景最有效的查询类型**：定义类问题（"什么是深模块？"）、概念比较（"战术性编程和战略性编程的区别？"）。这些正是学习笔记检索的主要用例。

3. **不需要"正确答案"这个特性的实际含义**：你不需要为 HyDE 维护一个"标准答案库"——LLM 随便编的答案已经够用了。这使得 HyDE 成为==零维护成本的检索增强==。

4. **验证了你在 [查询改写与扩展](../2-索引与检索/8-查询改写与扩展.md) 中的分析**：HyDE 确实对精确查询不适用——"CS61A 的作业截止日期"这种问题，用原始查询直接检索可能更好。
