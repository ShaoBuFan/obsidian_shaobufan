# RAPTOR · Recursive Abstractive Processing for Tree-Organized Retrieval

Sarthi et al. (Stanford), 2024. 分层索引的代表作。

## 核心问题

标准 RAG 把文档切成等长的 chunk，每个 chunk 独立嵌入和检索。这对于"一段话能回答"的问题是好的。但很多问题需要**跨多个 chunk 的综合理解**——"这本书讨论了哪几个核心主题？"没有一个单独的 chunk 能回答这个问题，因为它们各自只覆盖了一个主题。

==能否构建一个多层级的文档索引，让检索可以在不同的抽象层次上进行——从细节段落（底层）到全局主题（顶层）？==

## 方法

RAPTOR 构建一个**树状索引**：

1. **底层（Leaf Nodes）**：文档按固定长度切成 chunk，每个 chunk 嵌入
2. **聚类**：用 SBERT 嵌入 + 高斯混合模型（GMM）对 chunks 做软聚类。GMM 允许一个 chunk 属于多个类——因为一个段落可能同时涉及"模块深度"和"接口设计"
3. **摘要**：对每个聚类，用 LLM 生成一个结构化摘要——该聚类的核心主题是什么
4. **递归**：摘要作为上一层的节点，再次聚类 + 摘要。重复直到只剩一个或几个摘要——这就是整篇文档的全局摘要

```
                    ┌──── 全局摘要 ────┐          ← 顶层: "这本书的核心论点是..."
                    /        │          \
            ┌──摘要A──┐ ┌──摘要B──┐ ┌──摘要C──┐   ← 中层: 主题级
            /  |  \      /  \        /    \
           c1 c2 c3    c4 c5 c6    c7 c8 c9      ← 底层: chunk 级
```

**检索时怎么做？**

两种策略：
- **Tree Traversal**：从顶层往下逐层检索。顶层确定"哪个主题相关"，逐层细化到具体 chunk
- **Collapsed Tree**：把树的某一层（通常是底层 leaf chunks + 顶层全局摘要）展开成平列表，用标准向量检索。简单，但在顶层摘要上也能命中全局问题

> RAPTOR 的核心洞察：聚类 + 摘要的过程在建立一种**文档的语义层级结构**。这个结构不是人工定义的目录，而是从内容中自动涌现的。底层的 chunk 负责"这个细节是什么"，上层的摘要负责"这些细节在谈什么共同主题"。

## 关键实验

- 在长文档 QA（QASPER、QuALITY）上，RAPTOR 的 Collapsed Tree 检索策略显著优于标准 chunk-only 检索
- 全局性问题（"总结本文档的主要论点"）的准确率比基线提高约 20%
- GMM 软聚类 vs K-Means 硬聚类：软聚类的效果更好，因为一个段落确实可能同时属于两个主题

## 优势与局限

**优势：**
- 优雅地解决了"全局理解"和"细节检索"的矛盾——不同层级的节点服务不同粒度的查询
- GMM 软聚类避免了强行把一个段落归入单一主题的武断
- 树结构的构建完全自动化，不需要人工标注

**局限：**
- 构建成本高：需要大量 LLM 调用来生成摘要。一篇长文档可能需要几十到上百次 LLM 调用
- 树结构是静态的：文档更新后需要重建。对频繁更新的知识库不友好
- 聚类数量的选择对结果敏感——太少则摘要过于宽泛，太多则层级过深

## 对本项目的启发

1. **与你的三层架构（设计/08）的相似和不同**：RAPTOR 的"摘要层"和你的层2（定期摘要生成）思路一致，但 RAPTOR 的摘要是**聚类驱动**的（相似内容归在一起再总结），你的设计是**时间驱动**的（定期批处理）。聚类驱动对于发现"潜藏的主题关联"可能更有价值。

2. **对 Obsidian 笔记的适用性有限**：你的笔记本身就是结构化的（有标题层级）。RAPTOR 主要解决的是**无结构长文档**的索引问题。对于已经有 `## 标题` 做自然切分的笔记来说，RAPTOR 的聚类步骤可能没有太多增量价值——作者已经帮你"聚类"好了。但段落间的跨文件主题关联是 RAPTOR 可能提供增量价值的地方。

3. **Collapsed Tree 是最实用的策略**：把几层都展开成平列表做检索，简单有效。如果你的向量记忆库未来实现了层2摘要，检索时可以考虑"原始 chunks + 全局摘要"的 Collapsed Tree 式混合检索。
