# Agent 架构全景：ReAct 及其后继者

## 为什么 ReAct 不够

ReAct 解决了一个问题：让模型在推理和行动之间交替。但 ReAct 留下了三个缺口：

1. **规划短视**：ReAct 每步只思考下一步，没有全局计划。如果第 3 步发现第 1 步走错了方向，前面两步的 token 和时间全浪费了。
2. **一次性执行**：ReAct 执行完就结束了。如果最终答案错了，它不会反思"我哪里出了问题"并重试。
3. **线性思维**：ReAct 沿着一条路径走到底。如果当前 Thought 引导的路径是死胡同，它没有"回到分岔口试另一条路"的机制。

从 2023 到 2026 年，研究者针对这三个缺口提出了各自的解决方案——形成了 Plan-and-Solve、Reflexion、Tree-of-Thought 三大后继架构。

## 四大架构对比

### 核心机制差异

```
ReAct:          T₁ → A₁ → O₁ → T₂ → A₂ → O₂ → ... → Answer
Plan-Solve:     Plan(全局) → Execute(T₁→A₁→O₁) → Execute(T₂→A₂→O₂) → ...
Reflexion:      [Episode₁: ReAct → 自评 → 记忆存储]
                [Episode₂: ReAct(参考Episode₁记忆) → 自评 → 记忆更新] → ...
Tree-of-Thought: T₁ → [A₁a, A₁b, A₁c] → [评分] → 剪枝 → 扩展最优分支 → 回溯 → ...
```

### 完整对比

| 维度 | ReAct | Plan-and-Solve | Reflexion | Tree-of-Thought |
|------|-------|----------------|-----------|-----------------|
| **核心创新** | 推理与行动交替 | 先规划全局再执行 | 跨次尝试的自评学习 | 多路径搜索+回溯 |
| **解决 ReAct 什么问题** | —（基准） | 规划短视 | 一次性执行 | 线性思维 |
| **代表论文** | Yao et al. 2022 | Wang et al. 2023 | Shinn et al. 2023 | Yao et al. 2023 |
| **准确率（GPT-5 10 基准平均）** | 57.1% | 52.5% | 54.2% | — |
| **Token 开销（vs Direct）** | 4.0× | 6.9× | 9.4× | 极高（~71 次 LLM 调用） |
| **并行能力** | 无（完全串行） | 有（独立子任务可并行） | 无（需等上一轮结束） | 有（分支独立探索） |
| **实现复杂度** | 低 | 中 | 中 | 高 |
| **最适合的场景** | 开放域 QA、信息检索 | 结构化多步工作流 | 代码生成、高质量写作 | 数学证明、游戏、搜索问题 |
| **2026 年定位** | 安全默认选择 | 结构化工作流首选 | 质量优先的增强层 | 专项问题工具 |

### 各架构的独特优势

**ReAct —— 动态适应**
当任务路径不可预测、需要根据实时反馈调整策略时，ReAct 的灵活性无可替代。客服对话、故障排查、开放式研究查询都是它的主场。

**Plan-and-Solve —— 全局优化**
当任务有清晰的依赖结构时（如"先分析数据→再生成图表→最后写报告"），提前规划可以避免 ReAct 的"走一步看一步"造成的路径探索浪费。独立子任务可并行执行——延迟降低 ~3.7×。

**Reflexion —— 从失败中学习**
当第一次尝试几乎不可能完美时（代码调试需要 5-10 次迭代），Reflexion 的跨次学习能力让第二、第三次尝试比第一次显著提升。HumanEval 上达到 91% pass@1。

**Tree-of-Thought —— 搜索全局最优**
当问题的搜索空间足够小且需要确保找到最优解时（24 点游戏、填字游戏），多路径探索+剪枝+回溯比单路径试错更有效。

## 2026 年的关键演进（ReAct 之上）

### Pre-Act（2025）：结构化企业场景

Uniphore 提出的 Pre-Act 在 ReAct 循环之前插入一个**前置规划阶段**：

| | ReAct | Pre-Act |
|---|---|---|
| 推理方式 | 与行动交替 | 先制定完整计划，再逐步执行 |
| 合规审计 | 较难保证 | 内置合规嵌入 |
| 适用场景 | 开放域、零样本 | 结构化企业工作流（客服、审批） |

### Co-ReAct（2026 年 5 月）：评分标准驱动的步级引导

arXiv:2605.23590。引入==评分标准（Rubrics）作为每一步的决策协作者==。在每一步决策点注入评分标准，指导模型下一步应关注什么证据。使用 GRPO 训练评分标准生成器，在 DeepResearchBench 和 SQA-CS-V2 上实现一致性提升。

### ReAct V3 + 半形式化推理（2026）：强制逻辑证书

ServiceNow 的 ReAct V3 引入半形式化推理——模型必须构建"前提 → 执行轨迹 → 形式化结论"的逻辑证书。准确率从 78% 提升到 93%。同时引入并行执行模式——多个工具调用同时发起，用"证据门"替代串行等待。

### CodeAct（2025-2026）：从文本 Action 到代码 Action

CodeAct 将 Action 从 API 调用拓展为 Python 代码执行。"理解→编码→执行→验证"的闭环比"Thought→Action→Observation"更紧凑——因为代码本身就是一种高信息密度的 Action。

## 2026 年共识：没有银弹，只有路由器

[Select-then-Solve](https://arxiv.org/html/2604.06753v1)（2026 年 4 月）的发现具有总结意义：==没有任何单一范式在所有任务上最优==。ReAct 在 GAIA 上以 44pp 的优势碾压 Direct，但在代码生成上输给 Direct。

该论文提出的解决方案是一个**学习路由器**——根据任务特征选择最合适的范式，相比任何固定范式提升 ~3pp 准确率。

```
任务输入 → [路由器：分析任务特征] 
         → 选择范式(ReAct / Plan-Solve / Reflexion / Direct)
         → 执行
         → 输出
```

> 2026 年的 Agent 工程不是选择"最好"的架构，而是**根据不同任务动态选择合适架构**的元层次设计。

## 与 Function Calling 的关系

四大架构都是在 Function Calling 之上的**决策层抽象**。它们不替代 Function Calling——它们决定**何时、以何种策略**调用 Function Calling 来执行 Action。详见 [ReAct 与 Function Calling 的对比](../2-对比分析/ReAct与FunctionCalling的对比.md)。

## 思考题

1. Reflexion 的跨次学习依赖 episodical memory 的质量。如果模型在前一次尝试中进行了错误的自我评估（把对的说成错的），这个错误会污染后续尝试。你有什么机制可以防止这种"自我评估偏差"的传播？
2. 学习路由器选择范式——这个路由器本身也是一个 LLM 调用，增加了成本。在什么条件下路由器的收益（选对范式带来的准确率提升）超过它的成本？
3. Plan-and-Solve 在 2026 年基准上的准确率（52.5%）反而不如 ReAct（57.1%），但它的并行能力降低了延迟。在生产环境中，延迟和准确率之间应该如何权衡？有没有可能两者兼得？
