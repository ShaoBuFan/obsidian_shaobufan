# 库更新与同步

## 核心问题

向量库没有外键和级联更新。笔记改了 → 旧向量残留 → 检索到过时内容。

需要回答三个问题：**什么变了？变了的怎么更新？什么时候触发？**

对话的更新逻辑（只追加不修改）见 [05-对话入库](05-对话入库.md)。

## 变更检测（三种方式）

| 方式 | 做法 | 复杂度 | 适用 |
|------|------|--------|------|
| **全量重建** | 删光 collection，重新索引 | 最低 | 数据量 < 万级 |
| **文件哈希对比** | 每笔记存 md5，只更新变化的文件 | 中等 | 数据量增长后 |
| **块级哈希对比** | 每个 chunk 存 md5 | 最高 | 巨型库或协同编辑 |

> 本 vault 目前 24 章笔记 + 若干对话，全量重建只需几秒。先用方式 1，等数据量起来再切换到方式 2。

### 文件哈希对比的实现

```python
import hashlib, json, os

MANIFEST_PATH = "vector_memory_db/manifest.json"

def load_manifest():
    if os.path.exists(MANIFEST_PATH):
        return json.load(open(MANIFEST_PATH))
    return {}

def save_manifest(m):
    json.dump(m, open(MANIFEST_PATH, "w"), indent=2, ensure_ascii=False)

def detect_changes(file_list):
    """对比文件哈希，返回变更文件列表"""
    old_manifest = load_manifest()
    changed = []
    new_manifest = {}

    for fpath in file_list:
        md5 = hashlib.md5(open(fpath, "rb").read()).hexdigest()
        new_manifest[fpath] = md5
        if old_manifest.get(fpath) != md5:
            changed.append(fpath)

    # 检测已删除的文件
    deleted = set(old_manifest.keys()) - set(new_manifest.keys())

    save_manifest(new_manifest)
    return changed, list(deleted)
```

## 更新策略：整文件替换

以**文件**为最小更新单元（而非段落），原因：一段改了通常周边也受影响，整文件替换最干净。

ID 的设计是关键。用 `{filepath}#{chunk_index}` 作为 chunk id，天然保证同文件的 chunks 可精确定位和覆盖。

```python
def sync_file(filepath, collection, model):
    """同步单个文件：删旧 chunks → 重新切分 → 写新 chunks"""
    collection.delete(where={"source": filepath})

    text = read_markdown(filepath)
    chunks = chunk_text(text)

    embeddings = model.encode([c["text"] for c in chunks])
    collection.add(
        documents=[c["text"] for c in chunks],
        metadatas=[{**c["metadata"], "source": filepath,
                    "md5": file_md5(filepath)}
                   for c in chunks],
        ids=[f"{filepath}#{i}" for i in range(len(chunks))],
        embeddings=embeddings.tolist()
    )
```

## 触发时机

| 时机 | 机制 | 适用 |
|------|------|------|
| **手动** | `python3 sync.py` | 笔记变更后主动执行 |
| **MCP 工具** | `sync_notes` 工具，对话中按需调用 | Claude Code 内直接触发 |
| **Git Hook** | `post-commit` 触发同步 | vault 用 git 管理时自动运行 |
| **定时任务** | cron 每天跑一次 | 设置后就不用想了 |

推荐：MCP 工具 + 手动脚本。笔记变更频率不高，手动最实际。MCP 工具 `sync_notes` 定义见 [04-MCP服务设计](04-MCP服务设计.md)。

## 数据完整性保障

### 1. 部分写入失败

文件删了旧 chunk 但新 chunk 插入失败，导致该文件在库里"消失"。解决：先插入新 chunks，插入成功后再删旧的（原子性靠顺序保证）。

```python
def sync_file_safe(filepath, collection, model):
    new_ids = [f"{filepath}#{i}" for i in range(len(new_chunks))]
    # 1. 先插入新 chunks
    collection.add(ids=new_ids, documents=...)
    # 2. 再删除不属于 new_ids 的旧 chunks
    old = collection.get(where={"source": filepath})
    for old_id in old["ids"]:
        if old_id not in new_ids:
            collection.delete(ids=[old_id])
```

### 2. manifest 与实际不同步

手动删了笔记但 manifest 不知道，旧 chunks 会永远留在库里。定期运行**孤儿检测**：

```python
def cleanup_orphans(collection):
    """删除源文件已不存在的 chunks"""
    all_metadatas = collection.get()["metadatas"]
    for meta in all_metadatas:
        src = meta.get("source", "")
        if src and not os.path.exists(src):
            collection.delete(where={"source": src})
```

### 3. 嵌入模型换了

从 `all-MiniLM-L6-v2` 升级到更好的模型，所有旧向量作废，必须全量重建。建库时在 metadata 里记录模型名：

```python
metadata = {"embedding_model": "all-MiniLM-L6-v2"}
# 升级时检测
if meta["embedding_model"] != CURRENT_MODEL:
    force_rebuild = True
```
