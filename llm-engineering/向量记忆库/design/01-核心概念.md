# 核心概念

## 一句话定义

向量记忆库 = 把文字转成向量（embedding），检索时找语义最相近的片段，喂给 LLM 做上下文增强（RAG）。

## 核心管道

```
笔记/对话 → Embedding模型 → 向量 → 存入向量库
查询     → Embedding模型 → 向量 → 相似度搜索 → 最相关的K条 → 喂给LLM
```

## 向量不是"装"内容的容器

384 维向量只是一个**坐标/指纹**，用于定位语义相近的文本。

> 384 维向量 ≈ 图书馆索书号 —— 告诉你书在哪，不装书的内容。
> 原文 ≈ 书本身 —— 原封不动存在库里。

所以"384 维向量能容纳 700 万字吗"这个问题本身不成立。向量只参与**检索环节**，不参与存储。实际存储结构：

| 组成部分 | 估算（700万字、500字/段、384维） |
|----------|----------------------------------|
| 原文文本 | ≈ 14 MB（中文 UTF-8） |
| 向量索引 | ≈ 21.5 MB（14000段 × 384维 × 4字节 float32） |
| **总计** | **≈ 35 MB** |

## 相似度：余弦相似度

两个向量的夹角越小，语义越近。

```python
cosine_sim = np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
# 返回 [-1, 1]，1 = 完全相同方向，0 = 正交/无关，-1 = 完全相反
```

实际检索过程：

```python
query_vec = model.encode("闭包相关概念")        # 查询向量化
results = collection.query(query_vec, top_k=5)  # 在库里找最近的5个
# → 返回原文片段 + 元数据 + 相似度分数
```

## 向量数据库的职责

| 职责 | 说明 |
|------|------|
| 存储 | 向量 + 原文 + 元数据 |
| 索引 | 为海量向量建索引，避免每次全量扫描 |
| 检索 | 给定查询向量，快速返回 top-K 最近邻 |
| 过滤 | 按元数据筛选（如"只搜笔记，不搜对话"） |

具体工具选型见 [02-工具选型](02-工具选型.md)。
