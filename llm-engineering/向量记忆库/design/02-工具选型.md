# 工具选型

## 向量数据库对比

| 方案 | 定位 | 适合 |
|------|------|------|
| **NumPy 手写** | 理解原理，内存暴力搜索 | 学习阶段 |
| **FAISS** (Meta) | C++ 引擎，只存向量不存文档 | 百万级以上、需要极致性能 |
| **ChromaDB** | 封装 FAISS/hnswlib，向量+文档+元数据一体化 | 学习 + 小规模项目 |
| **Milvus** | 分布式生产级 | 大规模部署 |

> 推荐入门路线：NumPy 手写（理解本质）→ ChromaDB（第一个可运行系统）。

## FAISS 的两种索引策略

解决的核心问题：百万/亿级向量中快速找最近邻。暴力搜索 O(N×D) 太慢。

| 策略 | 思路 | 内存（百万条/384维） | 精度 |
|------|------|---------------------|------|
| Flat（暴力） | 全量逐一算 | 1.5 GB | 100% |
| **HNSW**（分层图） | 向量组织成图，沿边走 | **≈ 1.6 GB**（向量 + 图边） | 99%+ |
| IVF+PQ（聚类+压缩） | 先聚类再搜 + 向量压缩到 64 字节 | **≈ 100 MB** | 90-95% |

HNSW 内存大的原因：完全不压缩向量，还额外存图边（每个节点连约 32 个邻居）。但个人知识库万级段落只需几十 MB，完全不是问题。PQ 压缩是千万级以上才需要关心的。

## ChromaDB vs FAISS 的关系

ChromaDB 底层用的就是 hnswlib（HNSW 实现），和 FAISS 算法同源。区别：

| | FAISS | ChromaDB |
|------|------|------|
| 抽象层级 | C++ 引擎 + Python 绑定 | 完整数据库 |
| 存什么 | 只有向量 | 向量 + 文档 + 元数据 |
| 查询返回 | 向量 ID | 原文 + 元数据 + 距离 |
| 元数据过滤 | 无，自己另外存 | 内置 `where` 语法 |

> 对本项目：ChromaDB。不需要额外维护文档存储和向量索引的一致性。

## Embedding 模型选型

| 方案 | 模型 | 维度 | 特点 |
|------|------|------|------|
| **本地** | `all-MiniLM-L6-v2` | 384 | 免费、离线、CPU 可跑、几十 MB |
| **本地（中文强）** | `BGE-M3` | 1024 | 多语言效果好、支持稀疏+稠密混合检索 |
| **API** | OpenAI `text-embedding-3-small` | 1536 | 效果好、按量付费、需网络 |

推荐：入门用 `all-MiniLM-L6-v2`，中文效果不够再换 `BGE-M3`。

注意：换模型后所有旧向量作废，必须全量重建。建库时在 metadata 里记录模型名：

```python
metadata = {"embedding_model": "all-MiniLM-L6-v2"}
```

## 本项目推荐栈

```
ChromaDB + all-MiniLM-L6-v2（入门）
       ↓ 中文效果不够
ChromaDB + BGE-M3
       ↓ 数据量过万
ChromaDB 不变，切分策略从自然边界升级到 Max-Min
```
