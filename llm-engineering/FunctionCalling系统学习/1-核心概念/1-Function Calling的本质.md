# Function Calling 的本质

## 从"如果它不存在"出发

想象一个只能生成文本的 LLM。用户问："今天北京天气怎么样？"模型会怎么做？

它只能**猜**。它的训练数据截止于某个日期，它不知道"今天"是什么，它没有接入任何实时数据源。它能生成一段看起来像天气预报的文字，但那是幻觉——看起来合理，实际上毫无根据。

这就是 Function Calling 要解决的根本问题：==LLM 的知识是静态的、被冻结在训练截止日期的==。任何需要实时数据、私有数据、精确计算、外部操作的场景，纯文本生成都无法胜任。

> Function Calling 不是让 LLM"拥有"了调用工具的能力——它只是让 LLM 学会了用**结构化的方式表达**"我想调用哪个工具，传什么参数"。

这里面有一个新手最容易混淆的点：==模型只负责"表达意图"，不负责"执行"==。模型返回的是 `{"name": "get_weather", "arguments": {"city": "北京"}}`——一段结构化的文本。真正去调天气 API、查数据库、操作文件系统的，是你的代码。模型既没有权限也没有能力执行任何外部操作。

这个区分之所以重要，是因为它决定了安全模型：**你永远不应该把模型输出直接当作可执行指令**。即使模型生成了一个"调用 delete_database"的工具调用，你的代码也必须在执行前做验证、确认、权限检查。

> 把 Function Calling 理解为"模型的输出格式从自然语言变成了 JSON"虽然不完全准确，但比"模型可以执行工具了"这个误解好得多。

## 本质：这是一个协议，不是一种能力

如果用一个比喻来理解：

- **没有 Function Calling**：你给 LLM 发短信，LLM 回你一段自然语言。你需要自己从自然语言里解析出意图。
- **有 Function Calling**：你和 LLM 之间建立了一个**结构化协议**。你告诉 LLM 有哪些工具可用（工具定义），LLM 决定要不要用、用哪个、传什么参数（工具调用），你执行完把结果告诉 LLM（工具结果），LLM 基于结果生成最终回复。

这个协议的核心是一个三步循环：

```
用户输入 → [模型决定: 需要调用工具] → 返回 tool_calls
         → [你的代码执行工具] → 返回 tool_results
         → [模型基于结果生成回复] → 返回最终文本
```

## 为什么不能直接在 prompt 里描述工具？

这是新手最常问的问题："我在 system prompt 里告诉模型有哪些函数，让它返回 JSON 不就行了？"

在 Function Calling 出现之前，这确实是主流做法。但有两个致命缺陷：

1. **不可靠的格式**：自然语言描述的约束无法保证模型一定返回合法的 JSON。缺少花括号、多一个逗号、字段名拼写错误——这些都会让下游解析崩溃。
2. **不可靠的决策**：模型可能在不需要调用工具时调用工具，或者在应该调用时选择不调用。没有结构化的 `tool_choice` 机制来约束这种行为。

> Function Calling 的核心价值不在于"让模型知道有哪些工具"——prompt 也能做到这一点——而在于**通过训练和架构层面的约束，让工具调用的格式和决策变得可靠**。

更具体地说，Function Calling 提供了三个 prompt 无法提供的保障：

| 保障 | Prompt 方式 | Function Calling |
|------|-------------|------------------|
| JSON 结构合法性 | 不保证 | `strict: true` 下**完全保证** |
| 参数类型正确性 | 不保证 | Schema 约束下高度可靠 |
| 是否调用工具的决策 | 不可控制 | `tool_choice` 精确控制 |

## 工具调用的四种模式

根据控制权的分配，工具调用大致分为四种模式：

### 1. 自动模式（`tool_choice: "auto"`）

模型自行决定是否调用工具、调用哪个工具。这是最常用的模式，适合大多数对话式应用。

```
用户："帮我查一下北京的天气"  → 模型决定调用 get_weather({"city": "北京"})
用户："你好"                   → 模型决定不调用任何工具，直接回复
```

### 2. 强制模式（`tool_choice: "required"` / `"any"`）

要求模型必须调用至少一个工具。适用于你事先知道"这个问题一定需要工具"的场景。

> 强制模式的核心权衡：减少一次网络往返（模型直接输出工具调用，省略"要不要调用"的推理文本）vs. 失去决策质量（模型不再有机会在生成过程中判断"这个工具调用是否真的必要"）。不同平台对这个权衡的具体实现有差异——具体见各平台详解。

### 3. 指定工具模式（`tool_choice: {"type": "tool", "name": "xxx"}`）

强制模型调用某个特定工具。适用于明确的工具路由场景——比如"这条路只通一个工具"。

### 4. 禁用模式（`tool_choice: "none"`）

禁止模型调用任何工具，只生成纯文本。适用于你希望隔离工具调用上下文的场景。

## 为什么是 JSON Schema

两个平台不约而同地选择了 ==JSON Schema== 作为工具参数的定义语言。这不是巧合。

JSON Schema 是描述 JSON 数据结构的事实标准，它提供了：
- **类型系统**：`string`、`number`、`integer`、`boolean`、`array`、`object`、`null`
- **约束机制**：`enum`、`required`、`additionalProperties`
- **组合能力**：`anyOf`、`allOf`、`oneOf`
- **可扩展性**：通过 `$ref`/`$def` 复用定义

而且 JSON 是 LLM 训练数据中大量存在的格式，模型对 JSON 结构的理解和生成能力远强于其他格式（如 XML、YAML）。

> 选择 JSON Schema 定义工具参数，本质上是选择了**模型训练分布中最熟悉的格式**作为接口协议。

## 为什么工具调用发生在"生成过程中"而不是"生成之前"

一个常见的设计直觉是：既然模型在开始回复之前就应该知道自己是否需要工具，为什么不把工具调用作为一个"生成前的预检步骤"？

这个直觉的破绽在于它假设模型有 ==pre-generation metacognition==——在输出第一个字之前就已经知道自己知道什么。但自回归模型的架构并不支持这种能力。

### 从信息不对称的角度推导

模型在生成 token 之前，内部状态只是一束高维向量。这束向量包含了"关于用户问题的理解"，但它不包含一个可以查询的元认知标签——"我对此问题的确定性是 73%"。确定性是在生成过程中逐步浮现的。

具体来说：

1. 模型沿着某条 token 路径生成时，某些路径每个 token 的预测概率都很高——模型"很顺利"——这表明模型内部有足够的知识来支撑这条路径
2. 而另一些路径每个 token 都伴随着高熵、多候选——模型在"挣扎"——这表明模型对这个领域缺乏确定的知识
3. "是否需要调用工具"的决策本身就是这种 token 级别确定性评估的结果，而不是原因

> 工具调用不是一个"生成前的前置判断"，而是**生成过程中某条路径的自然终点**。当模型沿着"我不确定→我需要查一下→这个工具正好提供这个信息→调用它"的路径逐 token 生成时，`tool_call` 就是这个推理链的必然输出。

### 一个反事实测试

假设你设计了一个架构，在生成任何输出之前先让模型回答一个问题："你确定知道答案吗？(confidence: 0-100)"。这个架构能工作吗？

不能。因为回答这个元认知问题本身就需要生成 token——模型必须沿着"我在评估我是否知道 X"这条路径生成 token，才能得出"确定"或"不确定"的结论。==元认知在自回归架构中是生成过程的副产品，不是独立于生成过程的前置能力==。

### 这对 API 设计的启示

理解了这一点，两平台的 `tool_choice` 设计差异就有了更深的解释：

- **Anthropic 的 `tool_choice: "auto"` 保留推理链**：模型可以先生成"让我想想…"的文本，在推理过程中发现自己不确定，然后自然地转向工具调用。推理链就是元认知评估发生的载体。
- **Anthropic 的 `tool_choice: "any"` 跳过推理链**：模型被强制直接输出工具调用，跳过了"我是否真的需要这个工具"的 token 级别评估。这减少了延迟，但增加了"调用了不需要的工具"或"调用了错误的工具"的风险。
- **OpenAI 不文档化推理链行为**：但从功能上看，`tool_choice: "auto"` 同样允许推理空间，`tool_choice: "required"` 同样会压缩这个空间。

> 如果你的工具调用场景涉及模糊的、需要判断的用户意图（"帮我处理一下这个"），保留推理链是更好的选择。如果场景明确（"查天气"→一定需要 get_weather），跳过推理链可以减少延迟和 token 消耗。

## 思考题

1. 如果一个模型支持 Function Calling，但不支持并行工具调用，在高延迟工具场景下会有什么问题？你会如何设计一个绕过这个限制的方案？
2. 工具调用的 JSON Schema 定义和传统 API 的 OpenAPI/Swagger 定义有什么本质区别？一个是给模型看的，一个是给人类开发者看的——这种"受众差异"如何影响你的设计选择？
3. 如果工具调用的结果太大（比如返回了 10 万字的文档），直接放进 context 会撑爆上下文窗口。你会如何处理这种场景？
