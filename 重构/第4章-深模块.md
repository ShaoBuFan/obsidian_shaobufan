# 第4章 · 模块应该深

---

任何一个模块（函数、类、服务、任何抽象）都有两个部分：**==接口==**——外部使用者看到的东西，和**==实现==**——内部干的事情。

Ousterhout 用一个矩形来形象化这个关系。矩形的面积代表模块提供的功能价值，顶部边代表接口的复杂度。好模块是**==深的==**——顶部边短（接口简单），面积大（功能丰富）。差模块是**==浅的==**——顶部边长（接口复杂），面积小（没做什么事）。

> "Deep modules are those whose interfaces are much simpler than their implementations."

如果把接口看作模块向使用者收的"税"，实现看作模块提供的"服务"，那深模块就是税收低、服务好的政府。浅模块是高税收、低服务的烂政府。

## 接口不止是签名

接口有两个层面。**==形式接口==**是语言检查的那些——方法名、参数类型、返回值、异常声明。但更重要的是**==非形式接口==**——高层的使用约束、副作用、调用顺序要求。比如"这个方法必须在调用 init 之后才能用"、"这个操作不是线程安全的"——这些语言不帮你检查，但对调用者的认知负荷影响巨大。

> 一个模块的非形式接口往往比形式接口更复杂。注释和文档的作用，就是把非形式接口明确写下来（[第12章](第12章-为什么写注释.md)至[第15章](第15章-先写注释.md)会详细展开）。

## 深模块的极端例子：垃圾回收

最深的模块是那些**你完全感知不到**的模块。现代语言（Go、Java）的垃圾回收器就是最极端的好设计——接口为零（你不需要调用任何东西），实现却极度复杂（自动识别垃圾、释放内存、压缩碎片）。引入 GC 反而**减少了**系统的总接口——因为不再需要所有的 `free` 或 `delete` 调用。这是深度设计的极致示范。

Unix 文件 I/O 是另一个经典：`open`、`read`、`write`、`lseek`、`close`，五个函数，简洁的接口，几十年来保持不变。而底层实现从几十年前的简单磁盘读写进化到了现在包含 VFS 层、页缓存、预读、IO 调度、多设备驱动的复杂系统——几十万行代码。接口不变，实现可以任意进化。这正是深模块的威力。

> ==深模块==让最激进的内部改动不需要触碰调用者。接口=契约，实现=自由。

## 浅模块和 Classitis

浅模块的接口复杂度和实现复杂度差不多——它没帮你隐藏什么。Ousterhout 给出了一个极端但真实的例子，来自学生项目：

```java
private void addNullValueForAttribute(String attribute) {
    data.put(attribute, null);
}
```

这个方法的接口（方法名 + 参数）和它的一行实现复杂度相等。它没有提供任何有意义的抽象——你看完方法名去读实现，实现什么都没多告诉你。这是浪费。

更普遍的危害来自一个广为流传的教条——"类要小，方法要短"。Ousterhout 直接挑战了它。

> 太多的迷你类和方法会导致 ==Classitis==——大量的浅模块，整体的系统复杂性反而上升。因为你的认知负荷从"理解模块内部"转移到了"理解模块之间的关系"。拆得越细，连接点越多，网络越复杂。

一个 200 行的深方法（接口简单、实现复杂）远比 20 个 5 行的浅方法（每个方法名都是接口负担，每个都要追踪调用关系）更好。模块的价值不是长度决定的——是接口/实现的**==深度比==**决定的。

## Unix I/O vs Java I/O

Ousterhout 用 Java I/O 作为浅模块的经典案例。为了读一个文件，Java 让你这样写：

```java
new BufferedReader(new InputStreamReader(new FileInputStream(file)));
```

要理解三个类的嵌套——`FileInputStream` 怎么读字节、`InputStreamReader` 怎么解码成字符、`BufferedReader` 怎么加缓冲。这三个概念被**暴露在了调用者的面前**，而不是被隐藏在实现内部。而缓冲这种常见需求，应该是默认行为，不应该让调用者手动包装。

对比之下，Unix 的 `read()` 在你没有任何感知的情况下自动做缓冲、预读、缓存——这是深模块。

> 接口应该为最常见的用例优化。如果一个功能 90% 的使用者都需要，它不该是可选配置——它应该被默认藏进实现里。

## 思考题

1. 你的项目里有没有"为了实现一个简单功能，要组合很多小模块"的情况？像不像 Java I/O 的嵌套？能不能把它们包在一个深模块后面？
2. 你有没有为了"类要短"而把一个逻辑连贯的类拆成了好几个碎片？拆了之后调用者的知识负担变重了多少？
3. 团队最近一次新引入的模块——它是深的还是浅的？用"接口复杂度 / 实现复杂度"来估一下。
