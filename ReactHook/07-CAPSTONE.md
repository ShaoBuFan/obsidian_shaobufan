# Module 7: 综合项目——从阅读者到设计者的毕业考试

> **模块目标**：将前 6 个模块的知识整合，完成一个完整的、独立的设计到交付周期。

---

## R0: 前置检查

在开始综合项目之前，确认以下能力：

- [ ] 我能不查资料默写六步设计法
- [ ] 我能识别一个 Hook 使用了哪些模式（Wrapper/Composer/Extractor/Facade/Coordinator）
- [ ] 我能诊断和修复过期闭包、useEffect 误用、上帝 Hook 等反模式
- [ ] 我能为自定义 Hook 写行为测试
- [ ] 我能用第一性原理（Fiber 链表）解释 Hook 的规则

如果有任何一项不确定，回到对应模块复习后再开始。

## 项目 A：构建一个 Hook 库

### 目标

实现 8-10 个自定义 Hook，覆盖多种模式和类别，并提供测试和文档。

### 要求

至少包含以下 Hook（你可以自己补充更多）：

| Hook | 模式 | 类别 | 难度 |
|------|------|------|------|
| `useToggle` | Wrapper | 状态 | 简单 |
| `usePrevious` | Composer | 工具 | 简单 |
| `useDebounce` | Composer | 计时 | 中等 |
| `useLocalStorage` | Facade | 浏览器 API | 中等 |
| `useEventListener` | Extractor | DOM | 中等 |
| `useFetch` | Facade + Composer | 数据获取 | 中高 |
| `useMediaQuery` | Composer | DOM | 中等 |
| `useForm` | Coordinator | 表单 | 高 |
| （自选 1-2 个） | | | |

### 交付物

1. **源码**：每个 Hook 一个文件，带 TypeScript 类型定义
2. **测试**：每个 Hook 至少 3 个测试用例
3. **文档**：每个 Hook 的 JSDoc 注释，包含参数说明、返回值说明、使用示例
4. **演示**：一个简单的 Demo 页面展示每个 Hook 的效果

### 评分标准

| 维度 | 及格 | 良好 | 优秀 |
|------|------|------|------|
| 功能正确性 | 基本功能可用 | 边界情况处理 | 竞态/清理/SSR 全处理 |
| 代码质量 | 能跑 | 命名清晰、结构合理 | 读者能立刻理解设计意图 |
| 测试覆盖 | 有测试 | 核心逻辑有测试 | 边界+异步+清理全测试 |
| 设计决策 | 能用就好 | 能说出用了什么模式 | 能用六步法解释每个决策 |
| 文档 | 有 JSDoc | 有使用示例 | 有设计决策说明 |

### ⚡ 费曼检查 T1

> 录制一段 5 分钟的讲解，从你的 Hook 库中选一个最复杂的 Hook，用六步法解释它的设计过程。然后自己听一遍——有没有哪个决策的"为什么"解释得不清楚？

## 项目 B：重构一个复杂组件

### 目标

找一个至少有 200 行的 React 组件（自己的项目、开源项目的组件、或网上找一个），从中提取 3-5 个自定义 Hook，并证明每次提取都是合理的。

### 步骤

**1. 选择目标组件**

- 至少 200 行（越长越好）
- 包含多种关注点（数据获取、表单处理、DOM 操作等）
- 最好是你自己项目的代码

**2. 审计**

在重构前，标出组件中的所有关注点：

```
关注点 1: 数据获取（useEffect + fetch）
关注点 2: 表单验证（多个 useState + 条件逻辑）
关注点 3: 键盘快捷键（useEffect + addEventListener）
...
```

**3. 为每个关注点做设计决策**

| 关注点 | 是否提取？ | 为什么？ | 提取后的模式 |
|--------|-----------|----------|------------|
| 数据获取 | 是 | 复用 + 简化 | Facade |
| 键盘快捷键 | 是 | 关注点分离 | Extractor |
| 表单验证 | 待定 | 只有一个地方用 | - |

**4. 实现提取**

用六步法设计每个要提取的 Hook。

**5. 写测试**

每个提取出的 Hook 都要有测试。

**6. 对比**

做一个 before/after 的对比：

- 原组件的行数 vs 重构后的行数
- 原组件的圈复杂度 vs 重构后
- 代码可读性的主观评价

### 交付物

1. 原始组件代码（标注各关注点）
2. 重构后的组件代码
3. 提取出的 3-5 个 Hook（各自独立文件）
4. 每个 Hook 的测试
5. 设计决策文档（为什么提取 / 为什么不提取）

### 评分标准

| 维度 | 及格 | 良好 | 优秀 |
|------|------|------|------|
| 提取合理性 | 每个提取有理由 | 也有没提取的判断 | 每个决策有量化分析 |
| Hook 设计质量 | 功能可用 | 模式选择恰当 | 六步法可追溯 |
| 重构后组件可读性 | 没变差 | 明显变简洁 | 代码即文档 |
| 测试 | 有测试 | 核心行为有测试 | 回归测试确保不引入 bug |
| 设计决策文档 | 列出提取了什么 | 有选择理由 | 有拒绝提取的理由 |

## 项目 C：从规格设计 Hook 架构

### 目标

给你一个应用的**规格说明**（不是代码），你需要设计实现它所需的全部 Hook。这是终极挑战——从空白文件和纯文字需求开始。

### 规格：协作白板应用

你需要实现一个简单的协作白板应用的数据层。应用包含以下功能：

**画布管理**
- 用户可以创建多个画布
- 每个画布有一个名称和一组形状
- 支持切换画布、重命名画布、删除画布

**形状操作**
- 支持三种形状：矩形、圆形、文本
- 每个形状有：位置 (x, y)、大小 (width, height)、颜色、z-index
- 用户可以添加、移动、删除、选中形状
- 支持撤销/重做（至少 20 步历史记录）

**实时同步**
- 形状的变化需要同步到服务器（使用 REST API）
- 每 3 秒自动保存一次（debounce 模式）
- 网络断开时，操作进入离线队列，恢复连接后重放
- 多人协作时不要求冲突解决，但需要显示"有新版本可刷新"的提示

**UI 状态**
- 暗色/亮色主题切换
- 响应式布局：移动端和桌面端不同布局
- 快捷键：Ctrl+Z 撤销、Ctrl+Shift+Z 重做、Delete 删除选中形状、Ctrl+S 手动保存

### 你的任务

1. **分析规格**，识别所有需要自定义 Hook 的场景
2. **为每个 Hook 走一遍六步设计法**
3. **画出 Hook 之间的依赖关系图**
4. **实现至少 4 个核心 Hook**
5. **为实现的 Hook 写测试**

### 提示：从哪里开始？

先从顶层分解开始：

```
应用
├── 画布管理 → useCanvas, useCanvasList
├── 形状操作 → useShapes, useUndoRedo
├── 实时同步 → useAutoSave, useOfflineQueue, useSyncStatus
├── UI 状态  → useTheme, useMediaQuery, useKeyboardShortcuts
└── （交叉关注点）→ useHistory (?)
```

### 交付物

1. **Hook 架构图**：所有 Hook 及其关系
2. **六步设计文档**：至少 4 个核心 Hook 的完整设计过程（使用附录 D 的工作手册）
3. **实现代码**：至少 4 个核心 Hook
4. **测试**：已实现的 Hook 的测试
5. **依赖关系图**：Hook 之间的调用和依赖关系

### 评分标准

| 维度 | 及格 | 良好 | 优秀 |
|------|------|------|------|
| 架构设计 | 能找到明显的 Hook | 层次分明、职责清晰 | Hook 粒度恰到好处 |
| 设计过程 | 有设计文档 | 六步法完整填写 | 决策理由有深度 |
| 实现质量 | 核心逻辑工作 | 边界和清理完备 | 生产级质量 |
| 依赖管理 | Hook 之间能协作 | 没有循环依赖 | 依赖方向一致 |
| 整体完整性 | 4 个 Hook | 考虑了大部分场景 | 覆盖规格要求的所有能力 |

## 无论选哪个项目，做完后都要做的三件事

### 1. ⚡ 费曼检查 T1：5 分钟单 Hook 讲解

录制一段 5 分钟的讲解——从你的项目中选一个最复杂的 Hook，用六步法解释它的设计过程。然后自己听一遍——有没有哪个决策的"为什么"解释得不清楚？

### 2. ⚡ 费曼检查 T2：10 分钟完整架构讲解

录制一段 10 分钟的讲解（可以是对着空气说，也可以写出来）：

- 介绍你的项目中各 Hook 的设计思路
- 解释为什么某个 Hook 用了某个模式
- 说明你遇到过什么设计困难，怎么解决的
- 讲述一个你做错然后纠正的设计决策

然后自己听一遍。**在你出现"呃……"或解释不清的地方，那就是你还需要深入的地方。**

### 2. 写一份"我现在怎么看 Hook"

拿回你在 README 中写下的初始回答（或重新回答这三个问题）：

1. 用一句话解释：`useState` 返回的数组里有什么？
2. 为什么 Hook 不能放在 `if` 语句里？
3. 你最近一次想写自定义 Hook 但不知从何下手是什么场景？

对比你现在的回答和最初的学习前回答。差别在哪里？

### 3. 设计你自己的 Hook 库"模式卡"

做一套属于你自己的速查卡。每张卡包含：
- 模式名称
- 一句话描述
- 一个典型代码骨架
- 什么时候用 / 什么时候不用

这些卡不需要完美，它们是**你的**——基于你的理解、你的语言、你踩过的坑。

## 课程结语

你带着"能读懂 Hook，但设计不了"的问题来到这里。

如果这门课完成了它的使命，你现在应该拥有：

- **一个可重复的六步设计流程**——不需要灵感，只需要方法
- **一个底层心理模型**——Fiber 链表，不是魔法
- **一个模式词汇表**——五种模式帮你识别和设计
- **一套判断标准**——知道什么时候提取，什么时候不提取，什么时候拆
- **一套验证工具**——测试和审查让设计变可靠

设计能力不会因为"学完这门课"就自动完整。它会随着你设计更多 Hook 而持续成长。但你已经有了最重要的东西：**一个起始点不是空白的流程。**

当再次面对"我需要写一个自定义 Hook"时，你知道第一步做什么。这就够了。
