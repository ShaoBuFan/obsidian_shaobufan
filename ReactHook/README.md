# React Hooks: 从阅读者到设计者

## 这门课解决什么问题？

你正处于一个经典的**阅读-写作鸿沟**中：

> 我能读懂别人写的 Hook，理解每一行在做什么。但面对一个空白文件，要我从零设计一个 Hook——我做不出来。

这不是你的问题。阅读代码是被动理解，设计代码是主动创造。它们是两种不同的认知技能，需要的训练也完全不同。

这门课的核心目标只有一个：**让你从"能读懂 Hook"变成"能自信地从零设计 Hook"**。

## 这门课怎么教？

我们融合两种学习方法：

### 第一性原理

不满足于 API 文档层面的理解。把 Hook 拆解到最底层——JavaScript 闭包、函数调用、链表数据结构。当你理解了 Hook 在 React 内部**到底是什么**（而不是**怎么用**），设计决策就有了根基。

### 费曼学习法

> "如果你不能简单地解释一个东西，说明你还没真正理解它。"——理查德·费曼

每个模块都要求你做**教学检查**：用最简单的语言解释刚学的概念。如果解释不清楚，说明那里有知识缺口，回去填补它。

### 渐进式脚手架

你不会在第一页就被要求"从零设计一个 Hook"。相反：

1. **先看**：观察别人怎么设计（配套完整注释）
2. **再改**：在模板上填空、修改、扩展
3. **再对照**：用六步设计法工作手册引导，逐步减少提示
4. **最后独立**：从空白文件开始，自己设计

## 前置知识

- JavaScript 基础：闭包、作用域、数组方法、Promise
- React 基础：能用函数组件写一个简单的 Todo 应用
- 不需要：对类组件的深入了解、对 React 源码的了解

## 学习建议

### 不要跳模块

每个模块都建立在前一个模块之上。跳模块 = 跳过理解的关键环节。

### 先做费曼检查，再看下一节

每个费曼检查点用 ⚡ 标记。在你能用简单语言解释这个概念之前，不要往下走。

### 代码一定要手打

阅读代码给你的是一种**虚假的熟悉感**。只有亲手打出来的代码才真正属于你。每个练习都要自己敲。

### 善用工作手册

附录 D 提供了六步设计法工作手册模板。从 Module 3 开始，每次设计 Hook 都先填手册，再写代码。

### 允许自己犯错

设计能力无法通过"正确理解"来获得。它来自**不断地做决定、犯错、复盘**。课程中的每个 Bug 狩猎练习和逆向工程练习都是为了训练你的判断力。

## 五层审查机制

每个模块都经过 5 种不同角度的检查：

| 层级 | 类型 | 时机 | 做什么 |
|------|------|------|--------|
| R0 | 前置检查 | 模块开始前 | 3-5 道题回顾上一模块核心概念 |
| R1 | 费曼教学 | 每个重要概念后 | 用简单语言解释，识别理解缺口 |
| R2 | 实现检查 | 每个概念后 | 写代码验证理解 |
| R3 | 设计扩展 | 每个模式后 | 将模式应用到新场景 |
| R4 | 跨模块综合 | 模块结束时 | 连接本模块与前面模块的概念 |

## 开始之前

拿一张纸，写下你此刻对以下问题的回答（不需要完美，只需要诚实）：

1. 用一句话解释：`useState` 返回的数组里有什么？
2. 为什么 Hook 不能放在 `if` 语句里？
3. 你最近一次想写自定义 Hook 但不知从何下手是什么场景？

把答案放在旁边。学完 Module 0 后回来看你的第 2 个答案——你会发现理解深了很多。
