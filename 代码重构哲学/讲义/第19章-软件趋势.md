# 第19章 · 软件趋势

---

Ousterhout 把他的复杂度框架应用到当代软件开发的主流趋势上——逐个评估，不盲从，不服就怼。这一章是全书最具批判性的部分之一，他点名批评了几个广为流传的教条。

## 面向对象与继承

Ousterhout 区分两种继承。**==接口继承==**——父类定义签名不提供实现，子类各自实现。这是好的：同一个接口被多种实现复用，降低了认知负荷（学一个接口，理解所有实现）。实现越多，越"深"。**==实现继承==**——父类提供默认实现，子类覆盖。虽然减少了代码重复（变更放大问题），但制造了父类与子类之间的紧密耦合和信息泄露。最坏情况下，开发者需要完整理解整个类层次结构才能安全修改任何东西。

> "Although the mechanisms provided by OOP can assist in implementing clean designs, they do not, by themselves, guarantee good design." 用了 class ≠ 做了设计。组合优先于继承——这个古老的建议在这里得到了复杂度框架的理论支撑。

## 敏捷开发

敏捷的增量和迭代精神与本书哲学一致——你无法在前期足够好地可视化复杂系统以确定最佳设计。但 Ousterhout 的批评很尖锐：

> 敏捷让开发者聚焦在**功能**上，而不是**抽象**上。"先用最简专用方案，以后重构"——这在实践中变成了不投资设计的借口。开发的增量应该是**抽象**，不是功能。

## 单元测试

Ousterhout 强烈支持单元测试——不是为了设计，是因为它让**安全的重构**成为可能。没有测试套件时，开发者避免结构改进，复杂性累积。有好的测试时，开发者可以自信地重构。他引用了自己的经历：重写 Tcl 解释器为字节码编译器——已有单元测试套件的质量高到 alpha 版本发布后只发现了一个 bug。

## TDD（测试驱动开发）

尽管支持单元测试，Ousterhout **==明显批评 TDD==**：

> "The problem with test-driven development is that it focuses attention on getting specific features working, rather than finding the best design. This is tactical programming pure and simple."

TDD 太增量了——它引诱你为了通过下一个测试而草草堆叠代码，没有为"设计"分配明显的时间。他认可一种例外场景：修 bug 时先写一个复现 bug 的失败测试，再修——这证明你真的解决了问题。

## 设计模式

整体正面——解决常见问题的公认干净方案。但最大的风险是**过度应用**。不是每个问题都适合一个模式。用设计模式不自动改善系统——只有当它真正适合时才改善。

## Getter 和 Setter

Ousterhout 批评 Java 社区对 getter/setter 的过度使用：暴露实例变量违反信息隐藏、增加接口复杂度。getter/setter 是浅方法（通常一行），增加了混乱但没有提供有意义的抽象。最佳实践：尽量避免暴露实现数据。

## 终极问题

> 每当你遇到一个新的软件范式，问自己——"这真的有助于最小化大型软件系统的复杂性吗？"很多提议表面上好听，实际上让复杂性更糟。

---

## 思考题

1. 你的项目中有没有"为了复用"而使用的深层实现继承？有没有因为基类变更子类崩溃的经历？
2. 你的团队对敏捷的理解是"功能更快"还是"抽象更干净"？
3. 你用过 TDD 吗？有没有感觉它让你的注意力从设计转移到了"让测试通过"？
