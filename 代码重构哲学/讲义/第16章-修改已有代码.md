# 第16章 · 修改已有代码

---

软件系统的设计，在长期演化中，更多是由**==修改==**而非初始构思决定的。一个成熟系统的大部分代码是后来加上去或改出来的。所以，修改代码的方式直接决定了系统的长期命运。

> 当你完成一个改动后，系统应该拥有"如果从一开始就带着这个改动设计"它所会拥有的结构。不是"最小的改动让功能能跑"，而是"改动后系统比改动前更干净"。

这就是[第3章](第3章-战略性编程.md)"战略性编程"在修改场景下的应用。开发者在修改已有代码时往往出于恐惧而退回战术模式——对代码不熟悉，担心大改动引入新 bug，于是选择最小改动。但每一次最小改动都在引入特例、依赖和其他的复杂性增量。系统设计就这样一点一点地腐烂。

## Boy Scout 原则的编程版

> "如果你没有让设计变好，你很可能在让它变坏。"即使你的改动不需要重构，趁你在代码里时顺手修一两个设计瑕疵。每一次修改都应该让设计至少好一点点。

Ousterhout 也诚实地讨论了现实的约束——一个三个月的重构 vs 一个两小时的快速修复，在 deadline 下你不能永远选前者。他的建议：问自己"在给定约束下，这是我能做到的最好结果吗？"寻找中间地带——几乎一样干净、但能在几天内完成的方案。如果大重构眼下不可行，和老板达成共识在 deadline 后安排。每个组织都应该配置一定比例的资源用于清理和重构。

## 修改代码时怎么维护注释

注释和代码放在同一处——离它描述的代码越近越好。接口注释就在方法声明旁边，实现注释分布在方法内部每个阶段的上方。

> 不要把关键的设计理由只放在 commit message 里——后来的开发者很少翻提交记录。如果一个信息重要到值得写进 commit message，它大概也值得写进注释。

每条设计决策的注释只写一次，放在最显眼的位置。其他受影响的地方放简短的引用注释（"关于这个选择的解释，参见 xxx 的注释"）。不要在一个模块里重复记录另一个模块的设计决策。

**==提交前扫一遍 diff==**——确保每一处改动都在文档里有对应反映。这个习惯同时能帮你抓到遗留的调试代码和未完成的 TODO。

> 越抽象的注释越耐改。最容易被代码的小改动"废掉"的注释，是那些重复描述代码操作的注释。最耐改的注释，是那些描述高层意图和设计理由的注释。而后者恰好也是最有用的注释。

## 思考题

1. 你最近一次改老代码——是"最小改动"，还是"让代码比之前干净了一点"？
2. 你的 commit message 里有没有应该写进注释但没写进注释的设计理由？
3. 你的团队在 code review 时会检查"注释有没有跟着代码一起更新"吗？
