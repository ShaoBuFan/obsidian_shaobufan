# 第14章 · 命名

---

好的命名是软件设计里最被低估的技术。一个精确的名字就是一行精简到极致的注释（参见[第12章](第12章-为什么写注释.md)）——它在大脑里瞬间创建图像，而不需要读任何额外文字。

> "If someone sees this name in isolation, without seeing its declaration, its documentation, or any code that uses the name, how closely will they be able to guess what the name refers to?"

Ousterhout 用了一个他亲身经历的惨痛案例来开场。80 年代末在 Sprite 分布式操作系统中，变量 `block` 被同时用来表示**物理磁盘块号**和**逻辑文件块号**——两个完全不同的概念用了一个相近的名字。这导致了一个数据损坏 bug，花了**六个月**才找到。好几个人（包括 Ousterhout 自己）读过那段有问题的代码，都没发现异常——因为 `block` 对两种含义都"差不多对"。

> 不要满足于"差不多对"的名字。命名的目标是精确、无歧义、直觉化。

## 精确到什么程度

`getCount()`——什么的 count？如果叫 `getActiveIndexlets()`，你不需要读实现。`blinkStatus` 这个布尔值——"status"对一个布尔量太模糊了，`cursorVisible` 直接告诉你真和假分别代表什么。`result` 在一个 void 方法里——既误导（暗示有返回值），又不描述实际内容。

命名也可以**==过度精确==**：给接受任意范围的 `delete` 方法参数取名 `selection`——暗示了 UI 选中操作，而方法其实不知道也不在乎是不是来自 UI。`range` 更准确。

> 红牌信号：**==模糊的名字==**。如果一个名字宽泛到可以指很多不同东西，它不传递信息——它是在邀请误用。

> 红牌信号：**==名字难取==**。如果你找不到一个简洁、精确、直觉化的名字，可能不是因为你的语文不好——是底层实体本身的设计有问题。考虑重构：也许一个变量该拆成多个。

## 一致性

一致性命名（参见[第17章](第17章-一致性.md)） = 同一个概念在整个项目中永远用同一个词，而且那个词永远不用于其他概念。

当需要区分同一类的两个不同用途的变量（比如文件拷贝中的源和目的地），用区分性前缀：`srcFileBlock`、`dstFileBlock`。循环计数器永远用 `i` 为最外层、`j` 为第一层嵌套——这让读者能瞬间做出安全假设，不需要每次从头理解。

Go 语言的命名文化偏好极短的名字（经常是单字母），Ousterhout 表示了异议——他用 `RuneCount` 函数的两个版本（单字母 vs 描述性命名）做了对比，指出描述性版本同样可读甚至更可读。他同意 Go 社区的一条折中原则：**名字离它的声明越远，就应该越长。** 一个只在三行循环里用的变量叫 `i` 没问题。一个被一百行之外的代码引用的变量需要更长的、更自解释的名字。

---

## 思考题

1. 你的项目里有没有像 `block` 一样"一个名字指两个不同东西"的变量？试着回想一个因为歧义导致的 bug。
2. 有没有你一直觉得"名字不好但将就着用"的变量或方法？它的设计是不是本身有问题，导致一个好名字不存在？
3. 你的团队有命名约定吗？如果有——同一个概念在项目的所有文件里用了同一个词吗？
