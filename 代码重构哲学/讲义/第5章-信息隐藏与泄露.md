# 第5章 · 信息隐藏与泄露

---

==信息隐藏==是制造==深模块==（[第4章](第4章-深模块.md)）的核心技术。这个概念最早由 David Parnas 在 1972 年的论文中提出，Ousterhout 把它重新推到了设计方法论的中心位置。

> 每个模块应该封装少数几个"知识片段"——即设计决策。这些知识存在于模块的实现中，但不出现在接口中，对其他模块不可见。

所谓"设计决策"，不是指"变量名取什么"这种表面选择，而是指那些**将来可能会变的东西**：数据存储结构（B树还是哈希表？）、网络协议细节（TCP 重传策略）、文件格式解析方式、并发调度策略。这些东西今天怎么做的，明天可能因为性能、环境、需求而改变。如果你把这些决策封装在模块内部，未来改变它们时不需要触碰系统的其他部分。

有两个独立的好处。一是接口变简单——调用者看到的是一个抽象视图，认知负荷降低。二是演化变容易——对外部没有依赖，内部可以任意重构。

## 信息隐藏不等于 private 关键字

> 把变量声明为 private 不等于信息隐藏。如果你通过 getter/setter 暴露了私有数据的类型和语义，那数据的本质和 public 没有区别——调用者仍然知道它是什么类型、存了什么内容。

真正的信息隐藏发生在更高层：调用者不知道也不关心你是怎么存储的，只知道"我能通过接口得到我需要的结果"。

## 信息泄露：后门泄漏比表面泄漏更危险

==信息泄露==就是一个设计决策出现在了多个模块中。**==表面泄漏==**比较容易被发现——信息直接出现在模块的 API 签名里。**==后门泄漏==**更隐蔽：两个类各自独立实现，但都依赖了对同一件事的理解（比如都直接解析同一个文件格式、都知道某个远程服务的调用方式）。表面上各自的接口干净，但实际上修改那个共享的知识会同时影响两者。

> 如果信息泄露已经发生，两个补救方向：把相关类合并（如果它们小而紧密），或者提取一个新实体封装那段共享的知识——前提是你能找到一个简洁的接口来包装它。

## 时序分解：最常见的信息泄露来源

Ousterhout 指出了一个极其普遍的问题模式——按操作的**==时间顺序==**分解系统，而不是按**==知识领域==**。

> 一个典型的错误设计：把"读文件、修改内容、写文件"拆成 FileReader、FileModifier、FileWriter 三个类。问题是读和写都需要知道文件格式——这个知识被泄露到了两个不同类里。

正确的做法是把文件格式的知识封装在一个类里，内部处理读和写。执行顺序当然会在代码某处体现（通常在调用方），但它不应该决定模块的划分方式。

> "When designing modules, focus on the knowledge that's needed to perform each task, not the order in which tasks occur."

## 来自课堂的真实案例

Ousterhout 用斯坦福课程中学生实现的 HTTP 服务器为例，给出了几个具体的设计修正：

学生把请求处理拆成了"读取"类和"解析"类——典型的时序分解。修正方案是把它们合并成一个类，封装对 HTTP 请求格式的全部知识，使接口更简洁、模块更深。

另一组学生在参数处理上做得不错——隐藏了 URL 编码细节，合并了 header 参数和 body 参数。但 `getParams()` 返回了原始的 `Map<String, String>`——这暴露了内部数据结构。更好的设计是 `getParameter(name)` 和 `getIntParameter(name)`，把存储结构藏起来。

> "Classes should 'do the right thing' without being explicitly asked."

很多学生要求调用者显式指定 HTTP 版本和 Date 响应头——这些是每次响应都应该自动填充的。好设计应该提供合理的默认行为，调用者不需要为每次都一样的事情操心。

## 信息隐藏的边界

信息隐藏只在隐藏的东西"外部不需要"时才有价值。如果客观上外部确实需要知道某些信息（比如性能调优参数），强行隐藏只会让接口变得别扭——调用者总要想办法绕过你的设计。Ousterhout 的建议是：

> 目标是**最小化**外部需要的信息量，不是**清零**。识别哪些信息是外部必需的，确保它们被暴露——但要显式、干净地暴露，而不是通过后门泄露。

## 思考题

1. 你的项目里有没有"读写分离"的模块对——两个类共享对同一个数据格式的知识？这意味着信息泄露。
2. 有没有 `getXxx()` 返回了原始数据结构，让调用者可以任意操纵内部状态？试着重构成针对具体用途的方法。
3. 你最近拆过的一个模块是按"时间顺序"还是按"知识领域"？如果你不确定，列出每个子模块分别处理什么知识——如果有多于一个子模块共享同一种知识，那就是时序分解。
