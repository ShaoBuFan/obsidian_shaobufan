# 第10章 · 把错误定义掉

---

异常处理是软件系统中最严重的复杂性来源之一。Ousterhout 引用的数据：分布式数据密集型系统中，**90% 以上的灾难性故障是由不正确的错误处理造成的。** 异常处理代码极其难写，异常处理代码本身又制造更多异常的机会，而且异常分支几乎从不被执行到——"没执行过的代码就是不工作的代码"。

> "Throwing exceptions is easy; handling them is hard."

## 不是更好地处理异常，而是让异常消失

绝大多数开发者在面对异常时思考的是"怎样处理它"。Ousterhout 的建议更激进：**==重新设计接口的语义，让异常情况本身不存在。==**

这是他自己的教训。Tcl 语言中的 `unset` 命令是他设计的——原本删除一个变量，如果变量不存在就抛异常。他当时的逻辑是："你要删一个不存在的东西，这肯定是个 bug。" 但他错了。`unset` 最常见的用途是清理临时状态——你很少确切知道当前有哪些临时变量。这迫使每个调用者在 `unset` 外面包 try-catch，只为了忽略这个异常。Ousterhout 称这是 **Tcl 设计中最大的错误之一**。

> 正确的设计：`unset` 的语义应该是"确保这个变量不存在"——如果已经不存在，操作就已经完成了。把"错误情况"重新定义为"任务已经完成的正常情况"。

## 两个经典示范

Windows 删除文件：如果有任何进程打开了这个文件，Windows 拒绝删除。用户必须自己找出哪个进程持有了文件句柄——有时只能重启。这个异常被推给了最不该承担它的人——用户。

Unix 删除文件：标记文件名删除，立即返回。文件名消失（不会有新进程打开），但已经打开该文件的进程可以继续读写。数据在所有进程关闭后自动释放。这**==一次性定义掉了两个异常==**：删除不会因为文件被打开而失败，正在使用文件的进程不会遭遇新错误。

> 关键差距不在于"谁更好地处理了异常"，而在于 Unix 的设计把异常情况从根本上消除了。

Java 的 `substring`：索引越界抛 `IndexOutOfBoundsException`——调用者被迫在每次调用前手动检查边界（五六行代码做一件本该一行完成的事）。Python 的切片：超出范围的索引自动截断到有效范围——返回空字符串而不是报错。Python 的 API 更深（参见[第4章](第4章-深模块.md)）：接口更简单（不需要 try-catch 包装），实现反而更复杂（内部处理了边界情况）。

## 三个补充策略

当异常无法通过重新定义语义来消除时：

**==在底层蒙住异常。==** 把异常处理下沉到底层模块内部解决，不让上层感知。和"把复杂性往下拉"（[第8章](第8章-把复杂性往下拉.md)）一致。

**异常聚合。** 与其为每种异常写不同的处理逻辑，不如用一个通用处理机制替代大量分散的 handler。减少重复处理代码的量。

**直接崩溃。** 对于极少发生、不可恢复的错误（内存耗尽），打印诊断信息然后终止。试图处理它的复杂性不值得。这需要勇气——"让程序优雅处理一切"是另一种教条。有些场景下，干净地崩溃比混乱地挣扎更好。

---

## 思考题

1. 你的 API 里有类似 `unset` 的设计吗——一个操作的"正常结果"被当成"异常"在抛？
2. 有没有 try-catch 块只是为了"忽略"某个异常？能不能改 API 让这个异常不再发生？
3. 你有没有把异常往上抛是因为"不知道这里该怎么处理"？试着问自己：调用者真的比我更知道怎么处理吗？
