# 第6章 · 通用模块更深

---

一个反直觉的命题：**即使一个模块永远不会被复用，把它设计得比当前需求更通用一点，仍然是更好的设计。** 不是因为"将来可能用得上"，而是因为==通用化==的过程本身会迫使你产生更简洁、更深的接口。

> "General-purpose interfaces provide simpler and deeper modules than special-purpose ones."

关键限定词是"稍微"（somewhat）——不是让你做一个可以解决所有问题的超级框架。功能应该服务于当前需求，但**接口不应该被当前需求绑架**。

## 文本编辑器案例：本章最核心的演示

Ousterhout 用一个他课程中反复出现的例子来证明这个论点。学生开发 GUI 文本编辑器，文本类自然要支持删除操作。大多数学生写出了这样的接口：

```java
void backspace(Cursor cursor);
void delete(Cursor cursor);
void deleteSelection(Selection selection);
```

三个方法，分别对应三种 UI 操作。看起来直观——每个 UI 操作映射到一个方法。但这引入了什么？

**方法又多又浅。** 三个特殊用途方法，每个只对应一种 UI 手势。

**==信息泄露==。** UI 层的概念（backspace 键、光标、选择区）泄露到了文本层。文本类现在"知道"了 UI 的存在——而文本类根本不该在乎是键盘在删还是鼠标在删。

**==认知负荷==。** UI 开发者要学三个方法。文本层开发者要理解光标和选择区的抽象。两端的人都比实际需要的多学了东西。

**==假抽象==。** `backspace()` 看起来隐藏了"删除了哪些字符"的信息——但这个信息调用者实际上**需要**知道（UI 要更新显示）。隐藏重要的细节不是简化，是制造晦涩。

> "One of the most important elements of software design is determining who needs to know what, and when. When details are important, it is better to make them explicit and as obvious as possible."

## 改成通用接口

```java
void insert(Position position, String newText);
void delete(Position start, Position end);
Position changePosition(Position position, int numChars);
```

三个方法变成了两个（insert 和 delete），外加一个辅助方法（changePosition）。现在：

- 删除键的功能在 UI 层实现：`text.delete(cursor, text.changePosition(cursor, 1))`
- 退格键：`text.delete(text.changePosition(cursor, -1), cursor)`
- 文本层对 UI 一无所知——不知道 backspace、不知道 selection、不知道 cursor

接口变少了、概念变纯粹了、认知边界变清晰了。而且 backspace 的实现现在显式表达了它在做什么——"删除光标前一个字符"——代码本身变得显而易见。

这个案例的精髓在于：**通用化的过程让你区分了"本质操作"和"当前上下文"。** 删除的本质是对一个位置范围的文本操作，backspace 是调用者赋予这个操作的具体意义。把后者留给调用者，前者留给模块，边界就干净了。

## 三个自查问题

Ousterhout 给了评估通用化是否"适度"的标准：

> 一，能满足当前所有需求的最简接口是什么？减少方法数量但不减少能力。不要为了少几个方法而加一堆额外参数——那不叫简化。

> 二，这个方法会在多少种场景下被使用？如果只服务于单一用例（像 backspace 只服务于退格键），那是红牌。试着用一个通用方法替代好几个专用方法。

> 三，这个 API 对当前需求来说好用吗？如果你为了调用当前功能要写一大堆额外代码，那接口就太通用了。单字符 insert/delete 到处需要循环——太底层了，反而不方便。

三个问题围成一个区间：太专用是一端（backspace 映射到方法），太通用是另一端（单字符操作），中间是 sweet spot。

---

## 思考题

1. 你的项目里有没有方法名直接映射了一个 UI 操作或用户动作？这是信息泄露——底层模块不该知道上层在干什么。
2. 挑三个服务于"类似功能"的专用方法，试着用一个通用方法替代它们——调用方的代码是变简单了还是复杂了？
3. 有没有"为了通用而通用"的过度设计？接口是否因为通用化而变得比原来的专用接口更难用？
