# 第3章 · 代码能跑还不够

程序员的终极诱惑：代码跑通了，任务完成了，脑海里那个声音说"搞定，提交，下一个"。

这被称为**==战术性编程==**——你的所有决策服务于一个目标：让当前功能尽快跑起来。这种行为模式不是某个人的错，它来自人性的即时满足回路：写代码→跑通→成就感→继续下一个循环。问题是，这个循环的副产品在积累——每一个"先这样吧"的小妥协，都是一滴新的复杂性，滴进系统的池子里。

> "Tactical programming focuses on getting features working as fast as possible. You tell yourself a little complexity or a small kludge is okay if it gets the task done faster."

Ousterhout 描述了一个极端但普遍存在的角色：**==战术龙卷风==**（Tactical Tornado）——一个产出极高的开发者，代码写得比谁都快，管理层把他当英雄。但他经过的每一个模块都留下一地碎片，其他工程师被迫花时间清理。他是团队里"产出最高"的人，也是破坏力最大的人。

## 投资心态

战略性编程把写代码当成**==投资==**——你今天多花的每一分钟设计时间，是存进"可维护性"这个账户里的本金。它产生的利息，是未来所有修改省下的时间。

> "The primary goal of strategic programming is to produce a great design that also happens to work — not just working code."

Ousterhout 把战略性投资分成两类：**主动投资**和**被动投资**。主动投资是在写新代码之前花额外时间想清楚设计——尝试多种方案、考虑未来变化、写文档。被动投资是当你撞上现有设计的问题时，不停下来绕过去，而是停下来修好它。

两类投资缺一不可。只做主动投资 → 系统仍然会腐化（因为旧代码的设计问题会扩散）。只做被动投资 → 你一直在还债但没有为未来存钱。

## 到底投多少？

> 作者的建议：总开发时间的 ==10%-20%== 花在设计改进上。

这个数字既不会拖慢进度，又足够产生显著收益。关键转折点在 4-6 个月：前几个月战略性编程比战术性编程慢 10-20%，但等投资产生复利效应后，战略性编程的团队开始反超——而且差距随时间拉大。战术性团队在持续减速（因为复杂性累积），战略性团队在持续匀速甚至加速（因为系统保持干净）。

## Facebook 的教训

Ousterhout 用 Facebook 做了一个诚实的案例对比。Facebook 早期信奉"Move fast and break things"，工程师入职第一周就要推生产代码。这种极端的战术文化让 Facebook 在市场上拿了惊人的增长速度——但也导致代码库逐渐变得不稳定、难以理解。后来 Facebook 不得不把口号改成了"Move fast with solid infrastructure"。

而 Google 和 VMware 从一开始就注重代码质量和设计纪律，建立了可靠的基础设施，也吸引了最在乎设计品质的工程师。

> 初创公司如果走纯战术路线，不仅系统会腐烂，还会逼走最好的工程师——真正优秀的技术人才在乎设计品质。

## 核心循环

这一章建立了一个全书反复引用的因果链：

战术性编程 → 小妥协累积 → 复杂性增长 → 开发速度下降 → 管理层加压 → 更战术 → ==恶性循环==。

打破这个循环的方法只有一个：**承认代码能跑只是最低标准，不是完成的标准。** 完成的标准是：代码能跑，而且下一个人（包括六个月后的你自己）能轻松理解和修改它。

## 思考题

1. 你的团队（或你认识的组织）里有没有"战术龙卷风"？他的代码之后被别人重构了多少次？
2. 翻最近三个 PR——其中有几个改动是在"让代码比之前更干净"，几个是在"纯叠加功能"？
3. 如果你从明天开始在每一个任务上多花 10% 的时间做设计改进，你的 leader 会支持还是反对？用本章的长期回报逻辑来说服他。
