# 第7章 · 不同层，不同抽象

---

软件系统天然是分层的。上层使用下层提供的功能，每层为上层隐藏本层的复杂性。但什么决定了"一层"该在哪开始、在哪结束？

Ousterhout 给了一条简洁的规则：**相邻的层应该提供不同的抽象。** 如果你追踪一个操作从上到下的调用链，每一步的抽象都应该变化——你在和不同"概念层级"的东西打交道。

> 如果相邻两层说的是同一种语言、关心的是同一个概念，它们之中有一个是多余的。

文件系统是经典例子：最上层是文件的抽象（可变长度字节数组），中间层是固定大小磁盘块的缓存，底层是设备驱动把块移入移出存储设备。三层，三种完全不同的抽象——从"文件"到"块"到"扇区"。TCP 也是同理：上层是"可靠的字节流"，下层是"尽力而为的数据包投递（包可能丢失或乱序）"——两种完全不同的问题模型。

## 透传方法：假层的标志

==透传方法== = 什么都不干，只调用另一个签名相同的方法。在一个学生的项目中，15 个 public 方法里有 13 个是透传方法。

> 透传方法让类变浅——增加接口的复杂度，但不增加任何系统功能。同时制造不必要的依赖——底层方法签名的变化会沿调用链传播。

三种修复策略：直接暴露底层类给调用者（去掉中间层）、重新分配功能让每层做不同的事、如果职责分不开就直接合并。

透传方法并非永远错误——分发器（比如 HTTP 路由：根据 URL 选择具体处理器）提供了"选择"这个有价值的逻辑。同层级的多种实现共享相同接口（比如不同的磁盘驱动）也是在降低认知负荷——你学一个接口，适用于多个实现。关键是：这些方法通常**在同一层**，彼此不调用。

## 装饰器的陷阱

==装饰器==模式鼓励跨层 API 复制——装饰器包装一个对象，提供相同接口，加上一小点扩展。比如 `BufferedInputStream` 包装 `InputStream`，提供缓冲。

> 装饰器倾向于变浅：大量样板代码换来少量新功能。过度使用装饰器导致浅模块爆炸——几个浅装饰器不如一个深模块。

在决定使用装饰器之前，Ousterhout 建议先考虑四个替代方案：把功能直接加到原类（特别是通用功能）、和调用场景合并而不是独立出来、和已有装饰器合并成一个深类、或者做成一个完全独立的类（不共享接口）。

## 接口不该和实现一样

如果类的接口直接映射了内部数据结构，这个类往往很浅。Ousterhout 又用了文本编辑器案例——这次从不同的角度：

面向行的 API（`getLine`、`putLine`）暴露了内部存储方式，迫使每个调用者自己拆分和拼接行来处理字符级别的插入删除——复杂性散落四处。面向字符的 API（`insert(string, pos)`、`delete(start, end)`）把分行和拼接封装在内部——类变深了，上层代码变简单了。

> 接口抽象和内部表示之间的差距，本身就是有价值的功能。差距越大，模块越深。

## 透传变量

一个变量从顶层方法一直传到最底层，经过十几层中间方法的签名——即使只有最底层的方法真正需要它。==透传变量==给所有中间方法增加了不必要的认知负担，而且一旦变量类型或语义变化，所有签名都要改。

三种解法：把变量放在顶层和底层都能访问到的共享对象里、使用 Context 对象封装、或者在确实必要时接受全局变量（但要注意测试和多实例的问题）。

## 思考题

1. 你的 Controller-Service-Repository 三层中，有没有一层的大部分方法只是"转发"（透传方法）？
2. 用过装饰器模式吗？回过头看，那个装饰器是真的提供了有价值的抽象，还是制造了一个浅模块？
3. 有没有一个配置参数像穿针一样穿过十几层方法签名，到最后只有一个人用它？那是透传变量。
