# 第18章 · 代码应当显而易见

---

晦涩是复杂性的两大根源之一（[第2章](第2章-复杂性的本质.md)）。当一个新开发者面对你的代码时，重要的信息不显眼——他要么花大量时间追踪，要么猜错导致 bug。"显而易见"就是晦涩的反面。

> "Obvious is in the mind of the reader." 你觉得清楚的代码不一定真的清楚。判断"显而易见"的最可靠方式是 code review——如果有人说你的代码不显而易见，那它就是不显而易见。不管你自己觉得多清楚。

## 什么让代码更显而易见

好命名（[第14章](第14章-命名.md)）——精确、无歧义的名字让读者不需要钻进去读实现。一致性（[第17章](第17章-一致性.md)）——读者识别熟悉的模式，立刻做出安全的推论。策略性地使用空白——文档中参数之间用空行分隔让内容可扫描，方法内用空行分隔主要的逻辑块，语句中的空格让结构不被挤成一团。注释——当代码本身无法做到显而易见时，注释填补缺失的信息（回到[第13章](第13章-注释描述什么.md)）。

## 什么让代码更不显而易见

**==事件驱动编程。==** 事件处理器从不被直接调用——它们被事件系统间接调用。即使你找到了触发点，你也不知道运行时到底哪个具体函数会执行。缓解方式：在处理器函数的接口注释里写明它什么时候被触发。

**==泛型容器。==** `Pair<Integer, Boolean>` 的调用者只能用 `getKey()` 和 `getValue()`——这两个名字不透露任何关于**含义**的信息。定义一个专门的小类，字段名有意义：`voteCount` 和 `alreadyVoted` 比 `key` 和 `value` 强十倍。

> "Software should be designed for ease of reading, not ease of writing."

**==声明类型和实际类型不一致。==** `private List<Message> incomingMessageList = new ArrayList<>()`——只看声明你以为是任意 List，但实际代码依赖了 ArrayList 的性能特征或线程安全行为。和声明保持一致。

**==违反读者预期的代码。==** 大多数 Java 程序的 `main()` 返回后进程就退出了——如果你在构造函数里启动了后台线程让进程保持运行，这就是在违反读者的自然假设。如果你必须违反常规，注释必须大声说出来。

## 显而易见的三个路径

当读者面对代码时，他们缺少的是理解它所需要的信息。你可以通过三种方式让信息到位：**减少读者需要的信息量**（用抽象消除特例）、**利用读者已有的知识**（遵循常规、符合预期）、**把信息呈现在代码里**（好名字、策略性注释、有意识的格式）。

> 总目标：确保读者总能拥有理解你代码所需的信息——要么信息已经在他们脑子里（常规和习惯），要么信息就在他们眼前（代码和注释）。

## 思考题

1. 你的项目里有没有用 `Pair`/`Tuple`/`Map` 代替有意义的小类的地方？其中有哪些字段的含义只有"写的人"知道？
2. 你的代码有没有"表面看起来是一个行为，实际是另一个行为"的情况？读者怎么才能知道？
3. 最近一次 code review——有没有人说过"看不懂"？那段代码为什么"不显而易见"？
