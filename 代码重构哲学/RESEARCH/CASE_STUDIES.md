## 案例 1: Ousterhout vs Clean Code — PrimeGenerator 辩论

**来源**: [johnousterhout/aposd-vs-clean-code](https://github.com/johnousterhout/aposd-vs-clean-code) (2024-2025)

**背景**: Robert Martin 在 *Clean Code* 中将一个素数生成器 (Knuth 的经典算法) 拆分为 8 个极小方法作为"整洁代码"的范例。Ousterhout 以此作为反面案例。

**正例 (Ousterhout 的论证)**:
- 原始 Knuth 算法的单方法版本（约 20 行）实际上更易理解
- 所有逻辑在一个地方，不需要在方法间跳转
- 使用注释解释**为什么**这样优化（如"为什么用质数的平方作为起始倍数"）

**反例 (Martin 的分解)**:
- `isLeastRelevantMultipleOfLargerPrimeFactor` — 超长方法名
- `isNotMultipleOfAnyPreviousPrimeFactor` — 命名暗示是无副作用的谓词，但实际上修改了 `multiplesOfPrimeFactors`
- Martin 自己 18 年后无法立即理解自己写的代码
- 重新分解导致 3-4 倍性能退化

**教训**:
1. 方法分解的"单一职责"原则如果不以"降低整体复杂性"为目标，可能适得其反
2. 方法名不能替代注释——复杂的算法意图（如优化动机）只能通过注释表达
3. "浅模块+纠缠"比"长方法"更差

**相关原则**: Ch04 (深模块), Ch12 (注释), Ch13 (注释描述代码中难以理解的内容), Ch21 (决定什么是重要的)

## 案例 2: Java I/O — 多类症的经典案例

**来源**: Ch04.7, 多个博客 (2024-2025)

**问题**:
```java
FileInputStream fileStream = new FileInputStream(fileName);
BufferedInputStream bufferedStream = new BufferedInputStream(fileStream);
ObjectInputStream objectStream = new ObjectInputStream(bufferedStream);
```
需要 3 个对象执行一个最常见操作。前两个对象创建后再也不被直接使用。

**为什么这是"浅模块"**:
- `FileInputStream` 只提供原始 I/O，没有缓冲
- `BufferedInputStream` 添加缓冲（但几乎每个人都需要）
- `ObjectInputStream` 添加反序列化
- 每个类的接口复杂度 ≈ 其功能 → 三个浅模块

**解决方案**:
- 默认提供缓冲（需要的人太多）
- 对少数不需要的人，提供禁用方法
- "使常见情况尽可能简单"

**相关原则**: Ch04.7, Ch07.3 (装饰器)

## 案例 3: Unix I/O — 深模块的典范

**来源**: Ch04.4, 多个讨论

**接口**: 5 个系统调用 `open/read/write/lseek/close`
**实现**: 成千上万行代码处理：磁盘表示、目录结构、权限、文件访问调度、缓存、多种存储设备

**深度证明**:
- 50 年来接口未变，实现已彻底演进
- 默认顺序 I/O（最常见情况），随机访问可选
- Jimmy Koppel 的反驳：SibylFS 项目对 `open` 的形式化描述超过 3000 字，模型实现仅 40 行→ 质疑"深模块"的测量方式

**教训**: 深模块的价值在于接口的长期稳定性，但"接口有多简单"可能取决于观察角度。

## 案例 4: 编辑器撤销机制 — 分离通用与专用

**来源**: Ch06.7

**问题**: 学生将整个撤销机制放入文本类。导致文本类包含与文本无关的专用代码（选择撤销、光标撤销），信息在文本类和 UI 类之间泄露。

**解决方案**:
```java
public class History {
    public interface Action {
        public void redo();
        public void undo();
    }
    void addAction(Action action) {...}
    void undo() {...}
    void redo() {...}
}
```
- 通用机制（`History` 类）管理动作列表
- 专用机制（`History.Action` 实现）理解特定操作
- 分组策略（栅栏）由高层代码决定

**教训**: 通用/专用分离后，其余设计"自然而然地出现了"。

## 案例 7: RAMCloud Buffer 优化 — 性能与简洁的统一

**来源**: Ch20.4

**问题**: `Buffer::alloc` 的关键路径涉及 6 个条件测试 + 3 个方法调用（其中两个是浅的透传方法），性能有提升空间。

**改造**:
- 统一为一个方法，消除浅抽象层级
- 引入 `availableAppendBytes` 实例变量，用一个测试排除所有特殊情况（无空间、非内部块、无块）
- 关键路径代码从分散在 3 个方法中浓缩到 1 个方法内

**结果**:
- 速度提高约 2 倍（8.8ns → 4.75ns 对于 1 字节追加）
- 代码量减少 20%（1886 行 → 1476 行）
- 设计更简单、更深

**教训**: 好的性能设计和好的模块设计是同方向的。消除浅抽象、减少特殊情况、统一关键路径——这些都同时提高了性能和可读性。

**相关原则**: Ch07 (透传方法), Ch08 (下沉复杂性), Ch10 (消除特殊情况), Ch20 (围绕关键路径设计)

## 案例 8: Tcl "一切皆字符串" — Ousterhout 自己的深模块实验

**来源**: PLDB.io 访谈, Tcl Wiki, Tcl History

**设计**: Tcl 的基本哲学是"一切皆字符串"——数字、代码、列表、命令都只是字符串。含义由解释字符串的命令决定。这实现了"终极后期绑定"。

**作为深模块**:
- **接口**: 极简——少量语法规则（命令是空格分隔的词，一切皆字符串）
- **实现**: 处理变量、控制流、过程、I/O、事件等

**代价**: Ousterhout 承认 EIAS 有"性能惩罚"，即使经过大量工作，"Tcl 从未变得像 Python 等其他语言那样高效"。

**后续改进**: Tcl 8.0 添加了字节码编译器，Tcl_Obj 使用双表示（缓存原生表示+字符串形式）来减轻性能问题。

**教训**: Tcl 是 Ousterhout 设计哲学的活实验室——极简接口 + 深度实现，但性能代价真实且持久的。这验证了他的框架但也暴露了其局限：有时"简单接口"的代价在别的维度上体现。

## 案例 9: Uber Monorepo — 3000+ 微服务的变更放大

**来源**: Uber Engineering Blog (Aug 2025), InfoQ

**问题**: 一个看似简单的变更——升级所有 Go 服务使用的 RPC 库——瞬间波及 3000+ 微服务。分析 50 万次提交发现：
- 影响 >100 服务的提交占 1.4%（~14/天）
- 影响 >1000 服务的提交占 0.3%（~3/天）
- 平均爆炸半径: 247 个服务

**解决方案**: 跨切面服务部署编排层：
- 服务分层 (Tier 0-5, 关键到非关键)
- 渐进式分批部署 + 信号验证
- 24 小时最大窗口解锁所有批次
- 失败阈值自动暂停和回滚

**与 Ousterhout 框架的关系**: 完美的变更放大案例。共享库（通用代码）实际上创建了跨数千服务的依赖。这提出了深模块的一个悖论——过于深的共享模块反而创建了最大的变更放大风险。

## 案例 10: Booking.com — 20 年 Perl 单体的变更放大

**来源**: AWS re:Invent 2025, vFunction

**问题**: 20 年以上的 Perl 预订系统累计了 90+ 依赖（80% 硬依赖）。新开发者入职需 4 个月，小功能改动需跨多个领域。

**症状**（完美匹配 Ousterhout 的三个症状）:
- **变更放大**: 简单改动需理解多个领域的代码
- **认知负荷**: 4 个月入职仍不完整
- **未知的未知**: 测试周期数小时，回滚是常态

**解决方案**: Strangler Fig 模式 + AWS Step Functions + Lambda + DynamoDB + SNS/SQS 解耦

**结果**: 入职时间 4 月 → 1 月，根因分析降至数小时，预订路径 100% 在新系统上
